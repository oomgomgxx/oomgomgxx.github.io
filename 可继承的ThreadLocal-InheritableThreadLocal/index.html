可继承的ThreadLocal-InheritableThreadLocal

 2020-04-03   作者: TanDi

前言

  • 前段时间在Q群里看见有人问子线程如何继承父线程的ThreadLocal值，所以故作此记。

ThreadLocal小回顾

  • ThreadLocal 中有个内部类叫ThreadLocalMap，存储格式是键值对存储，ThreadLocal上的操作实质就是对ThreadLocalMap 操作。
  • 每个 Thread 对象中都持有一个与当前线程绑定的 ThreadLocalMap 引用，所以 ThreadLocal 也被叫做线程变量，是线程安全的，因为每个Thread都有独自的 ThreadLocalMap 引用。
  • 但需要注意的是如何正确使用ThreadLocal？
// 设置默认值
ThreadLocal<HashMap<String, Integer>> t 
    = ThreadLocal.withInitial(() -> new HashMap<String, Integer>());
// 或者
//t.set(new HashMap<String, Integer>());

// 而不是以下操作（会导致同步问题）
HashMap<String, Integer> map =  new HashMap<String, Integer>();
t.set(map);
// 因为map引用暴露所以如果有其他线程并发操作就会存在同步问题
map.put("xxx", 1);
  • 如果在线程池中使用 ThreadLocal 应该任务结束就主动调用 clear 方法清理 ThreadLocalMap，否则可能会因为线程重用的缘故而导致 Value 值无法被及时回收而导致内存泄露。

案例驱动源码浅析InheritableThreadLocal

案例代码

public class MyTest {

    public static void main(String[] args) throws Exception {
        CountDownLatch countDownLatch = new CountDownLatch(1);

        InheritableThreadLocal<String> threadLocal = new InheritableThreadLocal<>();
//        ThreadLocal<String> threadLocal = new ThreadLocal<>();
        threadLocal.set("hello"); // debug点

        new Thread(() -> {
            String str = threadLocal.get();
            // ThreadLocal=null, InheritableThreadLocal=hello
            System.out.println("子线程：" + str); 
            countDownLatch.countDown();
        }).start();

        // ThreadLocal=hello, InheritableThreadLocal=hello
        System.out.println("父线程：" + threadLocal.get()); 
        countDownLatch.await();
    }
}
  • 当使用 InheritableThreadLocal 时子线程是可以获取到在上级线程（父线程）中设入的值的。

观察一下 InheritableThreadLocal 的源码，其实很简单

public class InheritableThreadLocal<T> extends ThreadLocal<T> {

    protected T childValue(T parentValue) {
        return parentValue;
    }

    ThreadLocalMap getMap(Thread t) { 
       return t.inheritableThreadLocals;
    }

    void createMap(Thread t, T firstValue) { 
        t.inheritableThreadLocals = new ThreadLocalMap(this, firstValue);
    }
}
  • 可以看到 InheritableThreadLocal 只是重写了三个方法：childValue、getMap、createMap。其中只需要关注 getMap、createMap 即可。
  • getMap 用于获取当前线程的 ThreadLocalMap，而 createMap 则是创建 ThreadLocalMap。

接下来从 threadLocal.set(“hello”)；这句代码中进入代码观察一下

public void set(T value) {
    Thread t = Thread.currentThread();
    // 先尝试获取当前线程的ThreadLocalMap
    ThreadLocalMap map = getMap(t); 
    if (map != null)
        map.set(this, value);
    else
        // 如果没有获取到则直接创建一个并绑定到当前线程上
        createMap(t, value); 
}
  • 注意，以上有注释标注的两个方法否是被 InheritableThreadLocal 重写了的，因此 getMap(t) 实质获取的引用是 t.inheritableThreadLocals，而 createMap(t, value) 则是 t.inheritableThreadLocals = new ThreadLocalMap(this, firstValue);
  • 到这里可以得出这样的结论，就是如果使用的是 InheritableThreadLocal 而不是 ThreadLocal 的话，ThreadLocalMap是保存在 t.inheritableThreadLocals 上的。

ThreadLocal 源码

public class Thread implements Runnable {

    /* ThreadLocal values pertaining to this thread. This map is maintained
     * by the ThreadLocal class. */
    ThreadLocal.ThreadLocalMap threadLocals = null;

    /*
     * InheritableThreadLocal values pertaining to this thread. This map is
     * maintained by the InheritableThreadLocal class.
     */
    ThreadLocal.ThreadLocalMap inheritableThreadLocals = null;

    // 省略
}
  • 这段源码印证了上面的结论

重点注意，接下来看看实例化 Thread 时都做了些什么初始化动作，以下为Thread的初始化方法。

private void init(ThreadGroup g, Runnable target, String name,
                  long stackSize, AccessControlContext acc,
                  boolean inheritThreadLocals) {
    // 省略...
    Thread parent = currentThread();
    // 当父线程使用的是InheritableThreadLocals且不为null时
    // 就会继承父线程的ThreadLocalMap内容
    if (inheritThreadLocals && parent.inheritableThreadLocals != null)
        this.inheritableThreadLocals =
        ThreadLocal.createInheritedMap(parent.inheritableThreadLocals);

    // 省略...
}

static ThreadLocalMap createInheritedMap(ThreadLocalMap parentMap) {
    return new ThreadLocalMap(parentMap);
}

private ThreadLocalMap(ThreadLocalMap parentMap) { // 拷贝父线程的ThreadLocalMap
    Entry[] parentTable = parentMap.table;
    int len = parentTable.length;
    setThreshold(len);
    table = new Entry[len];

    for (int j = 0; j < len; j++) {
        Entry e = parentTable[j];
        if (e != null) {
            @SuppressWarnings("unchecked")
            ThreadLocal<Object> key = (ThreadLocal<Object>) e.get();
            if (key != null) {
                Object value = key.childValue(e.value);
                Entry c = new Entry(key, value);
                int h = key.threadLocalHashCode & (len - 1);
                while (table[h] != null)
                    h = nextIndex(h, len);
                table[h] = c;
                size++;
            }
        }
    }
}

最后看看案例代码中子线程的 String str = threadLocal.get(); 

public T get() {
    Thread t = Thread.currentThread();
    // getMap(t)被重写，所以getMap(t)实质获取的是t.inheritableThreadLocals
    ThreadLocalMap map = getMap(t);
    if (map != null) {
        ThreadLocalMap.Entry e = map.getEntry(this);
        if (e != null) {
            @SuppressWarnings("unchecked")
            T result = (T)e.value;
            return result;
        }
    }
    return setInitialValue();
}

小结

  • InheritableThreadLocal 之所以能够获取父线程的 ThreadLocalMap 内容是因为 Thread 在初始化实例时继承了父线程的 ThreadLocalMap 内容

问题

共享了父线程资源后是否带来安全隐患？

  从上面的分析我们可以得知，子线程在实例化 Thread 时会 copy 父线程的 ThreadLocalMap 内容，但此时我们需要警惕，如果保存在父线程的值是一个引用类型的话，那么 copy 时传递的就是一个引用地址，因此如果存在并发操作的话InheritableThreadLocal 会存在同步问题的隐患，所以建议在使用时加以注意，当然如果引用类型是JUC包下的原子类的话，因为原子类使用的是CAS所以能够保证线程并发安全。

  最后要说的是，其实我们使用 ThreadLocal 的初衷就是不想将变量共享，所以如果不是有特别的需求的话个人不太建议使用 InheritableThreadLocal 。
