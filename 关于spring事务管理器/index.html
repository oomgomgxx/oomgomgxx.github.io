关于Spring的事务管理

内容修订

  • 2020年4月5日 02:53:25 — 增加 AOP 事务织入源码分析 和 事务失效情况 等内容

知识小回顾

  • Spring支持两种事务定义，分别为：编程式事务 和 声明式事务
  • 声明式事务使用 Spring AOP 实现，而 Spring AOP 原理则是通过判断方法是否包含 Advisor 来生成动态代理
  • Spring除了抽象事务之外，还针对不同的ORM技术提供了各自的事务管理器，而事务管理器的作用其实就是ORM技术对事务getTransaction、commit、rollback等操作，用于配合 Spring AOP 实现事务织入操作

Spring对事务的抽象

PlatformTransactionManager

事务管理器抽象 - 事务的基本操作

TransactionDefinition

事务定义信息抽象 - 当前事务的元数据，作为 TransactionStatus 的配置信息

TransactionStatus

事务状态抽象 - 当前事务的状态，事务管理器需要根据 TransactionStatus 信息操作事务

源码阅读小知识

  • Spring AOP 依靠 动态代理 + 方法拦截器（位于org.aopalliance.intercept，并继承了Advice接口） 来织入 Advisor 逻辑。大致过程是通过生命周期中的 BeanPostProcessor#postProcessAfterInitialization 方法判断是否存在 Advisor ，如果存在则返回包含事务拦截器的代理。事务拦截器用于主要执行代理方法时对其进行拦截判断织入 Advisor 逻辑
  • 用户自定义的 JpaRepository 会被容器封装成一个FactoryBean
  • 反射中 Method 是 AnnotatedElement 的实现
  • Advisor 相当于一个切面中的方法
  • 一个切入点表达式 Pointcut 会对应生成一个 MethodMatcher 用于匹配是否需要织入操作
  • MethodMatcher用于匹配方法签名（cacheKey）从而得到需要织入 Advisce 的方法（Method/AnnotatedElement），而 AOP 操作需要使用特定的解析器将 Method 上的@Transactional注解的元信息封装为 TransactionAttributeSource
  • TransactionAttributeSource 用于获取 TransactionDefinition
  • AnnotationAwareAspectJAutoProxyCreator 提供用于获取 Advisors 和 创建代理 的功能
  • Jpa 使用的 EntityManager 会在容器初始化期间，Singleton Bean实例化之前优先被创建（LocalContainerEntityManagerFactoryBean 实现了接口 LoadTimeWeaverAware 得到的特殊待遇）

Repository是何时被实例化的？

实现 FactoryBean 接口是自定义 Bean 实例过程的一种方式，而它有如下优点

  • 屏蔽 Bean 实例化过程的复杂性
  • 只有真正调用 getObject 时 Bean 才会真正被实例化，所以有延时加载的效果

通常一些第三方库要接入 Spring Framework 时，都会通过实现 FactoryBean 来构建一个关键组件实例。例如 MyBatis 的 SqlSessionFactoryBean、Hibernate 的 LocalSessionFactoryBean、Ehcache 的 EhCacheManagerFactoryBean 等。

JtaTransactionManager

基本介绍

  • JTA事务是2pc协议的实现，可以保证强一致性

  • JakartaEE规范根据DTP（分布式事务处理模型）定制了 TX 协议（应用管理器和事务管理器）和 XA 协议（事务管理器和资源管理器）两种规范。即如果想在 Java 中实现二段提交，则至少需要一个事务管理器和一个资源管理器。而且资源管理器必须支持 XA 协议（譬如 MySQL 中 InnoDB 的 redo log 和 undo log）

  • Spring 中的 JtaTransactionManager 只是一个事务管理器的接口规范，而不是具体实现。但如果我们想要在 Spring 中使用 JTA 事务，则可以依赖如下实现：

    • atomikos
    • bitronix

如何判断是否需要jta事务？

  • 需要实现强一致性
  • 一个事务中包含多个数据源（数据源必须提供 XA 协议实现）

JTA事务的弊端？

  1. 事务时间长：协调者需要等待一阶段提交（预提交）中所有的事务参与者的响应才能进行下一阶段工作
  2. 造成资源锁定：资源在整个2PC阶段被锁定
  3. 虽然能够保证多个数据源的强一致性，但会降低系统的可用性和吞吐量

与2PC的区别

  • ChainedTransactionManager 并不能做到 2PC 的效果，它只能保证链中的一个事务出错，其后续的事务都回滚，而已经提高了的事务是不会回滚的

拓展

同步事务

  如果在 Spring 事务中需要进行同步处理，则同步块范围必须大于事务范围，否则无法达到同步效果。这是因为事务代理是环绕着目标方法来进行管理的，所以这时就可能会出现这么一种情况，即方法退出了（同步锁被释放）但事务还在进行，而又因为是 Singleton 域，所以并发时就会共享到同一个事务，从而导致混乱。所以在使用 synchronized 时，应该做到包裹着被代理的方法，而不是直接使用在方法上或方法内。
