关于Spring的事务管理
发表于 2019-01-22| 更新于 2020-04-30|编程Spring全家桶
评论数:

知识小回顾

  • Spring支持两种事务定义，分别为：编程式事务 和 声明式事务
  • 声明式事务使用 Spring AOP 实现，而 Spring AOP 原理则是通过判断方法是否包含 Advisor 来生成动态代理
  • Spring除了抽象事务之外，还针对不同的ORM技术提供了各自的事务管理器，而事务管理器的作用其实就是ORM技术对事务getTransaction、commit、rollback等操作，用于配合 Spring AOP 实现事务织入操作

Spring对事务的抽象

PlatformTransactionManager

事务管理器抽象 - 事务的基本操作

TransactionDefinition

事务定义信息抽象 - 当前事务的元数据，作为 TransactionStatus 的配置信息

TransactionStatus

事务状态抽象 - 当前事务的状态，事务管理器需要根据 TransactionStatus 信息操作事务

源码阅读小知识

  • Spring AOP 依靠 动态代理 + 方法拦截器 来织入 Advisor 逻辑。大致过程是通过生命周期中的 BeanPostProcessor#postProcessAfterInitialization 方法判断是否存在 Advisor ，如果存在则返回包含事务拦截器的代理。事务拦截器用于主要执行代理方法时对其进行拦截判断织入 Advisor 逻辑
  • 用户自定义的 JpaRepository 会被容器封装成一个FactoryBean
  • 反射中 Method 是 AnnotatedElement 的实现
  • Advisor 相当于一个切面中的方法
  • 一个切入点表达式 Pointcut 会对应生成一个 MethodMatcher 用于匹配是否需要织入操作
  • MethodMatcher用于匹配方法签名（cacheKey）从而得到需要织入 Advisce 的方法（Method/AnnotatedElement），而 AOP 操作需要使用特定的解析器将 Method 上的@Transactional注解的元信息封装为 TransactionAttributeSource
  • TransactionAttributeSource 用于获取 TransactionDefinition
  • AnnotationAwareAspectJAutoProxyCreator 提供用于获取 Advisors 和 创建代理 的功能
  • Jpa 使用的 EntityManager 会在容器初始化期间，Singleton Bean实例化之前优先被创建（LocalContainerEntityManagerFactoryBean 实现了接口 LoadTimeWeaverAware 得到的特殊待遇）

jms事务注意

  • jms和其他事务稍有不同，jms的事务是通过session来管理的，即每一个session就相当于一个事务；

  • 默认情况下jms并不受@Transactional标签控制，也就是说即使service方法报错也不会回滚；这是因为jms提供了两种对事务管理的方式：

    1. 什么都不配置，默认使用sesssion进行管理（jms原生事务）
    2. 依赖外部管理事务，如JmsTransactionManager、JtaTransactionManager等等

JtaTransactionManager

jta事务说明

  • jta事务是2pc协议的实现，可以保证强一致性；
  • 可以解决一个服务中出现多个事务的一致性问题；
  • JavaEE规范根据DTP(分布式事务处理模型，根据2pc协议制定的模型)提供了TX协议（应用管理器和事务管理器）和XA协议（事务管理器和资源管理器）两种规范；换句话来说，即如果想在java中实现二段提交，就需要借助相关的管理器来完成；而且需要主要的是资源管理器管理的数据资源也需要支持XA协议，如MySQL中InnoDB的redo log和undo log；
  • 对于JtaTransactionManager而言，因为实现的是2pc协议，因此在JavaEE中就需要实现XA协议，但spring中的JtaTransactionManager只是一个编程接口而并非是一个实现；
  • 在JavaEE中一般可以通过以下方式来获取TX协议和XA协议的实现和支持：
    • Jboos和WebLogic等重量级的服务器就提供：应用管理器、事务管理器、资源管理器的实现
    • 实现了XA协议的第三方工具库：atomikos、bitronix

如何判断是否需要jta事务

  • 需要实现强一致性
  • 一个服务方法中有两个或以上的不同类型的事务管理器（如多db就无需jta事务）

jta事务的弊端

虽然jta能够保证一个服务下使用了多个数据资源的事务一致性，但这是需要付出响应的代价的：

  1. 事务时间长：协调者需要等待一阶段提交（预提交）中所有的事务参与者的回应才能进行下一阶段工作
  2. 造成资源的全局锁：资源在整个一阶段提交中都是被锁定的
  3. 综合上面两点来说，对于系统则造成性能降低和吞吐量降低

与2PC的区别
