关于Spring的事务管理

 2019-01-22   作者: TanDi

知识小回顾

  • Spring支持两种事务定义，分别为：编程式事务 和 声明式事务
  • 声明式事务使用 Spring AOP 实现，而 Spring AOP 原理则是通过判断方法是否包含 Advisor 来生成动态代理
  • Spring除了抽象事务之外，还针对不同的ORM技术提供了各自的事务管理器，而事务管理器的作用其实就是ORM技术对事务getTransaction、commit、rollback等操作，用于配合 Spring AOP 实现事务织入操作

Spring对事务的抽象

PlatformTransactionManager

事务管理器抽象 - 事务的基本操作

package org.springframework.transaction;
import org.springframework.lang.Nullable;

public interface PlatformTransactionManager {
    // 获取事务
    TransactionStatus getTransaction(@Nullable TransactionDefinition definition)
            throws TransactionException;
    // 提交事务
    void commit(TransactionStatus status) throws TransactionException;
    // 回滚事务
    void rollback(TransactionStatus status) throws TransactionException;
}

TransactionDefinition

事务定义信息抽象 - 当前事务的元数据，作为 TransactionStatus 的配置信息

package org.springframework.transaction;
import java.sql.Connection;
import org.springframework.lang.Nullable;

public interface TransactionDefinition {

    // 事务的传播行为
    int PROPAGATION_REQUIRED = 0; 
    int PROPAGATION_SUPPORTS = 1;
    int PROPAGATION_MANDATORY = 2;
    int PROPAGATION_REQUIRES_NEW = 3;
    int PROPAGATION_NOT_SUPPORTED = 4;
    int PROPAGATION_NEVER = 5;
    int PROPAGATION_NESTED = 6;

    // 事务的隔离级别
    int ISOLATION_DEFAULT = -1; // 使用数据库默认级别
    int ISOLATION_READ_UNCOMMITTED = Connection.TRANSACTION_READ_UNCOMMITTED; // 读未提交
    int ISOLATION_READ_COMMITTED = Connection.TRANSACTION_READ_COMMITTED; // 读已提交
    int ISOLATION_REPEATABLE_READ = Connection.TRANSACTION_REPEATABLE_READ; // 可重复度
    int ISOLATION_SERIALIZABLE = Connection.TRANSACTION_SERIALIZABLE; // 序列化读取

    // 事务默认不超时
    int TIMEOUT_DEFAULT = -1;
    // 获取事务的传播行为
    int getPropagationBehavior();
    // 获取事务的隔离级别
    int getIsolationLevel();
    // 获取事务的超时时间定义
    int getTimeout();
    // 是否为只读事务
    boolean isReadOnly();
    @Nullable
    String getName();
}

TransactionStatus

事务状态抽象 - 当前事务的状态，事务管理器需要根据 TransactionStatus 信息操作事务


package org.springframework.transaction;
import java.io.Flushable;

public interface TransactionStatus extends SavepointManager, Flushable {
    // 是否为一个新的事务
    boolean isNewTransaction();
    // 是否有保存点
    boolean hasSavepoint();
    // 将事务设置为不可提交
    void setRollbackOnly();
    // 当前事务是否为一个不可提交的事务
    boolean isRollbackOnly();
    @Override
    void flush();
    // 事务是否以结束
    boolean isCompleted();
}

源码阅读小知识

  • Spring AOP 依靠 动态代理 + 方法拦截器 来织入 Advisor 逻辑。大致过程是通过生命周期中的 BeanPostProcessor#postProcessAfterInitialization 方法判断是否存在 Advisor ，如果存在则返回包含事务拦截器的代理。事务拦截器用于主要执行代理方法时对其进行拦截判断织入 Advisor 逻辑
  • 用户自定义的 JpaRepository 会被容器封装成一个FactoryBean
  • 反射中 Method 是 AnnotatedElement 的实现
  • Advisor 相当于一个切面中的方法
  • 一个切入点表达式 Pointcut 会对应生成一个 MethodMatcher 用于匹配是否需要织入操作
  • MethodMatcher用于匹配方法签名（cacheKey）从而得到需要织入 Advisce 的方法（Method/AnnotatedElement），而 AOP 操作需要使用特定的解析器将 Method 上的@Transactional注解的元信息封装为 TransactionAttributeSource
  • TransactionAttributeSource 用于获取 TransactionDefinition
  • AnnotationAwareAspectJAutoProxyCreator 提供用于获取 Advisors 和 创建代理 的功能
  • Jpa 使用的 EntityManager 会在容器初始化期间，Singleton Bean实例化之前优先被创建（LocalContainerEntityManagerFactoryBean 实现了接口 LoadTimeWeaverAware 得到的特殊待遇）

protected Object createProxy(Class<?> beanClass, @Nullable String beanName,
                             @Nullable Object[] specificInterceptors, TargetSource targetSource) {

    //省略...

    // ProxyFactory可以看做是一个用于创建AOP代理实例的装配类
    ProxyFactory proxyFactory = new ProxyFactory();
    proxyFactory.copyFrom(this);

getProxy方法实质调用了 DefaultAopProxyFactory#createAopProxy ，如下操作

public class DefaultAopProxyFactory implements AopProxyFactory, Serializable {

    @Override
    public AopProxy createAopProxy(AdvisedSupport config) throws AopConfigException {
        // 1. 判断是否配置了优化
        // 2. 判断是否配置ProxyTargetClass来强制使用Cglib
        // 3. 是不是没有实现接口
        // 以上随便满足一条则采用Cglib，否则使用JDK动态代理
        if (config.isOptimize() || config.isProxyTargetClass() || hasNoUserSuppliedProxyInterfaces(config)) {
            Class<?> targetClass = config.getTargetClass();
            if (targetClass.isInterface() || Proxy.isProxyClass(targetClass)) {
                return new JdkDynamicAopProxy(config);
            }
            return new ObjenesisCglibAopProxy(config); // 返回Cglib动态代理实例（继承形式）
        }
        else {
            return new JdkDynamicAopProxy(config); // 返回JDK动态代理实例（抽象接口形式）
        }
    }
}

回顾 FactoryBean 接口

  实现 FactoryBean 接口是实例化Bean的其中一种方式。FactoryBean可以屏蔽Bean实例化的复杂度，而且实例会在现实调用 getObject 方法才会真正实例化，所以有延时加载的效果。通常一些非Spring全家桶的第三方技术都会使用 FactoryBean 来构建Bean实例以接入Spring上下文。例如 MyBatis 的 SqlSessionFactoryBean，Hibernate 的 LocalSessionFactoryBean ，Ehcache的EhCacheManagerFactoryBean等等。

FactoryBean接口如下：

public interface FactoryBean<T> {
    String OBJECT_TYPE_ATTRIBUTE = "factoryBeanObjectType";
    // 获取实际的Bean(T)实例
    @Nullable
    T getObject() throws Exception;
    // 获取实例的实际类型
    @Nullable
    Class<?> getObjectType();
    // 是否单例。可以看到默认为true，即默认FactoryBean在单个容器中是单例的
    default boolean isSingleton() { return true; }
}

给进入 proceed 并添加条件断点以快速定位 TransactionInterceptor

跟随者步伐我们进入了一个名为 TransactionAspectSupport 的类。发现原来事务织入操作都是这个类完成的。方法源码如下：

protected Object invokeWithinTransaction(Method method, @Nullable Class<?> targetClass,
                                         final InvocationCallback invocation) throws Throwable {

    // If the transaction attribute is null, the method is non-transactional.
    // 获取当前事务的元数据，即 TransactionAttribute。如果不存在则无需添加事务
    // 而TransactionAttribute是的获取方式先前也稍微提到过，就是通过TransactionAttributeSource
    // 中两个Parser解析器通过对当前Method上的@Transactional注解解析出来了的
    TransactionAttributeSource tas = getTransactionAttributeSource();
    // TransactionAttribute中包含诸如传播行为和隔离级别的配置信息，即TransactionDefinition中的信息
    final TransactionAttribute txAttr = (tas != null ? tas.getTransactionAttribute(method, targetClass) : null);

    // 获取事务管理器
    final TransactionManager tm = determineTransactionManager(txAttr);
    PlatformTransactionManager ptm = asPlatformTransactionManager(tm);

    // 根据当前拦截的Method对象解析出Advisor的切入点
    // 格式如：org.springframework.data.jpa.repository.support.SimpleJpaRepository.save
    final String joinpointIdentification = methodIdentification(method, targetClass, txAttr);

    if (txAttr == null || !(ptm instanceof CallbackPreferringPlatformTransactionManager)) {

        // 【重点】使用当前的事务管理器实现来创建事务
        // TransactionInfo中封装了以下内容
        // * PlatformTransactionManager - 当前orm的事务管理器
        // * TransactionAttribute - @Transactional的元数据
        // * joinpointIdentification - 当前需要织入事务的目标方法签名
        // * transactionStatus - 事务的状态，用于控制事务commit和rollback
        TransactionInfo txInfo = createTransactionIfNecessary(ptm, txAttr, joinpointIdentification);

        Object retVal;
        try {
            // 以下操作相当于一个@Around方法，主要是返回在执行持久化操作期间的信息
            // 用于判断事务到底是commit还是rollback
            retVal = invocation.proceedWithInvocation();
        }
        catch (Throwable ex) {
            // 当前目标方法执行异常，进行事务回滚
            completeTransactionAfterThrowing(txInfo, ex);
            throw ex;
        }
        finally {
            //清楚事务信息
            cleanupTransactionInfo(txInfo);
        }

        // 没有发生异常，事务正常提交
        commitTransactionAfterReturning(txInfo);
        return retVal;
    }

    // 省略...
}

代码方式

编程式事务的优点在于可以进行局部事务而不用整个方法都使用事务。事务尽快结束就意味着数据库连接可以更快被释放。

@Service
public class PersonServiceCode {


    @Autowired
    private PersonRepository repository;

    @Autowired
    private PlatformTransactionManager transactionManager;

    public void save(Person person) {

        // 事务定义
        DefaultTransactionDefinition transactionDefinition 
            = new DefaultTransactionDefinition();
      transactionDefinition // 隔离级别
          .setIsolationLevel(TransactionDefinition.ISOLATION_READ_UNCOMMITTED);
        transactionDefinition // 传播行为
            .setPropagationBehavior(TransactionDefinition.PROPAGATION_REQUIRED);

        // 根据事务定义获取事务
        TransactionStatus transactionStatus 
            = transactionManager.getTransaction(transactionDefinition);

        try {
            repository.save(person);

            // 提交事务
            transactionManager.commit(transactionStatus);

        } catch (Exception e) {
            e.printStackTrace();
            // 回滚事务
            transactionManager.rollback(transactionStatus);
        }

    }
}

JmsTransactionManager

jms事务注意

  • jms和其他事务稍有不同，jms的事务是通过session来管理的，即每一个session就相当于一个事务；

  • 默认情况下jms并不受@Transactional标签控制，也就是说即使service方法报错也不会回滚；这是因为jms提供了两种对事务管理的方式：

    1. 什么都不配置，默认使用sesssion进行管理（jms原生事务）
    2. 依赖外部管理事务，如JmsTransactionManager、JtaTransactionManager等等

JtaTransactionManager

jta事务说明

  • jta事务是2pc协议的实现，可以保证强一致性；
  • 可以解决一个服务中出现多个事务的一致性问题；
  • JavaEE规范根据DTP(分布式事务处理模型，根据2pc协议制定的模型)提供了TX协议（应用管理器和事务管理器）和XA协议（事务管理器和资源管理器）两种规范；换句话来说，即如果想在java中实现二段提交，就需要借助相关的管理器来完成；而且需要主要的是资源管理器管理的数据资源也需要支持XA协议，如MySQL中InnoDB的redo log和undo log；
  • 对于JtaTransactionManager而言，因为实现的是2pc协议，因此在JavaEE中就需要实现XA协议，但spring中的JtaTransactionManager只是一个编程接口而并非是一个实现；
  • 在JavaEE中一般可以通过以下方式来获取TX协议和XA协议的实现和支持：
    • Jboos和WebLogic等重量级的服务器就提供：应用管理器、事务管理器、资源管理器的实现
    • 实现了XA协议的第三方工具库：atomikos、bitronix

如何判断是否需要jta事务

  • 需要实现强一致性
  • 一个服务方法中有两个或以上的不同类型的事务管理器（如多db就无需jta事务）

jta事务的弊端

虽然jta能够保证一个服务下使用了多个数据资源的事务一致性，但这是需要付出响应的代价的：

  1. 事务时间长：协调者需要等待一阶段提交（预提交）中所有的事务参与者的回应才能进行下一阶段工作
  2. 造成资源的全局锁：资源在整个一阶段提交中都是被锁定的
  3. 综合上面两点来说，对于系统则造成性能降低和吞吐量降低

      // 设置超时时间

      // 挂起一个事务

}

      // 获取当前XAResource实例的超时时间

    int prepare(Xid xid) throws XAException;
      // 开始 xid 事务参与者的事务处理工作
    void start(Xid xid, int flags) throws XAException;
    // 结束 xid 事务参与者的事务处理工作
    void end(Xid xid, int flags) throws XAException;

      // 以下为2pc的事务状态

SpringData提供的链式事务管理器

  在spring data commons中提供了一个叫ChainedTransactionManager的事务管理器。主要用于多事务场景时对事务进行编排。通俗点说就是按指定的顺序来提交事务，且可以做到当一个事务提交失败后在该事务后面的所有事务将会全部回滚而不会继续提交事务。

ChainedTransactionManager#commit 源码

    public void commit(TransactionStatus status) throws TransactionException {
        MultiTransactionStatus multiTransactionStatus = (MultiTransactionStatus) status;
        boolean commit = true;
        Exception commitException = null;
        PlatformTransactionManager commitExceptionTransactionManager = null;

        for (PlatformTransactionManager transactionManager : reverse(transactionManagers)) {// 遍历事务链按顺序进行提交
            if (commit) {
                try {
                    multiTransactionStatus.commit(transactionManager);
                } catch (Exception ex) {
                    commit = false;
                    commitException = ex;
                    commitExceptionTransactionManager = transactionManager;
                }
            } else {
                // after unsucessfull commit we must try to rollback remaining transaction managers
                multiTransactionStatus.rollback(transactionManager);
            }
        }

        // 省略...
    }

与2PC的区别

ChainedTransactionManager 并不能做到2pc的效果，它只能做到在事务链中某个节点事务回滚后位于该阶段后的事务都不会继续提交。

导致事务失效的几种情况

    • 如果定义在接口上，会采用 JDK 动态代理来织入事务操作

@Transactional方法被同类的非事务方法调用

如下：

@Component
public void Test{

    public void t1(){
        t2();
    }

    @Transactional
    public void t2(){
        // 事务操作
    }
}

  • 以上失效情况其实很好解释，因为 t1 方法并没有被 @Transactional 标注，所以 t1 方法并没有 TransactionAttributeSource 信息，因此方法在执行过程中并不会织入事务操作。

  • 解决方法有两种：

    • 添加@Transactional注解到 t1 方法上

    • 通过暴露当前代理实例，使用代理实例来调用，如下：

      @EnableAspectJAutoProxy(exposeProxy = true)
@Configuration
public void Test{

    public void t1(){
        //t2();
        Test proxy = (Test)AopContext.currentProxy();
        proxy.t2();
    }

    @Transactional
    public void t2(){
        // 事务操作
    }
}

手动处理异常而没有设置回滚标记

如下：

@Component
public void Test{

    @Transactional
    public void t(){
        try{
            int a = 1/0;// 异常点
        } catch (Exception e) {
            e.printStackTrace();
            // 手动捕获了异常而没有设置事务回滚点
        }
    }
}

  • 以上操作无法回滚是因为 Spring AOP 织入的事务是通过回滚标记（TransactionStatus#setRollbackOnly）来实现回滚操作的，而又因为 Spring AOP 在代理方法中已经添加了异常捕获操作来设置回滚点，所以如果开发者自己捕获了异常，就会导致 Spring AOP 定义捕获操作失效从而没法设置事务回滚标识

  • 解决方法有2种：

    • 使用 throws Exception 抛出异常而不是 catch

    • 在 catch 设置 TransactionStatus#setRollbackOnly ，如下：

      // 必须暴露当前代理类，@EnableAspectJAutoProxy(exposeProxy = true)
TransactionAspectSupport.currentTransactionStatus().setRollbackOnly();

事务方法抛出了检测性异常

  • 对于Spring事务而言虽然它有处理异常的能力并在捕获到异常时设置回滚点进行回滚操作，但是它只能捕获Error 和 RuntimeException 两种异常分支。换言之如果抛出的是检测性的异常（如IOException） Spring 就不能够设置回滚标记了，所以即使事务生效了也会导致无法回滚。问题代码如下：

  • 解决方案是手动设置回滚异常类型以代替默认的 TransactionAttribute：@Transactional(rollbackFor = Exception.class)

嵌套事务操作时错误设置事务传播方式

Spring支持的事务传播方式回顾

public enum Propagation {

    // 必须使用事务（默认）
    // 优先考虑加入到当前事务中，如果当前环境没有开启事务则自己开启一个新的事务
    REQUIRED(TransactionDefinition.PROPAGATION_REQUIRED),

    // 支持事务
    // 优先考虑加入到当前事务中，如果当前环境没有开启事务则不使用事务
    SUPPORTS(TransactionDefinition.PROPAGATION_SUPPORTS),

    // 强制使用事务
    // 优先考虑加入到当前事务中，如果当前环境没有开启事务则抛异常
    MANDATORY(TransactionDefinition.PROPAGATION_MANDATORY),

    // 必须开启新的事务
    // 如果当前存在事务环境，则先挂起当前事务，然后开启一个新的事务环境来执行当前操作
    REQUIRES_NEW(TransactionDefinition.PROPAGATION_REQUIRES_NEW),

    // 不支持事务
    // 如果当前存在事务环境，则先挂起当前事务，然后直接执行当前操作
    NOT_SUPPORTED(TransactionDefinition.PROPAGATION_NOT_SUPPORTED),

    // 强制不支持事务
    // 如果当前存在事务环境，则会抛异常，反之直接执行当前操作
    NEVER(TransactionDefinition.PROPAGATION_NEVER),

    // 必须使用事务(针对JTA事务)
    // 优先考虑加入到当前事务中，如果当前环境没有开启事务则使用PROPAGATION_REQUIRED模式执行
    // 但该模式只适用于 DataSourceTransactionManager 事务管理器类型
    NESTED(TransactionDefinition.PROPAGATION_NESTED);

    // 省略
}

嵌套事务使用了如下传播方式：

  • PROPAGATION_SUPPORTS
  • PROPAGATION_NOT_SUPPORTED
  • PROPAGATION_NEVER
