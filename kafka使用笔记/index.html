一篇文章解决Kafka的基本使用

 2019-09-08   作者: TanDi

拓展知识点

为了更好地学习和理解kafka，个人觉得有必要搞清楚一些概念

在开始之前，推荐书籍《Apache Kafka实战》和《Kafka权威指南》来作为kafka的入门书籍

什么是磁盘的顺序写入？

  所谓的磁盘顺序写入其实是相对与随机写入而言的，指的是在写入数据时磁盘的磁头不用寻道而可以一次将数据写完，因为省略了磁头寻道的时间，所以在速度来说是非常快的，甚至可以媲美内存的随机写入; 那什么情况下会出现顺序写入呢？如文件追加内容时（kafka就采用这种方式进行顺序写入，因为kafka存放数据的日志文件不能修改只能追加）

以下为磁盘、SSD、内存之间的访问速度对比图

图片源自：queue.acm.org

可以看出，磁盘的顺序访问是相当快的，所以kafka选择将内容写入磁盘

内存映射文件

  游戏就经常玩了，但不知道有没有这么一个疑问：游戏体积这么大，而内存这么小，那么操作系统到底是怎么运行游戏的呢？答案就是，操作系统会预先为程序生成一个内存镜像保存在磁盘中，在运行该程序时并不会真的将该程序全部加载进主存，而是当使用到某部分代码时才会加载进主存，而这个保存程序内存镜像的文件就是内存映射文件。操作可以可以直接将内存映射文件映射到对应的内存地址中，因此对于用户线程而言是可以直接访问到该内存地址的。而且操作系统会自动管理者块内存空间，例如操作完该内存中的数据后系统会自动将脏数据刷会文件中。

参考图

  在操作系统中一般会使用页来作为内存的基本单位。系统将用户进程中的虚拟内存地址划分为大小相等的页，这种页称为页面Kafka而物理内存地址中亦划分为大小相等的页，这种页称为页框。每个页面的大小为2的整数次幂，范围在512字节～1M，通常为4KB（32bit系统）和8KB（64bit系统）。在管理层面，操作系统使用一种叫页表的结构来映射虚拟内存地址和物理内存地址之间的关系（虚拟内存通常远大于虚拟内存）。

参考图

从上图我们不难看出，并不是所有虚拟内存地址都有与之对应的映射的。当程序访问到一个没有映射的虚拟地址时，就会发生一种叫缺页中断的操作，该操作会将物理内存地址中较为不常用的内存页替换成当前缺页的程序部分。大体步骤是系统会从内存映射文件中加载内容到主存上（物理内存地址），然后更新进程的虚拟内存空间（虚拟内存地址），最后恢复到缺页中断位置的操作。

  最后说一下mmap操作，mmap是一个系统函数，它能够实现文件到内存的映射（内存映射文件）。而kafka就是使用mmap将日志索引文件内容映射到内存的，所以在定位数据方面速度也是相当快的。

kafka只用mmap来映射索引是因为如果同时映射索引和日志的话就会存在一直性问题，而且mmap并不能保证百分百成功将内存中的脏数据刷回磁盘。例如并发操作时修改了日志并回刷成功，但因为某些原因索引回刷失败，这就导致了数据不一致性问题

上线程下文切换

线程上下文切换

  指的是在任务执行前，CPU寄存器和程序计数器会预先加载将要执行的任务内容。首先操作系统会先记录下正在执行的任务内容和状态，以便下次执行时恢复。保存成功会操作系统就会把将要执行的任务内容加载到CPU寄存器和程序计数器中（例如每条线程都有栈和程序计数器），而这个过程就被称为上下文切换（实质就是切换执行任务）。上下文切换是一个系统层面的调度操作，所以在上下文切换期间，用户程序将会进入呆滞状态，等到切换完成后才会继续运行。

用户态和内核态切换

  操作系统将线程（轻量级进程）的执行空间分为：用户空间 和 内核空间。

线程在用户空间时，因为用户空间权限有限所以线程在对资源操作上也是有限的。例如用户空间的线程不能直接访问除系统分配给自己的内存以外的内容，譬如：磁盘、网络模块 等，如何想要操作这部分资源就需要发起系统调用，让系统的内核线程来完成，而这时就会发生所谓的用户态和内核态切换操作。

举个例子：用户线程想要读取磁盘上文件的内容，这时就需要进行系统调用。系统调用会将读取磁盘的操作交给内核线程来完成，过程如下：

所以，应用做IO操作时并不是直接操作设备资源的，而是通过内核来去操作的。内核线程读取磁盘上的文件内容后将内容先保存到内核缓冲中，然后再将内核缓冲中的内容通过CPU Copy操作复制到用户空间，这样用户现线程会访问到磁盘文件中的内容。

那么从上图我们不难看出，在1次io操作中就出现了2次用户态和内核态之间的切换操作。上面已经提到过，上下文切换或用户态内核态切换时用户程序都会进入呆滞状态来等待系统调用完成。

简单总结一下

虽然上面提到用户态和内核态切换，但其实它也是线程上下文切换的一种操作。从用户程序的角度来看这是一个比较重量。虽然在单个上下文切看来并不需要花费多少时间，但是一旦发生大量的上下文切换，那么将会直接导致用户程序处理情况的能力下降。

零拷贝（Zero-copy）

零拷贝通常会在网络传输上使用的。指的是内核态直接将buffer（page buffer）中的数据通过RMA Copy 传输给网络模块，而网络模块就可以直接将其发送出去了，而无需从内核态切换给用户态再执行发送操作。

零拷贝例子

需要说明的是，零拷贝并不是不拷贝，而是不进行 CPU Copy 而已。零拷贝的好处在于减少了用户态和内核态的切换次数，这样就减少了用户程序的呆滞时间，因此提高了用户程序的处理能力。

为什么kafka性能高？

  • 数据操作方式为磁盘的顺序访问
  • 通过 Zero-copy 传输网络数据
  • 使用 mmap 将日志索引映射到内存，提高了数据的访问速度
  • 一个topic可以有多个partition，除了去中心化之外还提高了并发效率
  • 支持批量发送操作，只有数据大小到达batch.size才会发出，目的是提高操作的吞吐量

关于topic、partition、replica之间的关系

  通常生产者生产一个消息是存放在Broker的topic中的，但是topic只是逻辑上的概念，而消息实际存放的地方是partition（分区）。partition是一个有序的序列，在物理磁盘上对应着一个只能最追加内容的文件。

  其中，1个topic可以有1个或多个partition，换句话说就是1个topic的数据是分拆到1个或多个partition中的。 还有就是，kafka只能保证单个partition中的消息数据是有序的，如果1个topic有多个partition的话，那么就不能保证消息的有序性了，既 kafka 是不能保证全局有序的 。

  而 replica 是 partition 另外一个身份，因为它并不是简单的冗余。在 kafka 中 replica 分为两种：leader replica、follower replica。其中 follower replica 是不对客户端提供服务的，它的职责只用于同步 leader replica 数据。而 follower replica 又分为 ISR（in-sync replica，能够实时同步leader数据的replica）和 非ISR 两种。其中只有 ISR 能够参与 Leader 选举成为 leader replica，且在 borker 接收请求时，只有 leader replica 和 ISR 都接收到该请求时这条消息才会被broker认为是发送成功。实质上一开始所有的replica（包含leader replica）都属于 ISR，但运行后可能因为网络又或者其它的干扰因素，导致其它broker上的replica无法跟上复制 leader replica 数据的进度因此被踢出 ISR集合 ，但当被踢出 ISR集合的replica 跟上进度又会被重新纳入ISR集合。

为什么需要partition？

  有人可能会有这样的疑问，就是即然有了topic为什么要有partition这个概念呢？如 ActiveMQ 就没有partition概念。首选 kafka 主要为作为处理实时数据场景提供 高吞吐、低延迟 处理方案。partition 就是用来提 topic 的并发操作效率的，有着去中心化和负载均衡的效果

从上图不难看出，当topic有多个partition 和多个生产者的情况下，不同的生产者是可以针对不同的 partition 进行操作的。但如果这时只有一个partition 的话，并发操作时压力就会全部落在这个 partition 上，因此会降并发操作的性能。

为什么有segment？

1个 partition 多个 segment 主要是为了方便和提高查找数据的速度，因为每个 segment 文件名称中就包含了数据的起始 offset 信息，因此在二分查找时只需找到对应的 segment 就可以定位消息数据了，而并不需要扫描所有的消息数据。

topic offset 和 consumer offset

topic offset

当消息数据被写到 partition 时，kafka 会为其分配一个顺序的 offset ，这个 offset 就是消息数据在当前 partigion 中的偏移量。其中 offset 从0开始递增，在不同的 partition 中 offset 是互不干扰的。

consumer offset

consumer offset 和 topic offset 是不同的概念，这个千万不要混淆。consumer offset 是用于记录partition 消息被消费到那个位置的一个标记。consumer offset 可以用于发生 再均衡 操作后告诉新消费者应该从哪里开始消费。需要特别注意的是 consumer offset 并不是 kafka 管理的，而是由消费者（消费者组）自己管理的，所以消费者消费了一条消息就需要将 consumer offset 告诉 Kafka ，那么下次再消费又或者消费者宕机后再消费就可以知道上一次消费到那个位置了。这样做的好处是减轻了 Broker 维护数据的压力。

关于consumer offset的提交

上面已经提到，消费者是需要定期向 Broker 提交 consumer offset 以汇报自己的消费进度的，这个动作称为位移提交。针对位移提高操作在较新版本的 kafka 中会有1个专门对应的 topic（__consumer__offsets）与之对应，这个可以在 kafka 的日志文件夹下知道，这个topic一共有50个文件夹，也就是说有50个partition。

为什么需要这么多partition呢？这是因为位移提交是一个非常频繁的操作，因此利用 partition 的负载均衡特性可以减轻并发操作同一个 topic 带来的压力。

Java kafka Producer

生产者模型

注意：

Producer 的 send 方法是异步的，而将消息发送到 Broker 会有一条专门负责冲刷消息缓冲池的线程来完成。

  • ProducerRecord：消息数据对象
  • Serializer：用于序列化 ProducerRecord 以便进行网络传输
  • Partitioner：分区器。根据 ProducerRecord 中的 key 值来判断应将将当前消息数据发送给那个partition。例如 key 相同的消息数据会保存在同一个partition中。而当 ProducerRecord 没有指定 key 的话就会采用轮询分区方式来保存消息
  • 消息缓冲池：用于减少上下文切换次数，可以提高消息发送操作的吞吐量

producer interceptor

生产者拦截器，可以在消息序列化之前将其拦截来做一些消息定制化修改。

实现方式是实现 ProducerInterceptor 接口，并将其实现添加进生产者配置中即可（ProducerConfig.INTERCEPTOR_CLASSES_CONFIG）。

public interface ProducerInterceptor<K, V> extends Configurable {

    // 拦截消息
    public ProducerRecord<K, V> onSend(ProducerRecord<K, V> record);
    // 接收到broker ack应答调用
    // 当Exception为null时消息生产成功
    public void onAcknowledgement(RecordMetadata metadata, Exception exception);
    // 拦截器关闭时调用，发送完消息后关闭
    public void close();
}

自定义分区策略

上面已经提到过，Partitioner会根据 ProducerRecord 中的key来定向将消息数据持久化到指定partition中，但没有指定 key 的话，就会通过轮询方法来持久化消息数据。而如果默认的分区策略（轮询）无法满足业务需求的话就可以尝试自己动手实现分区策略，可以实现以下接口来实现：

public interface Partitioner extends Configurable, Closeable {

    // 分区逻辑实现
    public int partition(String topic, Object key, byte[] keyBytes, Object value, byte[] valueBytes, Cluster cluster);

    // Partitioner关闭，用于回收资源操作
    public void close();

}

常用配置参数

bootstrap.servers

  • kafka broker地址，host:ip，多个用逗号隔开

key.serializer

  • 指定 key 使用那种序列化器(必须使用全类名)

value.serializer

  • 指定 value 使用那种序列化器(必须使用全类名)

acks

  • 用于判断消息是否发送成功的依据

  • 值为0时，producer完全不理会 broker 的ack响应，发送完一条消息后马上发下一条消息，这时producer的吞吐量最高，但因为完全不理会 broker 的ack响应，所以就算出现发送失败的情况也不会做重试操作，因此会存在消息丢失的问题

  • 值为1时（默认值），producer会等待 leader broker 的ack响应，换句话就是只要 leader border 保存数据成功，producer 就会视为消息数据发送成功

  • 值为-1或all时，producer会等待集群中所有 broker 的ack响应，会等待所有 brokder 将消息持久化到日志后才会发送下一条消息。这种配置能够百分百成功持久化消息数据，但因吞吐量是3种配置中最低的

  • 注意这里提到的发送并不是指调用send方法，而是消息缓冲池满了后发送消息给 brokder 的操作

buffer.memory

  • 指定消息缓冲池的大小，默认为32M
  • 需要注意，该值尽量不宜设置得太小，如果设置得太小而 producer 执行 send 的频率又特别高的话，就会导致消息缓冲池很容易被撑爆，而消息缓冲池被撑满之后，producer线程的 send 操作时会阻塞的，因为需要等待brokder线程将消息缓冲池发送给Broker，既情况消息缓冲池后才可以继续将消息send进缓冲池

block.on.buffer.full

  • 缓冲池用尽后停止接受新的消息且不抛出异常

compression.type

  • 用于指定消息的压缩类型，默认为none，既不对消息进行压缩操作
  • 目前 kafka 支持的压缩格式有 gzip、snappy、lz4

retries

  • 消息重发机制

  • 消息发送失败的原因可能是 Broker 出现网络问题又或者时正处于 Leader 选举而导致的

  • 值0时（默认），表示不进行重试

  • 值为n时，表示可以重试n次

  • 要十分注意，重试操作可能会导致消息重复发送。例如因为网络原因 Brokder 保存消息成功但没有成功响应 Producer，而 Producer 因为没有收到 Broker 的 ack 从而进行了消息重发

  • 除此之外，开启重试操作还会出现消息乱序问题。例如a、b、c三条数据，a和c发送成功但b失败了，b重新发送，就导致原本的abc编程了acb

max.in.flight.requests.per.connection

  • 用于设置 producer 连接上允许发送的未应答请求数（默认为5）

  • 通俗点说就是在 Broker 未响应之前 producer 可以 send 多少条数据

  • 注意，如果开启了 retries 但又想保持重试消息要优先于之后的消息，那么可以将该值设置为1。

    • 举个栗子：当前有两条数据 xxx 和 ccc 要发送给 Broker ，但因为某些原因ccc成功发送而xxx失败进入了重试操作，而在 xxx 重试期间 producer 又给Broker发送了消息 nnn 且发送成功了，这时就算 xxx 重试成功，但显然已经乱需了，因为 nnn 应该在 xxx 后面。而将max.in.flight.requests.per.connection 设置为1，那么在重试发送 xxx期间只要broker没有应答，producer就不会发送nnn

retry.backoff.ms

  • 用于设置重试时间间隔（默认为100毫秒）

batch.size

  • 用于设置producer的send操作的batch值（默认为16k）

  • 但需要注意的是 producer 也并不是每次都等到 batch 值才会将其放入 buffer 中的。换句话来说，batch.size 只是一个上限制，既满了就必须放入 buffer ，但不要求一定要放满(具体参考linger.ms参数)

  • 通常可以增大 batch.size 来提高 send 操作的吞吐量，但是与此同时会加大内存的压力，因为不管怎么样batch.size 都会占据固定大小的内存

linger.ms

  • 用于设置消息发送的延时时间。默认值为0，既不进行延时发送

  • 如果你想要尽量填满 batch.size 以提高吞吐量，那么就可以适当地调高 linger.ms 值

  • 延时操作虽然能够提高消息的发送效率，但是实时性较强的业务场景下不建议使用

max.request.size

  • 用于设置消息数据的最大值（默认值差不多1M）

  • 如果需要发送的消息很大，就可以适当低调到该值，但建议对应着 Brokder 中的配置message.max.bytes（Brokder 能够接收消息的最大值）来设置

request.timeout.ms

  • 用于设置 Brokder 响应超时时间（默认为30s）
  • 既 producer 发送请求，broker要在30s内响应producer，否则会抛出 TimeoutException

partitioner.class

  • 用于指定自定义分区策略实现

interceptor.classes

  • 用于指定自定义消息拦截器实现

  • 和 listeners 配置类似，一般用于 IaaS 环境，既服务器有内网ip和外网ip（例如阿里云）时

  • 例如 kafka 处于阿里云服务器，默认情况下是无法通过外网访问到kafka的，解决方法是修改如下配置

  • listeners=PLAINTEXT://内网ip:9092
advertised.listeners=PLAINTEXT://外网ip:9092

  • 指定处理网络请求的线程数（默认为3）

  • 指定处理网络IO任务的线程数（默认为8）

  • 指定消息数据的持久化目录

  • 例子：log.dirs=/tmp/data 或者 log.dirs=/tmp/data,/tmp/data2

  • 建议：如果 kafka 服务器有多块磁盘，则推荐指定多个日志文件，因为这样可以用负载均衡的方法存储日志数据，可以大大提高写入操作的吞吐量

  • 是否允许非ISR成员参与 Leader 选举（默认值为false）

  • 如果设置为true，在一些较为极端的情况下，如 ISR 全部宕机后，非ISR被选举成为leader就会造成数据丢失

  • 用于设置日志文件的最大值。既当超过该值就会则会被清除（默认为-1 表示不启用该功能）

  • 该参数只对 producer 中的 acks 值-1 或 all 时才会起作用

  • 作用是设置必须应答成功的 replica 数。例如某个topic的 partition 有3个replica，而现在 min.insync.replicas = 2，则表示 kafka 集群只允许1台 broker 响应不成功，否者producer就需要进入重试操作

  • 指定 Broker 能够接收的消息的最大值（默认为差不多1M）

Java kafka Consumer

Consumer Group概念

为什么需要消费者组？

当 producer 生产消息的速度比 consumer 的消费速度快时，consumer 的消费能力就显得低下了。因此无法满足一些对实时性要求较高的业务。那么消费者组就是用来提高 consumer 的消费能力的

消费者组说明

消费者组订阅topic T1，其中 T1 的 partition 数量为4。当消费者组中仅有1个消费者时

消费者组订阅topic T1，其中 T1 的partition数量为4。当消费者组中有2个消费者时

消费者组订阅topic T1，其中T1的partition数量为4。当消费者组中有4个消费者时

消费者组订阅topic T1，其中T1的partition数量为4。当消费者组中有5个消费者时

2个消费者组同时订阅topic T1，其中 T1 的partition数量为4。

消费者组1，有4个消费者。消费者组2，有2个消费者。

小结：

  1. 相同 Consumer Group中的不同 Consumer，在同一时刻无法消费同一个 partition
  2. Consumer Group中的 Consumer可以同时消费多个 partition
  3. 即使 Consumer Group中的 Consumer数量多于 partition数量，也不会提高 Consumer Group的消费能力
  4. 多个 Consumer Group同时订阅同一个 topic 时是互不干扰

分区再均衡

是什么？

在消费者没超过partition数的情况下，消费者组中的每个 Consumer 都有一个自己负责消费的 partition 。但因为某些原因，例如有新的Consumer 加入到当前消费者组中，那么消费者组就会重置 Connsuemr 和 partition 的消费关系，而这个过程就叫做分区再均衡

什么情况会发生分区再均衡？

  • 有新的消费者加入到消费者组中
  • 消费者组中有消费者脱离，例如宕机
  • topic 修改了 partition 数量

如何确定消费者死亡？

首先 Broker 会作为消费者组的协调器，消费者在消费消息或提交offset时会想 Broker 发送心跳包。所以如果当消费者长时间没有给 Broker 发送心跳包的话就会被视为死亡，这时会发分区生再均衡操作

需要注意的事情：

  • 在再均衡期间，消费者无法消费消息，所以会造成短时间内消费者无法消费的情况
  • 当partition被重新分配给新的消费者时，新消费者会丢失消费状态直到再均衡结束为止，因此会拖慢消息的消费速度

常用参数

bootstrap.servers

  • kafka broker地址，host:ip，多个用逗号隔开

key.deserializer

  • 指定 key 使用那种序列化器（必须使用全类名）

value.deserializer

  • 指定value使用那种序列化器（必须使用全类名）

group.id

  • 指定 Consumer group 的名字，该名字必须在 Broker 中是唯一的

  • 默认值为空字符串，建议尽量取一个与业务有关或者有意义的名字，这样可以提高辨析度

session.timeout.ms（kafka0.10.1.0版本之前，非常重要）

  • 第1个含义：用于指定 Broker 协调器对 Consumer 停止发送心跳包这种行为的最大容忍限度（默认为3秒）。一旦超过该配置指定的事件，Consumer 就会被视为死亡，因此会进行分区再均衡操作
  • 第2个含义：Consumer 处理消息的最大时间限制（既两次poll之间的时间）。例如 session.timeout.ms = 60000 那么消费者处理消息必须在60s内完成，否则 Broker 协调器会将该 Consumer 视为跟不上消费进度并将其踢出消费者组，因此也会进行分区再均衡操作

max.poll.interval.ms

  • kafka0.10.1.0 版本之后 session.timeout.ms 进行了拆分，拆分之后 session.timeout.ms 只包含第1个含义，且默认值为10s

  • 而 max.poll.interval.ms 则是第2个含义（建议根据实际处理消息所需要的时间来设置，最好比实际所需的时间大一些）

auto.offset.reset

  • 该配置用于指定了消费者在读取一个没有偏移量(还没有消费提交offset)或偏移量无效（因消费者长时间失效，包含偏移量的记录已经过时并被删除）的分区时的应对手段。
  • Kafka提供以下 3 种消费模式：
    • latest（默认）：
      • 在偏移量无效的情况下，Consumer 会从最新插入的那条记录开始消费（在消费者启动之后插入的记录）
      • 偏移量有效的情况下，Consumer 会在最新的偏移量位置开始消费（既上一次提高的offset位置）
    • earliest：
      • 在偏移量无效的情况下，Consumer 会从 Partition 的起始位置开始消费
      • 偏移量有效的情况下，Consumer 会在最新的偏移量位置开始消费（既上一次提高的offset位置）
    • none：
      • 各个Partition都存在有效的 offset 时，会从有效的 offest 位置开始消费
      • 只要有一个分区不存在有效的 offset，则抛出异常

enable.auto.commit（非常重要）

  • 是否自动提交 Consumer offset（默认为true）

  • 为了尽量避免重复消费的情况出现，建议把它设为false，然后由程序来控制提交时机

  • 如果设为true，则可以通过配置 auto.commit.interval.ms 来控制提交offset的时间间隔

auto.commit.interval.ms

  • 该参数必须在 enable.auto.commit = true 时才生效;

  • 作用是控制自动提交 offset 的时间间隔（默认为 5000）

partition.assignment.strategy

  • 用于指定分区分配策略，通俗点来说就是如何将分区分配给消费者

  • 默认为range，既随机分配，还可以设置成 roundrobin 轮询分配

client.id

  • 用于给 Broker 设置客户端标识

fetch.max.bytes（非常重要）

  • 用于指定 Consumer 单次获取消息数据的最大字节

  • 如果实际业务中消息都很大的，那么就需要设置一个较大的值，否这无法获取数据

max.poll.records（非常重要）

  • 用于指定单次 poll 能够获取到的最大 record 数量（默认为500）
  • 建议：
    • 如果用户程序处理消息所需的时间较长，那么应该将该值设置得小一些，例如1，否则很容易被 Borker协调器认为消费能力过低从而被提出消费者组
    • 如果用户程序的处理能力较强，则可以设置大一些，这样有利于提高Consumer的消费能力

heartbeat.interval.ms（非常重要）

  • 用于指定消费者发送心跳包的时间间隔（默认为3000ms）

  • 该值必须必比Broker的 session.timeout.ms 配置小，通常不高于1/3，甚至可以更低

  • 可以简单理解 heartbeat.interval.ms 可以让 Broker协调器更快检测到当前消费者是否出现故障

手动提交offset

自动提交存在的问题

按照默认的offset提交频率5秒为例，假设现在消费者已经提交了offset，但过了3秒后（未够5秒提交offset）发生了分区再均衡操作，而这时因为不够 5 秒来提交 offset 且再均衡期间 Broker 会停止向客户端提供服务，所以在这3秒期间的消费是没能告诉给 Broker 知道的，也就是说这 3 秒内所消费的消息在再均衡后会被重复消费。该问题可以使用手动提交 offset 来解决，既程序可以在消费完后马上提交 offset 而无需等待到下一个5秒才提交offset，因此可以有效地防止数据丢失。

同步提交

while (true) {

    ConsumerRecords<String, String> records = consumer.poll(100);

    // 处理消息
    // 。。。

    try {
        consumer.commitSync();
    } catch (CommitFailedException e) {
        log.error("commit failed", e)
    }
}

同步提交是一个阻塞操作，既需要等待 Broker 应答后才能继续工作，换句话来说只要不是发生无法恢复的异常commitSync() 都能提交成功。但这样虽然能够保证 offset 的安全提交到Broker，但无疑会降低消费者的吞吐量

异步提交

用于解决同步提交导致吞吐量下降的问题

while (true) {

    ConsumerRecords<String, String> records = consumer.poll(100);

    // 处理消息
    // 。。。

    consumer.commitAsync();
}

需要注意的是 commitAsync() 不像 commitSync() 那样保证能够成功地将 offset 提交到 Broker，换言之就是可能会出现提交失败的情况。这时可以给 commitSync() 设置回调方法进行重试解决

while (true) {
    ConsumerRecords<String, String> records = consumer.poll(100);

    // 处理消息
    // 。。。

    consumer.commitAsync(new OffsetCommitCallback() {
        public void onComplete(
            Map<TopicPartition,
            OffsetAndMetadata> offsets, 
            Exception exception) {

            if (e != null){
                // 异常处理
            }
        }
    });
}

混合提交

try {
    while (true) {
        ConsumerRecords<String, String> records = consumer.poll(100);

        // 处理消息
        // 。。。

           // 异步提交
        consumer.commitAsync();
    }
} catch (Exception e) {
    log.error("Unexpected error", e);
} finally {
    try {
        // 同步提交，用于防止异步提交失败
        consumer.commitSync();
    } finally {
        consumer.close();
    }
}

混合提交可以进一步保证 offset 能够成功提交到Broker

@Configuration
public class KafkaConsumerConfig {

    @Autowired
    private KafkaProperties properties;

    @Bean
    public ConsumerFactory<String, String> kafkaConsumerFactory() {
        // 注入消费者配置
        Map<String, Object> map = properties.buildConsumerProperties();
        // 默认允许处理消息的最大时间，默认300s
        map.put(ConsumerConfig.MAX_POLL_INTERVAL_MS_CONFIG, 50000);
        return new DefaultKafkaConsumerFactory<>(map);
    }

    @Bean
    public KafkaListenerContainerFactory<ConcurrentMessageListenerContainer<String, String>> kafkaListenerContainerFactory() {
        ConcurrentKafkaListenerContainerFactory<String, String> factory = new ConcurrentKafkaListenerContainerFactory<>();
        factory.setConsumerFactory(kafkaConsumerFactory());
        // 消费完马上提交
factory.getContainerProperties().setAckMode(ContainerProperties.AckMode.MANUAL_IMMEDIATE); 
        // 异步提交
      factory.getContainerProperties().setSyncCommits(true);
        return factory;
    }

}

如何保证消息不丢失

数据丢失的核心原因是 Broker 没有保存成功

Producer配置

  • block.on.buffer.full = true

    消息缓冲池满后停止接收新的消息而不抛出异常，避免因为异常丢失数据

  • acks = all

    生产者生成消息后需要得到所有的 Broker 的应答，否则认为发送不成功，因此进行重试发送

  • retries = Integer.MAX_VALUE

    将重试次数设置到最大值，表示尽量重试

  • max.in.flight.requests.per.connection = 1

    因为开启了重试，所以将其设置为1来保证消息的顺序

  • 发送消失时，预防发送失败，使用回调函数来补救

Broker配置

  • unclean.leader.election.enable = false

    避免非ISR成员竞选为 Leader 从而导致数据丢失

  • replication.factor 设置大于等于 3

    尽量使用备份来避免分区单点故障

  • min.insync.replicas = 1

    表示最少有1个 replica 成功写入数据，否则视为持久化不成功（producer抛出异常）。该配置只有在producer设置了acks=all 才有效。（acks是应答，而 min.insync.replicas 是限制至少多少 replica 保存数据成功）

如何保证只消费1次

一条消息多次消费的原因：

  • offset 没能及时提交
  • 生产者开启了重试功能，导致消息重复生产
  • 消费者处理消息的时间过长，导致发生分区再均衡（和第一种类似）

其实对于消息重复消费而言，在消息系统层面是那以避免的，但这个问题可以通过对程序接口进行幂等处理来解决。

如何解决消息乱需问题

首先需要知道的是目前的 Kafka 并不支持全局（多个Partition之间）消息有序，只能保证单个 Partition 内是有序的。且当生产者开启重试功能后，Partition 内部也是可能发生乱序情况的。这时就需要将max.in.flight.requests.per.connection 设置为1来尽量降低乱序情况
