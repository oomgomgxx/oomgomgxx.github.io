拓展知识点

为了更好地学习和理解kafka，个人觉得有必要搞清楚一些概念

在开始之前，推荐书籍《Apache Kafka实战》和《Kafka权威指南》来作为kafka的入门书籍

什么是磁盘的顺序写入？

  所谓的磁盘顺序写入其实是相对与随机写入而言的，指的是在写入数据时磁盘的磁头不用寻道而可以一次将数据写完，因为省略了磁头寻道的时间，所以在速度来说是非常快的，甚至可以媲美内存的随机写入; 那什么情况下会出现顺序写入呢？如文件追加内容时（kafka就采用这种方式进行顺序写入，因为kafka存放数据的日志文件不能修改只能追加）

以下为磁盘、SSD、内存之间的访问速度对比图

图片源自：queue.acm.org

可以看出，磁盘的顺序访问是相当快的，所以kafka选择将内容写入磁盘

内存映射文件

  游戏就经常玩了，但不知道有没有这么一个疑问：游戏体积这么大，而内存这么小，那么操作系统到底是怎么运行游戏的呢？答案就是，操作系统会预先为程序生成一个内存镜像保存在磁盘中，在运行该程序时并不会真的将该程序全部加载进主存，而是当使用到某部分代码时才会加载进主存，而这个保存程序内存镜像的文件就是内存映射文件。操作可以可以直接将内存映射文件映射到对应的内存地址中，因此对于用户线程而言是可以直接访问到该内存地址的。而且操作系统会自动管理者块内存空间，例如操作完该内存中的数据后系统会自动将脏数据刷会文件中。

参考图

  在操作系统中一般会使用页来作为内存的基本单位。系统将用户进程中的虚拟内存地址划分为大小相等的页，这种页称为页面；而物理内存地址中亦划分为大小相等的页，这种页称为页框。每个页面的大小为2的整数次幂，范围在512字节～1M，通常为4KB（32bit系统）和8KB（64bit系统）。在管理层面，操作系统使用一种叫页表的结构来映射虚拟内存地址和物理内存地址之间的关系（虚拟内存通常远大于虚拟内存）。

参考图

从上图我们不难看出，并不是所有虚拟内存地址都有与之对应的映射的。当程序访问到一个没有映射的虚拟地址时，就会发生一种叫缺页中断的操作，该操作会将物理内存地址中较为不常用的内存页替换成当前缺页的程序部分。大体步骤是系统会从内存映射文件中加载内容到主存上（物理内存地址），然后更新进程的虚拟内存空间（虚拟内存地址），最后恢复到缺页中断位置的操作。

  最后说一下mmap操作，mmap是一个系统函数，它能够实现文件到内存的映射（内存映射文件）。而kafka就是使用mmap将日志索引文件内容映射到内存的，所以在定位数据方面速度也是相当快的。

kafka只用mmap来映射索引是因为如果同时映射索引和日志的话就会存在一直性问题，而且mmap并不能保证百分百成功将内存中的脏数据刷回磁盘。例如并发操作时修改了日志并回刷成功，但因为某些原因索引回刷失败，这就导致了数据不一致性问题

上线程下文切换

线程上下文切换

  指的是在任务执行前，CPU寄存器和程序计数器会预先加载将要执行的任务内容。首先操作系统会先记录下正在执行的任务内容和状态，以便下次执行时恢复。保存成功会操作系统就会把将要执行的任务内容加载到CPU寄存器和程序计数器中（例如每条线程都有栈和程序计数器），而这个过程就被称为上下文切换（实质就是切换执行任务）。上下文切换是一个系统层面的调度操作，所以在上下文切换期间，用户程序将会进入呆滞状态，等到切换完成后才会继续运行。

用户态和内核态切换

  操作系统将线程（轻量级进程）的执行空间分为：用户空间 和 内核空间。

线程在用户空间时，因为用户空间权限有限所以线程在对资源操作上也是有限的。例如用户空间的线程不能直接访问除系统分配给自己的内存以外的内容，譬如：磁盘、网络模块 等，如何想要操作这部分资源就需要发起系统调用，让系统的内核线程来完成，而这时就会发生所谓的用户态和内核态切换操作。

举个例子：用户线程想要读取磁盘上文件的内容，这时就需要进行系统调用。系统调用会将读取磁盘的操作交给内核线程来完成，过程如下：

所以，应用做IO操作时并不是直接操作设备资源的，而是通过内核来去操作的。内核线程读取磁盘上的文件内容后将内容先保存到内核缓冲中，然后再将内核缓冲中的内容通过CPU Copy操作复制到用户空间，这样用户现线程会访问到磁盘文件中的内容。

那么从上图我们不难看出，在1次io操作中就出现了2次用户态和内核态之间的切换操作。上面已经提到过，上下文切换或用户态内核态切换时用户程序都会进入呆滞状态来等待系统调用完成。

简单总结一下

虽然上面提到用户态和内核态切换，但其实它也是线程上下文切换的一种操作。从用户程序的角度来看这是一个比较重量。虽然在单个上下文切看来并不需要花费多少时间，但是一旦发生大量的上下文切换，那么将会直接导致用户程序处理情况的能力下降。

零拷贝（Zero-copy）

零拷贝通常会在网络传输上使用的。指的是内核态直接将buffer（page buffer）中的数据通过RMA Copy 传输给网络模块，而网络模块就可以直接将其发送出去了，而无需从内核态切换给用户态再执行发送操作。

零拷贝例子

需要说明的是，零拷贝并不是不拷贝，而是不进行 CPU Copy 而已。零拷贝的好处在于减少了用户态和内核态的切换次数，这样就减少了用户程序的呆滞时间，因此提高了用户程序的处理能力。

为什么kafka性能高？

  • 数据操作方式为磁盘的顺序访问
  • 通过 Zero-copy 传输网络数据
  • 使用 mmap 将日志索引映射到内存，提高了数据的访问速度
  • 支持批量发送操作，只有数据大小到达batch.size才会发出，目的是提高操作的吞吐量

关于topic、partition、replica之间的关系

  通常生产者生产一个消息是存放在Broker的topic中的，但是topic只是逻辑上的概念，而消息实际存放的地方是partition（分区）;

  partition是一个有序的序列，也就说明了为什么保存数据的日志文件不能修改只能追加 ;

  其中，1个topic可以有1个或多个partition，换句话说就是1个topic的数据分拆存放到1个或多个partition中; 还有就是，kafka只能保证单个partition中的消息数据是有序的，而如果1个topic有多个partition的话，并不能保证多个partition的有序性，既kafka是不能保证全局有序的 ;

  replica其实是partition另外一个身份，kafka防止partition中的日志数据丢失提供的冗余备份机制，其中replica主要分为两种：leader replica、follower replica;

  follower replica是不对客户端提供服务的，只负责同步leader replica中的数据，但需要注意的是多个replica时，replica肯定是分布在不同的broker上的，如果不是就没有意义了，所以基于partition和replica的缘故，kafka通常都是以集群的形式使用;

  而follower replica又分为 in-sync replica（ISR，能够实时同步leader数据的replica）和 非ISR ; 其中只有 ISR 能够选举成为 leader replica，且在borker接收消息请求时，只有在 leader replica 和 ISR 都接受到该请求，这条消息才会被broker认为发送成功; 实质上一开始所有的replica（包含leader replica）都属于 ISR集合，但经过运行后可能因为网络又或者其它的干扰因素，导致其它broker上的replica无法跟上复制 leader replica 数据的进度因此被踢出 ISR集合 ，但当被踢出 ISR集合的replica跟上进度又会被重新纳入ISR集合;

为什么需要partition？

  有些人可能会有这样的疑问，就是即然有了topic为什么要有partition的概念呢？如ActiveMQ就没有partition概念;

  首先kafka的目标是为处理实时数据提供一个统一、高吞吐、低延迟的平台; 而partition就是用来提高topic的操作吞吐量的，既为topic提供负载均衡的功能;

上图源自kafka官网，可以看出，当1和topic有多个partition时，生产者生产数据可以同时分别写入到同一个topic的不同partition中，但如果只有一个partition的话，并发写入时压力就全部落在这个partition上，因此会降低操作的性能;

1个partition多个segment主要是为了方便查找数据，因为每个segment的文件名称就包含了数据的offset信息，因此在二分查找时只需找到对应的segment即可，而不需要扫描所有的消息数据;

topic offset和consumer offset

topic offset

当消息数据被写到partition时就会分配一个顺序的offset，称为偏移量;

offset从0开始随着数据增加而递增，不同的partition的offset并不干扰;

consumer offset

消费者offset和topic offset不一样这个不要混淆;

消费者offset主要是用来记录分区中消息被消费到的位置，换句话来说是用来在发生再均衡后来告诉新的消费者应该从哪里开始消费的标记;

需要注意的是，在kafka中消费者offset是由消费者（消费者组）自己保存的，这样做的好处在于减轻了broker压力，如果让broker记录的话就会让broker带有状态。而且kafka consumer引入了checkpointing定期对offset进行持久话。

consumer offset提交

消费者会定期向broker提交offset来汇报自己的消费进度，这个动作称为位移提交;

在新版本的kafka中，有个专门用来保存新版consumer（就版保存在zookeeper中）的offset的内部topic（__consumer__offsets），这个也可以在kafka的日志文件夹下看到，一共有50个文件夹，既这个topic有50个partition，如下

这是因为利用partition的负载均衡特性来减轻同时提交offset时topic的压力;

Java kafka Producer

生产者模型

注意：Producer是异步发送的，除了当前调用send方法的线程外还会创建一条专门负责冲消息缓冲池取数据发送给broker的线程

  • ProducerRecord：消息
  • Serializer：序列化ProducerRecord对象进行网络传输，注意生产者和消费者的序列化器类型要一致，否则消费者无法进行反序列化操作
  • Partitioner：分区器，根据ProducerRecord中的key来判断保存在那个所保存到那个分区，如同一个key会保存在同一个分区，但ProducerRecord没有指定key的话则采用轮询分区方式保存消息
  • 消息缓冲池：用于减少用户态和内核态的切换次数，提高操作性能

producer interceptor

生产者拦截器，可以在消息序列化之前将其拦截，做一些消息的定制化修改;

继承以下接口，并添加进生产者的配置即可（ProducerConfig.INTERCEPTOR_CLASSES_CONFIG）;

public interface ProducerInterceptor<K, V> extends Configurable {

    // 拦截消息
    public ProducerRecord<K, V> onSend(ProducerRecord<K, V> record);
    // 接收到broker ack应答调用
    // 当Exception为null时消息生产成功
    public void onAcknowledgement(RecordMetadata metadata, Exception exception);
    // 拦截器关闭时调用，发送完消息后关闭
    public void close();
}

producer的分区策略

在默认情况下，如果ProducerRecord设置了key值，则Partitioner会进来确保具有相同key的消息发送到同一个partition上（根据key的hash值和分区数取模），而如果没有设置key值的话，则是通过轮询来将消息均匀地发送到不同的partition上;

如果默认的分区策略无法满足业务的使用，则可以尝试自己来实现分区策略，通过实现以下接口：

public interface Partitioner extends Configurable, Closeable {

    // 分区逻辑实现
    public int partition(String topic, Object key, byte[] keyBytes, Object value, byte[] valueBytes, Cluster cluster);

    // Partitioner关闭，用于回收资源操作
    public void close();

}

常用配置参数

[bootstrap.servers]

kafka broker地址，host:ip，多个用逗号隔开;

[key.serializer]

指定key使用那个序列化器，必须使用全类名;

[value.serializer]

指定value使用那个序列化器，必须使用全类名;

[acks]

producer判断消息是否发送成功的依据;

acks为0时，producer完全不理会broker的ack应答，发送完一条消息后马上发下一条，此时producer的吞吐量最高，但因为完全不理会broker的ack应答，所以就算发送失败也不会做重试操作，因此可能会丢失消息;

acks为1时（默认值），producer会等待leader broker的ack应到，换句话就是只要leader border保存数据成功（持久化到日志），producer就视为发送成功;

acks为-1或all时，producer会等待集群中所有broker的ack应到，既等所有的brokder将消息持久化到日志才进行下一条消息的发送，这时消息的持久性是最好的，但因为需要等待所有的broker应答，因此吞吐量也是3种中最低的;

还有就是，这里的”发送”并不是调用了send，而是将消息缓冲池中的数据发送给brokder这个发生

[buffer.memory]

指定消息缓冲池的大小，默认为32M;

如果修改该值，需要注意的是不要太小，因为如果producer的send速度过快，且超过了发送消息给brokder的线程的话，肯定是会把memory撑满的，撑满之后producer线程就会停止send操作来等待发送消息给brokder的线程腾出memory空间再进行send

[block.on.buffer.full]

缓冲池用尽停止接受新的消息而不抛出异常;

[compression.type]

用于指定消息的压缩类型，默认为none，既不对消息进行压缩;

目前kafka支持：gzip、snappy、lz4

[retries]

producer提供的消息重发机制;

消息发送失败的原因可能是broker瞬时之间因为网络问题又或者执行了leader选举而导致的;

retries为0时（默认），表示不进行重试;

retries为n时，表示可以重试n次;

但要注意，重试这个操作是可能会导致消息重复发送的，如因为网络原因，brokder保存消息成功，但没有成功响应producer;

还会出现消息乱序问题，这是因为producer会将消息发送请求缓存在内存中的原因，那么当重写时，原本因该在这么缓存在内存的消息请求前面的消息，就会变成在后面了因此而乱序了，所以如果开启retries功能的话需要将producer参数设置如：max.in.flight.requests.per.connection = 1（针对partition区内）

[max.in.flight.requests.per.connection]

用于设置producer单个连接上发送的未应答请求的最大数量，默认为5;

通俗点说就是在broker未相应之前producer可以send多少条数据;

需要注意的是，如果开启的retries但想要保持有序的话，那么需要将该值设置为1。

例子说明原因：将设max.in.flight.requests.per.connection=2，这是有两条数据 xxx 和 ccc 要发送给broker，但出于某些原因ccc成功而xxx没成功进入重试，而在重试期间数据 nnn 发送给了broker，这时就算 xxx 重拾成功，但显然乱需了，因为 nnn 应该在 xxx 后面。如果max.in.flight.requests.per.connection=1，那么发送了 xxx 之后，只要broker不应到，producer就不会发送nnn;

[retry.backoff.ms]

用于设置开启retries后的重试间隔时间，默认为100毫秒，当然这是在开启了重试机制前提下才有效的;

[batch.size]

batch.size用于设置存放将要发送到同一个paritition的batch值;

但需要紧记的是，producer也并不是每次都等到batch慢才放入buffer，换句话来说，batch.size是一个上限制，既满了就必须放入buffer，但不要求一定要放满(具体参考linger.ms参数);

batch.size默认值为16384（16k）;

通常可以增大batch.size来提高producer的吞吐量，但是与此同时会加大内存的压力，因为不管怎么样batch.size都会占据固定大小的内存;

[linger.ms]

linger.ms用于设置消息的发送延时时间，默认为0，既不进行延时发送;

如果你想要尽量填满batch.size，那么就可以适当地设置linger.ms来提高producer的吞吐量，但这样固然就会延迟了消息发送到broker的时间，所以linger.ms的设置需要按照实际的需要来权衡，既想要增大producer的吞吐量就设置，想要降低消息发送的延迟就不设置;

[max.request.size]

max.request.size用于设置producer发送请求的最大值，默认为 1048574 byte;

如果producer需要发送的消息很大，可以作出相应的调整，但建议对应brokder中的配置message.max.bytes（brokder能够接收消息的最大值）来设置

[request.timeout.ms]

request.timeout.ms用于设置broker接收到producer请求后的响应超时时间，默认为30s，既producer发送请求，broker要在30s内响应producer，告诉producer是否持久化成功，如果超时则抛出TimeoutException;

[partitioner.class]

partitioner.class用于指定自定义分区策略实现;

[interceptor.classes]

interceptor.classes用于指定自定义消息拦截器实现;

Broker常用配置参数

[broker.id]

指定kafka在集群中的id值，默认为-1;

需要注意的是，集群中的所有broker的id都必须唯一;

[zookeeper.connect]

zookeeper的地址，格式：ip:port

[listeners]

kafka broker监听（bind）的地址，默认采用情况使用java.net.InetAddress.getCanonicalHostName() 来配置;

格式：listeners = listener_name://host_name:port

例子：listeners = PLAINTEXT://your.host.name:9092

[advertised.listeners]

advertised.listeners和listeners的功能类似，一般用于IaaS环境，既kafka服务器有内网ip和外网ip（阿里云）;

如果kafka处于如阿里云这样的服务器，则如果不配置advertised.listeners是无法通过外网访问到kafka的，解决这个问题需要以下配置：

listeners=PLAINTEXT://内网ip:9092
advertised.listeners=PLAINTEXT://外网ip:9092

[num.network.threads]

指定处理网络请求的线程数，默认为3;

[num.io.threads]

指定执行网络请求的io线程数，默认为8;

[log.dirs]

指定保存日志文件（数据文件）的绝对路径;

例子：log.dirs=/tmp/data 或者 log.dirs=/tmp/data,/tmp/data2

建议，如果kafka服务器有多块磁盘的话，推荐指定多个日志文件，因为这样kafka可以负载均衡存储日志数据，多块磁盘可以同时写入，因此大大提高写入操作的吞吐量;

[unclean.leader.election.enable]

是否允许非ISR中的副本参与leader选举，默认值为false（版本0.11后），既不允许非ISR中的副本被选举为leader;

如果设置为true，在一些较为极端的情况下，如ISR全部宕机后，非ISR被选举成为leader就会造成数据丢失;

[delete.topic.enable]

是否允许删除topic，默认为true。

[log.retention.{hours|minutes|ms}]

用于设置日志文件的保存时间，默认情况下保存7天。

[log.retention.bytes]

用于设置保留日志的大小临界，既当超过该值就会则会被清除，默认为-1，既不启用该功能。

[min.insync.replicas]

该参数只针对producer参数acks为-1或all时才会起作用。

作用是指定必须应答成功写入的replicas最小数。如果partition有3个replica，min.insync.replicas = 2，则表示kafka集群只允许1台broker宕机不响应。

[message.max.bytes]

指定kafka能够接收的消息的最大值，默认为977KB。

Consumer Group概念

为什么需要消费者组？

因为当producer生产消息的速度比consumer速度快时，consumer就无法即使消费消息，对于一些实时要求较强的业务功能就无法满足需求了。

用图片了解消费者组

消费者组中仅有1个消费者;

消费者组订阅topic T1，其中T1的partition数量为4;

消费者组中有2个消费者;

消费者组订阅topic T1，其中T1的partition数量为4;

消费者组中有4个消费者;

消费者组订阅topic T1，其中T1的partition数量为4;

消费者组中有5个消费者;

2个消费者组同时订阅topic T1，其中T1的partition数量为4;

消费者组1中有4个消费者;

消费者组2中有2个消费者;

总结上图：

  1. 相同Consumer Group中的不同Consumer，在同一时刻无法消费同一个partition
  2. Consumer Group中的Consumer可以同时消费多个partition
  3. 即使Consumer Group中的Consumer数量多于partition数量，都不会提高Consumer Group的消费能力
  4. 多个Consumer Group同时订阅同一个topic互不干扰

分区再均衡

是什么？

通俗来说，再均衡指的是消费者与原来消费开的partition关系发生改变。

其实我觉得叫消费者再均衡更加贴切。

什么情况会发生再均衡？

  1. 消费者组加入新的消费者
  2. 消费者组中有消费者宕机
  3. topic修改partition数量

如何确定消费者死亡？

首先broker会作为消费者组的协调器，消费者在消费消息或提交offset时会发送心跳包。所以如果当消费者长时间（特定）不望broker发送心跳包，则会被broker视为死亡，因此就会发生再均衡。

需要注意写什么？

  • 在在均衡期间，消费者无法消费消息，所以造成短时间消费者组不可用
  • 当分区被重新分配给新的消费者时，该消费者会掉失消费状态知道重新回复位置，因此会托慢应用的消费速度

常用参数

[bootstrap.servers]

kafka broker地址，host:ip，多个用逗号隔开;

[key.deserializer]

指定key使用那个序列化器，必须使用全类名;

[value.deserializer]

指定value使用那个序列化器，必须使用全类名;

[group.id]

指定consumer group的名字，这个名字必须在broker中是唯一的;

默认之为空字符串，建议是尽量取一个与业务有关有意义的名字;

[session.timeout.ms]（kafka0.10.1.0版本之前）

非常重要的属性;

第1个含义：用于指定broker协调器对消费者故障时间的最大容忍程度，默认为3秒; 如当 session.timeout.ms = 60000 时，消费者组中某个消费者故障，必须在60s内与broker协调器通信，否则视为死亡，则会进行再均衡;

第2个含义：消费者处理消息的最大时间限制(两次poll之间的时间间隔)，例如 session.timeout.ms = 60000 那么消费者处理消息必须在60s完成。否则broker协调器会将该消费者视为跟不上消费进度，会将其踢出消费者组，因而会发生再均衡;

[max.poll.interval.ms]

kafka0.10.1.0版本之后，session.timeout.ms进行了分离，分离之后session.timeout.ms只包含第1个含义，且默认为10s，推荐尽量设置小一些;

而max.poll.interval.ms则是第2个含义，建议设置为比每条消息所需的时间稍微大一些;

[auto.offset.reset]

该配置指定了消费者在读取一个没有偏移量的分区(还没有消费提交offset)或者偏移量无效（因消费者长时间失效，包含偏移量的记录已经过时井被删除）的情况下该作何处理;

配置有3种：

  • latest（默认）：
    • 在偏移量无效的情况下，consumer将从最新的记录开始读取数据（在消费者启动之后生成的记录）;
    • 偏移量有效，在最新的偏移量开始;
  • earliest：
    • 在偏移量无效的情况下，consumer将从起始位置读取分区的记录;
    • 偏移量有效，在最新的偏移量开始;
  • none：
    • topic各分区都存在已提交的offset时，从提交的offest处开始消费；
    • 只要有一个分区不存在已提交的offset，则抛出异常

[enable.auto.commit]

非常重要，该配置指定了consumer是否自动提交偏移量，默认值是true;

为了尽量避免出现[重复消费]和[数据丢失]，可以把它设为false，然后由自 己控制何时提交偏移量;

如果把它设为true，还可以通过配置 auto.commit.interval.ms 属性来控制提交的频率;

[auto.commit.interval.ms]

该参数必须在 enable.auto.commit = true 时才生效;

功能是控制consumer自动提交offset的时间间隔，默认为5000;

[partition.assignment.strategy]

用于指定分区分配策略，通俗点来说就是如果将分区分配给对应的消费者;

默认为range，既随机分配，还可以设置成roundrobin，轮询分配;

[client.id]

用于给broker作为客户端表示;

[fetch.max.bytes]

非常重要，用于指定consumer单次获取的消息数据的最大字节，默认为52428800;

如果实际业务每条消息都很大，则必须设置一个较大的值，否这无法获取数据;

[max.poll.records]

非常重要，指定单次poll()能够获取到的最大record数量，默认为500;

建议：

  • 如果处理消息的逻辑需要较长时间，则尽量将该值设置小一些，如1;
  • 反之则可以设置大一些;

[heartbeat.interval.ms]

非常重要，设置消费者发送心跳的时间间隔，默认为3000;

该值必须必比session.timeout.ms小，通常不高于1/3;它可以调整的更低，以控制正常重新平衡的预期时间;

可以简单理解 heartbeat.interval.ms 可以让broker协调器更快检测到消费者故障;

手动提交offset

自动提交存在的问题

按照默认的offset提交频率5秒为例，假设现在消费者已经提交了offset，但过了3秒后（未够5秒提交offset）发生了再均衡，换句话来说就是因为不够5秒提交offset且再均衡期间Broker停止提供服务，所以在这3秒期间的消费是没有提交offset的，也就是说这3秒期间所消费的消息在再均衡后会被重复消费。当然这是比较极端的情况，这个问题可以使用手动提交offset来解决，可以在消费完后马上提交而不需要等待时间来提交;

同步提交

while (true) {

    ConsumerRecords<String, String> records = consumer.poll(100);

    // 处理消息
    // 。。。

    try {
        consumer.commitSync();
    } catch (CommitFailedException e) {
        log.error("commit failed", e)
    }
}

同步提交是一个阻塞操作，既需要等待Broker应答后才能继续工作，换句话来说只要不是发生无法恢复的异常commitSync()都能提交成功;

这样虽然保证了offset的提交安全，但无疑会降低消费者的吞吐量;

异步提交

用于解决同步提交降低吞吐量的问题

while (true) {

    ConsumerRecords<String, String> records = consumer.poll(100);

    // 处理消息
    // 。。。

    consumer.commitAsync();
}

需要注意的是commitAsync()不会像commitSync()那样重试提交，也就是说可能会提交失败，这个可以使用commitSync()的回调方法来处理（回调方法会在Broker响应后触发）;

while (true) {
    ConsumerRecords<String, String> records = consumer.poll(100);

    // 处理消息
    // 。。。

    consumer.commitAsync(new OffsetCommitCallback() {
        public void onComplete(
            Map<TopicPartition,
            OffsetAndMetadata> offsets, 
            Exception exception) {

            if (e != null){
                // 异常处理
            }
        }
    });
}

混合提交

try {
    while (true) {
        ConsumerRecords<String, String> records = consumer.poll(100);

        // 处理消息
        // 。。。

           // 异步提交
        consumer.commitAsync();
    }
} catch (Exception e) {
    log.error("Unexpected error", e);
} finally {
    try {
        // 同步提交，用于防止异步提交失败
        consumer.commitSync();
    } finally {
        consumer.close();
    }
}

@Configuration
public class KafkaConsumerConfig {

    @Autowired
    private KafkaProperties properties;

    @Bean
    public ConsumerFactory<String, String> kafkaConsumerFactory() {
        // 注入消费者配置
        Map<String, Object> map = properties.buildConsumerProperties();
        // 默认允许处理消息的最大时间，默认300s
        map.put(ConsumerConfig.MAX_POLL_INTERVAL_MS_CONFIG, 50000);
        return new DefaultKafkaConsumerFactory<>(map);
    }

    @Bean
    public KafkaListenerContainerFactory<ConcurrentMessageListenerContainer<String, String>> kafkaListenerContainerFactory() {
        ConcurrentKafkaListenerContainerFactory<String, String> factory = new ConcurrentKafkaListenerContainerFactory<>();
        factory.setConsumerFactory(kafkaConsumerFactory());
        // 消费完马上提交
factory.getContainerProperties().setAckMode(ContainerProperties.AckMode.MANUAL_IMMEDIATE); 
        // 异步提交
      factory.getContainerProperties().setSyncCommits(true);
        return factory;
    }

}

topic相关

[手动创建topic]

bin/kafka-topics.sh --create --zookeeper localhost:2181 --replication-factor 1 --partitions 1 --topic testx

参数注意：

  • partitions：指定topic的分区数

  • replication-factor：指定每个分区的副本数量，但不能超过broker数量; 如果设置为1其实就是只有partition而没有replica，换句话说这时replica（leader）就是partition; broker集群时建议设置大于1，不是的话broker宕机，该机器上的partition（replica）就不可用了。设置大于1，broker宕机该其它的replica就会选举成leader replica来提供服务。用下图来理解（两个broker的集群）

    bin/kafka-topics.sh –create –zookeeper localhost:2181 –replication-factor 2 –partitions 1 –topic testx

    从上图可以看出为什么replication-factor不能大于broker数量，原因很简单，就是多余。

[查看kafka中有那些topic]

bin/kafka-topics.sh --zookeeper localhost:2181 --list

[查看指定topic信息]

bin/kafka-topics.sh --zookeeper localhost:2181 --describe --topic testx

输出以下内容

bin/kafka-topics.sh –create –zookeeper localhost:2181 –replication-factor 1 –partitions 2 –topic testx

Topic:testx        PartitionCount:2    ReplicationFactor:1        Configs:
Topic: testx    Partition: 0        Leader: 0    Replicas: 0        Isr: 0

可以看出，testx这个topic有两个partition。因为replication-factor设置为1，所以leader replica是replica0，follower replica也是replica0，即Partition0和Partition1都不具备宕机可继续提供服务的功能，也因为这个缘故所以Isr为0，即没有其他replica可以在leader replica宕机之后接替leader replica的工作。

[动态修改分区的replication-factor数量]

首先需要创建一个json文件，添加上下面有关topic的元数内容，这里我将这个文件叫update-topic.json

    "version": 1,
    "partitions": [
        {"topic":"testx","partition":0,"replicas":[0,1,2]},

        {"topic":"testx","partition":2,"replicas":[0,1,2]}

上面的意思是将对应partition的replica分别放在brokerid为0，1，2这三台broker上，然后使用以下命令修改topic的分区数量

bin/kafka-reassign-partitions.sh --zookeeper localhost:2181 --reassignment-json-file ./update-topic.json --execute

问题配置

如何保证消息不丢失

数据丢失原因：broker没有保存成功

[producer配置]

  • block.on.buffer.full = true

    缓冲池满停止接受消息而不抛出异常，避免因为异常丢失数据

  • acks = all

    需要所有的follower应答，否则都不认为发送成功，但吞吐量会明显降低

  • retries = Integer.MAX_VALUE

    将重试次数设置到最大值，表示尽量重试

  • max.in.flight.requests.per.connection = 1

    因为开启了充实，所以将其设置为1来保证消息顺序

  • 使用send(消息，回调函数)方法发送

    预防发送失败，使用回调函数来补救

[broker配置]

  • unclean.leader.election.enable = false

    避免非ISR的replica竞选到leader而导致数据丢失

  • replication.factor >= 3

    尽量使用备份来避免分区单点故障

  • min.insync.replicas = 1

    表示最少有1个replica成功写入数据，否则视为保存不成功（producer抛出异常）。该配置只有在producer设置了acks=all才有效。（acks是应答，而min.insync.replicas是限制至少多少replica保存数据成功，这个应到什么内容是不冲突的）

如何保证只消费1次

一条消息多次消费原因：

  1. offset没有准确提交
  2. 生产者开启了重试导致消息重复生产
  3. 消费者处理消息的时间过长

所以在条件允许的情况下不开启重试发送功能，且将提交消费者offset的操作改为手动（最好为同步提交）和按照实际的业务情况来设置 max.poll.interval.ms

如何解决消息乱需问题

首先需要知道的是目前的kafka并不支持全局消息有序，即保持消息在所有的partition之间有序是做不到的，而kafka只能保证单个partition内有序。

而当开启了重试功能后，partition内部也是可能发生乱序情况的，这时就需要将max.in.flight.requests.per.connection 设置为1来保证在重试期间不会想broker发送新的数据而导致乱序。
