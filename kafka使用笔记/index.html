什么是磁盘的顺序写入？

  磁盘顺序写入是相对与随机写入而言的，指在写入数据时磁盘的磁头不用寻道而可以一次将数据写完。因为节省了磁头寻道的时间，所以在速度方面是非常快的，甚至可以媲美内存的随机写入。那什么情况下会出现顺序写入呢？如文件追加内容时（Kafka就采用这种方式来对日志文件进行顺序写入的，因为Kafka的日志文件不能修改只能追加）

内存映射文件

  游戏就经常玩了，但不知有没有过这样的疑问，就是游戏的体积这么大而内存这么小，它是怎样运行的呢？答案是操作系统会预先为游戏生成一个内存镜像并分为为小块存放在磁盘中，等到要运行游戏时再将当前需要用到的块加载到内存中运行。而这里用来存储内存镜像的文件实质就是内存映射文件。操作系统可以将内存映射文件直接映射到内存地址空间中，所以线程是可以直接通过操作内存来修改内存映射文件上的内存的，而操作系统则会定时地将脏数据落盘。

  内存通常会使用页来作为基本单位。操作系统会将用户进程中的虚拟地址空间以及物理内存地址空间划分为大小相等的页，其分别叫做页面和页框，而管理它们之间的映射关系的数据结构叫页表。通常每个页面的大小范围在 512 B～1M 之间，且为 2 的正数次幂。譬如常见的操作系统一般每个内存页的大小是 4K（32bit系统）或 8 K（64bit系统），还有就是 16 K（MySQL自己实现的内存页）等。

从上图不难看出并不是所有虚拟内存地址空间都有与之对应的物理内存地址空间的。这是因为虚拟内存的地址空间要比物理内存的大很多。而当程序访问到一个没有映射关系的虚拟内存地址时就会发生一种叫缺页中断的操作，然后操作系统就会将当前物理内存地址空间中较为不常用的内存页替换成当前所需但位于内存映射文件中的部分，然后再更新进程的虚拟内存并恢复到缺页中断位置继续操作。

  最后说一下 mmap 操作（Java 中的 MappedByteBuffer）。mmap 本身是一个操作系统函数，用户程序可以使用它来实现内存映射文件的功能，但要注意的是 mmap 并不是直接将文件映射到内存中的，它实际上会先将磁盘数据加载到 Page Cache 中再进行映射的，也就是说进程操作的数据会先落到 Page Cache 再由系统定时刷到磁盘，所以在一定程度上存在数据丢失的风险，而且因为需要将数据缓存到 Page Cache 再映射，因此 mmap 并不太适合操作体积较大的文件（大文件建议用直接IO）。例如在 Kafka 中只是通过 mmap 映射日志索引文件提高检索速度，而不是映射日志本身。

上下文切换和用户态内核态切换

线程上下文切换

  指任务交替执行时 CPU 寄存器和程序计数器需要先记录下当前正在执行任务的数据和状态（用于下次执行时恢复），然后再加载要执行任务的内容，而这个过程就叫线程的上下文切换。

  线程的上下文切换本身是系统的一个很正常的调度操作，但在上下文切换期间用户程序就相当于进入了呆滞状态，而需要等到切换完成后才可以继续运行。虽然单次上下文切换所需要的时间非常短暂，但是如果线程数量较多的话，就会导致大量的上下文切换从而消耗 CPU 资源以及直接影响用户程序的处理能力（导致吞吐量下降）。当然，可能这么说还不是够明白。举个不太恰当的例子，譬如现在有10 秒时间让用户程序用来处理客户端请求，但是因为线程太多导致切换时间花费了 6 秒，所以用户程序实质用来处理请求的时间就只下剩下 3 秒了。虽然这个例子不太恰当但却能够说明上下文切换的问题，也就说我们在编写多线程程序时应该做到尽可能地减少上下文的切换。

注：如果使用的是类 Unix 系统，则可以使用 vmstat 命令查看应用当前的上下文切换情况。

用户态和内核态切换

基本介绍

  在操作系统中线程的执行空间被分为用户空间 和内核空间两种。处于用户空间的线程是不能访问除属于自己之外的其他资源的。例如磁盘、网络块等这些都不能直接访问，而如果真要访问就需要用户线程发起系统调用（内核提供的接口），而系统调用本质就是从用户空间状态切换到内核空间状态，然后让内核线程来帮助完成这个访问操作，因为处于内核空间的线程拥有最高的资源访问权限。

  用户空间中的一条线程都有与之对应的内核线程，可能是一条或者多条（根据平台而定）。譬如 Java 线程就是一对一的关系。

零拷贝（Zero-copy）

  零拷贝通常会在网络传输上出现。指内核线程直接将内核缓冲中的内容通过 RMA Copy 传输给网络模块然后发送出去。这个过程无需再将数据从内核态拷贝回用户态再执行发送操作，所以零拷贝也可以减少切换开销。

上面已提到过，零拷贝并不是不进行拷贝，而是不进行 CPU 拷贝，既无需通过 CPU 拷贝将内核 Page Cache 数据拷回用户空间再进行发送操作。

在 Kafka 中，它使用该技术来实现了数据传输接口（TransportLayer） 。据传输接口内部通过 FileChannel 的 transferTo 方法来进行零拷贝操作，因为 transferTo 方法内部实质是 sendfile 系统指令。而需要特别注意的是，Kafka 零拷贝发送消息日志只在非 SSL 协议环境下适用。

Kafka是什么？

官方介绍

Apache Kafka是一个开放源的分布式事件流平台，成千上万的公司使用它来实现高性能数据管道，流分析，数据集成和关键任务应用程序。

什么是事件流，个人理解如下：

事件：特指某个上下文中被关注且已发生的事情

事件流：事件流向过程（从一端转移到另一端的）。譬如说数据库记录被更新，因此而产生了对应的更新事件，该事件通过 Kafka 事件流平台（作为介质）转移到了其他服务节点上的过程，就是一个CDC（capture data change）事件流

流处理：提供一系列操作手段（修改、持久化、特殊处理等），让数据在流动的过程中产生所需的变化

常见使用场景

  • 功能异步解耦
  • 消息订阅/发布
  • 日志收集
  • 数据中转，譬如上面提到的 CDC（capture data change）
  • 流量削峰，既使用Kafka进行任务缓冲以应对突发的流量剧增情况
  • 大数据流处理

注：以上只是部分 Kafka 的使用场景，其实不管是 Kafka 还是其他系统，使用场景都是根据系统所提供的特性而定的，简单地说就是了解系统特性的前提再发挥个人想象来决定如果使用系统，所以说 Kafka 的使用场景远不止上面这一些。

为什么kafka性能高？

  • 数据操作方式为磁盘的顺序访问
  • 通过 zero-copy 传输网络数据，减少了数据在用户态和内核态之间的复制消耗从而提高了传输效率
  • 使用 mmap 将日志索引映射到虚拟内存，提高了磁盘数据的访问速度
  • 一个 Topic 可以有多个 Partition，除了去中心化之外还提高了并发效率
  • 发送缓存。支持批量发送操作，Producer 只有数据大小到达batch.size才会真正发出，目的是提高操作的吞吐量
  • Broker 基于 Reactor 模式处理请求，一定程度提高了 IO 的处理能力
  • Kafka在消费完数据后它并不会马上删除，而是等到某个时间或者量批量删除，因此减少了不必要的操作，提高了吞吐量和降低了延时
  • 消息压缩。Kafka 允许在 Producer 和 Broker 这两个点对消息集合进行压缩
    • 这是一种时间换空间的做法，Producer 压缩数据可以提高传输效率，因此如果 Producer CPU 充裕的话建议开启该功能，且使用 zstd算法
    • 一般情况下 Producer 端开启了压缩功能 Broker 会接收后会直接保存（除非Broker开启了压缩且和Producer的压缩类型不一致才会解包再压缩），然后 Consumer 消费会将其解压再消费

Topic、Partition、Replica

基本介绍

  在Kafka中Topic只是逻辑概念，而物理层面实质是以 Partition 来进行日志存放的。Partition是一个序列，在物理磁盘上对应着一个文件夹，其中包含了一个或多个日志段。一个 Topic 可以有一个或多个 Partition，也就是说 Topic 的数据在多个 Partition 的情况下是分拆存放的，这是一种去中心化容灾的实现方案。 需要注意的是 Kafka 只能保证单个 Partition 内的消息是有序的，而不能保证全局有序（多个Partition）。

  Replica是Partition在副本层面上的概念，一个 Partition 可以有多个 Replica，但至少要有一个作为 Leader Replica 对外提供服务。在 Kafka 中 Replica 有两种，分别是Leader Replica和Follower Replica。Leader Replica 对客户端的提供读写服务。而 Follower Replica 正常情况下是不对外提供服务的（但 2.4 版本（KIP-392）后，追随者副本也允许提供读服务了），它的主要职责是拉取 Leader Replica 中的消息日志进行冗余备份，以及在 Leader Replica 发生故障后接替其职责。

  其中Follower Replica又分为 ISR（in-sync replica，实时同步Leader Replica的副本集）和 非ISR 两种类型。其区别在于默认情况下只有ISR集合成员能够参与Leader竞选，还有就是在 Borker 接收到客户端的写请求后，只有Leader Replica以及其他 ISR 集合成员都接收到后这个消息日志才会成功被提交，消息成功提交之前该写请求的消息日志是对消费者不可见的。而实质上一开始所有的 Replica 都属于 ISR 集合成员，但随着运行时间拉长，Broker 节点之间出现同步效率上的差异，导致某些 Broker 上的 Replica 无法跟上复制进度而被踢出 ISR（参考配置replica.lag.time.max.ms），而被踢出 ISR 的 Replica 只有追上复制进度后才会被重新纳入到 ISR 集合中。

为什么需要Partition？

  有人可能会有这样的疑问，既然有了 Topic 为什么还要有 Partition 这个概念呢？譬如 ActiveMQ 就没有Partition。这点需要从 Kafka 的主要使用场景角度来说，Kafka 通常被用作实时处理场景，为其提供 高吞吐、低延迟 的解决方案，而 Partition 就是用来提 Topic 的并发量的，同时有着去中心化容灾和负载均衡的效果

为什么有 segment 日志段？

一个 Partition 可以有多个 segment 日志段。目的是优化数据的检索效率，因为 segment 日志段文件名称是以某个范围内起始消息日志对应的offset来命名的，所以就可以利用二分查找来快速定位到需要的数据了，而不需要扫描所有的消息数据。

集群控制器

介绍

Kafka Controller的作用是协调 Broker 集群，一个集群当中只有一个 Broker Controller（通过Zookeeper竞选），它的职责如下：

  • Topic 管理
    • 创建、删除、分区（Leader选举）
  • 集群成员管理
    • 新增Broker、Broker主动关闭、Broker 宕机
  • Preferred 选举
    • 解决集群负载不均的问题
  • 重新分区
    • 指的是扩容或缩容集群节点后重新对 Partition 进行分配
    • 该操作可以通过kafka-reassign-partitions脚本完成
  • 提供元数据服务
    • 控制器会向其它 Broker 提供元数据服务
    • 控制器上保存了最全的集群元数据信息，其它 Broker 会定期接收控制器发来的元数据从而更新其内存中的缓存数据

    • 可以手动可以通过 kafka-preferred-replica-election 脚本完成优先副本选举

3）优先副本选举策略

  • 选举策略 PreferredReplicaPartitionLeaderElectionStrateg
  • 选择 AR 集合中第一个存活的副本作为优先副本

Leader Replica选举

1）什么时候会发起 Replica 选举？

  • 创建新分区
  • 原 Leader Replica 下线
  • Broker 被优雅关闭（kafka-server-stop.sh）

topic offset 和 consumer offset

topic offset

  当消息被写到 partition 时，kafka 会为其分配一个顺序的 offset。这个 offset 是当前消息在 partigion 中的偏移量。offset 从 0 开始递增，在不同的 partition 中 offset 是互不干扰的。

consumer offset

  consumer offset 记录的是 partition 的消费偏移量（即消费到那个位置）。conusmer 会在消费消息时（实际由consuemr自定义）向 Borker 提交自己当前的 consumer osffset，那么下次再消费同一个 partition 时就可以知道从哪里开始消费了。

消费者提交注意：

  上面提到消费者是需要定期向 Broker 提交 consumer offset 以确认 partition 被消费的进度的。在较新版本的Kafka Consumer中会对应一个专门的内部 Topic (__consumer__offsets，即提交消费者偏移量实质就是发送一条消息到这个 Topic 中。该 Topic 可以在日志文件夹下观察到，它一共有50个文件夹，即 50 个 Partition。

那为什么需要这么多 Partition 呢？因为位移提交是一个非常频繁的操作，这样可以减轻并发带来的压力。

拓展：

  • __consumer__offsets 这个内部主题不仅维护了消费者偏移量，还维护了消费者的元数据，由 Group Coordinator（消费者组协调者）管理
  • 早期 Kafka Consumer 中消费者偏移量是保存在 Zookeeper 中的，这样是为了降低 Broker 的管理成本从而提高集群的扩展性。但后来发现因为偏移量提交是一个频繁操作，而 Zookeeper 并不合适保存大数据量以及作频繁的数据修改，所以在后期新版本 Kafka Consumer 中消费者偏移量从 Zookeeper 迁移回了 Broker 管理。

HW和LEO

基本介绍

HW

  • 全称叫 Hight Watermark，一般称为高水位
  • HW 本质是一个偏移量，用于定义消息日志对 Consumer 的可见性
  • 在 Kafka 中一条消息如果对 Consumer 是可见的，则表示这条日志已经成功复制到所有的 Replica 上
  • Consumer 只能获取到 HW 前面的消息日志，即 HW 位置开始的日志（如果有）对于 Consumer 而言是不可见的，因为还没有同步完成
  • 即 Kafka 通过 HW 机制实现了对于 Consumer 而言的状态一致性（CAP 中的 C）
  • 可以简单理解 Partition 中的 HW 其实就是 Leader Replica 中的 HW，即以 Leader Replica 为准

LEO

  • 全称 LogEndOffset，记录了下一条 Producer 生成的日志的 offset
  • LEO 相当于是当前 Replica 最后一条日志的 offset + 1
  • 当 Leader Replica 中的 HW = LEO 时，表示所有的 Replica 已经同步完成
  • 重点注意
    • Leader Replica 在 Follower Replica 拉取数据时会顺带记录其 LEO 值。即 Leader 会保存所有远程副本的 LEO 值，这样做是因为 Leader 同收集回来的 LEO 值来定位 Partition 的 HW 值
    • 远程副本（Follower Replica）的 LEO 值是有 Leader Replica 来维护的

1）数据丢失问题

2）数据不一致

Leader Epoch 解决数据丢失和不一致问题

1）是什么？

Leader Epoch 它是 Leader 的任界值，它由两部分组成：

  • Epoch： 一个单调增加的版本号，每当 Leader 发生变更都会递增1，所以版本号较小的 Leader 会被认为是过期的 Leader，不会在行使 Leader 权力
  • Start Offset（起始位移）： 记录的第一条消息日志的偏移量

例如现在有两个 Leader Epoch，分别是 <0，0> 和 <1，20>。相当于说 <0，0> 这个 Leader 它是第一个 Leader，且记录了 20 条消息日志（因为偏移量是从0开始的）。

Leader Epoch 的主要作用是防止新 Leader 单纯根据 HW 来截断日志，从而造成数据丢失。

2）更新时机

  • Kafka Broker 会在内存中为每个分区都缓存一份 Leader Epoch 数据，同时还会定期地将这些信息持久化到一个 checkpoint 文件中

  • 当 Leader Replica 写入消息到磁盘时，Broker 会尝试更新这部分缓存。即如果是首次写入就向缓存中增加一个 Leader Epoch 条目，否则就不做更新

  • 一旦发生 Leader 变更，新的 Leader Replica 就会查询这部分缓存，取出对应的 Leader Epoch 的起始位移，以避免数据丢失和不一致的情况

自定义分区策略

上面已经提到过，Partitioner 会根据 ProducerRecord 中的 key 来决定将消息数据发送到哪个 Partition 上。但如果没有指定 key，则会按照默认策略来选择发送分区。

public interface Partitioner extends Configurable, Closeable {

常用配置参数

bootstrap.servers

  • kafka broker地址，host:ip，多个用逗号隔开

key.serializer

  • 指定 key 使用那种序列化器(必须使用全类名)

value.serializer

  • 指定 value 使用那种序列化器(必须使用全类名)

acks（重要）

  • 用于判断消息是否发送成功的依据

  • 0：Producer 完全不理会 Replica 是否已经日志落盘成功，发送完一个批消息后会马上发下一个。这时 Producer 吞吐量最高，但因为能确认日志是否落盘成功，所以存在丢失日志的风险

  • 1（默认）：Producer 只关注 Leader Replica 是否已经日志落盘成功。该配置级别是吞吐量和安全性的折中点

  • -1或all：Producer 会等待集群中所有 Replica 的日志落盘成功才会发送下一个批次的消息。消息能够确保落盘成功，但吞吐量最低

compression.type

  • 用于指定消息的压缩类型，默认为none，既不对消息进行压缩操作
  • 目前 kafka 支持的压缩格式有 gzip、snappy、lz4、zstd（Kafka 2.1 开始支持，建议以官方文档为准）

retries（重要）

  • 消息重发机制
  • 消息发送失败的原因可能是网络问题或正处于Leader选举导致
  • 值 0 时（默认），表示默认不进行重试。值为 n 时，表示可以重试 n 次
  • 要十分注意
    • 重试操作可能会导致消息重复发送。例如因为网络原因Brokder保存消息成功但没有成功响应，而Producer进行了消息重发
    • 开启重试操作还可能会出现消息乱序问题。例如a、b、c三条消息，a和c发送成功但b失败了，b重新发送。

retry.backoff.ms（重要）

  • 重试的时间间隔（默认为100毫秒）

max.in.flight.requests.per.connection

  • 在 Broker 未响应之前 producer 可以 send 多少条数据（默认为5）

  • 注意，如果开启了 retries 但又想重试消息要优先于之后的消息，就可以将该值设置为1。

    • 举个栗子：当前有两条数据 xxx 和 ccc 要发送，但因为某些原因 xxx 发送失败进入了重试操作，然而在重试期间 producer 又给发送了消息 nnn 且发送成功了，这时就算 xxx 重试成功，但显然已经乱需了。而将max.in.flight.requests.per.connection 设置为1，那么重试发送 xxx 期间因为 broker 还没应答，所以 send 就不会将 nnn 发送

batch.size（重要）

  • 发送消息的批次大小（默认16 k）
  • 适当地增加该值可以提高 Producer 的吞吐量，但会降低消息发送的实时性和加大内存消耗

  • 基于时间的 batch 打包策略，可配合 batch.size 使用

Consumer Group概念

为什么需要消费者组？

当 producer 生产消息的速度比 consumer 的消费速度快时，consumer 的消费能力就显得低下了，所以因此而无法满足时性实时消费性较高的业务场景（还与发送缓冲区有关）。而消费者组其实就是用来提高 consumer 消费能力而设计的，可以将消费者组视为一个多线程的 consumer 。

介绍

消费者组订阅 Topic T1，其中 T1 的 Partition 数量为4。

1）费者组中仅有1个消费者时

2）消费者组有2个消费者时

3）当消费者组有4个消费者时

4）当消费者组中的消费者数量超过了Partition数量时（不会提高消费能力）

5）当有 2 个消费者组同时订阅 Topic T1 时（消费者组之间互不影响）

小结

  1. 同 Consumer Group 中不同的 Consumer 在同一时刻无法消费同一个 Partition
  2. Consumer Group 中的 Consumer 可以同时消费多个 Partition
    • 不能消费不属自己负责的 Partition，由分区再均衡确定消费关系
  3. 即使 Consumer Group 的 Consumer 数量多于 Partition 数量，也不会提高消费能力
  4. 多个 Consumer Group 同时订阅同一个 Topic 时，互不干扰

分区再均衡

是什么？

  在消费者组的 Consumer 数量没超过 Partition 数量的情况下，每一个 Consumer 都能分配到一个或多个独自负责的 Partition 。但现在因为某些原因某个 Consumer 宕机了，这时消费者组就会根据分配策略来重新设置 Connsuemr 和 Partition 的消费关系。而这个过程就叫做分区再均衡。

注：分区再均衡和 JVM 的 STW 类似，在期间正常的 Consumer 是不能继续消费的，而需要等待分区再均衡结束。

什么情况下会发生分区再均衡？

  1. 消费者组成员数量发生变化

    • 譬如有新 Consumer 加入到 Group 中，又或是有 Consumer 宕机导致从 Group 中脱离时
    • Broker 会作为 Consumer Group 的协调器，Consumer 则需要定时地主动向协调者发送心跳包（在Kafka-0.10.1版本之后有专门的线程来发送心跳包），而如果 Consumer 长时间（session.timeout.ms指定，默认10秒）没有向协调者发送心跳包的话就会被协调者视为死亡从而导致被踢出Group 而导致分区再均衡

  2. Topic 的 Partition 数量发生变化

  3. 订阅的 Topic 数量发生变化

    • 譬如 Kafka 是支持使用正则表达式匹配订阅 Topic 然后消费的

如何尽量避免再均衡？

订阅的 Topic 数量发生变化、Topic 的 Partition 数量发生变化这两种情况是不能避免的，因为这时业务上的需要，所以我们应该关注的是消费者组成员数量发生变化这种情况。

譬如除了维护人员主动增加 Consumer 外，协调器在特定时间内如果没有收到 Consumer 的心跳包也会导致分区再均衡的发生。

为了降低这种情况的发生率，我们可以通过调节以下参数来做到：

  • session.timeout.ms
    • 该值默认为 10 秒，表示协调器必须在这段时间内收到心跳包，否者就将其踢除出消费者组（导致分区再均衡）
  • heartbeat.interval.ms
    • 用于限制 Consumer 发送心跳包的时间间隔
    • 注意该值必须小于 session.timeout.ms 才有意义
  • max.poll.interval.ms
    • 用于限制 Consuemr 消费消息的最长时间，该值默认为 5 分钟
    • 如果超过了该配置的时间，协调器就会认为该 Consumer 消费能力低下（通过 Lag 值判断），从而将其踢除出消费者组（导致分区再均衡）
    • 在 0.10.1 版本前 Consumer 是没有独立的心跳线程的，也就是说 Consumer 只有在 conmit/poll 操作时才会发送心跳包，所以如果这时 max.poll.interval.ms 就不能超过 session.timeout.ms，因为如果心跳包一旦迟了发送就会被被踢出费者组（导致分区再均衡）
    • 如果在消费消息期间发生了分区再均衡，那么 Consumer 在 commit 时将会发生CommitFailedException 异常，该异常是不可恢复的，它要表达的是分区再均衡导致 Parition 分配给了其它 Consumer，因此当前 Conusmer 并不能将偏移量再提交到当前 Partition。所以为了避免以上情况 Conusmer 就应该及时消费消息，例如增大max.poll.interval.ms或降低max.poll.records（一次poll所获取到的消息数量），再者就是使用多线程增加 Consumer 的消费能力

协调者指的是什么？

协调者（Coordinator）主要负责管理消费者组。

Coordinator 是一个专门为 Consumer Group 服务的角色（与 Group ID 对应）。

它有以下特点：

  • 每一个 Broker 都有自己的 Coordinator
  • 负责管理如下内容
    • 分区再均衡
    • 消费者偏移量
    • 消费者组成员

那么消费者是如何确立与自己对应的 Coordinator 的呢？有以下 2 个步骤：

  1. 确定当前消费者组所消费的 Topic 偏移量由哪一个 __consumer__offsets 的 Partition 来保存

    partitionId=Math.abs(groupId.hashCode() % offsetsTopicPartitionCount)

  2. 找出该 Partition 的 Replica Leader 所在的 Broker，而该 Broker 就是会成为当前这个消费者组的 Coordinator 了

常用参数

bootstrap.servers

  • kafka broker地址，host:ip，多个用逗号隔开

key.deserializer

  • 指定 key 使用那种序列化器（必须使用全类名）

value.deserializer

  • 指定value使用那种序列化器（必须使用全类名）

group.id

  • 指定集群唯一消费者组的ID

  • 建议尽量取一个与业务有关或者有意义的名字，可以提高辨析度

session.timeout.ms（重要）

  • 指定协调器对 Consumer 停止发送心跳包这种行为的最大容忍限度（默认为10秒）
  • 一旦超时 Consumer 就会被协调器踢出消费者组（导致分区再均衡）

max.poll.interval.ms（重要）

  • Consumer 处理消息的最大时间限制（两次 poll 之间的时间，默认为 5 分钟）

  • 建议根据实际消息的大小来设置该值

auto.offset.reset（重要）

  • 指定消费者的消费模式
  • 提供了三种消费模式：
    • latest（默认）：
      • 在偏移量无效时（没有提交过偏移量），会从最新插入的那条记录开始消费（即在消费者启动后插入的记录）
      • 在偏移量有效时（已经提交过偏移量），会从最新偏移量位置开始消费（即上一次提交的 offset 位置）
    • earliest：
      • 在偏移量无效时，会从起始位置开始消费
      • 在偏移量有效时，会从最新偏移量位置开始消费（即上一次提交的 offset 位置）
    • none：
      • 各个 Replica 都存在有效的 offset 时，从有效offest位置开始消费
      • 只要有一个 Replica 不存在有效的 offset（没有提交过、已过时或已被删除），则抛出异常

enable.auto.commit（重要）

  • 是否开启自动提交 offset 功能（默认为true，默认为 5000 ms）
  • 为了尽量避免重复消费建议设为 false，然后改由程序来控制提

auto.commit.interval.ms

  • 控制自动提交消费者偏移量的时间间隔
  • 该配置只在 enable.auto.commit = true 时才生效

partition.assignment.strategy

  • 指定分区分配策略，即如何将分区分配给消费者

  • 默认为 RangeAssignor，除此之外还有 RoundRobinAssignor、StickAssignor

client.id

  • 设置客户端ID

fetch.max.bytes（重要）

  • 指定单次获取消息数据的最大字节数
  • 如果实际业务中消息比较大，则应该将该值适当地调高，否则无法获取数据来消费
  • 调高该值有利于增加 Consumer 的吞吐量

max.poll.records（重要）

  • 用于指定单次 poll 能够获取到的最大 record 数量（默认为500）
  • 建议如果程序处理消息所需的时间较长（即Consumer消费能力不高的情况下），则该值应该设置得小一些（例如1），否则很容易被协调器提出消费者组导致从而导致分区再均衡

heartbeat.interval.ms（重要）

  • 指定消费者发送心跳包的时间间隔（默认为 3000 ms）
  • 该值必须必比 Broker 的 session.timeout.ms 配置小，否则没有意义。该值通常不高其于1/3，甚至可以更低
  • heartbeat.interval.ms 值越低，则可以让协调器更快地检测到当前消费者是否出现故障

手动提交offset

自动提交存在的问题

问题1：不能及时消费导致数据丢失

譬 Consumer 消费能力较弱，导致 offset 在未完成消费前被自动提交，而这时又刚好遇上Consumer 宕机就会导致数据丢失。因为这时 offset 已经提交，所以即使发生分区再均衡，其它 Consumer 也只会根据这个 offset 继续消费。

问题2：不能及时提交 offset 导致重复消费

譬如 Consumer offset 自动提交的频率为 5 秒。假设在前 3 秒已经消费了不少消息，但又因为未到 5 秒，所以暂时还无法提交 offset，而这时正好遇上宕机或者分区再均衡就会导致先前 3 秒内所消费的消息会被再次被消费。

如何解决以上两个问题呢？

  改为手动提交 offset 来解决。例如可以在真正处理完消息后才手动交 offset ，而不是在此之前或较久的之后提交。

同步提交

while (true) {
    ConsumerRecords<String, String> records = consumer.poll(100);
    try {
        //...
        consumer.commitSync(); // 同步提交
    } catch (CommitFailedException e) {
        log.error("commit failed", e)
    }
}

同步提交是一个阻塞操作，需要等待 Broker 应答后才能继续后续的操作。因此只要不是发生无法恢复的异常 commitSync() 都能提交成功，但这样无疑会降低消费者的消费吞吐量。

异步提交

用于解决同步提交导致吞吐量下降的问题

while (true) {
    ConsumerRecords<String, String> records = consumer.poll(100);
    //...
    consumer.commitAsync(); // 异步提交
}

注意，因为 commitAsync() 是异步的，所以并不能保证成功地将 offset 提交到 Broker。所以这时我们可以给 commitSync() 设置一个回调方法，当提交失败时让其可以进行重试操作。

while (true) {
    ConsumerRecords<String, String> records = consumer.poll(100);
    //...
    consumer.commitAsync(new OffsetCommitCallback() {
        public void onComplete(
            Map<TopicPartition,
            OffsetAndMetadata> offsets, 
            Exception exception) {
            if (e != null){
                // 异常处理
                // 重试
            }
        }
    });
}

混合提交

try {
    while (true) {
        ConsumerRecords<String, String> records = consumer.poll(100);
        //...
        consumer.commitAsync(); // 异步提交
    }
} catch (Exception e) {
    log.error("Unexpected error", e);
} finally {
    try {
        // 同步提交，用于防止异步提交失败
        consumer.commitSync();
    } finally {
        consumer.close();
    }
}

常见问题

如何保证消息不丢失

数据丢失的核心原因 Producer 没有确保 Broker 持久化日志成功。

Producer配置

  • block.on.buffer.full = true

    消息缓冲区满后，停止接收新的消息而不抛出异常（避免异常丢失数据）

  • acks = all

    Producer 需要确认所有 Broker 成功应答，否则认为发送不成功，进行重试发送

  • retries = Integer.MAX_VALUE

    将重试次数设置到最大值

  • max.in.flight.requests.per.connection = 1

    因为开启了重试，所以将其设置为 1，在 Broker 没应答前不发送数据

  • 不使用 producer.send(msg)，而是使用 producer.send(msg, callback) 手动预防失败

Broker配置

  • unclean.leader.election.enable = false

    避免非 ISR 成员成为为 Leader 而导致数据丢失

  • replication.factor 设置大于等于 3

    尽量使用备份来避免 Partition/Replica 单点故障

  • min.insync.replicas = 过半节点数

    表示至少有过半节点数的 Replica 写入成功才让 Leader 响应客户端

    该配置只有在 acks=all 才有效，建议设置过半节点数

Consumer配置

  • 采用手动提交偏移量，避免还没消费就提交了 offset
  • 消费者应该及时消费消息，尽量使用消费者组来提供消费能力

如何保证只消费1次

一条消息被多次消费有以下原因

  1. offset 没能及时提交，导致重复消费
    • 解决方法是手动提交偏移量，且尽量做到消费完就马上提交
  2. 生产者重试发送，导致相同的消息发送了多条
    • 可以直接关闭重试发送功能，或开启 Producer 消息幂等功能
    • 也可以代码层面解决

如何解决消息乱需问题

  • 目前而言 Kafka 是不不支持全局（多个Partition之间）有序的，而只能保证单个 Partition 内是有序的
  • 而且当 Producer 开启重试功能后，单个Partition 内部也是可能会乱序。这时可以将 max.in.flight.requests.per.connection 设置为 1 解决

关于 Producer 消息幂等性

在 Producer 方可以通过添加 enable.idempotence = true 配置参数来实现 Producer 发送消息的幂等性。但它有以下局限：

  1. 只能保证单个 Partition 消息的幂等
  2. 只能保证当前会话发送的消息的幂等

注意，当开启 enable.idempotence = true 时，它默认会设置如下配置

  1. acks = all
  2. retries = Integer.MAX_VALUE
  3. max.inflight.requests.per.connection = 1

关于 Kafka 的实时性

导致 Kafka 实时性不足的原因：

  • 生产者采用了异步 send 实现消息发送
  • 生产者采用了 batch、buffer 相关的机制提高生了发送吞吐量

如果要想提高实时性，则可以通过条件以上相关的参数来解决。又或者直接选用实时性较好的 RocketMQ 来解决。

监控 Kafka Consumer 的消费进度

注意：

  • 监控消费进度实质就是观察消费者组的 Lag 值。
  • Lag 指的是 Consumer 落后的进度。譬如生成了 1000 条消息，而 Consumer 只消费了 600 条，那么当前 Consumer 的 Lag 值就是 400，表示已经滞后了 400 条消息。所以 Lag 值越大就表示 Consumer 的处理能力越低

通常监控 Kafka 的手段有如下这些：

  • kafka-consumer-groups 脚本 (推荐)

    kafka-consumer-groups.sh --bootstrap-server <Kafka broker 连接信息 > --describe --group <group 名称>

  • Kafka Consumer API

  • JMX（推荐）

Kafka 的 Heap Size 应该如何设置

  1. 业界经验值 6G
  2. 系统正常情况下，通过手动触发 Full GC 并观察其存活的对象的总大小，再将该基础上添加 30~40% 倍来设置

Leader 副本状态总是为 -1 如何解决

手动删除 Zookeeper 中的/Controller节点，重新选举 Controller 。

因为重新选举 Controller 后，Controller 会为所有 Topic 的分区重新刷新状态。
