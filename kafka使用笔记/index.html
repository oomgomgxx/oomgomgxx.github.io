什么是磁盘的顺序写入？

  所谓的磁盘顺序写入其实是相对与随机写入而言的，指的是在写入数据时磁盘的磁头不用寻道而可以一次将数据写完，因为省略了磁头寻道的时间，所以在速度来说是非常快的，甚至可以媲美内存的随机写入; 那什么情况下会出现顺序写入呢？如文件追加内容时（kafka就采用这种方式进行顺序写入，因为kafka存放数据的日志文件不能修改只能追加）

内存映射文件

  游戏就经常玩了，但不知道有没有这么一个疑问：游戏体积这么大，而内存这么小，那么操作系统到底是怎么运行游戏的呢？答案就是，操作系统会预先为程序生成一个内存镜像保存在磁盘中，在运行该程序时并不会真的将该程序全部加载进主存，而是当使用到某部分代码时才会加载进主存，而这个保存程序内存镜像的文件就是内存映射文件。操作可以可以直接将内存映射文件映射到对应的内存地址中，因此对于用户线程而言是可以直接访问到该内存地址的。而且操作系统会自动管理者块内存空间，例如操作完该内存中的数据后系统会自动将脏数据刷会文件中。

参考图

  在操作系统中一般会使用页来作为内存的基本单位。系统将用户进程中的虚拟内存地址划分为大小相等的页，这种页称为页面Kafka而物理内存地址中亦划分为大小相等的页，这种页称为页框。每个页面的大小为2的整数次幂，范围在512字节～1M，通常为4KB（32bit系统）和8KB（64bit系统）。在管理层面，操作系统使用一种叫页表的结构来映射虚拟内存地址和物理内存地址之间的关系（虚拟内存通常远大于虚拟内存）。

参考图

从上图我们不难看出，并不是所有虚拟内存地址都有与之对应的映射的。当程序访问到一个没有映射的虚拟地址时，就会发生一种叫缺页中断的操作，该操作会将物理内存地址中较为不常用的内存页替换成当前缺页的程序部分。大体步骤是系统会从内存映射文件中加载内容到主存上（物理内存地址），然后更新进程的虚拟内存空间（虚拟内存地址），最后恢复到缺页中断位置的操作。

  最后说一下mmap操作，mmap是一个系统函数，它能够实现文件到内存的映射（内存映射文件）。而kafka就是使用mmap将日志索引文件内容映射到内存的，所以在定位数据方面速度也是相当快的。

kafka只用mmap来映射索引是因为如果同时映射索引和日志的话就会存在一直性问题，而且mmap并不能保证百分百成功将内存中的脏数据刷回磁盘。例如并发操作时修改了日志并回刷成功，但因为某些原因索引回刷失败，这就导致了数据不一致性问题

上线程下文切换

线程上下文切换

  指的是在任务执行前，CPU寄存器和程序计数器会预先加载将要执行的任务内容。首先操作系统会先记录下正在执行的任务内容和状态，以便下次执行时恢复。保存成功会操作系统就会把将要执行的任务内容加载到CPU寄存器和程序计数器中（例如每条线程都有栈和程序计数器），而这个过程就被称为上下文切换（实质就是切换执行任务）。上下文切换是一个系统层面的调度操作，所以在上下文切换期间，用户程序将会进入呆滞状态，等到切换完成后才会继续运行。

用户态和内核态切换

  操作系统将线程（轻量级进程）的执行空间分为：用户空间 和 内核空间。

线程在用户空间时，因为用户空间权限有限所以线程在对资源操作上也是有限的。例如用户空间的线程不能直接访问除系统分配给自己的内存以外的内容，譬如：磁盘、网络模块 等，如何想要操作这部分资源就需要发起系统调用，让系统的内核线程来完成，而这时就会发生所谓的用户态和内核态切换操作。

简单总结一下

虽然上面提到用户态和内核态切换，但其实它也是线程上下文切换的一种操作。从用户程序的角度来看这是一个比较重量。虽然在单个上下文切看来并不需要花费多少时间，但是一旦发生大量的上下文切换，那么将会直接导致用户程序处理情况的能力下降。

零拷贝（Zero-copy）

零拷贝通常会在网络传输上使用的。指的是内核态直接将buffer（page buffer）中的数据通过RMA Copy 传输给网络模块，而网络模块就可以直接将其发送出去了，而无需从内核态切换给用户态再执行发送操作。

零拷贝例子

需要说明的是，零拷贝并不是不拷贝，而是不进行 CPU Copy 而已。零拷贝的好处在于减少了用户态和内核态的切换次数，这样就减少了用户程序的呆滞时间，因此提高了用户程序的处理能力。

为什么kafka性能高？

  • 数据操作方式为磁盘的顺序访问
  • 通过 zero-copy 传输网络数据，减少了数据在用户态和内核态之间的复制消耗从而提高了传输效率
  • 使用 mmap 将日志索引映射到虚拟内存，提高了磁盘数据的访问速度
  • 一个 Topic 可以有多个 Partition，除了去中心化之外还提高了并发效率
  • 发送缓存。支持批量发送操作，Producer 只有数据大小到达batch.size才会真正发出，目的是提高操作的吞吐量
  • Broker 基于 Reactor 模式处理请求，一定程度提高了 IO 的处理能力
  • Kafka在消费完数据后它并不会马上删除，而是等到某个时间或者量批量删除，因此减少了不必要的操作，提高了吞吐量和降低了延时
  • 消息压缩。Kafka 允许在 Producer 和 Broker 这两个点对消息集合进行压缩
    • 这是一种时间换空间的做法，Producer 压缩数据可以提高传输效率，因此如果 Producer CPU 充裕的话建议开启该功能，且使用 zstd算法
    • 一般情况下 Producer 端开启了压缩功能 Broker 会接收后会直接保存（除非Broker开启了压缩且和Producer的压缩类型不一致才会解包再压缩），然后 Consumer 消费会将其解压再消费

topic、partition、replica之间的关系

  通常生产者生产一个消息是存放在 Broker 的 Topic 中的，但是 Topic 只是逻辑上的概念，而消息实际存放的地方是Partition（分区）。Partition 是一个有序的序列，在物理磁盘上对应着一个只能最追加内容的文件。其中 1个 Topic 可以有1个或多个 Partition，换句话说就是1个 Topic 的数据是分拆到1个或多个 Partition 中存放的。 还有就是，Kafka只能保证单个 Partition 中的消息数据是有序的而不能保证全局有序（多个Partition）。

  而Replica 指的是 Partition 的副本，而且它并不是简单的冗余。在 Kafka 中 Replica 分为 2 种：Leader Replica、Follower Replica。其中Follower Replica是不对客户端提供服务的，它的职责只用于同步 Leader Replica 数据，而 Follower Replica 又分为 ISR（in-sync replica，能够实时同步leader数据的replica）和 非ISR 两种。其中只有 ISR 能够参与 Partion/Replica Leader 的竞选，而且在 Borker 接收到客户端的持久化请求时，只有 Leader Replica 和 ISR 集合成员都接收到时，该消息日志才会被 Broker 认为是发送成功。而实质上一开始所有的 Replica 都属于 ISR 集合，但随和运行过程中出现同步数据效率上的差异，导致其它某些 Broker 上的 Replica 无法跟上某部分 Replica 的复制进度从而被踢出 ISR 集合。但注意，被踢出 ISR 集合的 Replica 如果跟上进度将会被重新纳入到 ISR 集合中。

为什么需要 Partition？

  有人可能会有这样的疑问，既然有了 Topic 为什么还要有 Partition 这个概念呢？譬如 ActiveMQ 就没有Partition。这点需要从 Kafka 的主要使用场景角度来说，Kafka 通常被用作实时处理场景，为其提供 高吞吐、低延迟 的解决方案，而 Partition 就是用来提 Topic 的并发量的，同时有着去中心化容灾和负载均衡的效果

什么是Kafak集群控制器

基本介绍

Kafka Controller 的作用是协调 Broker 集群，而且一个集群当中只有一个 Broker 会成为 Controller （通过 Zookeeper 竞选）。Kafak Controller 的作用有以下这些：

  • Topic 管理
    • 创建、删除、分区
  • 集群成员管理
    • 新增Broker、Broker主动关闭、Broker 宕机
  • Preferred 选举
    • 这是一种处理 Broker 负载不均的解决方案
    • 譬如因为原 Leader Replica 宕机从而导致 ISR 中的某个 Follower Replica 晋升为 Leader，而这个 Replica 所处的 Broker 原来已经有了其它 Topic 的 Leader Replica 了，也就是说现在单个 Broker 上有多个 Leader Replica，这样的话该 Broker 就的负载压力就会增大，而 Preferred 选举就是解决以上问题的，既重新选择一个合理的 Replica 来做 Leader。该操作可以通过kafka-preferred-replica-election脚本完成
  • 重新分配分区
    • 指的是扩容或缩容集群节点后重新对Partition进行分配
    • 该操作可以通过kafka-reassign-partitions脚本完成
  • 元数据服务
    • 控制器会向其它 Broker 提供元数据服务
    • 控制器上保存了最全的集群元数据信息，其它 Broker 会定期接收控制器发来的元数据从而更新其内存中的缓存数据

关于 Broker 的 Controller 选举

Controller 在整个集群中只有一个，它是通过 Zookeeper 提高的选举服务产生的。简单地说就是那个 Broker 先在 Zookeeper 中插入 znode 成功就成为 Broker Controller，在 Broker Controller 宕机后重新选举亦如是。

topic offset 和 consumer offset

topic offset

  当消息被写到 partition 时，kafka 会为其分配一个顺序的 offset ，这个 offset 就是消息在当前 partigion 中的偏移量。其中 offset 从 0 开始递增，在不同的 partition 中 offset 是互不干扰的。

consumer offset

  consumer offset 和 topic offset 是不同的概念，这个千万不要混淆。consumer offset 是用于记录 partition 消息被消费到那个位置的一个标记。conusmer 会在消费消息后向 borker 提交自己当前的 consumer osffset ，那么下次再消费时就可以从 broker 中知道上一次消费到什么位置了

关于consumer offset的提交

  上面已经提到，消费者是需要定期向 Broker 提交 Consumer offset 以汇报自己的消费进度的，这个动作称为位移提交。针对位移提交操作在较新版本的Kafka Consumer 中有1个专门的 Topic (__consumer__offsets)与之对应，也就是说提交消费者偏移量实质就是发送一条消息到这个 Topic 中。而这个 Topic 我们可以在 Kafka 的日志文件夹下观察到，它一共有50个文件夹，也就是说它有 50 个 Partition。

那为什么需要这么多 Partition 呢？这是因为位移提交是一个非常频繁的操作，利用 Partition 的负载均衡特性可以减轻并发操作同一个 Topic 时所带来的压力。

  注意，在早期的 Kafka Consumer 中消费者偏移量是保存在 Zookeeper 中的，这样做的愿意是为了降低 Broker 的管理成本从而提高 Boker 集群的伸缩性。但后来因为位移提交是一个频繁操作，而 Zookeeper 并不合适保存大数据量以及作频繁的数据修改，因此在后期的新版本 Kafka Consumer 中消费者偏移量从 Zookeeper 迁移会了 Broker 管理。

producer interceptor

生产者拦截器，可以在消息序列化之前将其拦截来做一些消息定制化修改。

实现方式是实现 ProducerInterceptor 接口，并将其实现添加进生产者配置中即可（ProducerConfig.INTERCEPTOR_CLASSES_CONFIG）。

public interface ProducerInterceptor<K, V> extends Configurable {

    // 拦截消息
    public ProducerRecord<K, V> onSend(ProducerRecord<K, V> record);
    // 接收到broker ack应答调用
    // 当Exception为null时消息生产成功
    public void onAcknowledgement(RecordMetadata metadata, Exception exception);
    // 拦截器关闭时调用，发送完消息后关闭
    public void close();
}

自定义分区策略

上面已经提到过，Partitioner会根据 ProducerRecord 中的key来定向将消息数据持久化到指定partition中，但没有指定 key 的话，就会通过轮询方法来持久化消息数据。而如果默认的分区策略（轮询）无法满足业务需求的话就可以尝试自己动手实现分区策略，可以实现以下接口来实现：

public interface Partitioner extends Configurable, Closeable {

    // 分区逻辑实现
    public int partition(String topic, Object key, byte[] keyBytes, Object value, byte[] valueBytes, Cluster cluster);

    // Partitioner关闭，用于回收资源操作
    public void close();

}

value.serializer

  • 指定 value 使用那种序列化器(必须使用全类名)

acks

  • 用于判断消息是否发送成功的依据

  • 0：Producer 完全不理会 Replica 的是否已经日志落盘（返回 ack响应），发送完一批消息后会马上发下一批消息，这时 Producer 的吞吐量最高，但因为完全不理会日志是否落盘，所以就算出现发送失败也不会进行重试操作，也就是说存在丢失日志的风险

  • 1（默认）：Producer 只等待 Leader Replica 所在的 Broker 将数据落盘成功（返回 ack响应）。换句话就是只要 Leader Replica 持久化日志成功，Producer 就会视消息数据发送成功

  • -1或all：Producer 会等待集群中所有 Replica 的日志落盘成功（返回 ack响应）才会发送下一批消息。当前配置模式能够百分百成功持久化日志，但吞吐量是3种配置中最低的

  • 注意这里提到的发送并不是指调用send方法，而是消息缓冲池满了后发送消息给 Brokder 的操作

  • 用于指定消息的压缩类型，默认为none，既不对消息进行压缩操作
  • 目前 kafka 支持的压缩格式有 gzip、snappy、lz4、zstd（Kafka 2.1 开始支持，建议以官方文档为准）

retries

  • 消息重发机制

  • 消息发送失败的原因可能是 Broker 出现网络问题又或者时正处于 Leader 选举而导致的

  • 值0时（默认），表示不进行重试

  • 值为n时，表示可以重试n次

  • 要十分注意，重试操作可能会导致消息重复发送。例如因为网络原因 Brokder 保存消息成功但没有成功响应 Producer，而 Producer 因为没有收到 Broker 的 ack 从而进行了消息重发

  • 除此之外，开启重试操作还会出现消息乱序问题。例如a、b、c三条数据，a和c发送成功但b失败了，b重新发送，就导致原本的abc编程了acb

max.in.flight.requests.per.connection

  • 用于设置 producer 连接上允许发送的未应答请求数（默认为5）

  • 通俗点说就是在 Broker 未响应之前 producer 可以 send 多少条数据

  • 注意，如果开启了 retries 但又想保持重试消息要优先于之后的消息，那么可以将该值设置为1。

    • 举个栗子：当前有两条数据 xxx 和 ccc 要发送给 Broker ，但因为某些原因ccc成功发送而xxx失败进入了重试操作，而在 xxx 重试期间 producer 又给Broker发送了消息 nnn 且发送成功了，这时就算 xxx 重试成功，但显然已经乱需了，因为 nnn 应该在 xxx 后面。而将max.in.flight.requests.per.connection 设置为1，那么在重试发送 xxx期间只要broker没有应答，producer就不会发送nnn

retry.backoff.ms

  • 用于设置重试时间间隔（默认为100毫秒）

batch.size（重要）

  • 用于设置 Producer.send 批量发送的数量。既多少条 Record 作为一个批次。该值增大可以提高 Producer 发送数据的吞吐量
  • 该值默认为 16 k，强烈建议生产环境适当地调大一些增加 Producer.send 操作的吞吐量，但要注意，与此同时会加大内存的压力

linger.ms（重要）

  • 基于时间的 batch 打包策略，可配合 batch.size 使用，相当于满足该值后就作为一个 batch，适当地提高该配置值可以让 batch 被填充得更加丰满，因为正常情况下（linger.ms为0时）batch 可能没满就被发送线程从缓冲池中取出将其发送了
  • 该值默认为 0 秒，既不开启该功能

  • 指定消息发送缓冲池的大小，默认为 32 M

Consumer Group概念

为什么需要消费者组？

当 producer 生产消息的速度比 consumer 的消费速度快时，consumer 的消费能力就显得低下了。因此无法满足一些对实时性要求较高的业务。那么消费者组就是用来提高 consumer 的消费能力的

消费者组说明

消费者组订阅 Topic T1，其中 T1 的 Partition 数量为4。当消费者组中仅有1个消费者时

消费者组订阅 Topic T1，其中 T1 的 Partition 数量为4。当消费者组中有2个消费者时

消费者组订阅 Topic T1，其中 T1 的 Partition 数量为4。当消费者组中有4个消费者时

消费者组订阅 Topic T1，其中 T1 的 Partition 数量为 4。当消费者组中有5个消费者时

2个消费者组同时订阅 Topic T1，其中 T1 的 Partition 数量为4。

消费者组1，有4个消费者。消费者组2，有2个消费者。

小结

  1. 同 Consumer Group 中的不同 Consumer，在同一时刻无法消费同一个 Partition
  2. Consumer Group 中的 Consumer 可以同时消费多个 Partition，但不能消费不属自己负责的 Partition
  3. 即使 Consumer Group 中的 Consumer 数量多于 Partition 数量，也不会提高 Consumer Group 的消费能力
  4. 多个 Consumer Group 同时订阅同一个 Topic 时，互不干扰

分区再均衡

是什么？

  在 Consumer Group 中的 Consumer 没超过 Partition 数的情况下，每一个 Consumer 都能分配到一个独自负责消费的 Partition 。但因为某些原因，譬如有新的 Consumer 加入到当前 Group 中，那么 Consumer Group 就会根据分配策略来重新设置 Connsuemr 和 Partition 的消费关系，而这个过程就叫做分区再均衡。

  注意，再均衡和 JVM 的 STW 类似，在再均衡期间 Consumer 是不能消费消息的，而且再均衡所需消耗的时间会随着 Consumer 的增多而增加。

什么情况会发生分区再均衡？

  1. Consumer Group 成员数量发生变化

    • 譬如有新 Consumer 加入到 Group 中，又或是有 Consumer 宕机导致从 Group 中脱离时
    • Broker 会作为 Consumer Group 的协调器，Consumer 则需要定时地主动向协调者发送心跳包（在Kafka-0.10.1版本之后有专门的线程来发送心跳包），而如果 Consumer 长时间（session.timeout.ms指定，默认10秒）没有向协调者发送心跳包的话就会被协调者视为死亡从而导致被踢出Group 而导致分区再均衡

  2. Topic 的 Partition 数量被修改

  3. Consumer Group 所订阅的 Topic 数量发生变化

    • 譬如 Kafka 支持使用正则表达式匹配订阅主题，这样就比较容易发生再均衡了

如何尽量避免再均衡？

首先上面提到的订阅Topic数量发生变化以及订阅Topic的Partition数量发生变化这种两种情况是不能避免的，因此我们应该尽量关注Consuemr Group成员数量发生变化这种情况。譬如除了维护人员主动增加 Consumer 外，上面提到的因为协调器在特定时间内没有收到 Consumer 的心跳包也会导致分区再均衡的发生，而避免这种情况发生可以通过调节一下3个参数来达到：

  • session.timeout.ms：该值默认为 10 秒，表示协调器必须在这段时间收到 Consuemr 的心跳包否者就会将其踢除出Group进而导致分区再均衡的发生
  • heartbeat.interval.ms：该配置用于限制 Consumer 发送心跳包给协调器的时间间隔。该值必须小于 session.timeout.ms 这样才能让协调器更高地感知 Consumer 是否存活
  • max.poll.interval.ms：该配置用于限制 Consuemr 消费消息的最长时间（默认为 5 分钟）。
    • 如果 Consumer 处理消息的时长超过了该配置那么协调器就会认为该 Consumer 消费能力低下从而将其踢出 Group 进而触发分区再均衡
    • 在Kafka-0.10.1版本之前 Consumer 是没有独立的线程来发送心跳包的，也就是说 Consumer 只有在 conmit 或 poll 时才会发送心跳包，所以如果使用的是 0.10.1 之前版本的 Kafka 的话 max.poll.interval.ms 就不能超过 session.timeout.ms 了，否者如果处理时间过长的话会被协调器认为已经挂掉了，会被踢出Group从而导致分区再均衡
    • 如果因为 Consumer 消费能力低下而导致分区在混合，那么 Consumer 在处理完消息 commit 时将会发生CommitFailedException 异常，该异常是不可恢复的，它要表达的是分区再均衡导致 Parition 分配给了其它 Consumer，因此当前这个 Conusmer 并不能提交到到当前 Partition 的 offset。避免以上情况的解决方案就是及时消费消息，例如增大 max.poll.interval.ms 值，又或者降低 max.poll.records 值（控制一次poll所获取到的消息数量），再者就是使用多线程增加 Consumer 的消费能力

Broker 中的协调者指的是什么？

在上文中提到，“协调者”会负责管理 Consumer Group ，而这个“协调者”实质就是指 Coordinator 。Coordinator 是一个专门为 Consumer Group 服务的功能，它有以下特点：

  • 每一个 Broker 都有自己的 Coordinator
  • 负责 Consumer Group 的再均衡操作
  • 负责消费者偏移量的管理
  • 负责 Consumer Group 成员的管理

那么 Consumer Group 是如何确立与自己对应的 Coordinator 的呢？有以下2部：

  1. 确定由 __consumer__offsets 这个 Topic 的哪一个分区来保存当前 Consumer Group 的数据

    partitionId=Math.abs(groupId.hashCode() % offsetsTopicPartitionCount)

  2. 找出该分区的 Leader 副本所在的 Broker，而该 Broker 就是当前 Conusmer Group 所对应的 Coordinator 了

  • 第1个含义：用于指定 Broker 协调器对 Consumer 停止发送心跳包这种行为的最大容忍限度（默认为10秒），一旦超过该配置指定的事件，Consumer 就会被视为死亡，因此会进行分区再均衡操作
  • 第2个含义：Consumer 处理消息的最大时间限制（既两次poll之间的时间，默认为 5 分钟）。例如 session.timeout.ms = 60000 那么消费者处理消息必须在60s内完成，否则 Broker 协调器会将该 Consumer 视为跟不上消费进度并将其踢出消费者组，因此也会进行分区再均衡操作

  • kafka 0.10.1.0 版本之后 session.timeout.ms 进行了拆分，拆分之后 session.timeout.ms 只包含第1个含义，且默认值为10s

  • 而 max.poll.interval.ms 则是第2个含义（建议根据实际处理消息所需要的时间来设置，最好比实际所需的时间大一些）

auto.offset.reset（重要）

  • 该配置用于指定消费者读取一个没有偏移量(还没有消费提交offset)或偏移量无效（因消费者长时间失效，包含偏移量的记录已经过时并被删除）的分区时的应对手段。
  • Kafka提供以下 3 种消费模式：
    • latest（默认）：
      • 在偏移量无效的情况下（例如最开始没有offset），Consumer 会从最新插入的那条记录开始消费（在消费者启动之后插入的记录）
      • 偏移量有效的情况下，Consumer 会在最新的偏移量位置开始消费（既上一次提交的offset位置）
    • earliest：
      • 在偏移量无效的情况下，Consumer 会从 Partition 的起始位置开始消费
      • 偏移量有效的情况下，Consumer 会在最新的偏移量位置开始消费（既上一次提交的offset位置）
    • none：
      • 各个 Partition 都存在有效的 offset时，会从有效的 offest 位置开始消费
      • 只要有一个分区不存在有效的 offset，则抛出异常

  • 为了尽量避免重复消费的情况出现，建议把它设为false，然后由程序来控制提交时机

  • 如果设为true，则可以通过配置 auto.commit.interval.ms 来控制提交offset的时间间隔

  • 该参数必须在 enable.auto.commit = true 时才生效;

  • 用于指定分区分配策略，通俗点来说就是如何将分区分配给消费者

  • 默认为range，既随机分配，还可以设置成 roundrobin 轮询分配

client.id

  • 用于给 Broker 设置客户端标识

fetch.max.bytes（重要）

  • 用于指定 Consumer 单次获取消息数据的最大字节

  • 注意，如果实际业务中消息都很大的，那么就需要设置一个较大的值，否则无法获取数据。除此之外提高该值还可以增加 Consumer 的吞吐量（增大吞吐量还可以使用多线程方式解决）

  • 用于指定单次 poll 能够获取到的最大 record 数量（默认为500）

手动提交offset

自动提交存在的问题

问题1：Consumer 不能及时消费导致数据丢失

譬如程序处理消息较慢（既Consumer消费能力较弱），导致 offset 在未消费完消息就自动提交了，而这时刚好遇上 Consumer 宕机就会导致数据丢失。因为 offset 已经提交，所以分区再均衡后就会根据该 offset 后的消息开始消费，而其实这个 offset 根本就没有消费成功

问题2：Consumer 不能及时提交 offset 导致重复消费

譬如 Consumer offset 自动提交的频率为 5 秒。现在假设 Conusmer 在前 3 秒已经消费了不少消息，但又因为未到 5 秒而无法自动提交 offset，而这时正好遇上宕机或者分区再均衡的话就会导致先前 Consumer 3 秒内所消费的消息再次被消费，因为 offset 根本没有提交，所以就相当于没有消费这些消息

如何解决以上两个问题呢？

  可以通过手动提交 Consumer offset 来解决。例如我们可以在处理完数据之后手动交 offset ，而不是在此之前或较久的之后提交。

  出现问题1是因为 Consumer 消费消息的能力较低而导致 offset 提前提交了而后来又没有处理消息成功而导致的，那么我们只需要做到只有处理消息成功后才提交 offset 这个点问题就可以解决了。

  而问题2中未能及时提交 offset 的情况从另一个角度来说是无法根治的，因此极端情况下即使我们做到处理完消息后马上手动提交offset也不能保证在这个极为短暂的时间间隙里 Consuemr 不会宕机，所以处理完消息后及时提高 offset 只是一种较好的习惯而并不是解决重复消费的方案。因此我们可以得出这么一个结论就是从消息系统层面是无法保证不会重复消费的，解决这个问题只能是在 Consumer 端的业务代码层面解决。

同步提交

while (true) {

    ConsumerRecords<String, String> records = consumer.poll(100);

    // 处理消息

    try {
        consumer.commitSync();
    } catch (CommitFailedException e) {
        log.error("commit failed", e)
    }
}

同步提交是一个阻塞操作，既需要等待 Broker 应答后才能继续工作，换句话来说只要不是发生无法恢复的异常commitSync() 都能提交成功，但这样虽然能够保证 offset 的安全提交到Broker，但无疑会降低消费者的吞吐量

异步提交

用于解决同步提交导致吞吐量下降的问题

while (true) {

    ConsumerRecords<String, String> records = consumer.poll(100);

    // 处理消息

    consumer.commitAsync();
}

需要注意的是 commitAsync() 不像 commitSync() 那样保证能够成功将 offset 提交到 Broker，换言之就是可能会出现提交失败的情况，这时可以给 commitSync() 设置回调方法进行重试解决

while (true) {
    ConsumerRecords<String, String> records = consumer.poll(100);

    // 处理消息

    consumer.commitAsync(new OffsetCommitCallback() {
        public void onComplete(
            Map<TopicPartition,
            OffsetAndMetadata> offsets, 
            Exception exception) {

            if (e != null){
                // 异常处理
            }
        }
    });
}

混合提交

try {
    while (true) {
        ConsumerRecords<String, String> records = consumer.poll(100);

        // 处理消息

           // 异步提交
        consumer.commitAsync();
    }
} catch (Exception e) {
    log.error("Unexpected error", e);
} finally {
    try {
        // 同步提交，用于防止异步提交失败
        consumer.commitSync();
    } finally {
        consumer.close();
    }
}

混合提交可以进一步保证 offset 能够成功提交到 Broker 也兼顾了吞吐量。

@Configuration
public class KafkaConsumerConfig {

    @Autowired

    @Bean
    public ConsumerFactory<String, String> kafkaConsumerFactory() {
        // 注入消费者配置
        Map<String, Object> map = properties.buildConsumerProperties();
        // 默认允许处理消息的最大时间，默认300s
        map.put(ConsumerConfig.MAX_POLL_INTERVAL_MS_CONFIG, 50000);
        return new DefaultKafkaConsumerFactory<>(map);
    }

    @Bean
    public KafkaListenerContainerFactory<ConcurrentMessageListenerContainer<String, String>> kafkaListenerContainerFactory() {
        ConcurrentKafkaListenerContainerFactory<String, String> factory = new ConcurrentKafkaListenerContainerFactory<>();
        factory.setConsumerFactory(kafkaConsumerFactory());
        // 消费完马上提交
factory.getContainerProperties().setAckMode(ContainerProperties.AckMode.MANUAL_IMMEDIATE); 
        // 异步提交
      factory.getContainerProperties().setSyncCommits(true);
        return factory;
    }

}

Zookeeper 相关

Kafka 和 Zookeeper 的关系。其中图中 “直角方块” 代表持久性节点 而 “圆角方块” 代表临时节点

  • ids 节点维护的是集群中的 broker 的唯一标识，其中包含了每个 broker 的元信息
  • topics 节点维护的是 Topic 的 Partitions 信息，其中数字为特定分区的编号，而 state 由 Leader 创建且保存的是当前 Leader 和 其它 ISR 成员的信息，一旦 Leader 不可用该节点会被新 Leader 从新创建

问题配置

如何保证消息不丢失

数据丢失的核心原因是 Broker 没有保存成功，当然换个角度是 Producer 没有发送成功。而还需要特别注意的是 Producer中的 send 方法是异步的，既即使 send 返回后也不能保证消息发出去了，而只是发送到了缓冲中

Producer配置

  • block.on.buffer.full = true

    消息缓冲池满后停止接收新的消息而不抛出异常，避免因为异常丢失数据

  • acks = all

    生产者生成消息后需要得到所有的 Broker 的应答，否则认为发送不成功，因此进行重试发送

  • retries = Integer.MAX_VALUE

    将重试次数设置到最大值，表示尽量重试

  • max.in.flight.requests.per.connection = 1

    因为开启了重试，所以将其设置为1来保证消息的顺序

  • 发送消失时，预防发送失败，使用回调函数来补救。即不使用 producer.send(msg)，而是使用 producer.send(msg, callback)

Broker配置

  • unclean.leader.election.enable = false

    避免非ISR成员竞选为 Leader 从而导致数据丢失

  • replication.factor 设置大于等于 3

    尽量使用备份来避免分区单点故障

  • min.insync.replicas = 1

    表示至少有1个 Replica 成功写入数据并响应 Leader 才算写入成功，否则视为持久化不成功（producer抛出异常）。该配置只有在 Producer 设置了acks=all 才有效，建议设置过半节点数

Consumer配置

  • 采用手动提交 offset 而不是自动提交
  • 消费（处理）完后才提交 offset ，而不是拿到消息还没处理就提交 offset
  • Consumer 应该及时消费消息

如何保证只消费1次

一条消息多次消费的原因：

  • offset 没能及时提交。譬如处理完消息后没有提交offset就宕机了
  • 生产者开启了重试功能，导致消息重复生产

其实在上面已经提高过，对于消息被重复消费这个问题目前而言在消息系统层面是无法解决的，而唯一解决的办法就是从 Consumer 端的业务代码层面解决

如何解决消息乱需问题

首先需要知道的是目前的 Kafka 并不支持全局（多个Partition之间）消息有序，只能保证单个 Partition 内是有序的。且当生产者开启重试功能后，Partition 内部也是可能发生乱序情况的。这时就需要将max.in.flight.requests.per.connection 设置为1来尽量降低乱序情况

关于 Kafka Producer 的幂等性

在 Producer 方可以通过添加 enable.idempotence = true 配置参数来实现 Producer 发送消息的幂等性。但它有以下局限：

  1. 只能保证单个 Partition 消息的幂等，既如果 Topic 有多个会话就不好使了
  2. 只能保证当前会话发送的消息的幂等，既如果 Producer 重新创建新的会话就不好使了，譬如重启Producer

对实时性要求较高，建议使用 RoketMQ

因为 Kafka 采用了异步 Producer.send 以及 发送缓冲来提高消息发送的吞吐量，因此在实时性访问并没有 RocketMQ 那么好。如果系统业务对消息实时性要求较高，那么个人建议这时应该使用 RocketMQ 而不是 Kafka。换句话说你可以简单理解如果需要高吞吐量就选择 Kafka ，而需要实时性高就选择 RocketMQ。

监控 Kafka Consumer 的消费进度

注意：监控消费进度实质就是观察消费者组的 Lag 值。Lag 指的是 Consumer 落后的进度。譬如 Producer 生成1000 条消息，而 Consumer 值消费了 600 条，那么当前 Consumer 的 Lag 值就是 400，表示 Conusmer 已经滞后了 400 条消息。因此 Lag 值越大就表示 Consumer 的处理能力越低

监控方式有以下 3 种：

  • kafka-consumer-groups 脚本 (推荐)

    kafka-consumer-groups.sh --bootstrap-server <Kafka broker 连接信息 > --describe --group <group 名称 >

  • Kafka Consumer API

  • JMX 监控 Consumer（推荐）
