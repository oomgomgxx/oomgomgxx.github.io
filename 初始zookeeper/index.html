基本介绍

简介

  • Zookeeper 是一个分布式协调服务。简单说就是可以利用 Zookeeper 的数据特点来协调分布式服务运行。譬如 Dobbo 利用 Zookeeper 提供的命名服务（既DNS服务）来实现服务的注册和发现
  • 保证节点状态一致性优先的前提下，Zookeeper 集群只要过半节点能够通信就能正常提供服务
  • Zookeeper 在内存中存储的数据模型叫 Data Tree ，实质就是和文件系统目录结构一样呈现树状。在 Data Tree 上的节点被称为 ZNode ，ZNode 除了保存子节点之外，自身还可以存储数据，而且带有Version版本号等元数据信息
  • Zookeeper 并不适合存储大量数据。也就是说数据应该尽量少，而且这些数据对于分布式服务协调来说是至关重要的
  • Zookeeper 并不适合高并发写的场景。这是因为写操作只能在 Leader 节点上进行，除此之外 Leader 节点在处理一个写请求的同时，当前 Leader 节点是不能再处理其它读写请求的，因为它是（会话）串行(FIFO形式)处理请求的，所以 Zookeeper 相当于降低了系统的可用性来提高一致性。这也是为什么 Kafka 不再使用 Zookeeper 来保存消费者偏移量的原因，因为消费者提交偏移量是一个非常频繁的写操作
  • 在 Zookeeper 中，同一个会话的操作是先进先出（FIFO）按顺序被操作的

关于节点

节点类型

zookeeper节点的基本类型：

  1. 持久(persistent)节点
  2. 临时(ephemeral)节点（会话级别）
  3. 顺序节点

组合节点类型：

  1. 持久(persistent)节点
  2. 临时(ephemeral)节点
  3. 持久顺序节点
  4. 临时顺序节点

acl策略

作用：对节点进行访问控制

使用说明

[支持的认证策略]
world：默认，所有会话可以访问
auth：auth策略。使用前需要先用 authadd 添加认证策略
digest：用户密码策略。会话访问节点时需要正确的用户和密码
ip：ip绑定策略。会话访问节点时需要是特定的IP才能访问

[支持的访问权限（默认crwda，表示有所有操作权限）]
CREATE： 创建
READ： 读取
WRITE： 写入
DELETE： 删除
ADMIN： 管理

acl策略使用

1）ip模式

create /test ip:127.0.0.1:crwda

2）digest模式

注意：设置节点的 digest 时，需要对密码部分进行编码。

使用 org.apache.zookeeper.server.auth.DigestAuthenticationProvider ，譬如：

java -cp ./lib/:./ org.apache.zookeeper.server.auth.DigestAuthenticationProvider 账号:密码 

create /test/t3 't3'
setAcl /test/t3 digest:test:V28q/NynI4JI3Rk54h0r8O5kMug=:crwda
addauth digest test:test

3）auth模式

auth 和 digest 差不多，但 auth 模式需要先使用 addauth 添加权限再操作，其次是 auth 模式下无需对密码进行编码

addauth digest aaa:aaa
setAcl /test/t4 auth:aaa:aaa:crwda

zookeeper

依赖

<dependency>
    <groupId>org.apache.zookeeper</groupId>
    <artifactId>zookeeper</artifactId>
    <version>3.4.6</version>
</dependency>

缺点

  • watcher无法复用
  • 没有提供连接失败重试的实现
  • 没有提供创建多级节点的实现

基本使用

public class TestZookeeper {
    private static final Logger LOGGER = LoggerFactory.getLogger(TestZookeeper.class);

    // 因为Zookeeper连接操作时异步的，所以需要等待其连接成功后，否则主线程会先关闭
    private static final CountDownLatch countDownLatch = new CountDownLatch(1);

    public static ZooKeeper connect(String server, int timeout, Watcher watcher) throws Exception {
        return new ZooKeeper("127.0.0.1:2181", 20000, watcher);
    }

    public static void main(String[] args) throws Exception {

        Watcher masterWatcher = watchedEvent -> {
            // 判断连接状态
            if (watchedEvent.getState() == Watcher.Event.KeeperState.SyncConnected) {
                LOGGER.info("连接成功");
                countDownLatch.countDown();
            }
        };

        Watcher createWatcher = watchedEvent -> {
            // 判断操作类型
            if (watchedEvent.getType() == Watcher.Event.EventType.NodeCreated) {
                LOGGER.info("创建节点成功");
            }
        };

        ZooKeeper zooKeeper = connect("127.0.0.1:2181", 5000, masterWatcher);
        countDownLatch.await();

        // 根节点
        if (zooKeeper.exists("/test", createWatcher) == null) {
            // 创建所有会话可操作的持久化节点
            zooKeeper.create("/test", null, ZooDefs.Ids.OPEN_ACL_UNSAFE, CreateMode.PERSISTENT);
        }

        // 创建临时节点
        zooKeeper.create("/test/t1", "t1".getBytes(), ZooDefs.Ids.OPEN_ACL_UNSAFE, CreateMode.EPHEMERAL);
        zooKeeper.create("/test/t2", "t2".getBytes(), ZooDefs.Ids.OPEN_ACL_UNSAFE, CreateMode.EPHEMERAL);

        // 添加认证
        zooKeeper.addAuthInfo("digest", "aaa:aaa".getBytes());

        // 设置acl - digest
        List<ACL> digest = TestZookeeper.getAcls("digest", DigestAuthenticationProvider.generateDigest("aaa:aaa"));
        // version是乐观所的期望值，如果与zookeeper中的不一直则会操作失败,当版本号设置为-1时，忽略节点的版本号
        Stat stat = zooKeeper.setACL("/test/t1", digest, -1);
        LOGGER.info(zooKeeper.getACL("/test/t1", stat).toString());

        // 设置acl - ip
        List<ACL> ip = TestZookeeper.getAcls("ip", "127.0.0.1");

        Stat t2Stat = zooKeeper.exists("/test/t2", null);
        t2Stat = zooKeeper.setACL("/test/t2", ip, t2Stat.getVersion());
        LOGGER.info(zooKeeper.getACL("/test/t2", t2Stat).toString());
        byte[] data = zooKeeper.getData("/test/t2", null, t2Stat);
        LOGGER.info(new String(data));

        TimeUnit.SECONDS.sleep(10);
        zooKeeper.delete("/test/t1", -1);

        TimeUnit.SECONDS.sleep(30);
        zooKeeper.close();
    }


    public static List<ACL> getAcls(String scheme, String id) {
        ACL auth = new ACL(ZooDefs.Perms.ALL, new Id(scheme, id));
        final List<ACL> aclList = new ArrayList<>();
        aclList.add(auth);
        return aclList;
    }
}

基本使用

public class TestCurator {
    private static final Logger LOGGER = LoggerFactory.getLogger(TestZookeeper.class);

    private static final String SERVERS = "127.0.0.1:2181";
    private static final String ROOT_PATH = "/test";

    public static CuratorFramework createConn(String servers, 
                                           int timeout,  // 超时时间
                                           int n, // 重试次数
                                           int sleepMsBetweenRetries) { // 重试间隔 

        // 重试策略
        RetryNTimes retryNTimes = new RetryNTimes(n, sleepMsBetweenRetries);

        // 连接Zookeeper
        CuratorFramework client = CuratorFrameworkFactory.builder()
                .connectString(SERVERS) // Zookeeper地址
                .sessionTimeoutMs(timeout) // 连接超时
                .retryPolicy(retryNTimes)  // 重试策略
                .build();

        // 连接
        client.start();

        return client;
    }

    public static void main(String[] args) throws Exception {

        // 获取连接
        CuratorFramework client = createConn(SERVERS, 10000, 10, 5000);

        // 创建一个临时（根）节点，并初始内容为空
        if (client.checkExists().forPath(ROOT_PATH) == null) {
            client.create().withMode(CreateMode.EPHEMERAL).forPath(ROOT_PATH);
        }

        // 注册可重用监听
        PathChildrenCache cache = new PathChildrenCache(client, ROOT_PATH, true);
        cache.start();
        PathChildrenCacheListener cacheListener = (client1, event) -> {
            System.out.println("事件类型：" + event.getType());
            if (null != event.getData()) {
                System.out.println("节点数据：" + event.getData().getPath() + " = " + new String(event.getData().getData()));
            }
        };
        cache.getListenable().addListener(cacheListener);


        // 递归创建一个临时节点
        // 递归创建：即使没有根节点都可以创建成功
        client.create().creatingParentContainersIfNeeded().withMode(CreateMode.EPHEMERAL).forPath("/test/t1","t1".getBytes());
        TimeUnit.SECONDS.sleep(10);
        // 如果主节点是临时的话，就不能构建其子节点。
        //client.create().creatingParentContainersIfNeeded().withMode(CreateMode.EPHEMERAL).forPath("/test/t1/t11","t11".getBytes());

        // 设置acl操作
        List<ACL> ip = TestZookeeper.getAcls("ip", "127.0.0.1");
        client.setACL().withACL(ip).forPath("/test/t1");
        System.out.println(client.getACL().forPath("/test/t1"));
        TimeUnit.SECONDS.sleep(30);

        int version = client.checkExists().forPath(ROOT_PATH).getVersion();

        // 递归删除子节点
//        client.delete().deletingChildrenIfNeeded().forPath("/test/t1");
        // 指定版本号
        client.delete().deletingChildrenIfNeeded().withVersion(version).forPath("/test/t1");

        client.close();
    }

    public static List<ACL> getAcls(String scheme, String id) {
        ACL auth = new ACL(ZooDefs.Perms.ALL, new Id(scheme, id));
        final List<ACL> aclList = new ArrayList<>();
        aclList.add(auth);
        return aclList;
    }

}

数据持久化了解

简介

  在 Zookeeper 中负责数据持久化和启动数据恢复的组件叫ZKDatabase。ZKDatabase 是 Zookeeper 的内存数据库，主要负责会话管理、事务日志、生成DataTree快照以及重启时恢复数据等工作。

  Zookeeper 的持久化方案和 Redis 非常相似，分别有事务日志（类似AOF）和DataTree快照（类似RDB）两种。ZKDatabase 会在事务产生时将日志写入到事务日志缓冲后再由系统负责定时冲刷到磁盘，以及在特定时间生成 DataTree 快照并将其持久化。

为什么是 server2 做了 Leader？

  首先服务器并不是同时启动的，在当前案例中我们先启动了 server1，再启动 server2，但因为 server2 的集群节点 myid 值比 server1 大，而且这时 server1 + server2 已经够过半票数了，所以 server2 就自然成为 Leader 了。虽然随后 server3 加入到集群，但因为集群中已经有了 Leader，因此 server3 就只能成为 Follower 了。

4）查看集群配置

3.5.x 版本配置变动

新增重要特性

新增 Dynamic Reconfiguration （动态配置）功能，解决手动扩容时可能导致已提交事务被覆盖从而丢失数据的问题。

配置变动

  • 3.5.3 版本开始，动态配置默认关闭，需要通过reconfigEnabled=true开启

  • 3.5.0 版本开始，Zookeeper 默认为单节点模式（Standalone），在该模式下无法对其进行扩展，因此如果要部署集群，则需要在配置中设置standaloneEnabled=false

  • 3.5.0 版本开始，原来提供客户端服务的 clientPort（服务端口）、clientPortAddress（服务地址） 两个配置项被弃用，而被修改为如下格式：

    # 格式
server.<positive id> = <address>:<port1>:<port2>[:role];[<client port address>:]<client port>
# role 有两种，分别是participant(参与者)和observer(观察者)，不设置默认为participant
# client address相当于原来的clientPortAddress，既提供客户端服务的IP地址，默认值和address一致
# client port相当于原来的clientPort，既提供客户端服务的端口

# 例子（集群）=======================
# 其中1234为集群通信端口，而1235为选举端口
server.1 = 192.168.0.201:1234:1235;2181
server.2 = 192.168.0.202:1234:1235:participant;2181
server.3 = 192.168.0.203:1234:1235:participant;2181
server.4 = 192.168.0.205:1234:1235:observer;2181
# 以上配置表示server.1、server.2、server.3为普通节点，可以参与选举投票。而server.4为Observer节点，只用于备份和扩展读操作，不参与选举投票

启用动态配置例子

zookeeper.cfg

  • 静态配置，不可变
tickTime=2000
initLimit=10
syncLimit=5
reconfigEnabled=true
standaloneEnabled=false
dataDir=/zookeeper/data/zookeeper1

# 动态配置项文件
# 动态配置如果变动，则由Zookeeper推送，静态配置不变
dynamicConfigFile=/zookeeper/conf/zookeeper_1.dynamic

zookeeper_1.dynamic

server.1=125.23.63.23:2780:2783:participant;2181
server.2=125.23.63.24:2780:2783:participant;2181
server.3=125.23.63.25:2780:2783:participant;2181

动态扩容例子

  1. 假设现在集群中已经有了 3 个节点（集群节点必须都配置 reconfigEnabled=true 和 standaloneEnabled=false）
  2. 现在打算给该集群新增1个节点，但在此之前必须使用 Zookeeper 提供的 superDigest 产生一个超级用户权限，这是使用动态配置的前提。因此需要执行以下两个步骤的操作：
    1. 使用java -cp ./lib/*:./* org.apache.zookeeper.server.auth.DigestAuthenticationProvider super:passwd 得出编码后的账号密码信息。譬如执行后得到super:passwd -> super:gACzJ4L2A0F2ygTno5HQnfabuik=
    2. 然后将第一步得到的内容配置到 Zookeeper 中。譬如可以通过设置环境变量来实现，即直接执行export SERVER_JVMFLAGS="-Dzookeeper.DigestAuthenticationProvider.superDigest=super:gACzJ4L2A0F2ygTno5HQnfabuik=" 即可
  3. 在第二步操作过后，重启当前机器上的 Zookeeper 节点（集群中的其中一个节点），然后在重启完成后使用命令行客户端连接上，执行如下两个命令：
    1. 先执行addauth digest super:passwd进行权限认证
    2. 然后通过动态修改配置往集群中增加一个节点（在这之前新节点应该是正在运行的）。执行命令reconfig -add server.4=125.23.63.26:2780:2783:participant;2181 即可

常见 Zookeeper 使用场景

配置管理

将共享配置存放到 ZooKeeper 节点中。

应用在启动时到 ZooKeeper 中获取节点信息，同时也可以在其节点上添加 Watch 来监听配置内容的变更以实现动态更新配置。

全局ID

利用 ZooKeeper 的顺序节点和 API 特性来实现。

譬如 create() 一个名称为 id- 的顺序节点，该方法就会返回一个诸如id-0000000005的节点名称。那么就可以使用这个节点名称来做全局 ID 了。除此之外，还可以在 id-0000000005 的基础上增加一些额外的信息。譬如 type-id-0000000005 以实现更富有内涵的全局 ID。

主备切换( 热备 )

  • 方法1

  利用临时节点。例如两台服务器启动时先成功创建节点的就先提供服务，而创建节点失败的服务器则则需要在该节点上添加一个 Watch 来监听节点的消失，因为一旦该节点消失就可以顶替已经宕机的服务器来继续对外提供服务了

  • 方法2

  利用临时顺序节点。例如多台服务器在启动时，在持久性节点 instances 下创建临时顺序节点。创建完成后判断自己的序号是否为最小，如果是则对外提供服务，即 RUNNING 状态。如果不是则为 STANDBY ，并监听 instances 节点的变化来切换自己的状态

Master选举

Master 选举和上面主备切换的方法1是一样的。既先成功创建节点的就为 Master，而 Slave 则是在该节点上添加Watch ，一旦发现节点消失则顶替 Master 继续提供服务。

命名服务（DNS服务）

利用临时节点来存储服务提供者 IP 地址。

譬如可以先创建一个持久性节点 /services/tiket-services 用于聚集服务列表，然后服务提供者在启动时将自己注册到该节点上。例如注册时可以创建一个临时节点，名称为address-127.0.0.1:8080。然后服务消费者通过获取持久节点 /services/tiket-services 下的子节点列表再进行服务调用。

当然，具体的实现方式千差万别，但基本原理是一样的。

分布式锁

  • 排它锁

利用临时节点和Watch（和上面提到的Master选举操作是一样）。

譬如获取锁时可以在持久节点/exclusive_lock中创建一个临时节点lock。那么即使是并发创建也只会有一个客户端创建成功（成功即表示获取到锁），而没有成功的客户端就在临时节点lock上添加 Watch 来监听其状态编号，一旦该节点消失或被主动删除（释放）后就再竞争创建该节点（竞争锁）。

而之所以使用临时节点原因是避免获取到锁的客户端因为特殊原因而无法主动释放锁而造成死锁。

  • 共享锁

利用临时顺序节点和Watch。

例如获取锁时可以在持久节点/shared_lock中创建一个临时顺序节点，但这时和排它锁就稍有区别了。

1）读锁获取过程：

  1. 创建临时顺序节点。例如名称为主机IP-R-。如果创建成功 Zookeeper 就会生成一个名称诸如为 主机IP-R-0000000001 的临时顺序节点
  2. 然后客户端程序需要再获取/shared_lock下的子节点列表
  3. 最后判断是否获取锁成功，会有以下种情况放生：
    1. 通过判断，发现当前创建的节点的序列号最小，则获取共享锁成功
    2. 如果不是最小，则需要接着判断在比自己序号小的节点中是否存在写节点，如果不存在则获取共享锁成功
    3. 如果比自己序号小的节点中存在写节点，则获取共享锁失败，需要进入等待状态，并在距离自己最近且比自己序号小的写节点上添加 Watch 监听其释放，当触发事件时则表示获取共享锁成功（因为已经没有写锁了，只剩下共享锁）

2）写锁获取过程：

  1. 创建临时顺序节点。例如名称为主机IP-W-。如果创建成功 Zookeeper 就会生成一个名称诸如为主机IP-W-0000000001 的临时顺序节点
  2. 然后客户端程序需要再获取/shared_lock下的子节点列表
  3. 最后判断是否获取锁成功，会有以下种情况放生：
    1. 通过判断，自己的序号是否为最小，如果是则获取写锁成功
    2. 如果自己的序号不是最小，则进入等待状态，并在距离自己最近且比自己序号小的节点上添加 Watch 监听其消失，当触发事件时则表示获取写锁成功
