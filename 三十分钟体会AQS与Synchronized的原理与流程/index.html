Synchronized原理

简介

  synchronized 是 Java 较早的同步机制，原理是通过系统调用互斥量来实现同步操作的。synchronized 应着两个互斥指令，分别是代表加锁操作的lock指令以及代表解锁操作的unlock指令。但需要注意的是这两个指令在 JVM 字节码中有另一种表示形式，譬如用 monitorenter 代表 lock，用 monitorexit 代表 unlock 。

  在 JDK 1.6 之前 synchronized 被称为重量级锁，这样称呼它的原因是因为线程在竞争锁时需要从用户态切换到内核态，而且在竞争锁期间需要由内核来维护线程状态的切换操作。因此，如果在多线程高并发场景就会出现大量的系统调用（用户态和内核态切换）造成性能损耗。

  在 JDK 1.5 引入的 Lock 自旋锁是在 Java 语言层面实现的，竞争锁操作使用 CAS (CPU原子指令)完成。这里需要重点提一下的是 Lock 自旋锁同样会阻塞（LockSupport.park）而产生用户态切换到内核态切换操作，但 Lock 相较与 synchronized 来说就较为轻量了。这样说的原因有两点，第一点是使用 Java 语言以及 CAS 操作代替了 synchronized 的内核态竞争，虽然 CAS 也是个系统调用但是相较与 synchronized 的互斥量粒度更细更高效；第二点是 Lock 会尽可能地通过自旋来避免线程被阻塞挂起，而会出现竞争锁失败就马上由内核挂起的情况。

  最后要说的是 synchronized 在 JDK 1.6 版本中得到了较大的改进。在上面已经提到 synchronized 之所以重是因为竞争锁操作需要由内核来完成，既每次竞争锁时线程都需要马上从用户态切换到内核态再进行操作，且线程状态将由内核调度维护。而 JDK 1.6 中对 synchronized 的优化点主要集中在如何避免线程进入内核态后被阻塞，因此线程的阻塞和唤醒需要由内核完成，这样的操作对于用户态层面代码来说是较为低效的。譬如引入了如下优化项：

  • 锁消除：尽量无谓的锁操作
  • 锁粗化：减少锁竞争
  • 锁升级：引入无锁、偏向锁、轻量级锁、重量级锁等不同的锁阶段以尽量避免线程进入内核竞争锁
  • 自旋锁（CAS）：自旋锁操作位于轻量级锁和重量级锁的获取过程中，用于避免线程被内核挂起
  • 自适应自旋：自旋锁默认情况下最大自旋数为10，一旦超过阈值则会发生锁膨胀或被内核挂起。自适应则是动态调借该阈值以得出一个更合理的自旋阈值

锁优化

锁消除

  指编译器经过分析后对锁进行消除的操作。譬如虽然代码片段中加了 synchronized ，但实际并不存在并发操作，这时去掉编译器会将 synchronized 块去除以避免无谓的加锁操作。

public String concatString(String s1, String s2, String s3) {
    return s1 + s2 + s3;
}

例如上面这个案例在 JDK 1.6 之前会在前端编译 javac 优化为 StringBuffer ，而在 JDK 1.6 开始则改为了 StringBuilder，而两者的区别在于前者的方法都被 synchronized 修饰，而后者不是。

锁粗化

  锁粗化和锁消除类似，指的是经过分析后发现连续的代码片段中出现了重复加锁操的操作，因此将 synchronized 块范围扩大的过程。

StringBuffer sf = new StringBuffer();
sf.append(s1);
sf.append(s2);
sf.append(s3);

例如上面这个案例中因为 StringBuilder 的 append 方法是同步的，因此锁粗化操作会将多个 synchronized 块合并成一个以减少无畏的加锁操作，譬如下面这样：

StringBuffer sf = new StringBuffer();
synchronized (sf) {
    sf.append(s1);
    sf.append(s2);
    sf.append(s3);
}

自旋锁

  自旋通俗点说其实就是循环地等待某个条件满足为止的意思。而自旋锁这个功能其实在 JDK 1.4 就已经引入了，主要是为了尽量避免因为线程竞争所失败而被内核挂起。但自旋本身是一种较为消耗 CPU 资源的操作，所以如果线程执行任务所需的时间较长的话，就可能会出现 CPU 使用率急速上升的情况，且严重的话甚至会引起卡顿感。因此在这种场景下自旋反而没有直接阻塞来得高效，所以 JVM 提供了两个配置项用来控制自旋操作，分别是自旋开关-XX:+UseSpinning （默认开启）和阈值设置 -XX:PreBlockSpin （默认10次）。

  而自旋锁在 JDK 1.6 之后得到了改进，其优化操作主要针对自旋阈值，即引入了自适应自旋功能。该功能可以动态调节自旋阈值。例如它会根据上一次自旋获取锁成功的案例所使用的自旋次数来调节下一次的自旋的阈值以提高成功率；又或者经过分析后得出自旋获取锁的成功几率非常低，因此 JVM 有可能会直接关闭自旋操而直接进入阻塞线程。

锁升级

  个人认为锁升级是 JDK 1.6 对 synchronized 监控锁最大的优化项。默认情况下，锁升级分为4个阶段：无锁 -> 偏向锁 -> 轻量级锁 -> 重量级锁(膨胀锁)。其中偏向锁是可以通过 JVM 配置项关闭的（-XX:-UseBiasedLocking）

了解锁升级之前应该先了解对象头

可能你听过或被问过这样一个问题，就是“一个对象实例有多大？”。首先对象实例的大小时不固定的，譬如在默认开启压缩指针（-XX：+UseCompressedOops）的情况下：

  • 引用占4个字节
  • 布尔类型占1个字节
  • short和char占2个字节
  • float和int占4个字节
  • long和dubble占8个字节

以上这些是固定的，但是唯有对象头的大小会根据操作系统的位数而变动，譬如64位操作系统对象头为16个字节（开启压缩指针后为12个字节），而32位操作系统则是8个字节。所以说对象实例是有基本大小的（16/12/8），而实际的大小需要根据实际情况而定。

对象头是对象实例结构的组成部分，它主要由两部分组成：

  • Mark Word：记录了锁标记、锁偏向模式、哈希值、实例GC年龄等信息
  • Class Word：存放方法区入口（Class实例）的地址

其中 Mark Word 是锁升级的关键，或者说它是 synchronized 监控锁的关键，因为 synchronized 锁定的目标是实例（Mark Word）而不是代码片段。synchronized 通过修改 Mark Word 信息来实现线程的持锁状态。

大致了解对象头之后接下来看看 4 种锁的区别

  • 无锁：锁没有被线程持有，这时 Mark Word 值为[ 对象哈希值 | GC年龄 | 0 | 01 ]
  • 偏向锁：
    • 偏向的意思是锁偏心于某条线程，这时 JVM 仅需要判断实例对象头 MarkWord 中的线程标识是否等于当前要获取锁的线程标识（ID值）即可，如果相同则无需进行同步操作，这时的效率相当于无锁状态，这也是为什么无锁和偏向锁的锁标记都是01的原因，因为两者的处理方式基本相同
    • 引入偏向锁的目的是为了消除在没用锁竞争时的同步互斥量操作
    • 当偏向锁出现两条及以上的线程竞争时，持锁线程会先撤销锁偏向，然后锁会升级为轻量级锁
    • 偏向锁 Mark Word 值为 [ JavaTheadId | epoch | GC年龄| 0 | 01 ]
  • 轻量级锁：
    • 轻量级锁和偏向锁很像，两者都是为了消除在没有锁竞争时的同步互斥量操作，而区别在于轻量级锁采用的是CAS，而偏向锁则只需判断线程标识
    • 当轻量级锁出现两条及以上的线程竞争时，锁会升级为重量级锁，而这个过程成为锁膨胀，因此重量级锁又叫做膨胀锁
    • 轻量级锁 Mark Word 值为 [ ptr | 00 ]，其中 ptr 是持锁线程栈的 BasicLock 指针（BasicLock是JVM内部创建的锁，用于实现轻量级锁）
  • 重量级锁：
    • 当锁处于轻量级时，如果有线程自旋次数到达阈值后依然没发获取到这把锁的话，锁就会进入膨胀阶段晋升为重量级锁
    • 重量级锁在介绍 synchronized 时已经说过了，其实就是 JDK 1.5 的模样，既线程一旦竞争锁失败就会被内核挂起
    • 重量级锁 Mark Word 值为 [ ptr | 10 ]

void ObjectSynchronizer::fast_enter(Handle obj, BasicLock* lock, bool attempt_rebias, TRAPS) {

 // 如果正在使用偏向锁，则先撤销偏向，如何将锁升级为轻量级锁（slow_enter）
 if (UseBiasedLocking) {

    // 不在安全点
    if (!SafepointSynchronize::is_at_safepoint()) {
      // 撤销后再偏向
      BiasedLocking::Condition cond = BiasedLocking::revoke_and_rebias(obj, attempt_rebias, THREAD);
      if (cond == BiasedLocking::BIAS_REVOKED_AND_REBIASED) {
        return;
      }

    // 在安全点，撤销偏向
    } else {
      assert(!attempt_rebias, "can not rebias toward VM thread");
      BiasedLocking::revoke_at_safepoint(obj);
    }

    assert(!obj->mark()->has_bias_pattern(), "biases should be revoked by now");
 }

 // 偏向锁已经撤销，锁操作升为轻量级
 slow_enter (obj, lock, THREAD) ;
}

轻量级锁

// BasicLock是线程栈LockRecord段中的组成部分
void ObjectSynchronizer::slow_enter(Handle obj, BasicLock* lock, TRAPS) {

  // 获取锁对象的对象头
  markOop mark = obj->mark();

  // 该对象头没有被锁定
  if (mark->is_neutral()) {

    // 将锁对象头信息转移到线程栈LockRecord段的displaced_header中
    lock->set_displaced_header(mark);

    // 使用CAS尝试获取轻量级锁
    // 尝试将当前线程栈LockRecord段地的指针内容替换为锁对象头的指向
    if (mark == (markOop) Atomic::cmpxchg_ptr(lock, obj()->mark_addr(), mark)) {
      TEVENT (slow_enter: release stacklock) ;
      return ; // 加锁成功
    }

  // 当前轻量级锁存在锁竞争

  // 当前操作的线程为持锁线程，进行重入
  } else if (mark->has_locker() && THREAD->is_lock_owned((address)mark->locker())) {
    assert(lock != mark->locker(), "must not re-lock the same lock");
    assert(lock != (BasicLock*)obj->mark(), "don't relock with same BasicLock");
    // 可以看到轻量级锁的重入操作只是置空了displaced_header
    //因为displaced_header只在加锁时起暂存作用，方式加锁失败
    lock->set_displaced_header(NULL); 
    return;
  }

  // 当前线程并非持锁线程，即这时出现了两条及以上的线程进行锁竞争

  // 重置线程栈BasicLock
  lock->set_displaced_header(markOopDesc::unused_mark());
  // 进入锁膨胀阶段
  ObjectSynchronizer::inflate(THREAD, obj())->enter(THREAD);
}

重量级锁（轻量级锁的膨胀过程）

// 进入监控锁
void ATTR ObjectMonitor::enter(TRAPS) {

  // 获取当前操作线程的指针
  Thread * const Self = THREAD ;
  void * cur ;

  // &_owner为持锁线程的地址
  // 尝试使用CAS操作将当前线程变为持锁线程，成功则表示加锁成功
  // 返回持锁线程的指针
  cur = Atomic::cmpxchg_ptr (Self, &_owner, NULL) ;

  // 获取失败（可能锁正在升级）
  if (cur == NULL) {
     // Either ASSERT _recursions == 0 or explicitly set _recursions = 0.
     assert (_recursions == 0   , "invariant") ;
     assert (_owner == Self, "invariant") ;
     // CONSIDER: set or assert OwnerIsThread == 1
     return ;
  }

  // 获取成功，且当前线程为“上一任持锁线程”，进行锁重入计数
  if (cur == Self) {
     _recursions ++ ;
     return ;
  }

  // 获取成功，但当前线程是第一次获取到该锁
  if (Self->is_lock_owned ((address)cur)) {
    _recursions = 1 ;
    _owner = Self ; 
    OwnerIsThread = 1 ;
    return ;
  }

  // 获取失败，因为有其他线程持有该锁，意味需要进行锁竞争
  assert (Self->_Stalled == 0, "invariant") ;
  Self->_Stalled = intptr_t(this) ;

  // 线程进入队列之前，先尝试一轮自旋，若成果直接返回
  // 注意：TrySpin操作是“自适应自旋”，既它会自动调节自旋次数阈值
  if (Knob_SpinEarly && TrySpin (Self) > 0) { // mutex.cpp
     assert (_owner == Self      , "invariant") ;
     assert (_recursions == 0    , "invariant") ;
     assert (((oop)(object()))->mark() == markOopDesc::encode(this), "invariant") ;
     Self->_Stalled = 0 ;
     return ;
  }

  assert (_owner != Self          , "invariant") ;
  assert (_succ  != Self          , "invariant") ;
  assert (Self->is_Java_thread()  , "invariant") ;

  // 获取当前Java线程的指针
  JavaThread * jt = (JavaThread *) Self ;

  assert (!SafepointSynchronize::is_at_safepoint(), "invariant") ; // 不在安全点
  assert (jt->thread_state() != _thread_blocked   , "invariant") ; // 没有阻塞
  assert (this->object() != NULL  , "invariant") ; 
  assert (_count >= 0, "invariant") ;

  // Prevent deflation at STW-time.  See deflate_idle_monitors() and is_busy().
  // Ensure the object-monitor relationship remains stable while there's contention.
  Atomic::inc_ptr(&_count);

  EventJavaMonitorEnter event;

  { 
    // 将当前Java线程的状态修改为Blocked
    JavaThreadBlockedOnMonitorEnterState jtbmes(jt, this);
    // 切换为内核线程
    OSThreadContendState osts(Self->osthread());
    ThreadBlockInVM tbivm(jt);

    Self->set_current_pending_monitor(this);

    // 重量级锁竞争
    // TODO-FIXME: change the following for(;;) loop to straight-line code.
    for (;;) {
      jt->set_suspend_equivalent();

      // 重量级锁竞争逻辑
      EnterI (THREAD) ;

      if (!ExitSuspendEquivalent(jt)) break ;

      _recursions = 0 ;
      _succ = NULL ;
      exit (false, Self) ;

      jt->java_suspend_self();
    }
    Self->set_current_pending_monitor(NULL);
  }

  Atomic::dec_ptr(&_count);
  assert (_count >= 0, "invariant") ;
  Self->_Stalled = 0 ;


  // ...

  if (ObjectMonitor::_sync_ContendedLockAttempts != NULL) {
     ObjectMonitor::_sync_ContendedLockAttempts->inc() ;
  }
}

重量级锁竞争

void ATTR ObjectMonitor::EnterI (TRAPS) {
    Thread * Self = THREAD ;
    assert (Self->is_Java_thread(), "invariant") ;
    assert (((JavaThread *) Self)->thread_state() == _thread_blocked   , "invariant") ;

    // 尝试获取锁
    if (TryLock (Self) > 0) { // 返回值大于1表示尝试成功
        assert (_succ != Self              , "invariant") ;
        assert (_owner == Self             , "invariant") ;
        assert (_Responsible != Self       , "invariant") ;
        return ;
    }

    DeferredInitialize () ;

    // 乐观地再尝试一次“自适应自旋”
    if (TrySpin (Self) > 0) {
        assert (_owner == Self        , "invariant") ;
        assert (_succ != Self         , "invariant") ;
        assert (_Responsible != Self  , "invariant") ;
        return ;
    }

    // 重量级锁自旋失败，线程将进入同步队列（实质是链表）进行阻塞操作
    assert (_succ  != Self            , "invariant") ;
    assert (_owner != Self            , "invariant") ;
    assert (_Responsible != Self      , "invariant") ;

    // Enqueue "Self" on ObjectMonitor's _cxq.
    //
    // Node acts as a proxy for Self.
    // As an aside, if were to ever rewrite the synchronization code mostly
    // in Java, WaitNodes, ObjectMonitors, and Events would become 1st-class
    // Java objects.  This would avoid awkward lifecycle and liveness issues,
    // as well as eliminate a subset of ABA issues.
    // TODO: eliminate ObjectWaiter and enqueue either Threads or Events.
    //
    // 返回线程的代理节点
    // 该同步队列由链表结构实现
    ObjectWaiter node(Self) ;
    Self->_ParkEvent->reset() ;
    node._prev   = (ObjectWaiter *) 0xBAD ; // 当前(线程)节点的前驱节点
    node.TState  = ObjectWaiter::TS_CXQ ; // 当前(线程)节点的状态

    // Push "Self" onto the front of the _cxq.
    // Once on cxq/EntryList, Self stays on-queue until it acquires the lock.
    // Note that spinning tends to reduce the rate at which threads
    // enqueue and dequeue on EntryList|cxq.
    ObjectWaiter * nxt ;
    // 先将当前竞争锁线程的代理节点加入到同步队列头
    for (;;) {
        // 头插法
        node._next = nxt = _cxq ; // 一开始_cxq为null
        // 替换原来的同步队列，成功后打断循环
        if (Atomic::cmpxchg_ptr (&node, &_cxq, nxt) == nxt) break ;
        // 在CAS替换队列失败时也不忙尝试获取锁
        if (TryLock (Self) > 0) {
            assert (_succ != Self         , "invariant") ;
            assert (_owner == Self        , "invariant") ;
            assert (_Responsible != Self  , "invariant") ;
            return ;
        }
    }

    // 加入同步队列后，尝试申请成为“责任线程”
    // 在同步队列中只有一条责任线程
    // 责任线程的特点是会被定时唤醒，主要用于轮询持锁线程_owner是否已经释放锁
    // 主要申请条件是当前线程（节点）没有子节点，且_EntryList中没有正在获取尝试获取锁的线程
    // 所以因为头插法的缘故第一条进入同步队列的线程会率先作为责任线程
    if ((SyncFlags & 16) == 0 && nxt == NULL && _EntryList == NULL) {
        // Try to assume the role of responsible thread for the monitor.
        // CONSIDER:  ST vs CAS vs { if (Responsible==null) Responsible=Self }
        Atomic::cmpxchg_ptr (Self, &_Responsible, NULL) ;
    }

    TEVENT (Inflated enter - Contention) ;
    int nWakeups = 0 ;
    int RecheckInterval = 1 ;

    for (;;) {
        // 再次尝试获取锁
        if (TryLock (Self) > 0) break ;
        assert (_owner != Self, "invariant") ;

        // 获取锁失败且当前没有责任检测，则尝试申请成为责任线程
        if ((SyncFlags & 2) && _Responsible == NULL) {
           Atomic::cmpxchg_ptr (Self, &_Responsible, NULL) ;
        }

        // 注意：以下就是重量级锁“重”的地方，线程将会阻塞，操作由系统调度

        // 如果当前线程是“责任线程”，则进行超时阻塞
        if (_Responsible == Self || (SyncFlags & 1)) {
            TEVENT (Inflated enter - park TIMED) ;
            Self->_ParkEvent->park ((jlong) RecheckInterval) ;
            // Increase the RecheckInterval, but clamp the value.
            RecheckInterval *= 8 ;
            if (RecheckInterval > 1000) RecheckInterval = 1000 ;
        // 如果当前线程不是“责任线程”，则一直阻塞直到被唤醒
        } else {
            TEVENT (Inflated enter - park UNTIMED) ;
            Self->_ParkEvent->park() ;
        }

        // 被唤醒后先尝试一次获取锁，成功直接返回
        if (TryLock(Self) > 0) break ;

        // 尝试获取锁失败，即表示锁已经被其他线程持有
        // 因此记录无用的唤醒次数
        TEVENT (Inflated enter - Futile wakeup) ;
        if (ObjectMonitor::_sync_FutileWakeups != NULL) {
           ObjectMonitor::_sync_FutileWakeups->inc() ;
        }
        ++ nWakeups ;

        // 如果不是虚假唤醒，通常会得到 _succ == Self，则尝试进行“自适应自旋”
        if ((Knob_SpinAfterFutile & 1) && TrySpin (Self) > 0) break ;

        if ((Knob_ResetEvent & 1) && Self->_ParkEvent->fired()) {
           Self->_ParkEvent->reset() ;
           OrderAccess::fence() ;
        }

        if (_succ == Self) _succ = NULL ;

        // Invariant: after clearing _succ a thread *must* retry _owner before parking.
        OrderAccess::fence() ;
    }

    assert (_owner == Self      , "invariant") ;
    assert (object() != NULL    , "invariant") ;

    UnlinkAfterAcquire (Self, &node) ;
    if (_succ == Self) _succ = NULL ;

    assert (_succ != Self, "invariant") ;
    if (_Responsible == Self) {
        _Responsible = NULL ;
        OrderAccess::fence(); // Dekker pivot-point
    }

    if (SyncFlags & 8) {
       OrderAccess::fence() ;
    }
    return ;
}

释放锁

// 释放监控锁
void ATTR ObjectMonitor::exit(bool not_suspended, TRAPS) {

   // 获取当前释放锁线程的指针
   Thread * Self = THREAD ;

   if (THREAD != _owner) { 
     if (THREAD->is_lock_owned((address) _owner)) {
       assert (_recursions == 0, "invariant") ;
       _owner = THREAD ;
       _recursions = 0 ;
       OwnerIsThread = 1 ;
     } else {
       TEVENT (Exit - Throw IMSX) ;
       assert(false, "Non-balanced monitor enter/exit!");
       if (false) {
          THROW(vmSymbols::java_lang_IllegalMonitorStateException());
       }
       return;
     }
   }

   // 释放锁重入标记
   // 注意：如果锁重入了，调用一次 ObjectMonitor::exit 并不会释放锁，必须递减到0
   if (_recursions != 0) {
     _recursions--;        // this is simple recursive enter
     TEVENT (Inflated exit - recursive) ;
     return ; // 递减后退出方法返回
   }
   if ((SyncFlags & 4) == 0) {
      _Responsible = NULL ;
   }

   for (;;) {

      // 虽然 _recursions = 0 但线程还持有锁标识
      assert (THREAD == _owner, "invariant") 

      if (Knob_ExitPolicy == 0) { 
         OrderAccess::release_store_ptr (&_owner, NULL) ;   // drop the lock
         OrderAccess::storeload() ;                         // See if we need to wake a successor
         if ((intptr_t(_EntryList)|intptr_t(_cxq)) == 0 || _succ != NULL) {
            TEVENT (Inflated exit - simple egress) ;
            return ;
         }
         TEVENT (Inflated exit - complex egress) ;
         if (Atomic::cmpxchg_ptr (THREAD, &_owner, NULL) != NULL) {
            return ;
         }
         TEVENT (Exit - Reacquired) ;
      } else {

         if ((intptr_t(_EntryList)|intptr_t(_cxq)) == 0 || _succ != NULL) {
            OrderAccess::release_store_ptr (&_owner, NULL) ;   // drop the lock
            OrderAccess::storeload() ;

            // Ratify the previously observed values.
            if (_cxq == NULL || _succ != NULL) {
                TEVENT (Inflated exit - simple egress) ;
                return ;
            }

            if (Atomic::cmpxchg_ptr (THREAD, &_owner, NULL) != NULL) {
               TEVENT (Inflated exit - reacquired succeeded) ;
               return ;
            }
            TEVENT (Inflated exit - reacquired failed) ;
         } else {
            TEVENT (Inflated exit - complex egress) ;
         }
      }

      guarantee (_owner == THREAD, "invariant") ;

      ObjectWaiter * w = NULL ;
      int QMode = Knob_QMode ; // Knob_QMode 默认为 0

      if (QMode == 2 && _cxq != NULL) {
          w = _cxq ;
          assert (w != NULL, "invariant") ;
          assert (w->TState == ObjectWaiter::TS_CXQ, "Invariant") ;
          ExitEpilog (Self, w) ;
          return ;
      }

      if (QMode == 3 && _cxq != NULL) {
          w = _cxq ;
          for (;;) {
             assert (w != NULL, "Invariant") ;
             ObjectWaiter * u = (ObjectWaiter *) Atomic::cmpxchg_ptr (NULL, &_cxq, w) ;
             if (u == w) break ;
             w = u ;
          }
          assert (w != NULL              , "invariant") ;

          ObjectWaiter * q = NULL ;
          ObjectWaiter * p ;
          for (p = w ; p != NULL ; p = p->_next) {
              guarantee (p->TState == ObjectWaiter::TS_CXQ, "Invariant") ;
              p->TState = ObjectWaiter::TS_ENTER ;
              p->_prev = q ;
              q = p ;
          }

          ObjectWaiter * Tail ;
          for (Tail = _EntryList ; Tail != NULL && Tail->_next != NULL ; Tail = Tail->_next) ;
          if (Tail == NULL) {
              _EntryList = w ;
          } else {
              Tail->_next = w ;
              w->_prev = Tail ;
          }

      }

      if (QMode == 4 && _cxq != NULL) {
          w = _cxq ;
          for (;;) {
             assert (w != NULL, "Invariant") ;
             ObjectWaiter * u = (ObjectWaiter *) Atomic::cmpxchg_ptr (NULL, &_cxq, w) ;
             if (u == w) break ;
             w = u ;
          }
          assert (w != NULL              , "invariant") ;

          ObjectWaiter * q = NULL ;
          ObjectWaiter * p ;
          for (p = w ; p != NULL ; p = p->_next) {
              guarantee (p->TState == ObjectWaiter::TS_CXQ, "Invariant") ;
              p->TState = ObjectWaiter::TS_ENTER ;
              p->_prev = q ;
              q = p ;
          }

          // Prepend the RATs to the EntryList
          if (_EntryList != NULL) {
              q->_next = _EntryList ;
              _EntryList->_prev = q ;
          }
          _EntryList = w ;

          // Fall thru into code that tries to wake a successor from EntryList
      }

      w = _EntryList  ;
      if (w != NULL) {
          assert (w->TState == ObjectWaiter::TS_ENTER, "invariant") ;
          ExitEpilog (Self, w) ;
          return ;
      }

      // 将同步队列转移给w
      w = _cxq ; 
      if (w == NULL) continue ;

      // 以下操作相当于置空_cxq
      for (;;) {
          assert (w != NULL, "Invariant") ;
          ObjectWaiter * u = (ObjectWaiter *) Atomic::cmpxchg_ptr (NULL, &_cxq, w) ;
          if (u == w) break ;
          w = u ;
      }
      TEVENT (Inflated exit - drain cxq into EntryList) ;

      assert (w != NULL              , "invariant") ;
      assert (_EntryList  == NULL    , "invariant") ;

      if (QMode == 1) {
         // QMode == 1 : drain cxq to EntryList, reversing order
         // We also reverse the order of the list.
         ObjectWaiter * s = NULL ;
         ObjectWaiter * t = w ;
         ObjectWaiter * u = NULL ;
         while (t != NULL) {
             guarantee (t->TState == ObjectWaiter::TS_CXQ, "invariant") ;
             t->TState = ObjectWaiter::TS_ENTER ;
             u = t->_next ;
             t->_prev = u ;
             t->_next = s ;
             s = t;
             t = u ;
         }
         _EntryList  = s ;
         assert (s != NULL, "invariant") ;


      } else {
         // QMode == 0 or QMode == 2
         _EntryList = w ; // w = _cxq
         ObjectWaiter * q = NULL ;
         ObjectWaiter * p ;
         for (p = w ; p != NULL ; p = p->_next) {
             guarantee (p->TState == ObjectWaiter::TS_CXQ, "Invariant") ;
             p->TState = ObjectWaiter::TS_ENTER ; 
             p->_prev = q ;
             q = p ;
         }
      }

      if (_succ != NULL) continue;

      w = _EntryList  ;
      if (w != NULL) {
          guarantee (w->TState == ObjectWaiter::TS_ENTER, "invariant") ;
          ExitEpilog (Self, w) ; // 唤醒 _EntryList 队列中头一条线程
          return ;
      }
   }
}

唤醒线程

// 释放监控锁时唤醒线程
void ObjectMonitor::ExitEpilog (Thread * Self, ObjectWaiter * Wakee) {
   assert (_owner == Self, "invariant") ;

   // Exit protocol:
   // 1. ST _succ = wakee
   // 2. membar #loadstore|#storestore;
   // 2. ST _owner = NULL
   // 3. unpark(wakee)

   _succ = Knob_SuccEnabled ? Wakee->_thread : NULL ;
   ParkEvent * Trigger = Wakee->_event ;
   Wakee  = NULL ;

   // 解除锁关联
   OrderAccess::release_store_ptr (&_owner, NULL) ;
   OrderAccess::fence() ;  // ST _owner vs LD in unpark()

   // 代码执行安全点
   if (SafepointSynchronize::do_call_back()) {
      TEVENT (unpark before SAFEPOINT) ;
   }

   // 唤醒队列（_EntryList）中的头部线程
   Trigger->unpark() ; 

}

抽象队列同步器（AQS）

简介

  在 JDK 1.5 版本中除了新增了 Instrumentation 这个有趣的 API 外还加入了大神道格利（Doug Lea） 编写的 JSR-166（一整个 juc 包）。该包中除了 CAS 操作之外最为重要的就是 AbstractQueuedSynchronizer。准确点来说 AQS 其实并不是和 synchronized 对等的，因为 AQS 只负责提供同步方案(主要围绕着关键字段state进行操作)，而 Lock 只是基于 AQS 的其中一个体现。譬如在 juc 包中除了 Lock 之外还有诸如 CountDownLatch 、Semaphore 这样的同步工具类。

  因为当前文章主要讨论的是锁，所以接下来会通过 Lock 源码来理解它是如何使用 AQS 实现锁功能的。但在此之前需要了解一些事，就是 juc 包下的工具类基本上很少使用 synchronized 来实现同步操作，而大多数使用的是 CAS（比较和交换），CAS 本质上来说其实就是乐观锁的操作，只不过它是通过一条 CPU 原子指令来实现的。而在 Java 中有一个较为低级别的类叫 Unsafe ，CAS 操作均由该类实现，且除了 CAS 操作之外 Unsafe 还提供线程阻塞和唤醒等功能，这些操作在 juc 中都被大量使用。

Lock和Synchronized对比

区别

性能差别

自 JDK 1.6 对 synchronized 进行优化过后，其实两者在性能方面差别并不大，所以个人认为无需多虑。这点可以在 JDK 1.8 的 ConcurrentHashMap 中体会到。譬如说 ConcurrentHashMap 使用 CAS + synchronized 组合取代了原来的段锁，而没有用 CAS + Lock 这个组合。当然，很大原因是因为 Lock 本身也是 CAS 实现的，所以问题就在于如果 ConcurrentHashMap 光使用 CAS 来实现同步的话，那么在多线程高并发环境就会导致严重的 CPU 消耗，因此我们可以简单地得出两个结论：

  1. synchronized 性能不差
  2. synchronized 可以一定程度地通过消耗时间来换取 CPU 资源

如何选择

  • Lock 是纯粹的自旋锁，通过自旋操作避免了不必要的用户态和内核态的切换开销。但自旋本身对 CPU 消耗较大，所以在多线程高并发场景如果存在大量自旋操作的话，很容引发 CPU 使用率急速上升，甚至可能引起卡顿现象
  • 虽然 synchronized 也存在自旋操作，但自旋次数一般很少，且受自适应机制影响，因此 CPU 消耗并没有 Lock 高。所以，如果是多线程高并发场景下且线程执行任务所需的时间较长，则建议使用 synchronized 而不是 Lock
  • Lock 灵活性较高且粒度也细，所以如果有特别需要，譬如读多写少的场景则建议选择 Lock 的读写锁来实现同步，因为这样可以获得更高的并发性能
