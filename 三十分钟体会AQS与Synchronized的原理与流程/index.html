Synchronized原理

简介

  synchronized 是 Java 中相对于 AQS 较早的同步方式，原理是通过 互斥指令 实现的同步。synchronized 操作其实对应着两个互斥指令，分别是代表加锁操作的 lock 指令和代表解锁操作的 unlock 指令，但需要注意的是这两个指令在 JVM 字节码中有另外的表述方式，例如用 monitorenter 代表 lock 指令，用 monitorexit 代表 unlock 指令。

  在 JDK 1.6 之前 synchronized 被称为重量级锁。它重的表现在于一旦竞争锁失败线程将会被挂起，当锁被持有者释放时又会被唤醒，而这两个操作均由内核调度来完成。因为涉及到内核调度，所以需要从用户态切换到内核态才能做到，在这期间用户程序相当于处在呆滞状态。在人看来这段时间虽然非常短暂，但是如果发生大量用户态切换到内核态操作，无疑消耗会积小成多从而间接导致用户程序请求处理能力降低，而且线程挂起这个动作并不会释放线程持有的资源，这就是 synchronized 重的地方。但从 JDK 1.6 开始 synchronized 有了较大的改进，例如引入了多项锁优化操作，譬如 锁消除、锁粗化、锁升级 、自旋锁 等等。而我们需要重点明白的就是，这些优化操作的目标全都是为了减少(用户态切换到内核态)线程挂起唤醒而做的。

锁优化

锁消除

  指的是经分析得知虽然代码片段加了 synchronized 同步但是真实运行情况并不会发生并发操作，这时去掉 synchronized 的过程就称为锁消除。例如：

public String concatString(String s1, String s2, String s3) {
    return s1 + s2 + s3;
}

以上操作在 JDK 1.6 之前会在(javac)前端编译优化为 StringBuffer ，而在 JDK 1.6 开始被优化为 StringBuilder。两者的区别在于前者操作是同步的，其方法都被 synchronized 修饰，而后者不是。

锁粗化

  锁粗化和锁消除类似，指的是经过分析后发现连续的代码片段中发生重复加锁操作后将 synchronized 提升范围的过程。例如：

StringBuffer sf = new StringBuffer();
sf.append(s1);
sf.append(s2);
sf.append(s3);

上面已经提到过，StringBuilder 的方法是同步的，这时会提升 synchronized 代码块范围而不会为每个append方法加锁，伪代码如下

StringBuffer sf = new StringBuffer();
synchronized (sf) {
    sf.append(s1);
    sf.append(s2);
    sf.append(s3);
}

自旋锁

  自旋通俗点说其实就是循环的意思，既循环尝试获取锁的操作就叫自旋锁。该功能其实在 JDK 1.4 就已经引入了。其主要的目的就是为了降低线程被内核挂起的几率。但这种理论是建立在持锁线程在获取到锁后会在短时间内释放锁资源的基础上说才行得通的。换句话来说，就是如果线程执行任务所需要的时间较长，就会导致其它申请锁的线程不停地自旋从而导致 CPU 使用率极速上升，同样会影响用户程序性能，这时反而没有阻塞高效。因此通常自旋操作都是有个次数阈值的，既自旋到一定次数后线程还是没能获取到锁资源的话就会挂起。在 JVM 中提供了两个配置项来设置自旋锁，分别是自旋开关-XX:+UseSpinning （默认开启）和阈值设置 -XX:PreBlockSpin （默认10次）。

  自旋锁在 JDK 1.6 之后得到了改进。优化操作主要针对自旋阈值，为其引入了自适应自旋功能。该功能可以动态调节自旋阈值。例如根据上一次成功获取到锁的自旋次数作为参考值来设置阈值。又或者分析得出自旋成功获得锁的可能性较低时会直接关闭自旋操从而直接进入阻塞操作。

锁升级

  锁升级是 JDK 1.6 对 synchronized 最大的优化。默认情况下锁升级分为4个阶段：无锁 -> 偏向锁 -> 轻量级锁 -> 重量级锁(膨胀锁)。其中偏向锁是可以通过 JVM 参数关闭的(-XX:-UseBiasedLocking)。

了解锁升级之前应该先了解对象头

对象头是对象实例结构的组成部分，主要由两部分组成：

  • Mark Word：记录 锁标记 、锁偏向模式、实例GC年龄
  • Class Word：存放指向方法区对象类型实例的指针

其中 Mark Word 是锁升级的关键，或者说它是 synchronized 同步锁的关键。因为 synchronized 锁定的是实例 ，准确一些它锁定的是实例对象头的 Mark Word 。synchronized 就是通过修改 Mark Word 信息来实现锁分配的。如下为对象头的基本结构：

大致了解对象头之后接下来看看 4 种锁的区别

  • 无锁：锁没有被线程持有，这时 Mark Word 值为[ 对象哈希值 | GC年龄 | 0 | 01 ]
  • 偏向锁：
    • 偏向的意思是锁偏心于某条线程，这时 JVM 仅需要判断 MarkWord 中的线程标识是否等于当前要获取锁的线程标识（ID值）即可，如果相同则无需进行同步操作，这时效率相当于无锁状态（可以看到无锁和偏向锁的锁标记是一样的，都是01，既 JVM 对两者的处理方式大致相同）
    • 偏向锁的目的是消除在没用锁竞争的情况下的同步互斥量
    • 当偏向锁出现锁竞争时，就会升级为轻量级锁
    • 偏向锁 Mark Word 值为 [ JavaTheadId | epoch | GC年龄| 0 | 01 ]
  • 轻量级锁：
    • 轻量级锁和偏向锁较为类似，两者都是为了消除在没有锁竞争时的同步互斥量的。区别之处是轻量级锁采用的是 CAS自旋，而偏向锁则只需判断线程标识即可
    • 当轻量级锁出现锁竞争时，就会升级为重量级锁。该过程称之为锁膨胀，因此重量级锁又叫膨胀锁
    • 轻量级锁 Mark Word 值为 [ ptr | 00 ]，其中 ptr 是持锁线程栈的 Lock Record 指针
  • 重量级锁：
    • 当轻量级锁自旋次数到达阈值后依然没有获取到锁资源时就会膨胀为重量级锁
    • 重量级锁在介绍 synchronized 时已经说过了，其实就是 JDK 1.5 的模样，既当线程竞争锁失败会被挂起
    • 重量级锁 Mark Word 值为 [ ptr | 10 ]

抽象队列同步器（AQS）

简介

  在 JDK 1.5 这个版本中除了新增了 Instrumentation API 之外还加入了大神道格利(Doug Lea) 编写的 JSR-166（一整个 juc 包），该包中除了 CAS 操作之外最为重要的就是 AbstractQueuedSynchronizer。准确点来说 AQS 其实并不是和 Synchronized 对等的，因为 AQS 只负责提供抽象的同步方案(围绕着关键字段state进行操作)，而实现者只需要继承 AQS 并复写其模板方法即可实现特定的同步功能。再通俗点说 Lock 只是 AQS 的一种体现，例如在 juc 包中除了 Lock 之外还有诸如 CountDownLatch 、Semaphore 这样的同步工具类。

  因为当前文章是围绕锁展开的，所以接下来会通过 Lock 源码作为开端对 AQS 进行浅析。但在此之前还需要了解一些事。就是 juc 包下的工具类基本上很少使用 synchronized ，而大多数使用的是 CAS，可以说 CAS 支撑着整个 juc 包也不为过。CAS 中文意思叫比较和交换，其实就是乐观锁操作，只不过它是一条原子 CPU 指令，所以它是天然同步的。在 Java 中有一个较为低级别的类叫 Unsafe ，CAS 操作都是通过该类实现了。 除了CAS 之外 Unsafe 还提供线程阻塞和唤醒等操作，这些操作在 juc 中都被大量使用。

特性区别

性能对比

  • 自从 JDK 1.6 以后对 synchronized 进行了优化以后，其实两则性能上区别不大。网上也有一些人做了些测试报告，但是个人认为只能作为一个参考，因为场景不同的话性能是有所出入的
  • 除此之外，还可以在 JDK 1.8 的 ConcurrentHashMap 中体会到，例如它摒弃了段锁改用 CAS + Synchronized 而不是 Lock ，所以可以间接得出结论就是 Synchronized 的性能其实并不差，因此无需过于纠结两者的性能

如何选择

  • Lock 也被称为自旋锁，它通过自旋操作来避免线程的挂起唤醒操作。而采用自旋需要特别注意的是，在较大的并发场景中如果存在大量自旋操作的话，就会导致 CPU 使用率急速增长从而影响用户程序性能。虽然 synchronized 中也存在自旋操作，但它的自旋次数是有阈值的，而 Lock 并没有，因此 Lock 的自旋消耗要比 synchronized 大，所以如果线程执行任务的时间较长的话建议采用 synchronized 而不是 Lock
  • Lock 灵活性较高和粒度也相对较低，所以如果业务有特别需要，例如读读并行就可以使用Lock的读写锁功能来细化锁的粒度来提高并发性能
