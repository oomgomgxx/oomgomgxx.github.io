关于控制反转和依赖注入

  那么在谈注入方式之前先聊一聊关于控制反转和依赖注入的区别。主要是因为看到网上很多人说它们是同一个东西，而我个人并不这样理解。

以下内容片段摘自网上的某篇文章

可以看到文章作者的理解是依赖注入其实是IOC的另一种说法，除此还提及到控制反转是指获得依赖的方式反转了。那我的质疑就是 IoC 和 DI 其实不是同一个级别的东西，还有就是 IoC 不仅仅体现在依赖注入。还有就是不要因为他提到马丁福勒就彻底相信他了，我们应该时刻保持着怀疑的态度来看待问题。

  那我会先谈谈自己的见解，然后再用源码来证实一下自己的说法。因为口说无凭，事实胜于雄辩。我所理解的 Ioc 其实是一种思想，且并不只是在 Spring 中出现。例如GOF的23种计模式中的观察者模式也运用了控制反转的思想，主要体现在由可观察对象来通知观察者而不是观察者主动轮询可观察对象，既这时后观察者是从主动变为被动。除了编程之外其实生活中也有 IoC 思想的场景，例如著名的 好莱坞原则 就是 IoC 的体现。在 Spring 中，IoC 主要体现在实例维护上，开发者将维护实例职责交由Spring来完成，这就是Spring的 IoC 思想。而需要重点情况出的实例维护到底包含哪些内容？大致有这些：配置管理、实例创建与销毁、实例的依赖查找、实例的依赖注入等等，以上提到的都是 IoC 的职责。所以说IoC 是不等同于依赖注入的。通俗地说，依赖注入仅仅只是 Spring IoC 中的一项功能，如果依赖注入脱离了 IoC 的话，那么只是一个没有”上下文”的普通操作而已。

  除此之外，很多人都只会提到依赖注入而忽略了依赖查找。其实(手动)依赖查找和依赖注入是存在区别的。(手动)依赖查找主要是根据 BeanDefinition 查找实例，而在查找过程中如果发现没有对应的 BeanDefinition 则会抛出 NoSuchBeanDefinitionException 异常。依赖注入包含(自动装配)依赖查找和注入依赖2个动作，且(自动装配)依赖查找还有别与前面的手动依赖查找。例如下面这个案例：

// 手动依赖查找
@Test
public void test() {
    // 会抛出 NoSuchBeanDefinitionException 
    applicationContext.getBean(BeanFactory.class);
}

// 依赖注入（自动装配依赖查找 + 注入依赖）
@Autowired
private BeanFactory beanFactory; // 可以获取到 BeanFactory 实例

得到上面两种结果的原因很简单，上面已经提到手动依赖查找是根据 BeanDefinition 来查找实例的，而 BeanFactory 是 Spring 容器的内置Bean，换句话说它并不是由解析配置得到的，所以容器中不存在与之对应的 BeanDefinition，因此手动依赖查找方式会抛出 NoSuchBeanDefinitionException 异常。

多说一句，除了 BeanFactory 之外，ResourceLoader、ApplicationEventPublisher、ApplicationContext 等都是内置 Bean（可以查看AbstractApplicationContext#prepareBeanFactory方法）。

依赖注入 和 依赖查找 的基本流程图

最后可以得出这样的结论：

  • 控制反转包含 依赖注入 和 依赖查找
  • 控制反转 != 依赖注入 != 依赖查找

Spring支持的几种依赖注入方式

简介

  在Spring中支持 4 种依赖注入方式，分别是：构造器注入、字段注入、方法注入(包含Setter方法)、接口方法回调注入。在网上看到有很多人误以为方法注入和Setter注入是区分开的，但实时并非如此，如果有看上一章内容的话，在流程图中可以看到，@Autowired注解处理过程中其实只区分是 Field 和 Method，而不会区分 普通Method 还是 Setter Method，这点可以看看以下源码：

而上面提到的 4 种依赖注入方式又可以按照依赖注入时机划分为 3 大类型：

第1类：构造器注入(@Bean标注的实例工厂方法类似)。依赖会在创建实例时被注入

第2类：方法注入。依赖会在创建实例后，填充属性时被注入

第3类：接口方法回调注入。依赖会在创建实例和填充属性后的初始化阶段注入

注入方式详解

构造器注入

基本介绍

构造器注入指的是通过重载构造方法来实现注入配置，使用构造器注入无需额外添加注解配置。例子如下：

@Configuration
public class TestConfig {

    private PersonRepo personRepo;

    public TestConfig(PersonRepo personRepo) {
        this.personRepo = this.getPersonRepo();
    }
}

但需要重点注意的是，如果在实例化过程中容器检测到实例的构造器存在歧义则会调用默认构造方法来完成实例化，而如果这时没有提供默认构造方法的话就会抛出没有找到方法异常。那么什么叫存在歧义呢，例子如下：

@Configuration
public class TestConfig {

    private PersonRepo personRepo;
    private TransactionTemplate transactionTemplate;

    public TestConfig() {
    }

    public TestConfig(PersonRepo personRepo) {
        this.personRepo = this.getPersonRepo();
    }

    public TestConfig(PersonRepo personRepo, TransactionTemplate transactionTemplate) {
        this.personRepo = this.getPersonRepo();
        this.transactionTemplate = transactionTemplate;
    }
}

上面存在两个重载的构造方法，因此在实例化时 Spring 是不知道该选用哪个的，因此在这种情况下就会选用默认构造方法来完成实例化过程，而如果这时没有提供默认构造器就会抛异常。

源码分析

源码位置：AbstractAutowireCapableBeanFactory#createBeanInstance

protected BeanWrapper createBeanInstance(String beanName, RootBeanDefinition mbd, @Nullable Object[] args) {
    // Make sure bean class is actually resolved at this point.
    Class<?> beanClass = resolveBeanClass(mbd, beanName);

    // Lambda 方法创建。例如Spring5后支持下面这种注册Bean的操作
    // context.registerBean("mySecondService", MyService.class, () -> new MyService());
    Supplier<?> instanceSupplier = mbd.getInstanceSupplier();
    if (instanceSupplier != null) {
        return obtainFromSupplier(instanceSupplier, beanName);
    }

    // 实例工厂方法创建
    if (mbd.getFactoryMethodName() != null) {
        return instantiateUsingFactoryMethod(beanName, mbd, args);
    }

    // 构造器创建
    if (resolved) {
        if (autowireNecessary) {
            return autowireConstructor(beanName, mbd, null, null);
        }
        else {
            return instantiateBean(beanName, mbd);
        }
    }
    Constructor<?>[] ctors = determineConstructorsFromBeanPostProcessors(beanClass, beanName);
    if (ctors != null || mbd.getResolvedAutowireMode() == AUTOWIRE_CONSTRUCTOR ||
        mbd.hasConstructorArgumentValues() || !ObjectUtils.isEmpty(args)) {
        return autowireConstructor(beanName, mbd, ctors, args);
    }

    // Preferred constructors for default construction?
    ctors = mbd.getPreferredConstructors();
    if (ctors != null) {
        return autowireConstructor(beanName, mbd, ctors, null);
    }

    // No special handling: simply use no-arg constructor.
    return instantiateBean(beanName, mbd);
}

其中实例工厂方法和(重载)构造器这两种创建 Bean 实例的方式的依赖注入时机是一样，都会在创建Bean 后马上注入依赖。要证明这一点其实很简单，我们可以在 Bean 生命周期的属性填充操作之前拦截实例观察，看看是否已经存有依赖就可以了。例子如下：

@Configuration
public class TestConfig {

    private PersonRepo personRepo;
    private TransactionTemplate transactionTemplate;

    public TestConfig(PersonRepo personRepo, TransactionTemplate transactionTemplate) {
        this.personRepo = personRepo;
        this.transactionTemplate = transactionTemplate;
    }
}

方法注入和字段注入

基本介绍

方法注入依赖和字段注入依赖的时机是一样的，都是在 Bean 实例化之后的属性填充操作中完成依赖注入的。从这点可以看出注入依赖的时机相较于构造器注入来说要稍晚一些。而且方法注入和字段注入依赖的是注解，因此需要 InstantiationAwareBeanPostProcessor 类型的注解处理器来完成注入工作。Spring中常见的依赖注入注解处理器有以下几个：

接口方法回调注入

基本介绍

接口方法回调注入主要依赖的是 Spring 提供的 Aware 接口。例如 Spring 的几个内置 Bean 都有相应的的接口，如下：

  • ApplicationContextAware：用于获取内置的 ApplicationContext 实例
  • BeanFactoryAware：用于获取内置的 BeanFactory 实例
  • ApplicationEventPublisherAware：用于获取内置的 ApplicationEventPublisher 实例
  • ResourceLoaderAware：用于获取内置的 ResourceLoaderAware实例

可以看出 Spring 除了自动装配之外，还提供了Aware接口来暴露容器的内置bean。

接口方法回调注入依赖的时机是最晚的，位于 Bean 生命周期的初始化阶段。

将上一次的测试案例修改为接口方法回调注入，如下：

@Configuration
public class TestConfig implements ApplicationContextAware {

    private PersonRepo personRepo;
    private TransactionTemplate transactionTemplate;

    @Override
    public void setApplicationContext(ApplicationContext applicationContext) throws BeansException {
        this.personRepo = applicationContext.getBean(PersonRepo.class);
        this.transactionTemplate = applicationContext.getBean(TransactionTemplate.class);
    }
}

该如何选择注入方式

  • 经过上面的分析我们已经得知几种注入方式的主要区别是依赖注入的时机不同
  • 构造器注入依赖的时机最早，它能够确保在Bean实例化之后就马上注入依赖。一般如果依赖注入后是不可变的，就推荐使用构造器注入。而且据了解旧版本的 Spring 官方文档中是推荐使用构造器注入的（新版本没留意到）。除此之外使用构造器注入还可以提前暴露循环依赖问题（下一章会谈到）。但构造器注入的缺点是如果要注入多个依赖的话那么构造方法就会变得复杂，易读性较差。
  • 方法注入和字段注入这两种方式注入依赖的时机是一样的。如果依赖注入后还存在重新注入的情况那么推荐使用方法注入。
  • 接口方法回调注入的依赖注入时机是最晚的。需要手动进行依赖查找用得较少。
  • 总的来说如果没有特别场景的话使用哪种注入方式都是一样的，而我个人更倾向字段注入，主要是因为方便和易读性较好，特别是使用Lombok省略手写Setter、Getter方法之后。

如何解决构造器注入的循环依赖问题

基本介绍

  循环依赖指的是：A -> B -> A。如果出现这种情况而又没有做任何处理，则会在实例化时出现死循环。在 Spring 中该问题是通过提前暴露信息来中断死循环的。通俗点说就是在创建 Bean 时将创建标识暴露到创建中对象池，那么其它对象实例化时依赖该实例的话就可以在创建中对象池里得知这个依赖处于正在创建状态，如此一来就可以中断死循环了。例如Spring发生死循环后会抛出 BeanCurrentlyInCreationException 异常。

  这里需要强调一下，关于循环依赖出现死循环一般只出现在使用构造器注入时，因为构造器注入依赖的时机最早，它要求创建Bean后就要马上注入依赖，而这时的依赖可能还没完成创建，因此大大增加了死循环的概率，因此这种情况下只能抛出 BeanCurrentlyInCreationException 异常。

那么如何解决循环依赖问题呢？

  • 使用方法注入或字段注入。这两种方式会提前暴露一个 ObjectFactory 实例来提供给其它实例进行依赖操作。ObjectFactory 和 FactoryBean 即可一样有懒加载效果，只有调用 getObject 方法后才会返回真正实例化
  • 使用接口方法回调注入
  • 构造器注入时，采用 @Lazy 注解来延迟依赖使用
