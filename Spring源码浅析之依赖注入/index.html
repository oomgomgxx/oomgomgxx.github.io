关于控制反转和依赖注入

什么是控制反转和依赖注入？

谈注入方式之前我们先聊聊关于控制反转（IOC）和依赖注入（DI）的区别。主要是因为在网上看到很多人说两者是同一个东西，而个人并不这样理解。以下内容片段摘自网上的某篇文章

可以看到文中作者的理解是依赖注入其实是IOC的另一种说法。而个人其实并不是这样理解，因为 IoC 和 DI 其实并不是同级别的概念，譬如 IoC 是包含 DI 的，且不仅仅有 DI。

  那先聊聊到底什么是控制反转？很多人聊控制反转都拿 Spring 容器举例，而其实用到控制反转思想的地方不仅仅如此。譬如设计模式中的观察者模式、著名的好莱坞原则都是控制反转的体现。而在 Spring 中，控制反转主要体现在上下文管理上，开发者不仅可以将 Bean 的维护工作交给容器，还能使用容器来进行资源管理、依赖查找、依赖注入、事件传播等操作，所以“依赖注入其实是IOC的另一种说法”这种说法不是错误的，而准确点说应该是“依赖注入是控制反转的一种体现”。

但口说无凭，事实胜于雄辩。我所理解的 Ioc 其实是一种思想，且并不只是在 Spring 中出现。例如GOF的23种计模式中的观察者模式也运用了控制反转的思想，主要体现在由可观察对象来通知观察者而不是观察者主动轮询可观察对象，既这时后观察者是从主动变为被动。除了编程之外其实生活中也有 IoC 思想的场景，例如著名的 好莱坞原则 就是 IoC 的体现。在 Spring 中，IoC 主要体现在实例维护上，开发者将维护实例职责交由Spring来完成，这就是Spring的 IoC 思想。而需要重点情况出的实例维护到底包含哪些内容？大致有这些：配置管理、实例创建与销毁、实例的依赖查找、实例的依赖注入等等，以上提到的都是 IoC 的职责。所以说IoC 是不等同于依赖注入的。通俗地说，依赖注入仅仅只是 Spring IoC 中的一项功能，如果依赖注入脱离了 IoC 的话，那么只是一个没有”上下文”的普通操作而已。

  在这再多说一句，依赖注入功能其实是依赖查找功能的超集，可以简单理解依赖注入操作时是隐含着依赖查找的，而且需要重点注意的是，依赖注入中的依赖查找和主动依赖查找并不相同。具体体现在主动依赖查找只会查找有BeanDefinition的Bean，而注入中的依赖查找还可以查找到 Spring 容器内部创建的 bean (譬如BeanFactory)。

上面的手动依赖查找之所以会抛出 NoSuchBeanDefinitionException 是因为 BeanFactory 是 Spring 容器的内部创建的 Bean ，因此在手动依赖查找是并不会找到 BeanFactory 的 BeanDefinition，因此才会抛出这个异常。而除了 BeanFactory 之外，ResourceLoader、ApplicationContext、Environment、ApplicationEventPublisher 等都是内置 Bean（参考AbstractApplicationContext#prepareBeanFactory）。

依赖注入和依赖查找的基本流程

小结：

  • 控制反转包含 依赖注入 和 依赖查找
  • 控制反转 <> 依赖注入 <> 依赖查找

Spring支持的几种依赖注入方式

简介

  Spring 支持 4 种依赖注入方式，分别是构造器注入、字段注入、方法注入(包含Setter方法)、接口方法回调注入。而在网上看到有人将方法注入和Setter注入分开其实是不对的，这点可以在 @Autowired 注解处理器的源码中得到答案。

除了上面按行为来划分注入方式之外，其实也可以按照注入时机来划分：

  • 构造器注入：依赖会在创建实例时被注入

  • 方法注入和字段注入：依赖会在创建实例后的填充属性阶段被被注入（populateBean）

  • 接口方法回调注入:依赖会在创建实例以及填充属性后的初始化阶段被注入（initializeBean）

注入方式详解

构造器注入

基本介绍

构造器注入指的是通过带参构造器或@Bean 注解方法来产生实例并注入依赖的方式。因为构造器注入是 Spring 默认的依赖注入方式，因此在使用构造器注入时我们并不需要添加额外的注解配置

如何选择依赖注入方式？

  • 经过上面的简单分析，我们已经得知到几种依赖注入方式主要的区别在于注入时机
  • 构造器注入
    1. 注入依赖的时机最早，它能够确保其依赖在 Bean 实例化时就注入
    2. 如果依赖在注入后不打算改变，且依赖数量不多的情况下，推荐使用构造器注入，因为这样甚至可以免去setter、getter的实现
    3. Spring 官方推荐，原因是构造器注入可以做到“强制性注入”已提前暴露问题。譬如在使用 Bean 时你不需要再担心依赖是否已经被注入
    4. 但当你需要循环引用时，构造器注入无法实现而会抛异常，因此这是一个选择非构造器注入方式注入依赖
  • 方法（设值）注入和字段注入
    1. 注入依赖的时机在 Bean 实例化之后的填充属性阶段进行
    2. 在一个 Bean 的依赖数量较多及依赖注入后可能会变动时推荐使用
    3. 能够解决循环依赖问题
  • 接口方法回调注入
    • 依赖注入时机在实例化以及属性填充后进行，是几种注入方式最晚的
    • 因为注入时需要手动进行依赖查找，且有一定的局限性，所以并不推荐使用

为什么构造器无法解决循环依赖

什么是循环依赖？

  循环依赖指的是A -> B -> A这样的依赖关系 。因为 A 实例化时需要依赖 B，因此 B 会被实例化，但 B 在实例化时又需要依赖 A 因此就会进入死循环导致栈溢出。

而在 Spring 中虽然不会直接抛出 StackOverflowError ，但也会抛出 BeanCurrentlyInCreationException 来告诉开发者当前发生了循环依赖。

    需要注意的是在 Spring 中，只有使用构造器注入时发生循环依赖才会抛 BeanCurrentlyInCreationException ，这是因为使用构造器注入的依赖被容器认为是急需的依赖（这时如果发生循环依赖，就会因为死循环而导致两个Bean都不能完成实例化），而对于字段注入、方法注入 、接口注入等方式 Spring 则做了内部处理，因为这时的 Bean 已经成功实例化。

    Spring 解决非构造器注入引起的循环依赖问题的思路是在 Bean A 实例化时暴露一个 ObjectFactory 到 DefaultSingletonBeanRegistry.singletonFactories 中，而该接口的 getObject 方法封装的是获取或创建当前 Bean A 的逻辑实现（如果马上执行 getObject 可获得当前还没注入依赖的 Bean A 实例）。这时假设发生了循环依赖，Bean A 在填充属性时就会去实例化 Bean B（Bean B 亦会暴露一个 ObjectFactory），而 Bean B 因为又依赖了 Bean A，因此 Bean B 在获取依赖查找时可以从 DefaultSingletonBeanRegistry.singletonFactories 中的 ObjectFactory 获取到 Bean A 的引用从而解决解决了无法获取到已经实例化的 Bean A 依赖问题，进而解决了循环依赖。
