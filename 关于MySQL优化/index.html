MySQL优化入门
发表于 2019-06-20| 更新于 2020-04-30|编程数据库
评论数:

前言

  • 关于优化方面个人认为是一个比较复杂的环节，而且并不能一概而论；

  • 数据库优化需要结合实际的使用场景，一步一个脚印地针对出现的问题来进行解决，这就是数据库的优化；

  • 因为优化的手段也有很多，如最基本的SQL优化、表优化、数据库本身的优化、数据库所在机器的优化 等等，如果是应用层面还可以通过 缓存 、流量削峰 、 请求限流 等等手段减轻数据库压力

常用数据库优化手段

  • SQL优化

    • 添加索引
    • 进行执行计划分析优化
    • 必要时使用存储过程

  • 加入缓存层以降低数据库压力

  • 表优化

    • 水平拆分
    • 垂直（纵向）拆分
    • 分区

  • 数据库优化

    • 水平拆分
    • 垂直（纵向）拆分
    • 主从复制
    • 数据库集群（用的比较少）

那么什么情况下分表，什么情况下分库呢？

一般都是数据量大考虑分表，并发高就考虑分库。

但是需要注意的是，能不拆就不拆，能少拆就少拆，因为拆得越多越细会大大提高后期维护成本。因此分库分表其实是没有办法中的办法。

执行计划

是什么？

  举个简单的例子：select * form t1 join t2 on t1.id = t2.t1_id在这个连接查询中，到底是先访问t1表呢？还是先访问t2表？而先访问t1表和先访问t2表这两个选择就代表了两个不同的执行计划；

  再通俗点说，就是 执行计划 = 执行引擎的处理方法 ，通常我们会说查看或分析一下某SQL语句的执行计划，其实就是查看和分析一下数据库对这条SQL语句的处理方法，然后再通过数据库的特性来去对其进行调整优化；

  但需要明确的是，SQL优化，其实是围绕的如何更高效地使用索引为进行的，所以如果没有索引，就没法谈优化

了解执行计划

参考文档/文章：

  • https://dev.mysql.com/doc/refman/8.0/en/optimization.html

  • https://yq.aliyun.com/articles/688754

  • id：SQL查询id；特点是id相同，由上而下执行；id不同，大的先执行

  • select_type：SQL查询类型，常见有以下类型：

    • simple：简单查询，不包含如 子查询、union 等复杂查询的查询

    • primary：包含子查询的sql语句的外层查询

    • subquery：包含子查询的sql语句的内层查询

    • union：属于当前分析的sql语句的 union/union all 部分查询

    • derived：衍生查询（表示使用了临时表），如子查询中使用了 in 或 union/union all

  • partitions：所匹配到的分区，如果为null表示当前查询为非分区查询

  • table：表名

  • type：索引操作的类型，常见有以下类型：

    • system：只有一条数据的系统表 或 衍生表只有一条数据的主查询
    • const：使用了unique或primary key作为查询条件，且只返回了一个结果
    • eq_ref：对于每个索引键的查询，返回匹配唯一行数据，即查询的内容数据是唯一的，如unique；一般为关联查询
    • ref：对于每个索引键的查询，返回匹配的所有行（0，多）；即通过索引查询后没有进行额外的筛选操作；一般为关联查询
    • range：进行了区间查询
    • index：进行了索引全扫描，扫描了所有索引
    • all：当前语句进行了全表扫描
    • 其他：https://dev.mysql.com/doc/refman/8.0/en/explain-output.html
    • 效率：system > const > eq_ref > ref > range > index > all

  • possible_keys：表示当前查询中可能使用到的索引，是一种预测，不一定准确

  • key：实际使用到的索引

  • key_len：索引的字节长度，可以用该值来判断组合索引是否全用上了；

    计算：如一张utf8(一个字符3个字节)表中name char（20）是索引，那么key_len为3x20=60，又因为name可以为null（规定用1个字节表示），所以为60+1=61（如果是varchar的话，因为不是定长，系统中非定长用2个字节表示）

  • rows：这次查询所扫描的记录数，该值越少越好

  • filtered：在rows中过略掉的百分比值，即当前结果筛掉了多少rows才得到，该值越大越好

  • extra：额外信息，常见有以下类型(如果同时出现，则表示部分使用了不同的类型)

    • use index：进行了覆盖索引查询，即查询了索引就直接找到了结果，没有进行回表操作，性能最好
    • use index dondition：表示存储引擎对查询的数据进行的优化
    • using where：使用了索引，但进行了回表操作
    • using temporary：使用了临时表，原因是根据索引进行条件查询，但又没有使用查询条件中的索引进行分组所导致的，以下操作会可能使用临时表，尽量避免
      • group by
      • distinct
      • union
    • using join buffer：表示引擎使用了连接缓存
    • using filesort：进行了文件排序(额外排序)，通常伴随order by出现，原因是根据索引进行条件查询，但又没有使用查询条件中的索引进行排序所导致的；还有一种情况就是跨列使用组合索引导致的，如 组合索引(a1,a2,a3,a4)，那么where a1.. and a2 .. and a4 order by a3 不属于跨列，按照最左前缀原则虽然a2到a4跨列导致a4索引实现会进行using where回表之外，a1.. a2 … order by a3 是符合组合索引使用顺序的，但如果where a1.. and a2 order by a4，因为a2到a4跨列，索引失效，而a4又作为排序条件，所以因为没有使用到where中的索引排序而需要额外的using filesort

  • ref：指明当前表所参照的字段；如在关联查询中为null，则可能没有使用到索引

    • const：表示key依赖的是一个常量
    • fucn：表示key依赖的是一个函数返回的值

MySQL复制

是什么

  • 指的是将一个数据库（主库）上的数据复制到另一个数据库（从库）中

  • 默认情况下复制是异步的

  • 根据配置可以选择性复制某个数据库或者表

复制基本原理

以一主一从为例子，如下图：

  1. 首先从库会创建一条IO线程去读取主库的二进制日志binlog
  2. 主库发现从库的连接后会创建一条新的线程来处理从库的复制请求
    • 首先会向本地的binlog事件提供从库的信息
    • 告知从库已经准备好复制工作
  3. 从库读取binlog并将内容写入中继日志中（复制延迟主要在这里，例如网络慢）
  4. 从库启动SQL线程将中继日志的内容入读库中

可以看出MySQL的复制并非直接传输数据而是复制binlog，这是因为binlog中记录了所有修改数据库的SQL操作

主从复制配置

基本步骤

  1. 在主库中开启二进制日志（这是复制的大前提）
  2. 在主库中创建一个专门用来做复制操作的用户账号
  3. 设置从库的唯一id（server_id）
  4. 在slave中远程备份master
  5. 在slave中恢复从master备份的数据
  6. 使用show master status查看主库状态
  7. 在slave中执行 change master to 命令指向主库（根据第4的信息配置）
