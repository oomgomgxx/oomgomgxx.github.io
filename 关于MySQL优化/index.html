前言

  • 关于优化个人认为是一个比较复杂的环节，不能一概而论

  • 数据库优化需要结合实际的使用场景，一步一个脚印地针对出现的问题来解决

  • 优化手段有很多，如最基本的SQL优化、表结构优化、数据库服务器软硬件优化等等，如果站在应用层面还可以通过 缓存 、流量削峰 、 请求限流 、资源限流等方案减轻其数据库的访问压力

那么什么情况下分表，什么情况下分库呢？

个人理解一般是数据量大考虑分表，负载高就考虑分库。但能不拆就不拆，且能少拆就少拆，因为拆得越多越细会大大提高后期维护成本，所以分库分表其实是没有办法中的办法。

执行计划

是什么？

  执行计划指的是数据层对 SQL 语句的执行方案。例如在连接查询中到底是先执行左表还是先执行右表呢？这都是执行计划的一部分即执行计划 = 执行引擎的处理方法

字段意思

  • id：查询id

    • id相同时，由上而下执行
    • id不同时，大的先执行

  • select_type：SQL 的查询类型

  • simple：简单查询。不包含如子查询、union等操作

  • primary：子查询的外层查询

    • subquery：子查询的内层查询
    • union：并集查询

  • derived：子查询中的衍生查询（表示使用了临时表）。例如使用了 in 或 union/union all

  • partitions：所匹配到的分区，如果为null表示当前查询为非分区查询

  • table：表名

  • type：常见索引操作类型

    • system：系统表主查询
    • const：使用了 unique 或 primary key 作为查询条件，且只返回了一个结果
    • eq_ref：查询的内容是唯一的
    • ref：通过索引查询后没有进行额外的筛选操作
    • range：进行了区间查找
    • index：扫描了所有索引
    • all：进行了全表扫描
    • 效率：system > const > eq_ref > ref > range > index > all。一般优化到 ref、range 就不错了

  • possible_keys：当前查询中可能使用到的索引（预测值，不准确）

  • key：实际使用的索引

  • key_len：索引长度

    • 可用该值来判断组合索引是否全用上
    • 计算方式：
      • 在 MySQL 中，5.X 开始 varchar 中的值表示字符而不是字节
      • 例如假设一张 utf8 （一个字符3个字节）的表中 msg char（20）是索引，那么 key_len 就为 3x20=60，又因为 msg 可以为 null（规定用1个字节表示），所以从长度为 60+1=61

  • rows：查询所扫描的记录数。该值越少越好

  • filtered：在 rows 中被过略掉的数据的百分比值。该值越大越好

  • extra：查询的额外信息

    • use index：进行了覆盖索引查询。没有进行回表操作，性能最好
    • use index dondition：存储引擎对查询的数据进行了优化
    • using where：使用了索引，但存在回表操作
    • using temporary：使用了临时表
      • 通常是因为使用了索引作为条件进行查询，但没有使用相同索引条件作为分组查询条件所导致的
      • 以下常见操作默认会使用临时表
        • group by（如果与 order by 一起使用，则尽量保证用相同的列作为条件）
        • distinct
        • union
    • using join buffer：引擎使用了连接缓存
    • using filesort：进行了文件排序（额外排序）
      • 通常伴随 order by 出现。原因是使用了索引进行条件查询，但没有使用相同索引条件作为排序条件所导致的
      • 还有一种特殊情况就是跨列使用组合索引导致的。譬如 where a1.. and a2 order by a4 操作，其中组合索引 （a1,a2,a3,ab），而在这次查询中因为 a2 到 a4 跨了列而导致 a4 失效，但 a4 又了作为了排序条件，所以 a4 相当于不属于当前使用到的组合索引，因为就需要额外的 using filesort 操作（因为索引本身有序，但搜索引擎觉得 a4 是无序的）

  • ref：当前表所参照的字段。例如在关联查询中该值为 null，则可能没有使用到索引

    • const：表示索引（key）依赖的是一个常量
    • fucn：表索引（key）依赖的是一个函数返回的值

水平拆分和垂直拆分

基本概念

有些数据库是支持表分区的（如MySQL），即通过物理拆分将数据存储到不同的物理分区。例如可以利用分区功能将一个表的数据保存到不同的硬盘上就可以利用其分区策略快速等位到数据以提高查询效率

例子 - 表的水平拆分

准备工作

-- 用户表
create table user(
    u_id int primary key auto_increment,
    u_name varchar(10)
) DEFAULT charset = utf8 ; 
-- 插入用户数据
insert into user(u_name) values('张三'),('李四'),('王五');

-- 用户订单表
create table uorder(
    o_id int primary key auto_increment,
    u_id int,
    o_desc varchar(100)
) DEFAULT charset = utf8 ; 

-- 批量插入模拟用订单数据
-- userid 用户id
-- produce 订单产品
-- num 插入数量
CREATE PROCEDURE pro_insert_data(IN userid INT,IN produce VARCHAR(10), IN num INT)
BEGIN
    DECLARE temp INT;
    SET temp = 1;
    REPEAT
        insert into uorder(u_id, o_desc) values(userid, CONCAT(produce,temp));
        SET temp = temp + 1;
    UNTIL temp>num END REPEAT;
END

-- 插入订单数据
call pro_insert_data(1, 'manhua', 20);
call pro_insert_data(2, 'wanju', 30);
call pro_insert_data(3, 'fushi', 30);

拆分工作

1）现在打算对 uorder 进行拆分

  • 因为这张表的关联字段是 u_id，所以需要使用 u_id 作为分拆策略字段
  • 因为 uorder 表中共有80条记录，所以现在打算将 uorder 拆分成 2 个，即 uorder0、uorder1，其中每张表各存 40 条记录
  • 分区定位策略为 hash（u_id）%2 或 hash（u_id）&2-1

2）按照策略将数据转移（可使用程序操作）到分拆表中

-- 拆分0表
create table uorder0(
    o_id int primary key auto_increment,
    u_id int,
    o_desc varchar(100)
) DEFAULT charset = utf8 ; 

-- 拆分1表
create table uorder1(
    o_id int primary key auto_increment,
    u_id int,
    o_desc varchar(100)
) DEFAULT charset = utf8 ;

MySQL复制

是什么

  • 指将一个数据库（主库）上的数据冗余到到另一个数据库（从库）中

  • 在 MySQL 中默认情况下复制是异步的

  • 根据配置可以有选择性地指定需要复制的库或表

基本原理

以一主一从为例

  1. 首先从库会创建一条IO线程去读取主库的 binlog
  2. 主库在发现从库连接后，会创建一条复制处理线程来处理从库的复制请求
    • 首先会向本地 binlog 提交从库的信息
    • 告知从库已经准备好复制工作
  3. 从库读取 binlog 并将内容写入中继日志中（复制延迟主要在这里）
  4. 从库启动SQL线程将中继日志的内容入读库中

主从复制配置

基本步骤

  1. 在主库中开启 binlog
  2. 在主库中创建一个专门用来做复制的用户账号
  3. 设置从库的唯一id（server_id）
  4. 在 slave 中远程备份 master
  5. 在 slave 中恢复 master 备份的数据
  6. 使用show master status查看主库状态
  7. 在 slave 中执行 change master to 命令指向主库

注意事项

  • 应该先备份 master 数据，如果不进行备份就只能同步到在start slave开始的数据，而不能同步到之前的操作
  • 从节点在恢复 master 数据之前，应该使用相同的 schema 来创建库和表结构，否则可能会导致同步失败
  • 同步失败解决参考：https://blog.csdn.net/xiaoxinla/article/details/7679578

使用MyCat实现读写分离

基本概念

通过 MyCat 来代理 master 和 slave 两个数据节点。这对于程序而言是无感知的，程序只需要正常执行 sql 操作即可，因为数据库中间件可以屏蔽数据库层的复杂性。

<?xml version="1.0"?>
<!DOCTYPE mycat:schema SYSTEM "schema.dtd">
<mycat:schema xmlns:mycat="http://io.mycat/">

    <!-- mycat schemas -->
    <schema name="mycat_masterdb" checkSQLschema="false" sqlMaxLimit="100" dataNode="dn1" ></schema>

    <!-- mycat datanodes -->
    <dataNode name="dn1" dataHost="localhost1" database="masterdb" />

    <!-- 数据库节点配置 -->
    <!--
        balance:
            0：不用读写分离
            1：读操作发送到当前writeHost下的readHost和备用writHost
            2：writeHost和readHost共同分摊读操作
            3：所有读操作都发送给writeHost下的readHost，比较合适与一主多从
    -->
    <dataHost name="localhost1" maxCon="1000" minCon="3" balance="1" writeType="0" dbType="mysql" dbDriver="native" switchType="2"  slaveThreshold="100">

        <!-- 心跳测试 -->
        <heartbeat>show slave status</heartbeat>

        <!-- 写库（master） -->
        <writeHost host="mysql57-master" url="localhost:3307" user="root" password="123456" >
            <!-- 读库（slave） -->
            <readHost host="mysql57-slave" url="localhost:3308" user="root" password="123456" />
        </writeHost>
    </dataHost>

</mycat:schema>

JpaTrasactionConfig

@Configuration
@EnableJpaRepositories(basePackages = "com.td.dbtest.jpa", enableDefaultTransactions = false)
public class JpaTransactionConfig {
    /**
     * Jpa默认的事务管理对读操作添加了readOnly=true事务
     * 而MyCat只要是处于事务的sql都会走write库，所以需要修改参数 enableDefaultTransactions = false
     * 除此之外，还可以在sql前添加 /*balance*\/ 来解决
     */
}

service

@Service
public class PersonService {
    @Autowired
    private PersonRepository repository;

    @Transactional(propagation = Propagation.NOT_SUPPORTED)
    public List<Person> findAll(){
        return repository.findAll();
    }

    @Transactional
    public void save(Person person) {
        repository.save(person);
    }
}
