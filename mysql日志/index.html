MySQL日志

 2019-06-17   作者: TanDi

日志的作用

在代码中我们经常使用日志来去记录和输出系统的一些执行信息，我们可以通过这些日志来观察系统的执行过程和状况，不管是开发时用于调试又或者用来定位错误，这都是日志所带给我们的便利；

而对于数据库而言，日志更加是比不可少的组成部分，因为数据库本身的理念就是存储数据，保证数据的持久化；那么一旦数据库出了问题又或者丢失了一些数据时该怎么办？mysql中就提供了诸如二进制日志来对数据库的变更操作进行记录，一旦出了什么问题就可以通过二进制日志来去恢复；

除了针对恢复数据的二进制日志外，mysql还提供了：慢查询日志、错误日志、通用查询日志等等一些负责不同任务的日志；

除了数据库(mysql server)本身，一些存储引擎也有自己的日志，如InnoDB的redo日志，就可以做到减少磁盘访问的效果；

因此了解日志个人认为也是非常有必要的，就算不能彻彻底底地弄懂它也至少对其有基本的认识；

对于学习技术而言，个人的感受是，俯瞰技术全貌怎么都比片面认识来的实际；

MySQL中日志的种类

二进制日志(归档日志)

作用

  1. 二进制日志记录的是所有更改数据的语句
  2. 使用二进制日志的最大原因是二进制日志可以最大可能恢复数据库，因为二进制日志包含数据库服务启动后的所有更改数据的操作
  3. 二进制日志默认是关闭的，需要手动开启

查看二进制日志是否开启

-- 是否开启
show variables like '%log_bin%';
-- 二进制日志文件的名字
show binary logs;

开启二进制日志

官方文档：https://dev.mysql.com/doc/refman/8.0/en/replication-options-binary-log.html

在mysql的ini文件中添加或修改配置

Mysql 5.7

[mysqld]
server-id = 1
log-bin = mysql-bin-log

重启mysql服务，因为像我一样使用docker容器的话只需要重启容器即可

其中000001这个后缀会随着mysql的重启而增加1，即二进制文件会从mysql启动时开始记录；

使用二进制日志恢复数据

目录表的状态如下

mysql> delete from book;
Query OK, 2 rows affected (0.02 秒)

mysql> select * from book;
空的数据集 (0.01 秒)

现在我的目标是将删除 delete from book;的数据恢复回来；

恢复之前需要到二进制文件中找到这条删除数据的命令锁执行的时间点；

... 省略

可以看到时间点是：190627 13:50:29，将格式转换为：2019-06-27 13:50:29

使用同样的方法找恢复数据的起点事件(不设置起点的话默认从mysql启动开始恢复)

我这里的插入数据的起点事件为：190627 13:37:21，将格式转换为：2019-06-27 13:37:21

使用 mysqlbinlog 命令恢复数据

mysqlbinlog --start-datetime="2019-06-27 13:37:21" --stop-datetime="2019-06-27 13:50:29" mysql-bin-log.000001 mysql-bin-log.000002 | mysql -uroot -p123456

通用查询日志

作用

记录用户的所有操作，包括 启动关闭服务器、查询、更新操作

查看通用查询日志是否开启

mysql> show variables like '%general_log%';
+------------------+---------------------------------+
| Variable_name    | Value                           |
+------------------+---------------------------------+
| general_log      | OFF                             |
| general_log_file | /var/lib/mysql/8ddb8a0a85bb.log |
+------------------+---------------------------------+
2 行于数据集 (0.02 秒)

开启通用查询日志

在mysql配置文件添加如下配置

[mysqld]
# General
# close 0, open 1, default 0
general-log=1
general_log_file="general-log"

重启服务

慢查询日志

作用

  1. 慢查询日志记录的是超过指定时间的SQL语句
  2. 通过慢查询日志可以找出 执行时间长、执行效率低 的语句，从而可以进行针对性的优化

需要注意的是，虽然叫慢查询日志，但是只要超过设定时间的sql就会记录起来，即慢查询并非单指select操作

查看慢查询日志是否开启

mysql> show variables like '%query%';
+------------------------------+-------------------+
| Variable_name                | Value             |
+------------------------------+-------------------+
| slow_query_log               | OFF               |
+------------------------------+-------------------+
13 行于数据集 (0.02 秒)

或者使用 show variables like ‘%query%’;

开启慢查询日志

方式1：配置文件开启

在mysql配置文件中添加如下配置

# slow-query-log
# close 0, open 1, default 0
slow-query-log = 1
slow_query_log_file = "my_slow-query-log"
# 超过 2s 的查询都会被记录
long_query_time = 2

重启服务

方式2：设置全局变量开启

CREATE PROCEDURE pro_insert_data(IN num INT)
BEGIN
    DECLARE temp INT;
    SET temp = 1;
    REPEAT
        insert into book(bname) values(CONCAT('book',temp));
        SET temp = temp + 1;
    UNTIL temp>num END REPEAT;
END

错误日志

对于错误日志，感觉没什么特别要说的，用于记录mysql运行过程中的错误，默认是开启的；

也可以通过mysql配置文件指定错误日志的生成路径和名字

[mysqld]
log_error = "error_log"

InnoDB的redo、redo

redo log

redo log是InnoDB事务日志的组成部分（事务日志 = redo log + 日志缓冲区）。用于记录页更改的物理情况，功能是保证事务的完整性。

与MySQL bin log的区别：

  • bin log用于记录修改数据的操作，在事务提交之前写到磁盘
  • redo log用于记录页的物理情况。内容先写入日志缓冲区，随后默认每隔1s刷新到磁盘日志文件中
  • redo log属于写前日志，即修改操作之前会先记录到redo log中

undo段

  • undo段主要用来保证事务的回滚
  • 功能和redo log相反，用于记录页修改前的状态
  • undo并不位于日志文件，而是在数据库内部的一个内存段中
