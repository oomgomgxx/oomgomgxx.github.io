日志的种类

二进制(归档)日志

作用

记录修改数据的操作

查看是否开启

二进制日志默认是关闭的，需要手动开启

-- 是否开启
show variables like '%log_bin%';
-- 输出二进制日志文件的名字
show binary logs;

开启日志

官方文档：https://dev.mysql.com/doc/refman/8.0/en/replication-options-binary-log.html

1）在 mysql 的 ini 文件中添加或修改配置（Mysql 5.7）

[mysqld]
server-id = 1
log-bin = mysql-bin-log

2）重启 mysql 服务

3）然后可以看到在目录中会多了日志文件

其中 000001 这个后缀会随着 mysql 的重启而递增。

查看日志

在此之前我在数据库中做了如下操作

mysql> create database test;
Query OK, 1 rows affected (0.01 秒)

mysql> use test;
Query OK, 0 rows affected (0.01 秒)

mysql> create table book(id int primary key auto_increment, bname varchar(30) not null);
Query OK, 0 rows affected (0.04 秒)

mysql> insert into book(bname) values('book1'),('book2');
Query OK, 2 rows affected (0.01 秒)

mysql> delete from book;
Query OK, 2 rows affected (0.02 秒)

进入 docker 容器使用 mysql 提供的 mysqlbinlog 来读取二进制日志中的内容

docker exec -it 容器名称 bin/sh

根据二进制日志恢复数据

1）先删除表中的数据

mysql> delete from book;
Query OK, 2 rows affected (0.02 秒)

mysql> select * from book;
空的数据集 (0.01 秒)

2）恢复 delete from book;前的数据

在恢复之前我们需要到二进制日志文件中找到删除数据命令的时间点，如下

...
#190627 13:50:29 server id 1  end_log_pos 1051 CRC32 0xa331f579     Delete_rows(删除了数据): table id 103 flags: STMT_END_F

BINLOG '
pckUXRMBAAAAMgAAAOIDAAAAAGcAAAAAAAEABHRlc3QABGJvb2sAAgMPAh4AAAXI3Sk=
pckUXSABAAAAOQAAABsEAAAAAGcAAAAAAAEAAgAC//wBAAAABWJvb2sx/AIAAAAFYm9vazJ59TGj
...

可以看到时间点是：190627 13:50:29（2019-06-27 13:50:29）

3）用同样的方法，找到要恢复数据的起点时间（如果不设置起点默认从 mysql 启动位置开始恢复）

时间点是：190627 13:37:21（2019-06-27 13:37:21）

4）使用 mysqlbinlog 命令恢复数据

mysqlbinlog --start-datetime="2019-06-27 13:37:21" --stop-datetime="2019-06-27 13:50:29" mysql-bin-log.000001 mysql-bin-log.000002 | mysql -uroot -p123456

5）查看结果

mysql> select * from book;
+----+-------+
| id | bname |
+----+-------+
| 1  | book1 |
| 2  | book2 |
+----+-------+
2 行于数据集 (0.02 秒)

通用查询日志

作用

记录用户的所有操作，包括启动关闭服务器、查询、更新等信息

查看是否开启

mysql> show variables like '%general_log%';
+------------------+---------------------------------+
| Variable_name    | Value                           |
+------------------+---------------------------------+
| general_log      | OFF                             |
| general_log_file | /var/lib/mysql/8ddb8a0a85bb.log |
+------------------+---------------------------------+
2 行于数据集 (0.02 秒)

开启日志

1）在 ini 配置文件添加如下配置

[mysqld]
# General
# close 0, open 1, default 0
general-log=1
general_log_file="general-log"

2）重启服务

慢查询日志

作用

  • 记录超过指定时间的 SQL 语句

  • 可以利用慢查询日志定位执行时间长、效率低 的语句，从而可以做到有针对性地优化

查看是否开启

mysql> show variables like '%query%';
+------------------------------+-------------------+
| Variable_name                | Value             |
+------------------------------+-------------------+
| slow_query_log               | OFF               |
+------------------------------+-------------------+
13 行于数据集 (0.02 秒)

开启日志

方式1：配置文件开启

1）在 mysql 配置文件中添加如下配置

# slow-query-log
# close 0, open 1, default 0
slow-query-log = 1
slow_query_log_file = "my_slow-query-log"
# 超过 2s 的查询都会被记录
long_query_time = 2

2）重启服务

方式2：设置全局变量开启

1）查看是否开启了慢查询日志

2）开启慢查询日志

3）将日志的输出定向为 Table 中，因为默认是输出到 file 的

观察日志

1）在此之前我使用存储过程往 book 表添加 250000 条数据

CREATE PROCEDURE pro_insert_data(IN num INT)
BEGIN
    DECLARE temp INT;
    SET temp = 1;
    REPEAT
        insert into book(bname) values(CONCAT('book',temp));
        SET temp = temp + 1;
    UNTIL temp>num END REPEAT;
END
mysql> use test;
Query OK, 0 rows affected (0.01 秒)

mysql> call pro_insert_data(250000);
Lost connection to MySQL server during query

mysql> select count(1) from book;
+----------+
| count(1) |
+----------+
| 250001   |
+----------+
1 行于数据集 (0.05 秒)

可以看到已经成功插入了250000条数据

2）执行更新操作

mysql> update book set bname = 'asasf5461werw321sdfqwasd';
Query OK, 250001 rows affected (3.55 秒)

在上面我们将慢查询定义为了 2s，可以看到当前这个更新操作已经超过了2s

3）查看慢查询日志

错误日志

用于记录 mysql 运行过程中的错误，默认是开启的。但可以通过配置文件指定日志的生成路径和名字

[mysqld]
log_error = "error_log"

InnoDB的redo、redo

redo log

redo log是InnoDB事务日志的组成部分（事务日志 = redo log + 日志缓冲区）。

用于记录数据页更改的物理情况，主要是保证事务的一致性。

与 bin log 的对比：

  • bin log 是 server 层面的日志，用于记录修改数据的操作，在事务提交之前写入磁盘
  • redo log 是 InnoDB 层面的日志，用于记录数据页的物理情况。内容先写入日志缓冲区，随后默认每隔1s刷新到磁盘日志文件中
  • redo log 属于写前日志，即修改操作之前就会先记录，且在事务操作期间不断积累

undo段

  • undo 段主要用来保证事务的回滚，以及作为隔离级别读取一致性的基础
  • 功能和 redo log 相反，用于记录数据页修改前的状态
  • undo 并不位于日志文件，而是在数据库内部的一个内存段中
