关于引用的小知识

public class Test {
    public static void main(String[] args) throws InterruptedException {

        // 软引用生命周期：内存不足的前提下第二次gc前。即第一次gc回收过后，依然不能满足所需内存才会回收软引用
        SoftReference<Person> softReference = new SoftReference<>(new Person("Sam"));
        System.gc();
        System.out.println(softReference.get() == null ? "对象已被回收" : "对象依然存活");

        System.out.println("-------------------- 软引用 end -------------------------");

        // 弱引用生命周期：从对象诞生到下一次GC之前
        WeakReference<Person> weakReference = new WeakReference<>(new Person("Sam"));
        System.out.println(weakReference.get() == null ? "对象已被回收" : "对象依然存活");
        System.gc();
        System.out.println(weakReference.get() == null ? "对象已被回收" : "对象依然存活");

        System.out.println("-------------------- 弱引用 end -------------------------");

        // 虚引用生命周期：该等级的引用不会对引用本身产生任何副作用，只用于监听引用是否被回收
        Person sam = new Person("Sam");
        ReferenceQueue<Person> queue = new ReferenceQueue<>();
        PhantomReference<Person> personPhantomReference = new PhantomReference<>(sam, queue);
        // 注意：虚引用无法像其他引用一样 get() 获取到关联的引用实例
        // 主动将引用入队
        //personPhantomReference.enqueue();

        new Thread(() -> {
            while (queue.poll() == null) { // 当引用被回收就会入队
                try { TimeUnit.SECONDS.sleep(1); } catch (InterruptedException e) { }
                System.out.println("对象依然存活");
            }
            System.out.println("对象已被回收");
        }).start();

        TimeUnit.SECONDS.sleep(3);
        sam = null; // 消除强引用让gc回收
        System.gc();

        // 等待gc
        TimeUnit.SECONDS.sleep(3);

        /**
         * 输出结果：
         * 
         * 对象依然存活
         * -------------------- 软引用 end -------------------------
         * 对象依然存活
         * 对象已被回收
         * -------------------- 弱引用 end -------------------------
         * 对象依然存活
         * 对象依然存活
         * 对象依然存活
         * 对象已被回收
         */
    }
}

拓展内容1：三种引用的使用场景

软引用：适合做缓存

弱引用：适合临时数据。例如 JDK 中的 WeakHashMap 以及接下来聊的 ThreadLocalMap

虚引用：监控对象回收

拓展内容2：FinalReference

除了以上3种引用之外，还有一种特别的引用FinalReference。FinalReference接口是包级别的，说白了就是对外不公开只是内部使用的接口。

JVM会在垃圾回收之前（第一次标记之后）将实现了Object的finalze方法的实例包装为FinalReference放进ReferenceQueue中进行统一调用，而完成这项工作的精灵线程叫Finalizer线程。

ThreadLocal 和 Thread 之间的关系

  • 在ThreadLocal 中有个内部类叫 ThreadLocalMap，其存储格式为键值对存储

  • 操作 ThreadLocal 实质就是对 ThreadLocalMap 操作

  • 每个 Thread 对象中都有一个与当前线程所管理的 ThreadLocalMap，所以 ThreadLocal 之所以被叫做线程变量，其实就是这个原因，因为 ThreadLocalMap 是线程独占的并不存在并发问题

  • 一个 Thread（ThreadLocalMap）中可以存储多个 ThreadLocal

    • 添加线程变量：ThreadLocalMap#set(ThreadLocal, Object)
    • 获取线程变量：ThreadLocalMap#getEntity(ThreadLocal)

  • 如果要在线程池中使用 ThreadLocal，则应该在任务结束时主动调用 ThreadLocal#remove() 将当前 ThreadLocal 在 Thread（ThreadLocalMap）中删除，否则可能会因为线程重用的缘故而导致 Value 值无法被及时回收从而导致内存泄露。

Entry类的内存泄露问题

Entry类

static class Entry extends WeakReference<ThreadLocal<?>> {
    Object value;
    Entry(ThreadLocal<?> k, Object v) {
        super(k);
        value = v;
    }
}

为什么要将 key 定义为弱引用？

  个人认为原因是为了更好地回收 ThreadLocal 与之对应的 value 值。因为 ThreadLocalMap 的 set/get 方法是具有一定的清理能力的，而其清理操作主要针对 key 为 null 的 Entry（参考下面的源码分析）。大致步骤是先往数组左边检索，然后再从左往右清理其 key 为 null 的 Entry。

  在上面我们已经得知了弱引用实例的生命周期范围，所以简单地说就是如果当 ThreadLocal 用完后被手动置空的话（即丢失了强引用），那么这个 ThreadLocal 就只能活到下一次gc，而当 ThreadLocal 被回收掉，那么在下一次再调用 set/get 方法时就会被发现 key 为 null 然后其value就会被清除。

  现在我们假设如果 key 被定义为强引用看看又会发生什么事情。key如果为强引用，也就是说即使在程序中手动置空 ThreadLocal 引用 gc 也不能将其回收，主要是因为 ThreadLocalMap 中还强引用着这个已经“用完”的 ThreadLocal，因此对于 JVM 而言宁可抛出 OOM 都不会回收强引用的其空间。但我们现在明确知道这个 ThreadLocal（以及其关联的Value）已经“用完”了，但其空间却不能回收，这时如果 Thread 的生命周期非常长的话，就会导致内存泄露问题的发生。所以看到网上有些人说是因为弱引用而导致内存泄露，这显然是不正确的。

为什么会出现内存泄露？

  ThreadLocalMap 的生命周期与 Thread 绑定，所以当 Thread 被回收后 ThreadLocalMap 很自然就被回收了，所以并不会出现什么内存泄露的问题。明白这一点很重要，因为问题的关键点就在 Thread 的生命周期被延长后，才导致 ThreadLocalMap 存在内存泄露的风险。譬如线程池中的线程可能长期驻留在内存中，因此导致 ThreadLocalMap 中的内容无法被回收。

  虽然 ThreadLocalMap 的 set/get 方法具有一定的清理能力的，但这里需要强调一下这种清理能力是有限度的，因为它并不能保证所有 key 为 null 的 Entry 都被完全清理掉。例如在 ThreadLocalMap#set(ThreadLocal, Object) 中，ThreadLocal没有发生冲突或发生了冲突但在当前下标的右边不存在key为null的Entry时，是不会往左遍历的。也就是说在这两种情况下左边即使有 key 为 null 的 Entry 都不会被发现和清理，所以就会导致内存泄露。

如何避免内存泄露？

  发生内存泄露问题的本质是 Thread 生命周期被延长而导致 ThreadLocalMap 生命周期也被延长，且在操作过后 ThreadLocalMap 并没有发现 key 为 null 的 Entry，这时就会导致内存泄露问题。所以最好的解决方法就是：

  1. 尽量不要在线程池中使用 ThreadLocal
  2. 使用完 ThreadLocal 后要主动调用 ThreadLocal 的 remove 方法，将自己在当前 Thread 的 ThreadLocalMap 中移除来主动释放空间

set

private void set(ThreadLocal<?> key, Object value) {
    Entry[] tab = table;
    int len = tab.length;
    int i = key.threadLocalHashCode & (len-1);

    // 出现了Hash冲突
    for (Entry e = tab[i];
         e != null;
         e = tab[i = nextIndex(i, len)]) { // 注意：从当前下标往右遍历

        // 返回遍历到的ThreadLocal
        ThreadLocal<?> k = e.get();

        // 发现key是同一个ThreadLocal，直接替换即可
        if (k == key) {
            e.value = value;
            return;
        }

        // 遍历时发现了key（ThreadLoca）为null，替换并执行清理操作
        if (k == null) {
            // 用当前设入的 ThreadLocal 将其替换掉
            // 注意，这时会进行一次全散列表清理（从左往右）
            replaceStaleEntry(key, value, i);
            return;
        }
    }

    // 【没有发生冲突】或【发生了冲突但从i开始散列表右边不存在key为null的Entry】
    // 那么就将当前Entry保存到散列表的最右边
    tab[i] = new Entry(key, value);
    int sz = ++size;

    // 判断是否需要rehash
    // cleanSomeSlots：从i开始往右清理哪些key为null的Entry
    // !cleanSomeSlots(i, sz)：为true时表示i开始往右并没有需要清理的Entry，即没法腾出空间
    // sz >= threshold，即表示已经到达容许限制
    if (!cleanSomeSlots(i, sz) && sz >= threshold)
        rehash(); // 扩容
}
  • 可以注意到，当【没有发生冲突】或【发生了冲突但从i开始散列表的右边不存在key为null的Entry】时，是不会往左遍历的，所以这时左边即使有 key 为 null 的 Entry 也不会被发现，即存在内存泄露风险
  • 处于右边且 key 为 null 的 Entry 之所以能够被清理是因为每一次 set 操作都会进行一次 cleanSomeSlots，而 cleanSomeSlots 是往右清理的

rehash和expungeStaleEntries

private void rehash() {
    // 压缩散列表
    expungeStaleEntries();
    // 如果这时size已经到达阈值，则扩容为原来的2倍
    if (size >= threshold - threshold / 4)
        resize(); 
}

private void expungeStaleEntries() {
    Entry[] tab = table;
    int len = tab.length;
    // rehash操作会对散列表进行一次全表清理
    for (int j = 0; j < len; j++) {
        Entry e = tab[j];
        // 从散列表最左边且key为null的Entry位置开始清理
        if (e != null && e.get() == null)
            expungeStaleEntry(j);
    }
}

replaceStaleEntry

private void replaceStaleEntry(ThreadLocal<?> key, Object value,
                               int staleSlot) {
    Entry[] tab = table;
    int len = tab.length;
    Entry e;

    // 从当前冲突位置staleSlot（不包含staleSlot）开始往左查找key为null的Entry
    int slotToExpunge = staleSlot;
    for (int i = prevIndex(staleSlot, len);
         (e = tab[i]) != null;
         i = prevIndex(i, len))

        if (e.get() == null)
            // slotToExpunge记录了散列表最左边且key为null的Entry下标
            slotToExpunge = i; 

    // 再往右遍历（不包含staleSlot）
    for (int i = nextIndex(staleSlot, len);
         (e = tab[i]) != null;
         i = nextIndex(i, len)) {

        ThreadLocal<?> k = e.get();

        // 从staleSlot（不包含staleSlot）往右遍历时发现了相同的ThreadLocal
        if (k == key) {
            e.value = value;

            // 将当前key为null的Entry替换到i位置
            // 同时将当前保存的ThreadLocal所在的Entry赋值给staleSlot
            tab[i] = tab[staleSlot];
            tab[staleSlot] = e;

            // 避免先前往右遍历时发现staleSlot真好是要清除的对象
            // 所以将当前往右遍历到的i设置为清除的目标
            if (slotToExpunge == staleSlot)
                slotToExpunge = i;

            // 从slotToExpunge往右清除（slotToExpunge是位于最左边key为null的Entry的下标）
            // 回顾：slotToExpunge为哈希表最左边的且key为null的Entry
            cleanSomeSlots(expungeStaleEntry(slotToExpunge), len);
            return;
        }

        // If we didn't find stale entry on backward scan, the
        // first stale entry seen while scanning for key is the
        // first still present in the run.
        if (k == null && slotToExpunge == staleSlot)
            slotToExpunge = i;
    }

    // If key not found, put new entry in stale slot
    tab[staleSlot].value = null;
    tab[staleSlot] = new Entry(key, value);

    // If there are any other stale entries in run, expunge them
    if (slotToExpunge != staleSlot)
        cleanSomeSlots(expungeStaleEntry(slotToExpunge), len);
}

expungeStaleEntry

// 移除指定下标的entry和该下标后key为null的entry
private int expungeStaleEntry(int staleSlot) {

    // 当前entry[]信息
    Entry[] tab = table;
    int len = tab.length;

    // 将其置空后再size减1
    // 置空的目的是解除entry的引用好让gc将其回收
    tab[staleSlot].value = null;
    tab[staleSlot] = null;
    size--;

    Entry e;
    int i;

    // 这个置空的entry后面是否还有entry？
    for (i = nextIndex(staleSlot, len); (e = tab[i]) != null; i = nextIndex(i, len)) {
        ThreadLocal<?> k = e.get();

        // 有entry且key为null？
        if (k == null) {
            // 将它一同置空
            e.value = null;
            tab[i] = null;
            size--;

            // 有entry但key不为null
        } else {
            // 获取当前ThreadLocal对应Entry的下标
            int h = k.threadLocalHashCode & (len - 1);
            // 如果h不等于当前i则表示存在ThreadLocal冲突的情况，即i和h冲突
            if (h != i) {
                // 置空当前i下标的Entry（tab[i]的值位于当前局部变量e）
                tab[i] = null;
                // 如果当前冲突的Entry（h）不会null则往后找位置来存放原来位于i的Entry
                while (tab[h] != null) 
                    h = nextIndex(h, len);
                // 将当前i的Entry存放到右边，总体来说是将Entry往右移动
                tab[h] = e
            }
        }
    }

    return i;
}

cleanSomeSlots

private boolean cleanSomeSlots(int i, int n) {
    boolean removed = false;
    Entry[] tab = table;
    int len = tab.length;

    // 从i开始往右清理哪些key为null的Entry
    do {
        i = nextIndex(i, len);
        Entry e = tab[i];
        if (e != null && e.get() == null) {
            n = len;
            removed = true;
            i = expungeStaleEntry(i);
        }
    } while ( (n >>>= 1) != 0); 
    return removed;
}

getEntry

private Entry getEntry(ThreadLocal<?> key) {
    int i = key.threadLocalHashCode & (table.length - 1);
    Entry e = table[i];
    // entry[]中存在该值，且正好是要查询的key
    if (e != null && e.get() == key)
        return e;
    else
        // 表示存在哈希冲突，当前要找的Entry不在当前位置，则需要往右查找
        return getEntryAfterMiss(key, i, e);
}

getEntryAfterMiss

private Entry getEntryAfterMiss(ThreadLocal<?> key, int i, Entry e) {
    Entry[] tab = table;
    int len = tab.length;

    while (e != null) {
        ThreadLocal<?> k = e.get();
        // 找到对应的Entry直接返回
        if (k == key)
            return e;
        // 在找Entry途中发现到存在key为null的Entry则进行清理
        if (k == null)
            expungeStaleEntry(i);
        else
            // 当前下标没有找到想要的结果，接着往右遍历
            i = nextIndex(i, len);
        e = tab[i];
    }

    // 不存在该key对应的Entry
    return null;
}
