软引用、弱引用、虚引用之间的基本区别

// 【软引用】只在内存不足才会被回收
public static void testSoftReference() {
    ReferenceQueue<Person> queue = new ReferenceQueue<>();
    SoftReference<Person> reference = new SoftReference<>(new Person("Sam"), queue);

    System.out.println("第1次gc");
    System.gc();
    System.out.println(queue.poll() == reference?"对象已被回收":"对象依然存活");

    System.out.println("第2次gc");
    System.gc();
    System.out.println(queue.poll() == reference?"对象已被回收":"对象依然存活");
}

// 【弱引用】实例的生命周期在第二次gc之前
public static void testWeakReference() {
    ReferenceQueue<Person> queue = new ReferenceQueue<>();
    WeakReference<Person> reference = new WeakReference<>(new Person("Sam"), queue);

    System.out.println("第1次gc");
    System.gc();
    System.out.println(queue.poll() == reference?"对象已被回收":"对象依然存活");

    System.out.println("第2次gc");
    System.gc();
    System.out.println(queue.poll() == reference?"对象已被回收":"对象依然存活");
    /*
        第1次gc
        对象依然存活
        第2次gc
        对象已被回收
        * */
}

// 【虚引用】随时被gc回收，但不一定经历gc就会被回收，知识飘忽不定

Entry类的内存泄露问题

Entry类

static class Entry extends WeakReference<ThreadLocal<?>> {
    /** The value associated with this ThreadLocal. */
    Object value;
    Entry(ThreadLocal<?> k, Object v) {
        super(k);
        value = v;
    }
}

为什么要将 key 定义为弱引用

  原因是为了更好地回收对应的 value 值。这样说的原因是因为 ThreadLocalMap 的 set 和 get 方法是具有一定的清理能力的，其清理操作主要是针对 key 为 null 的 Entry（参考下面的源码分析），步骤是先往链表左边检索然后再从左往右清理其 key为 null 的 Entry。

  这里需要再次强调和回顾一下 ThreadLocal、ThreadLocalMap、Thread 三者的关系，因为这是最重要的理论基础。ThreadLocalMap 虽然是 ThreadLocal 的内部类，但实质 ThreadLocalMap是作用在 Thread 实例上的。简单地说每个 Thread 实例都有对应的一个 ThreadLocalMap ，而 ThreadLocalMap 可以用来存储多个 ThreadLocal 实例，这也是为什么有些人将 ThreadLocal 叫作线程变量的原因。

  在上面我们已经得知了有关弱引用实例的生命周期范围，因此简单地说是当 ThreadLocal 用完被手动置空的话（即丢失了强引用）那么这个 ThreadLocal 实例就只能活到第二次 gc 了。然而 ThreadLocal 被回收掉的话那么在下一次再调用 ThreadLocalMap 的 set 或 get 方法时其对应的 value 值就会被发现然后被置空释放。

  接下来我们假设 key 为强引用会发生什么事情。如果 key 被定义为强引用也就是说即使用户程序中手动置空了 ThreadLocal 实例 gc 都不会将其回收掉，因为对于强引用而言 JVM 宁可不惜抛出 OOM 都不会去回收被强应用的实例空间。所以结论就是如果 key 被定义为强引用就导致 ，ThreadLocal 在 Thread 生命周期结束之前无法被回收，从而间接地导致 value 也没法被回收，这样一来的话就会导致所谓的内存泄露。所以看到网上有些人说是弱引用导致的内存泄露，这显然是不正确的，因为弱引用有利于置空 Entry 的 key 从而避免内存泄露。

为什么出现内存泄露

  首先需要说的是 ThreadLocalMap 的生命周期与 Thread 绑定。所以当 Thread 被回收 ThreadLocalMap 很自然也就会一同回收，因此并不会出现什么内存泄露的问题。所以当前问题的关键点就是Thread 的生命周期范围，既如果 Thread 生命周期很长的话 ThreadLocalMap 就会存在内存泄露的风险。那么什么一种清理下会延长 Thread 的生命周期呢？答案就是线程池。在线程池中线程是可能长期驻留的，所以如果在线程池中使用 ThreadLocal 的话就会存在内存泄露的风险。

  还需要注意的是，虽然 ThreadLocalMap 的 set/get 方法具有一定的清理能力的，但需要强调这种清理能力是有限度的，因为它并不能保证所有 key 为 null 的 Entry 都被清理掉。例如 set 方法中如果当前操作的 ThreadLocal【没有发生冲突】或【发生了冲突但从当前下标开始散列表的右边不存在 key 为 null 的Entry】时，是不会往左遍历的。换句来话就是在这样的情况下左边即使有 key 为 null 的 Entry 都不会被发现然后清理掉。

最后可以得出如下结论：

  • 在线程池中使用 ThreadLocal 会存在内存泄露的风险
  • 虽然 ThreadLocalMap 的 set/get 方法有一定的清理能力，但我们不能依靠它来解决内存泄露问题
  • 之所以发生内存泄露的根本原因是 ThreadLocalMap 没有发现 key 为 null 的 Entry 从而无法置空 value 的引用，因此 gc 就没办法会都掉 value 的内存，因为 value 和 key 不一样，value 是强引用的。

如何解决内存泄露

  从上面我们得知发生内存泄露的主要原因是因为 ThreadLocalMap 没有发现 key 为 null 的 Entry 。所以解决内存泄露隐患的最好解决方案就是在使用完 ThreadLocal 之后主动地调用 ThreadLocal 的 remove 方法来明确让 ThreadLocalMap 释放对应的 value 空间。
  除此之外在网上我看到有人建议将 ThreadLocal 修饰成 static，我想这人大概的想法是提升 ThreadLocal 的生命周期范围使得其得到更好的管控。但事实上这种操作对内存泄露问题没有半点帮助，这是因为不管 ThreadLocal 的生命周期有多广，请不要忘记 ThreadLocal 本质上是一个线程变量，既即使是同一个 ThreadLocal 在面对不同的 Thread 时它所对应的 ThreadLocalMap 都是不同的，因此还是建议养成良好的习惯，既用完就显示调用 remove 释放 value 内存。

set

private void set(ThreadLocal<?> key, Object value) {
    Entry[] tab = table;
    int len = tab.length;
    int i = key.threadLocalHashCode & (len-1);

    // 出现了Hash冲突
    for (Entry e = tab[i];
         e != null;
         e = tab[i = nextIndex(i, len)]) { // 注意：这时是往右遍历
        ThreadLocal<?> k = e.get();

        // 同一个key直接替换即可
        if (k == key) {
            e.value = value;
            return;
        }

        // 发生了哈希冲突，且发现了某个key为null的冲突Entry
        if (k == null) {
            // 用当前的 ThreadLocal 将其替换掉
            // 注意这时会进行一次全散列表清理（从左往右）
            replaceStaleEntry(key, value, i);
            return;
        }
    }

    // 【没有发生冲突】或【发生了冲突但从i开始散列表右边不存在key为null的Entry】
    // 那么就将当前ThreadLocal的Entry保存到散列表最右边
    tab[i] = new Entry(key, value);
    int sz = ++size;

    // 判断是否需要rehash
    // cleanSomeSlots：从i开始往右清理哪些key为null的Entry
    // !cleanSomeSlots(i, sz)：为true时表示i开始往右并没有需要清理的Entry，即没法腾出空间
    // sz >= threshold，即表示已经到达容许限制
    if (!cleanSomeSlots(i, sz) && sz >= threshold)
        rehash();
}
  • 重点注意：当【没有发生冲突】或【发生了冲突但从i开始散列表的右边不存在key为null的Entry】时，是不会往左遍历的，换句话就是左边即使有 key 为 null 的 Entry 也不会被发现，这时就存在内存泄露的风险。而处于右边的 key 为 null 的 Entry 能够被清理是因为每一次 set 操作都会进行一次cleanSomeSlots 操作，因为 cleanSomeSlots 是往右清理的。

rehash()和expungeStaleEntries()

private void rehash() {
    // 压缩哈希表
    expungeStaleEntries();

    // 如果这时size已经到达阈值，则扩容为原来的2倍
    if (size >= threshold - threshold / 4)
        resize(); 
}

private void expungeStaleEntries() {
    Entry[] tab = table;
    int len = tab.length;
    // rehash操作会对哈希表进行一次全表清理
    for (int j = 0; j < len; j++) {
        Entry e = tab[j];
        // 从哈希表最左边的key为null的Entry开始清理
        if (e != null && e.get() == null)
            expungeStaleEntry(j);
    }
}

replaceStaleEntry

private void replaceStaleEntry(ThreadLocal<?> key, Object value,
                               int staleSlot) {
    Entry[] tab = table;
    int len = tab.length;
    Entry e;

    // 从当前冲突位置staleSlot（不包含staleSlot）开始往左查找key为null的Entry
    int slotToExpunge = staleSlot;
    for (int i = prevIndex(staleSlot, len);
         (e = tab[i]) != null;
         i = prevIndex(i, len))

        if (e.get() == null)
            // slotToExpunge记录了哈希表最左边且key为的Entry下标
            slotToExpunge = i; 

    // 往左遍历完后再往右遍历（不包含staleSlot）
    for (int i = nextIndex(staleSlot, len);
         (e = tab[i]) != null;
         i = nextIndex(i, len)) {

        ThreadLocal<?> k = e.get();

        // 从staleSlot（不包含staleSlot）往右遍历时发现了相同的ThreadLocal
        if (k == key) {
            e.value = value;

            // 将当前key为null的Entry替换到i位置
            // 同时将当前保存的ThreadLocal所在的Entry赋值给staleSlot
            tab[i] = tab[staleSlot];
            tab[staleSlot] = e;

            // 避免先前往右遍历时发现staleSlot真好是要清除的对象
            // 所以将当前往右遍历到的i设置为清除的目标
            if (slotToExpunge == staleSlot)
                slotToExpunge = i;

            // 从slotToExpunge往右清除（slotToExpunge是位于最左边key为null的Entry的下标）
            // 回顾：slotToExpunge为哈希表最左边的且key为null的Entry
            cleanSomeSlots(expungeStaleEntry(slotToExpunge), len);
            return;
        }

        // If we didn't find stale entry on backward scan, the
        // first stale entry seen while scanning for key is the
        // first still present in the run.
        if (k == null && slotToExpunge == staleSlot)
            slotToExpunge = i;
    }

    // If key not found, put new entry in stale slot
    tab[staleSlot].value = null;
    tab[staleSlot] = new Entry(key, value);

    // If there are any other stale entries in run, expunge them
    if (slotToExpunge != staleSlot)
        cleanSomeSlots(expungeStaleEntry(slotToExpunge), len);
}

expungeStaleEntry

// 移除指定下标的entry和该下标后key为null的entry
private int expungeStaleEntry(int staleSlot) {

    // 当前entry[]信息
    Entry[] tab = table;
    int len = tab.length;

    // 将其置空后再size减1
    // 置空的目的是解除entry的引用好让gc将其回收
    tab[staleSlot].value = null;
    tab[staleSlot] = null;
    size--;

    Entry e;
    int i;

    // 这个置空的entry后面是否还有entry？
    for (i = nextIndex(staleSlot, len); (e = tab[i]) != null; i = nextIndex(i, len)) {
        ThreadLocal<?> k = e.get();

        // 有entry且key为null？
        if (k == null) {
            // 将它一同置空
            e.value = null;
            tab[i] = null;
            size--;

            // 有entry但key不为null
        } else {
            // 获取当前ThreadLocal对应Entry的下标
            int h = k.threadLocalHashCode & (len - 1);
            // 如果h不等于当前i则表示存在ThreadLocal冲突的情况，即i和h冲突
            if (h != i) {
                // 置空当前i下标的Entry（tab[i]的值位于当前局部变量e）
                tab[i] = null;
                // 如果当前冲突的Entry（h）不会null则往后找位置来存放原来位于i的Entry
                while (tab[h] != null) 
                    h = nextIndex(h, len);
                // 将当前i的Entry存放到右边，总体来说是将Entry往右移动
                tab[h] = e
            }
        }
    }

    return i;
}

cleanSomeSlots

private boolean cleanSomeSlots(int i, int n) {
    boolean removed = false;
    Entry[] tab = table;
    int len = tab.length;

    // 从i开始往右清理哪些key为null的Entry
    do {
        i = nextIndex(i, len);
        Entry e = tab[i];
        if (e != null && e.get() == null) {
            n = len;
            removed = true;
            i = expungeStaleEntry(i);
        }
    } while ( (n >>>= 1) != 0); 
    return removed;
}

getEntry

private Entry getEntry(ThreadLocal<?> key) {
    int i = key.threadLocalHashCode & (table.length - 1);
    Entry e = table[i];
    // entry[]中存在该值，且正好是要查询的key
    if (e != null && e.get() == key)
        return e;
    else
        // 表示存在哈希冲突，当前要找的Entry不在当前位置，则需要往右查找
        return getEntryAfterMiss(key, i, e);
}

getEntryAfterMiss

private Entry getEntryAfterMiss(ThreadLocal<?> key, int i, Entry e) {
    Entry[] tab = table;
    int len = tab.length;

    while (e != null) {
        ThreadLocal<?> k = e.get();
        // 找到对应的Entry直接返回
        if (k == key)
            return e;
        // 在找Entry途中发现到存在key为null的Entry则进行清理
        if (k == null)
            expungeStaleEntry(i);
        else
            // 当前下标没有找到想要的结果，接着往右遍历
            i = nextIndex(i, len);
        e = tab[i];
    }

    // 不存在该key对应的Entry
    return null;
}
