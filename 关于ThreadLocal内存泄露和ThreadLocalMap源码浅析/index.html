Entry类的内存泄露问题

Entry类

为什么要将 key 定义为弱引用

  原因是为了更好地回收对应的 value 值。这样说的原因是因为 ThreadLocalMap 的 set 和 get 方法是具有一定的清理能力的，其清理操作主要是针对 key 为 null 的 Entry（参考下面的源码分析），步骤是先往链表左边检索然后再从左往右清理其 key为 null 的 Entry。

  这里需要再次强调和回顾一下 ThreadLocal、ThreadLocalMap、Thread 三者的关系，因为这是最重要的理论基础。ThreadLocalMap 虽然是 ThreadLocal 的内部类，但实质 ThreadLocalMap是作用在 Thread 实例上的。简单地说每个 Thread 实例都有对应的一个 ThreadLocalMap ，而 ThreadLocalMap 可以用来存储多个 ThreadLocal 实例，这也是为什么有些人将 ThreadLocal 叫作线程变量的原因。

  在上面我们已经得知了有关弱引用实例的生命周期范围，因此简单地说是当 ThreadLocal 用完被手动置空的话（即丢失了强引用）那么这个 ThreadLocal 实例就只能活到第二次 gc 了。然而 ThreadLocal 被回收掉的话那么在下一次再调用 ThreadLocalMap 的 set 或 get 方法时其对应的 value 值就会被发现然后被置空释放。

  接下来我们假设 key 为强引用会发生什么事情。如果 key 被定义为强引用也就是说即使用户程序中手动置空了 ThreadLocal 实例 gc 都不会将其回收掉，因为对于强引用而言 JVM 宁可不惜抛出 OOM 都不会去回收被强应用的实例空间。所以结论就是如果 key 被定义为强引用就导致 ，ThreadLocal 在 Thread 生命周期结束之前无法被回收，从而间接地导致 value 也没法被回收，这样一来的话就会导致所谓的内存泄露。所以看到网上有些人说是弱引用导致的内存泄露，这显然是不正确的，因为弱引用有利于置空 Entry 的 key 从而避免内存泄露。

为什么出现内存泄露

  首先需要说的是 ThreadLocalMap 的生命周期与 Thread 绑定。所以当 Thread 被回收 ThreadLocalMap 很自然也就会一同回收，因此并不会出现什么内存泄露的问题。所以当前问题的关键点就是Thread 的生命周期范围，既如果 Thread 生命周期很长的话 ThreadLocalMap 就会存在内存泄露的风险。那么什么一种清理下会延长 Thread 的生命周期呢？答案就是线程池。在线程池中线程是可能长期驻留的，所以如果在线程池中使用 ThreadLocal 的话就会存在内存泄露的风险。

  还需要注意的是，虽然 ThreadLocalMap 的 set/get 方法具有一定的清理能力的，但需要强调这种清理能力是有限度的，因为它并不能保证所有 key 为 null 的 Entry 都被清理掉。例如 set 方法中如果当前操作的 ThreadLocal【没有发生冲突】或【发生了冲突但从当前下标开始散列表的右边不存在 key 为 null 的Entry】时，是不会往左遍历的。换句来话就是在这样的情况下左边即使有 key 为 null 的 Entry 都不会被发现然后清理掉。

最后可以得出如下结论：

  • 在线程池中使用 ThreadLocal 会存在内存泄露的风险
  • 虽然 ThreadLocalMap 的 set/get 方法有一定的清理能力，但我们不能依靠它来解决内存泄露问题
  • 之所以发生内存泄露的根本原因是 ThreadLocalMap 没有发现 key 为 null 的 Entry 从而无法置空 value 的引用，因此 gc 就没办法会都掉 value 的内存，因为 value 和 key 不一样，value 是强引用的。

如何解决内存泄露

  从上面我们得知发生内存泄露的主要原因是因为 ThreadLocalMap 没有发现 key 为 null 的 Entry 。所以解决内存泄露隐患的最好解决方案就是在使用完 ThreadLocal 之后主动地调用 ThreadLocal 的 remove 方法来明确让 ThreadLocalMap 释放对应的 value 空间。
  除此之外在网上我看到有人建议将 ThreadLocal 修饰成 static，我想这人大概的想法是提升 ThreadLocal 的生命周期范围使得其得到更好的管控。但事实上这种操作对内存泄露问题没有半点帮助，这是因为不管 ThreadLocal 的生命周期有多广，请不要忘记 ThreadLocal 本质上是一个线程变量，既即使是同一个 ThreadLocal 在面对不同的 Thread 时它所对应的 ThreadLocalMap 都是不同的，因此还是建议养成良好的习惯，既用完就显示调用 remove 释放 value 内存。
