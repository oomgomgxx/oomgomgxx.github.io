前言

推荐入门书籍：《Redis开发与运维》和《Redis实战》

Redis 命令参考：http://doc.redisfans.com/

Redis 文档：http://www.redis.cn/documentation.html

lua入门：https://moonbingbing.gitbooks.io/openresty-best-practices/

是什么？

Redis是一款C语言开发的键值对NoSQL

Redis具有以下特性：

  • 开源
  • 基于内存
  • 单线程
  • 支持多种数据类型
  • 基于 epoll 的多路复用IO模型
  • 跨平台且支持多种语言客户端

入门案例

下载和安装

官网地址：https://redis.io/

[下载并解压]

wget http://download.redis.io/releases/redis-5.0.5.tar.gz # 下载redis源码包
tar -zxvf redis-5.0.5.tar.gz -C ~/apps/   # 解压到指定位置

[安装]

cd ~/apps/redis-5.0.5 # 切换到解压的目录下
make # 编译源码（需要gcc-c++提供编译支持）
mkdir ～/app/redis # 创建一个用于存放安装内容的文件夹
make install PREFIX=/home/用户名称/apps/redis  #　安装到该文件夹下，注意要绝对路径
cp redis.conf ../redis # 将配置文件复制一份给安装好的redis

服务端的3种启动方式

[默认启动]

./bin/redis-server

[配置启动]

./redis-server --port 7777

[配置文件启动] - 推荐

bin/redis-server redis.conf # 可以在配置文件种添加 daemonize yes 配置以后台形式运行

停止服务端

bin/redis-cli shutdown

客户端连接服务端

bin/redis-cli # 或者添加参数指定： -h localhost -p 6379
# string类型基本操作
localhost:6379> set name zhangsan # 设置一个string类型的key value
OK
localhost:6379> get name # 根据key来获取value
"zhangsan"
localhost:6379>

实时监控

/bin/redis-cli monitor

常用key命令

  • keys：打印符合给定pattern的key
127.0.0.1:6379> keys *
1) "name"
127.0.0.1:6379> keys na*
1) "name"

  • scan、sscan、hscan、zscan 等命令可用于代替较为耗时的范围操作（譬如 keys、lrange、smembers 等等），其区别如下：

    • keys、lrange、smembers 等范围操作实时性强，但阻塞时间也长
    • scan 系列命令阻塞时间短，但有以下两个缺点
      1. 可能返回重复数据，去重操作需要应用程序解决
      2. 不能保证返回数据集合的实时性。譬如操作中途数据被新增或修改，scan命令可能无法马上感知到
    127.0.0.1:6379> scan 0
1) "17"
2)  1) "key:12"
    2) "key:8"
    3) "key:4"
    4) "key:14"
    5) "key:16"
    6) "key:17"
    7) "key:15"
    8) "key:10"
    9) "key:3"
    10) "key:7"
    11) "key:1"

redis 127.0.0.1:6379> scan 17
1) "0"
2) 1) "key:5"
   2) "key:18"
   3) "key:0"
   4) "key:2"
   5) "key:19"
   6) "key:13"
   7) "key:6"
   8) "key:9"
   9) "key:11"
  • dbsize：查询当前redis中有多少key
127.0.0.1:6379> set name1 zhangsan
OK
127.0.0.1:6379> set name2 lisi
OK
127.0.0.1:6379> dbsize
(integer) 2
  • exists：检查指定key是否存在
127.0.0.1:6379> exists name1
(integer) 1
127.0.0.1:6379> exists name3
(integer) 0
  • del：删除指定的key
127.0.0.1:6379> del name1
(integer) 1
127.0.0.1:6379> exists name1
(integer) 0
  • expire 和 pexpire：设置key的过期时间，单位默认为s和ms
127.0.0.1:6379> expire name2 30
(integer) 1
127.0.0.1:6379> pexpire name1 10000
(integer) 1
  • persist：清除key的过期设定
127.0.0.1:6379> set name1 zhangsan
OK
127.0.0.1:6379> expire name1 60
(integer) 1
127.0.0.1:6379> ttl name1
(integer) 57
127.0.0.1:6379> persist name1
(integer) 1
127.0.0.1:6379> ttl name1
(integer) -1
  • ttl 和 pttl：返回key的存活时间，ttl返回s，pttl返回ms
127.0.0.1:6379> set name1 zhangsan
OK
127.0.0.1:6379> expire name1 30
(integer) 1
127.0.0.1:6379> ttl name1
(integer) 23
127.0.0.1:6379> pttl name1
(integer) 20462
  • flushdb 和 flushall：删除当前db的所有key和删除所有db的所有key
127.0.0.1:6379> flushdb
OK
127.0.0.1:6379> flushall
OK
  • rename：修改key的名字
127.0.0.1:6379> keys *
(empty list or set)
127.0.0.1:6379> set name1 zhangsam
OK
127.0.0.1:6379> rename name1 name100
OK
127.0.0.1:6379> keys *
1) "name100"
  • type：查看key的数据类型
127.0.0.1:6379> type name100
string
  • object encoding：查看数据类型的内部编码
127.0.0.1:6379> object encoding name100
"embstr"

Redis支持的 6 种数据类型

图片源自：《Redis开发与运维》

string

基于动态字符串

  • 长度根据保存字符串而定，一般为8、16、32、64个字节
  • 可保存二进制字节流
  • 在 Redis 中所有key都是动态字符串类型
  • Redis 没有沿用C语言的字符串是因为想构建一个可修改的字符串，譬如append操作
  • set 和 get：设置和取值
127.0.0.1:6379> set name1 zhangsan
OK
127.0.0.1:6379> get name1
"zhangsan"
  • append：追加值
127.0.0.1:6379> append name1 -helloworld
(integer) 19
127.0.0.1:6379> get name1
"zhangsan-helloworld"
  • incr：自增
127.0.0.1:6379> set num1 1
OK
127.0.0.1:6379> incr num1
(integer) 2
127.0.0.1:6379> get num1
"2"
  • decr：自减
127.0.0.1:6379> get num1
"2"
127.0.0.1:6379> decr num1
(integer) 1
127.0.0.1:6379> get num1
"1"
  • incrby 和 decrby：增加和自减指定的值
127.0.0.1:6379> set num1 10
OK
127.0.0.1:6379> incrby num1 90
(integer) 100
127.0.0.1:6379> get num1
"100"
127.0.0.1:6379> decrby num1 80
(integer) 20
127.0.0.1:6379> get num1
"20"
  • setnx：只有当key不存在的时候才会设置成功
127.0.0.1:6379> get num1
"1"
127.0.0.1:6379> setnx num1 100
(integer) 0 # 因为num1存在所以设置结果为0（false）
127.0.0.1:6379> get num1
"1"
  • setex：强制设入值并指定超时时间
127.0.0.1:6379> get num1
"1"
127.0.0.1:6379> setex num1 60 1000 # 存在则覆盖
OK
127.0.0.1:6379> get num1
"1000"
127.0.0.1:6379> ttl num1
(integer) 52
  • mset 和 mget：批量设置和批量获取
127.0.0.1:6379> mset name1 zhangsam name2 lisi name3 wangwu
OK
127.0.0.1:6379> mget name1 name2 name3
1) "zhangsam"
2) "lisi"
3) "wangwu"
  • strlen：值长
127.0.0.1:6379> mget name1 name2 name3
1) "zhangsam"
2) "lisi"
3) "wangwu"
127.0.0.1:6379> strlen name1
(integer) 8
127.0.0.1:6379> strlen name2
(integer) 4
  • getset：设置新值并返回旧值（覆盖式）
127.0.0.1:6379> getset name4 zhaoliu
(nil)
127.0.0.1:6379> getset name4 zhaoliu
"zhaoliu"
127.0.0.1:6379> getset name4 zhaoliu1
"zhaoliu"
127.0.0.1:6379> getset name4 zhaoliu2
"zhaoliu1"
  • setrange 和 getrange：获取和设置指定下标的字符（覆盖式）
127.0.0.1:6379> set name helloworld
OK
127.0.0.1:6379> getrange name 0 4
"hello"
127.0.0.1:6379> setrange name 5 zhangsan
(integer) 13
127.0.0.1:6379> get name
"hellozhangsan"

list（有序）

基于双向链表或压缩列表

  • 压缩队列是一种节约内存的特殊双端链表。两端操作为O(1)，push/pop
  • 双向链表最优操作为O(1)（push/pop），平均为O(n)
  • 集合元素较多，或元素长度较长时使用双向链表
  • rpush 和 lpush：向右插入值和向左插入值
127.0.0.1:6379> rpush nums 1 2 3
(integer) 3
127.0.0.1:6379> lrange nums 0 -1
1) "1"
2) "2"
3) "3"
127.0.0.1:6379> lpush nums 0
(integer) 4
127.0.0.1:6379> lrange nums 0 -1
1) "0"
2) "1"
3) "2"
4) "3"
127.0.0.1:6379> rpush nums 4
(integer) 5
127.0.0.1:6379> lrange nums 0 -1
1) "0"
2) "1"
3) "2"
4) "3"
5) "4"
  • linsert：在指定元素前或后插入元素
127.0.0.1:6379> lrange nums 0 -1
1) "0"
2) "1"
3) "2"
4) "3"
5) "4"
127.0.0.1:6379> linsert nums before 0 hello # before/after
(integer) 6
127.0.0.1:6379> lrange nums 0 -1
1) "hello"
2) "0"
3) "1"
4) "2"
5) "3"
6) "4"
  • lindex：获取指定下标的元素
127.0.0.1:6379> lindex nums 0
"hello"
  • llen：获取列表的长度
127.0.0.1:6379> llen nums
(integer) 6
  • rpop 和 lpop：弹出右边和弹出左边的元素
127.0.0.1:6379> lrange nums 0 -1
1) "hello"
2) "0"
3) "1"
4) "2"
5) "3"
6) "4"
127.0.0.1:6379> lpop nums
"hello"
127.0.0.1:6379> rpop nums
"4"
127.0.0.1:6379> lrange nums 0 -1
1) "0"
2) "1"
3) "2"
4) "3"
  • lrem：删除指定的元素
    • count > 0 ：从头起删除count个
    • count < 0 ：从尾起删除count个
    • count = 0 ：删除所有
127.0.0.1:6379> rpush nums 1 1 1 2 2 2 3 3 3
(integer) 9
127.0.0.1:6379> lrem nums 3 2 # 从头开始删除3个2
(integer) 3
127.0.0.1:6379> lrange nums 0 -1
1) "1"
2) "1"
3) "1"
4) "3"
5) "3"
6) "3"
127.0.0.1:6379> lrem nums -3 3 # 从尾开始删除3个3
(integer) 3
127.0.0.1:6379> lrange nums 0 -1
1) "1"
2) "1"
3) "1"
  • ltrim：修剪列表
127.0.0.1:6379> rpush nums 1 1 1 2 2 2 3 3 3
(integer) 9
127.0.0.1:6379> ltrim nums 3 5 # 保留下标从[3,5]的元素
OK
127.0.0.1:6379> lrange nums 0 -1
1) "2"
2) "2"
3) "2"
  • lset：修改指定下标的元素
127.0.0.1:6379> lrange nums 0 -1
1) "2"
2) "2"
3) "2"
127.0.0.1:6379> lset nums 0 1
OK
127.0.0.1:6379> lrange nums 0 -1
1) "1"
2) "2"
3) "2"
  • blpop 和 brpop：阻塞式弹出
127.0.0.1:6379> lrange nums 0 -1
1) "1"
2) "2"
3) "2"
127.0.0.1:6379> blpop nums 60 # 阻塞式弹出（左），超时时间为60s
1) "nums"
2) "1"
127.0.0.1:6379> lrange nums 0 -1
1) "2"
2) "2"
127.0.0.1:6379> brpop nums 60 # 阻塞式弹出（右），超时时间为60s
1) "nums"
2) "2"
127.0.0.1:6379> lrange nums 0 -1
1) "2"

hash

  1. 基于散列表或压缩列表
  2. 压缩队列是一种节约内存的特殊双端链表。两端操作为O(1)，push/pop
  3. 散列表最优的情况下时间复杂度为O(1)
  4. 集合元素较少，或元素长度较短时使用压缩列表
  • hset 和 hget：设置和获取值
127.0.0.1:6379> hset person:1 name zhangsan # 设置person1的name
(integer) 1
127.0.0.1:6379> hset person:1 age 23 # 设置person1的age
(integer) 1
127.0.0.1:6379> hget person:1 name
"zhangsan"
127.0.0.1:6379> hget person:1 age
"23"
  • hdel：删除指定key的属性
127.0.0.1:6379> hget person:1 age
"23"
127.0.0.1:6379> hdel person:1 age
(integer) 1
127.0.0.1:6379> hget person:1 age
(nil)
  • hlen：获取指定key的属性长度
127.0.0.1:6379> hlen person:1
(integer) 1
127.0.0.1:6379> hset person:1 age 23
(integer) 1
127.0.0.1:6379> hlen person:1
(integer) 2
  • hmset 和 hmget：批量设入属性和获取属性
127.0.0.1:6379> hmset person:1 name zhangsan age 23
OK
127.0.0.1:6379> hmget person:1 name age
1) "zhangsan"
2) "23"
  • hexists：判断指定key的属性是否存在
127.0.0.1:6379> hmget person:1 name age
1) "zhangsan"
2) "23"
127.0.0.1:6379> hexists person:1 name
(integer) 1
127.0.0.1:6379> hexists person:1 city
(integer) 0
  • hkeys：获取指定key的所有属性名称
127.0.0.1:6379> hkeys person:1
1) "name"
2) "age"
  • hvals：获取指定key的所有属性值
127.0.0.1:6379> hkeys person:1
1) "name"
2) "age"
127.0.0.1:6379> hvals person:1
1) "zhangsan"
2) "23"
  • hincrby：指定属性自增
127.0.0.1:6379> hkeys person:1
1) "name"
2) "age"
127.0.0.1:6379> hincrby person:1 age 10
(integer) 33
127.0.0.1:6379> hget person:1 age
"33"
  • hstrlen：值长
127.0.0.1:6379> hget person:1 age
"33"
127.0.0.1:6379> hstrlen person:1 age
(integer) 2

set

基于散列表或整数集合

  • 散列表最优的情况下时间复杂度为O(1)
  • 集合只有正数或集合不大时，使用整数集合
  • sadd 和 smembers 和 sismember：添加元素、列出所有元素、是否存在某个元素
127.0.0.1:6379> sadd myset a b c d
(integer) 4
127.0.0.1:6379> smembers myset
1) "c"
2) "d"
3) "b"
4) "a"
127.0.0.1:6379> sismember myset c
(integer) 1
  • srem：移除集合中的元素
127.0.0.1:6379> sadd myset a b c d
(integer) 4
127.0.0.1:6379> smembers myset
1) "c"
2) "d"
3) "b"
4) "a"
127.0.0.1:6379> srem myset b
(integer) 1
127.0.0.1:6379> smembers myset
1) "c"
2) "d"
3) "a"
  • scard：获取集合当前元素个数
127.0.0.1:6379> sadd myset a b c d
(integer) 4
127.0.0.1:6379> scard myset
(integer) 4
  • srandmember：随机返回集合中指定个数的元素
127.0.0.1:6379> srandmember myset 2
1) "a"
2) "b"
127.0.0.1:6379> srandmember myset 2
1) "d"
2) "b"
  • spop：随机弹出一个集合元素
127.0.0.1:6379> spop myset
"b"
127.0.0.1:6379> smembers myset
1) "c"
2) "d"
3) "a"
  • sinter：求集合之间的交集
127.0.0.1:6379> sadd myset a b c d e
(integer) 5
127.0.0.1:6379> sadd myset2 e d f g
(integer) 4
127.0.0.1:6379> sinter myset myset2
1) "d"
2) "e"
  • sunion：求集合之间的并集（去重）
127.0.0.1:6379> sadd myset a b c d e
(integer) 5
127.0.0.1:6379> sadd myset2 e d f g
(integer) 4
127.0.0.1:6379> sunion myset myset2
1) "c"
2) "d"
3) "g"
4) "a"
5) "b"
6) "f"
7) "e"
  • sdiff：求集合之间的差集
127.0.0.1:6379> sadd myset a b c d e
(integer) 5
127.0.0.1:6379> sadd myset2 e d f g
(integer) 4
127.0.0.1:6379> sdiff myset myset2
1) "c"
2) "b"
3) "a"
  • smove：将一个元素移动到另外的集合
127.0.0.1:6379> sadd myset a b c
(integer) 3
127.0.0.1:6379> smembers myset2
(empty list or set)
127.0.0.1:6379> smove myset myset2 b
(integer) 1
127.0.0.1:6379> smembers myset2
1) "b"

zset（有序）

基于压缩列表或跳表

  • 压缩队列是一种节约内存的特殊双端链表。两端操作为O(1)，push/pop

  • 跳表的平均操作为O(logn)，最差时为O(n)

  • 集合元素较多或元素较长，使用跳表

  • zadd 和 zrange：添加元素、返回指定下标区间的元素
127.0.0.1:6379> zadd myzset 1 a 2 b 3 c # 排序分数越小越靠前
(integer) 3
127.0.0.1:6379> zrange myzset 0 -1
1) "a"
2) "b"
3) "c"
  • zcard：返回集合元素的个数
127.0.0.1:6379> zcard myzset
(integer) 3
  • zscore：返回元素的排序分数
127.0.0.1:6379> zadd myzset 10 a 1 b 2 c
(integer) 3
127.0.0.1:6379> zrange myzset 0 -1
1) "b"
2) "c"
3) "a"
127.0.0.1:6379> zscore myzset a
"10"
  • zincrby：增加元素的分数（没有元素先创建）
127.0.0.1:6379> zscore myzset b
"2"
127.0.0.1:6379> zincrby myzset 10 b
"12"
127.0.0.1:6379> zscore myzset b
"12"
  • zrevrange：获取指定下标范围分数从高到低排序的元素
127.0.0.1:6379> zincrby likecount 2 num1  # 文章编号为num1的文章获取了2个赞
"2"
127.0.0.1:6379> zincrby likecount -1 num1  # 文章编号为num1的文章减少了1个赞
"1"
127.0.0.1:6379> zadd likecount 1 num2 # 文章编号为num2的文章获取了1个赞
(integer) 1
127.0.0.1:6379> zincrby likecount 1 num1  # 文章编号为num1的文章减少了1个赞
"2"
127.0.0.1:6379> zrevrange likecount 0 9 # 按照分数的排序获取0到9下标的元素
1) "num1"
2) "num2"
  • zcount：返回指定分数范围的元素个数
127.0.0.1:6379> zcount myzset 0 5
(integer) 3
  • zrangebyscore：返回指定分数范围的元素
127.0.0.1:6379> zcount myzset 0 5
(integer) 3
127.0.0.1:6379> zrangebyscore myzset 0 5
1) "a"
2) "c"
3) "d"
  • zremrangebyscore：删除指定分数范围的元素
127.0.0.1:6379> zremrangebyscore myzset 0 5
(integer) 3
127.0.0.1:6379> zrangebyscore myzset 0 5
(empty list or set)
  • zrank：返回元素的排序下标
127.0.0.1:6379> zadd myzset 10 a 1 b 2 c
(integer) 3
127.0.0.1:6379> zrange myzset 0 -1

事务相关

简介

  • Redis 事务和传统关系型数据的 ACID 事务不同，它只能保证原子性，且不提供回滚操作
  • Redis 提供的原子性概念是：事务中的命令要么全部被执行，要么全部都不执行。不难理解，Redis 的原子性并并不是 ACID 中的原子性（要么都成功，要么都失败），它强调的是执行与否
  • Redis 提供以下命令实现事务操作：
    • multi：该命令执行后表示客户端将要提交一系列操作，而这些操作将会在 exec 时按队列形式一同执行。注意，命令在 exec 前会先暂存到队列中
    • watch：乐观锁，用于监听key。即如果 exec 之前发现 key 被修改，事务将失败
    • discard：丢弃暂存队列中的命令
    • exec：执行暂存队列中的命令

基本用法

multi 和 exec 用法

127.0.0.1:6379> multi
OK
127.0.0.1:6379> set num 1
QUEUED
127.0.0.1:6379> incr num
QUEUED
127.0.0.1:6379> exec
1) OK
2) (integer) 2

127.0.0.1:6379> set num 1
OK
127.0.0.1:6379> watch num # 监听num
OK
127.0.0.1:6379> multi
OK
127.0.0.1:6379> incr num 
QUEUED

# 客户端2 ----
127.0.0.1:6379> incr num
(integer) 2

# 客户端1 ----
127.0.0.1:6379> incr num 
QUEUED
127.0.0.1:6379> exec
(nil) # num被修改事务失效
127.0.0.1:6379> get num
"2"

不难看出，如果 exec 之前入队的命令发生错误，则会抛弃该命令，而 exec 依然能够执行所有正常入队的命令

数据持久化策略

Redis 提供了两种数据持久化方案来，分别是：RDB（默认）、AOF。两者主要的区别在于 RDB 持久化的是内存快照，而 AOF 持久化的是修改数据的命令操作（类似于 MySQL 的二进制日志）

RDB（默认）

RDB，即内存快照持久化方案，采用写时复制实现，因为属于全量复制操作，所以频繁进行 RDB 备份容易导致 Redis 性能下降，而且数据量越大RDB备份消耗越高。

触发 RDB 的方法有2种：

  • 手动触发
  • 自动触发

手动触发

save命令

  • 阻塞当前服务线程后持久化内存中快照，直到持久化完成

bgsave命令

  • 当前服务线程 Fork 出一条子线程来做持久化操作。但要注意，Fork 操作是阻塞的，但相对于 save 来说 bgsave 效率更高
  • 注意： Fork 持久化 RDB 的用时是和数据量是正相关的

自动触发

情况1：满足服务端配置的 save 属性条件后会自动触发

save 900 1 # 表示900s内触发1次修改，则自动触发bgsave
save 300 10 # 表示300s内触发10次修改，则自动触发bgsave
save 60 10000 # 表示60s内触发10000次修改，则自动触发bgsave
# 设置多个save条件是为了适应不同的情况，为了在更加合适的条件下持久化数据

情况2：主从复制情况下，从节点进行全量复制时自动触发bgsave

情况3：客户端执行debug reload命令刷新 Redis 时自动触发bgsave

情况4：没有开启AOF的情况下，客户端执行 shutdown 命令时自动触发bgsave

RDB优缺点

  • 恢复速度比 AOF 快
  • 可以代表某一个时间点的数据状态，且因为是紧凑的二进制文件所以较为节省空间
  • 因为是条件触发，所以在没满足持久化条件前出现异常则会丢失数据
  • Fork操作属于系统操作，因此频繁的 bgsave 会导致 Redis 性能下降
  • 可能会出现旧版本 Redis 无法兼容新版本 RDB 文件的情况，从而导致无法恢复数据的问题

AOF

AOF（append only file）是一种写后日志，优点是不会阻塞当前命令执行（但会阻塞后面命令）以及不会记录错误的命令，但因为是写后日志所以存在丢失更新风险

aof流程

aof使用

默认情况下 AOF 是没有启用的，需要在服务端配置文件中做如下配置

appendonly yes # 开启AOF功能
appendfilename "appendonly.aof" # 指定AOF文件

# 同步策略
# appendfsync always # 每条命令都立即fsync到文件中。数据做到高可靠，但影响性能
appendfsync everysec # 每1秒从缓存中fsync一次。可靠和性能的折中，但如果落盘不及时就会丢失1秒内所有的操作
# appendfsync no # 交给操作系统处理。性能最好，但可能性相对最低

使用注意

  • everysec 相对于 always 来说属于延时写入，这样有利于提高操作系统缓冲区使用率，其原理是调用系统write 命令后就会立即返回而无需进行 fsync 操作，因此能够一定程度提高服务的吞吐量。

  • always 表示每条修改数据的命令执行后都需要马上持久化。这样做能够尽可能地降低丢失数据的风险，但频繁的持久化操作会造成较大的开销，而且不能有效地使用缓冲因，此吞吐量会低于 everysec 模式

关于aof重写

随着时间的推移 aof 文件会变得非常大，因此 Redis 提供了aof重写操作以缩减 aof 备份数据文件的大小。做法是：去除无效命令、命令合并、去除超时数据 等操作

aof重写触发条件：

  • 手动触发：执行 bgrewriteaof 命令
  • 自动触发：
  • auto-aof-rewrite-percentage：当前文件大小和上一次重写后的文件大小的比值大于该值时触发
    • auto-aof-rewrite-min-size：大于该属性值时触发

aof 重写流程

当发生aof重写时主进程程会 Fork 出一条后台子进程以及产生一个aof重写时缓冲，其中aof重写时缓冲专门用来保存在重写过程中产生的新命令，而子进程则用于处理旧的aof文件以及将结果追加到新aof文件中。子进程通过信号通知主线程，主线程得知信息后就会将aof重写时缓冲和旧aof文件的处理结果内容追加到新aof文件中。

为什么单线程还那么快？

原因

  • 基于内存操作
  • 使用全局哈希表进行定位，其时间查询复杂度可达O(1)
    • 注意，Redis采用全局哈希表定位元素，而“元素”并非真实值而是一个指针，因此实际操作的时间复杂度还要看具体（指针）对应的元素所用的数据结构
  • 数据结构操作高效。除了范围查询这种较为耗时的操作外（O(n)），其他操作一般可达到O(1)，譬如单一的元素操作以及统计操作等，而范围查询也可以采用SCAN系列命令代替，因为SCAN可以将发批量数据分为小批量操作从而减少阻塞时间
  • 单线程写操作，避免了高并发时大量的上下文切换以及锁资源竞争带来的耗时
  • 采用 Reactor 线程模型，即基于多路复用IO模型提高网络请求的处理效率
  • 客户端和服务端采用简单高效的 RESP 协议通信

注意

  1. 虽然 Redis 6.x 引入了多线程来处理网络IO，但 Redis 在执行写操作时依然是由主线程单独来完成的，而网络IO处理线程只负责处理读操作和回写操作
  2. 在 Redis 6.x 前，我们一直说 Redis 是单线程的，其实这里的单线程指的是 Redis 在执行命令时是单线程的，而并非指 Redis 的所有操作都只有一条线程来完成。这并不难理解，譬如持久化、节点数据同步等都会额外fork出新的进程来处理的（在Linux中线程被称为轻量级进程）

关于分布式缓存

缓存剔除策略

  1. 算法剔除（FIFO、LRU、LFU、最小TTL）
  2. 超时剔除（expire命令）
  3. 主动更新（代码控制，推荐）

相关的配置

配置最大可用内存。默认为0，表示不设置限制

maxmemory <bytes>

指定当 maxmemory 满了时，使用什么策略来淘汰缓存

maxmemory-policy noeviction

  • volatile-lru：从具有过期设置（expire）的 key 中，使用 LRU 剔除（最近最少使用）

  • volatile-lfu：从具有过期设置（expire）的 key 中，使用 LFU 剔除（最少命中）

  • volatile-random：从具有过期设置（expire）的 key 中，随机剔除

  • volatile-ttl：从具有过期设置（expire）的 key 中，剔除 ttl 时间较小的

  • allkeys-lru（推荐）：使用 LRU 剔除缓存后再保存新的缓存

  • allkeys-lfu：使用 LFU 剔除缓存后再保存新的缓存

  • allkeys-random：随机剔除缓存

  • noeviction：不剔除，只在写操作时返回错误告知

常用缓存剔除/淘汰算法

指定每次对比样本的数量

maxmemory-samples 5
  • 因为 Redis 实现的LRU、LFU、以及最小TTL等算法都不是精准的（这时为了减轻数据淘汰对缓存性能的影响），因此提供了 maxmemory-samples 用于指定剔除缓存时用来对比的样本数
  • Redis 会在第一次剔除缓存时生成样本集合，然后让将要剔除的缓存与集合中访问时间戳最小的缓存进行对比，如果少于则剔除。（Redis的LRU实现默认会记录每个数据的最近一次访问的时间戳 ）
  • 该值越大越精准，但也更消耗CPU

指定副本节点复制缓存时是否忽略 maxmemory 设置

replica-ignore-maxmemory yes
  • 非常重要的是，Redis 5.x开始replica-ignore-maxmemory默认为yes，表示副本在复制时将会忽略主节点的maxmemory设置，换句话来说就是副本节点不会对剔除缓存。

缓存容量设置多少合适？

根据二八定律，可将缓存大小设置为总数量的20%~30%。

二八定律指的是在总数量中通常是20%占据了80%的访问量，而其余的80%虽然是大多数，但访问率却不高。

分布式缓存需要注意问题

缓存穿透

缓存穿透指的是访问了不存在的数据，导致间接绕过了缓存层（因为数据根本不存在）。出现这种情的通常原因是发生了误操作删除了数据或恶意攻击。

解决方法：

  • DB没数据时，让其缓存空值，避免再访问数据库
    • 最好给这个空值设置超时时间，这样可以避免大量空值占用空间
  • 使用布隆过滤器进行缓存预判
    - 原理是在缓存数据时，将缓存对应的 key 值通过哈希函数将其映射到布隆过滤器上，以示存在该缓存。然后在查询时，使用同样的哈希函数对 key 进行映射判断，如果已存在则访问缓存或数据库，如果不存在则直接返回请求而不访问缓存和数据库
    - 注意，布隆过滤器通常采用一个或多个哈希函数对 key 值进行映射。而因为存在哈希冲突的情况，因此存在一定的误判性，即可能出现不存在 key 的缓存，也能访问缓存层和数据的情况。而且这种误判可能性会随着数据库增大而增大
    - 在 Google Guava 工具库中提供了对 BloomFilter 的相关实现
    - [BloomFilter 原理展示](https://llimllib.github.io/bloomfilter-tutorial/)
    • 对请求进行合法性检测，排除恶意请求

无底洞问题

无底洞问题指的是当时 Facebook 用的 Memcache 节点数已多达3000个了，开发人员发现水平扩容节点不但没有得来想要的性能，反而降低了性能。

其原因主要是 key 散乱在不同的缓存节点上，导致了较大的网络的开销。所以预防这个问题基本思路是，减少网络通讯的次数以及让相关的缓存落在相同的节点上，所以可以使用 管道、mget、m* 这样的命令来解决。

缓存击穿（热点key失效）

缓存击穿这个名字个人认为并不恰当。缓存击穿指的是当大量并发访问同一个key的时候，这个key恰好失效了，导致所有并发请求落到数据库从而带来访问压力。因此“热点key失效”这个称呼更加合适。

解决的思路：

  • 恢复缓存时采用互斥处理，令其他线程进入等待，让最先发现缓存失效的线程将其恢复
  • 热点缓存不过期，但需要主动更或定时更新缓存以保持一致性

缓存雪崩

缓存雪崩指的是缓存层失效。譬如宕机或是大量缓存同时失效，从而导致大量请求落入数据库

图片源自《Redis开发与运维》

解决方法：

  • 缓存层的高可用，避免缓存层节点不可用
  • 错开缓存的超时时间，避免设置同样的超时值
  • 当出现雪崩时，后端进行服务降级，让非核心数据返回预设值
  • 当出现雪崩时，后端进行请请求限流保护系统
  • 提前测试缓存雪崩后的应对能力

关于Redis的内存碎片问题

什么是内存碎片和造成内存碎片的原因是什么

内存碎片可以简单理解为无法使用的内存空缺。产生内存碎片的原因主要有2个：

  1. 受系统的内存分配器影响，例如频繁进行大小不定的内存分配且（例如大量修改value值，且value值前后大小差异较大）
  2. 大量增删操作，因为删除key后并不会马上回收内存。

拓展知识

  1. 内存池是一种内存分配方式。通常我们习惯直接使用new、malloc等API申请分配内存，这样做的缺点在于：由于所申请内存块的大小不定，当频繁使用时会造成大量的内存碎片并进而降低性能。— 百度百科

  2. 因为Redis没有实现自己的内存池，因此需要依赖系统自身的内存分配器（如Linux上的glibc库）来进行内存的管理。换句话来说，操作系统内存分配器性能和处理内存碎片的能力对Redis有一定能影响。

  3. 在较新版本的Redis中使用 Jemalloc代替了系统默认的内存分配器。（在Redis5.x中Jemalloc也得到了升级，对内存分配和碎片整理更加高效）

  4. 主动回收内容有两种情况：key过时（expire）、内容到达maxmemory触发驱逐策略

查看是否存在内存碎片

命令：info memory

图片源自《Redis开发与运维》

可以看到内存碎片的公式为：内存碎片 = 进程占用的物理内存总量 / 实际数据内存占用量

该值最好的状态是接近1，该值越大表示存在内存碎片越多。

解决内存碎片问题方案

Redis4.x 版本之前，只能通过重启 Redis 解决，即回收再分配

Redis4.x 版本之后，可通过设置activedefrag yes开启自动整理内存碎片功能，此后 Redis 会尽最大努力解决内存碎片问题。

        return connection.set("hi".getBytes(), "test setnx".getBytes(),
                              Expiration.seconds(30), // 过期时间
                              RedisStringCommands.SetOption.SET_IF_ABSENT); // 设置NX
    });

    System.out.println(execute);
}
  • StringRedisTemplate继承自RedisTemplate
  • StringRedisTemplate默认使用StringRedisSerializer进行序列化，而RedisTemplate默认使用JdkSerializationRedisSerializer
