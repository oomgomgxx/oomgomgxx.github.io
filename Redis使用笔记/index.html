前言

推荐入门书籍：《Redis开发与运维》和《Redis实战》

Redis 命令参考：http://doc.redisfans.com/

Redis 文档：http://www.redis.cn/documentation.html

lua入门：https://moonbingbing.gitbooks.io/openresty-best-practices/

是什么？

Redis是一款C语言开发的键值对（key-value）NoSQL。

Redis具有以下特性：

  • 开源
  • 基于内存
  • 单线程
  • 支持多种数据类型
  • 跨平台且支持多种语言客户端

入门案例

下载和安装

官网地址：https://redis.io/

[下载并解压]

wget http://download.redis.io/releases/redis-5.0.5.tar.gz # 下载redis源码包
tar -zxvf redis-5.0.5.tar.gz -C ~/apps/   # 解压到指定位置

[安装]

cd ~/apps/redis-5.0.5 # 切换到解压的目录下
make # 编译源码（需要gcc-c++提供编译支持）
mkdir ～/app/redis # 创建一个用于存放安装内容的文件夹
make install PREFIX=/home/用户名称/apps/redis  #　安装到该文件夹下，注意要绝对路径
cp redis.conf ../redis # 将配置文件复制一份给安装好的redis

服务端的3种启动方式

[默认启动]

./bin/redis-server

[配置启动]

./redis-server --port 7777

[配置文件启动] - 推荐

bin/redis-server redis.conf # 可以在配置文件种添加 daemonize yes 配置以后台形式运行

停止服务端

bin/redis-cli shutdown

客户端连接服务端

bin/redis-cli # 或者添加参数指定： -h localhost -p 6379
# string类型基本操作
localhost:6379> set name zhangsan # 设置一个string类型的key value
OK
localhost:6379> get name # 根据key来获取value
"zhangsan"
localhost:6379>

实时监控

/bin/redis-cli monitor

常用key命令

keys：打印符合给定pattern的key

127.0.0.1:6379> keys *
1) "name"
127.0.0.1:6379> keys na*
1) "name"

dbsize：查询当前redis种有多少key

127.0.0.1:6379> set name1 zhangsan
OK
127.0.0.1:6379> set name2 lisi
OK
127.0.0.1:6379> dbsize
(integer) 2

exists：检查指定key是否存在

127.0.0.1:6379> exists name1
(integer) 1
127.0.0.1:6379> exists name3
(integer) 0

del：删除指定的key

127.0.0.1:6379> del name1
(integer) 1
127.0.0.1:6379> exists name1
(integer) 0

expire和pexpire：设置key的过期时间，单位默认为s和ms

127.0.0.1:6379> expire name2 30
(integer) 1
127.0.0.1:6379> pexpire name1 10000
(integer) 1

persist：清除key的过期设定

127.0.0.1:6379> set name1 zhangsan
OK
127.0.0.1:6379> expire name1 60
(integer) 1
127.0.0.1:6379> ttl name1
(integer) 57
127.0.0.1:6379> persist name1
(integer) 1
127.0.0.1:6379> ttl name1
(integer) -1

ttl和pttl：返回key的存活时间，ttl返回s，pttl返回ms

127.0.0.1:6379> set name1 zhangsan
OK
127.0.0.1:6379> expire name1 30
(integer) 1
127.0.0.1:6379> ttl name1
(integer) 23
127.0.0.1:6379> pttl name1
(integer) 20462

flushdb和flushall：删除当前db的所有key和删除所有db的所有key

127.0.0.1:6379> flushdb
OK
127.0.0.1:6379> flushall
OK

rename：修改key的名字

127.0.0.1:6379> keys *
(empty list or set)
127.0.0.1:6379> set name1 zhangsam
OK
127.0.0.1:6379> rename name1 name100
OK
127.0.0.1:6379> keys *
1) "name100"

type：查看key的数据类型

127.0.0.1:6379> type name100
string

object encoding：查看数据类型的内部编码

127.0.0.1:6379> object encoding name100
"embstr"

stream（redis 5.x）

xadd：添加stream

格式：xadd key ID field string

# 使用自动生成id：*
127.0.0.1:6379> xadd test:person * name zhangsan
"1571756285397-0"
127.0.0.1:6379> xadd test:person * age 18
"1571756296247-0"
# 自定义id
127.0.0.1:6379> xadd test2:person 0-1 name lisi age 19
"0-1"

xlen：查看key的长度

127.0.0.1:6379> xlen test:person
(integer) 2

xdel：按照key和id删除一个stream

127.0.0.1:6379> xdel test2:person 0-1
(integer) 1

xread：查看指定stream【后】的1个或多个stream

格式：xread [COUNT count] [BLOCK ms] streams key [key…] id [id…]

# 查看key为test:person且id为1571756285397-0后的stream
127.0.0.1:6379> xread streams test:person 1571756285397-0
1) 1) "test:person"
   2) 1) 1) "1571756296247-0"
         2) 1) "age"
            2) "18"
# 添加一个stream
127.0.0.1:6379> xadd test:person * city qy
"1571757314013-0"
# 查看key为test:person且id为1571756285397-0后的stream
127.0.0.1:6379> xread streams test:person 1571756285397-0
1) 1) "test:person"
   2) 1) 1) "1571756296247-0"
         2) 1) "age"
            2) "18"
      2) 1) "1571757314013-0"
         2) 1) "city"
            2) "qy"     
# 查看指定数量的stream
127.0.0.1:6379> xread COUNT 1 streams test:person 1571756285397-0
1) 1) "test:person"
   2) 1) 1) "1571756296247-0"
         2) 1) "age"
            2) "18"

xrange：查看范围stream

格式：xrange key startid endid

特殊字符：+表示最大，-表示最小

# 查看从1571756285397-0到1571756296247-0
127.0.0.1:6379> xrange test:person 1571756285397-0 1571756296247-0
1) 1) "1571756285397-0"
   2) 1) "name"
      2) "zhangsan"
2) 1) "1571756296247-0"
   2) 1) "age"
      2) "18"
# 从1571756285397-0开始查看
127.0.0.1:6379> xrange test:person 1571756285397-0 +
1) 1) "1571756285397-0"
   2) 1) "name"
      2) "zhangsan"
2) 1) "1571756296247-0"
   2) 1) "age"
      2) "18"
# 查看所有

事务相关

  redis的事务和传统的ACID事务不同，redis提供的事务只能保证命令的要么都执行要么都不执行，且事务就算出错了也没办法自动回滚。

  redis提供了multi和exec两个命令来表示事务。用法如下：

# 第1次事务
127.0.0.1:6379> multi
OK
127.0.0.1:6379> set num 1
QUEUED
127.0.0.1:6379> incr num 1
(error) ERR wrong number of arguments for 'incr' command
127.0.0.1:6379> incr num
QUEUED
127.0.0.1:6379> exec
(error) EXECABORT Transaction discarded because of previous errors.
# 第2次事务
127.0.0.1:6379> multi
OK
127.0.0.1:6379> set num 1
QUEUED
127.0.0.1:6379> incr num
QUEUED
127.0.0.1:6379> exec
1) OK
2) (integer) 2

可以看出，如果事务途中出现错误那么事务就会丢失（不成功），其次是redis的事务其实只是相当于将单个命令转为单个队列（包含多个命令）执行。

  其中，redis还提供了一个watch命令，其中能类似于乐观锁，用于监听key的改变，一旦监听到key的改变事务就会失败。用法如下：

127.0.0.1:6379> set num 1
OK
127.0.0.1:6379> watch num # 监听num
OK
127.0.0.1:6379> multi
OK
127.0.0.1:6379> incr num 
QUEUED
# 客户端2 ----
127.0.0.1:6379> incr num
(integer) 2
# 客户端2 ----
127.0.0.1:6379> incr num 
QUEUED
127.0.0.1:6379> exe
(nil) # 因为watch监听到由其他客户端修改num，所以当前事务失效
127.0.0.1:6379> get num
"2"

  当然，除了上面提到的 multi 和 exec 之外，redis还支持lua脚本。一段lua脚本相当于一条redis命令，且lua脚本内置redis函数，可以直接在lua脚本种调用redis命令。

127.0.0.1:6379> set str redis
OK
127.0.0.1:6379> eval 'return "hello - " .. redis.call("get",KEYS[1])' 1 str
"hello - redis"

除了eval命令可以执行lua脚本之外，redis还提供了evalsha，evalsha和eval的区别在于evalsha可以将lua脚本传递给服务端让其得到复用从而客户端降低每次都需要发送lua脚本的开销。

使用场景

两种持久化策略

  Redis提供了两种持久化数据方案来避免进程退出而导致内存中的数据丢失。分别是：RDB（默认）、AOF。两者主要的区别在于RDB持久化的是内存镜像，而AOF持久化的是修改数据的命令操作（类似于MySQL的二进制日志）。

RDB（默认）

触发RDB的方法分为：手动触发和自动触发；

[手动触发]

  • save命令：阻塞当前redis服务线程来持久化内存中的数据直到持久化完成。
  • bgsave命令：redis服务线程fork出一条子进程来做持久化操作。但需要注意的是fork这个动作是阻塞的，不过显然的是bgsave比save在生产环境中更加高效。

[自动触发]

情况1：满足服务端配置的save属性条件自动触发，例如：

save 900 1 # 表示900s内触发1次修改，则自动触发bgsave
save 300 10 # 表示300s内触发10次修改，则自动触发bgsave
save 60 10000 # 表示60s内触发10000次修改，则自动触发bgsave
# 设置多个save条件是为了适应不同的情况，为了在更加合适的条件下持久化数据

情况2：主从复制情况下，从节点进行全量复制时自动触发bgsave

情况3：客户端执行debug reload命令刷新redis时自动触发bgsave

情况4：没有开启AOF的情况下，客户端执行shutdown命令时自动触发bgsave

[指定RDB文件存储位置]

方法1：配置属性文件

dir ./ # 保存到redis当前目录下（默认）

方法2：在运行时客户端执行 config set dir {路径} 指定，例如

127.0.0.1:6379> config set dir ./ # 指定路径
OK
127.0.0.1:6379> config set dbfilename rdb.rdb # 指定文件名称
OK

[RDB流程]

[RDB的优缺点]

  • 恢复速度比AOF快
  • 可以代表某一个时间点的数据状态，且因为是紧凑的二进制文件所以较为节省空间
  • 因为是条件触发，所以在没满足持久化条件前出现异常则会丢失数据
  • fork属于重量级的操作，频繁的bgsave会导致redis性能下降
  • 可能出现旧版本无法兼容新版本的rdb文件导致无法恢复数据的问题

AOF

AOF，append only file

[aof流程]

[aof使用]

默认情况下AOF是没有启用的，需要在服务端配置文件种进行如下设置：

appendonly yes # 开启AOF功能
appendfilename "appendonly.aof" # 指定AOF文件

# 同步策略
# appendfsync always # 每条命令都立即fsync到文件中
appendfsync everysec # 只需调用系统的write后就立即返回，并开启一条aof线程，每1秒从缓存中fsync一次
# appendfsync no # 交给操作系统处理，30秒内fsync一次

[aof注意]

  • everysec相对于always来说，属于延时写入，这样有利于提高操作系统缓冲区使用率，且因为调用系统write命令后就返回而无需等待fsync，因此而提高了服务端的吞吐量。

  • 设置成always，则表示每条修改数据的命令都需要马上持久化到硬盘，显然这是非常低效的，所以不推荐配置成always。但always可能将数据丢失的风险降到最低。

  • 理论来说everysec最大可能丢失1s的命令（aof缓冲的层面上）

  • 还需要注意的是，如果系统fsync操作缓慢则会影响主线程的服务，导致主线程阻塞，原因如下：

可以看出，主线程会对比上一次系统fsync的操作时间，如果发现上一次fsync操作时间大于2秒的话，主线程就会阻塞停止服务来腾出些许时间给系统fsync，这样做的好处在于避免大量fsync操作延迟导致命令没有切实写入到硬盘，换句话就是通过这样的方式进一步确保数据的持久化。

Redis重启加载流程

为什么单线程还那么快？

  • 基于内存的读写操作
  • 单线程避免了上下文切换和资源竞争的消耗
  • 采用网络IO多路复用技术（epoll） 和 事件处理；类似与Netty属于非阻塞IO，减少了网络IO的时间消耗
  • 客户端和服务端采用高效的RESP协议（Redis序列化协议）进行通信。（协议特点：易于实现、解析速度快、人可读懂）

[注意]

因为Redis是单线程的缘故，因此不会存在多条命令同时执行的情况，换句话来说就是要一条一条地执行，正因这样就必须注意命令如果执行时间过长就会阻塞后面的命令（Redis阻塞），这是个非常严重的问题，所以Redis的设计理念其实是面向快速执行的场景，即每条命令都是简短快速的。所以尽量避免诸如keys、hgetall这样的命令使用。

Redis的缓存更新策略

  1. 算法剔除（FIFO[无]、LRU、LRU、最小TTL）
  2. 超时剔除（expire命令）
  3. 主动更新（推荐，例如CRUD数据库时同时跟新缓存）

Redis中算法剔除相关的配置

配置redis最大可用内存。默认为0，即不设置限制直到耗尽主内存

maxmemory <bytes>

指定当maxmemory满了的时候，使用什么策略来管理内存

maxmemory-policy noeviction
  • volatile-lru：从具有过期设置（expire）的key中使用近似的LRU剔除（最近最少使用）缓存后再保存新的缓存
  • volatile-lfu：从具有过期设置（expire）的key中使用近似的LFU剔除（最近最少命中）缓存后再保存新的缓存
  • volatile-random：从具有过期设置（expire）的key中随机剔除缓存后再保存新的缓存
  • volatile-ttl：从具有过期设置（expire）的key中剔除ttl时间较小的缓存后再保存新的缓存
  • allkeys-lru：使用近似的LRU剔除缓存后再保存新的缓存
  • allkeys-lfu：使用近似的LFU剔除缓存后再保存新的缓存
  • allkeys-random：随机剔除缓存后再保存新的缓存
  • noeviction：不剔除，只在写操作是返回错误告知

需要注意的是LFU是在Redis 4.x加入的，以下图片源自网络

指定每次对比样本的数量

maxmemory-samples 5

因为Redis中实现的LRU、LFU、最小TTL等都不是精准的算法。maxmemory-samples用于指定剔除时对比的样本数。官方注释表示，设置为10非常接近真是的算法所达的精准度，但是会消耗更多的CPU资源

非常重要的是，Redis 5.x开始replica-ignore-maxmemory默认为yes，表示副本在复制时将会忽略主节点的maxmemory设置，换句话来说就是副本节点不会对剔除缓存。

分布式缓存需要注意问题

缓存穿透

缓存穿透指的是访问了不存在的数据导致间接绕过了缓存层（因为数据根本不存在）;

举个例子，根据id=nihao来获取对象信息，但数据库并没有id=nihao的对象，这时如果有人故意攻击，拿着一个不存在的id不断请求应用，这就会导致所有请求落在数据库上，从而间接导致缓存层失效;

为了解决这种问题有2种方法：

  • 缓存空值：访问数据库时如果数据库没有数据，则返回一个null给缓存层，避免拿着不存在的值不停地访问应用从而造成压力。同时最好给这个空值设置超时时间，这个可以避免大量空值占用空间。

  • BloomFilter(布隆过滤器)：基于比特向量。原理是先将数据的key（例如某个字段）通过哈希计算并保存到集合中。那么请求到来时先检查BloomFilter集合中是否可能存在该值。存在则查找数据库，不存在直接返回。但这里注意的是BloomFilter是存在误判问题的，即BloomFilter并非百分百准确，换句话来说BloomFilter只能减少无畏的数据库查询。在Google Guava工具库中提供了对BloomFilter的先关实现。

    BloomFilter原理展示： https://tandi960505.github.io/bloomfilter-tutorial/zh_CN/

无底洞问题

无底洞问题由Facebook所发现，当时Facebook的Memcache节点数已多达3000个，开发人员发现水平扩容节点不但没有得来想要的效果，反而降低了性能。

其原因主要是key散乱在不同的缓存节点上导致了网络的开销。

所以预防这个问题基本思想就是，减少网络通讯的次数和让相关的key落在相同的节点上。所以可以使用管道、mget、m* 这样的命令来解决。

图片源自《Redis开发与运维》

缓存击穿（热点key失效）

其实这个名字个人觉得不是很恰当，缓存击穿指的是当大量并发访问同一个key的时候，这个key恰好失效了，导致所有并发请求落到数据库;

解决的思路：

  • 进行互斥处理来等待先发现key为空的线程将该key重新放入缓存后其他线程再获取。(多服务节点下可使用分布式锁)
  • key不过期。使用后台线程来定时更新缓存

缓存雪崩

缓存雪崩类似于缓存击穿。指的是缓存层突然失效。

图片源自《Redis开发与运维》

解决方法：

  • 保证缓存层的高可用
  • 后端请求限流
  • 提前测试缓存雪崩后的应对能力

关于Redis的内存碎片问题

什么是内存碎片和造成内存碎片的原因

内存碎片可以简单理解为无法使用的内存空缺。

产生内存碎片的原因主要有2个：

  1. 受系统的内存分配器影响，例如频繁进行大小不定的内存分配且（例如大量修改value值，且value值前后大小差异较大）
  2. 大量增删操作，因为删除key后并不会马上回收内存。

拓展知识

  1. 内存池是一种内存分配方式。通常我们习惯直接使用new、malloc等API申请分配内存，这样做的缺点在于：由于所申请内存块的大小不定，当频繁使用时会造成大量的内存碎片并进而降低性能。— 百度百科

  2. 因为Redis没有实现自己的内存池，因此需要依赖系统自身的内存分配器（如Linux上的libc库）来进行内存的管理。换句话来说，操作系统内存分配器性能和处理内存碎片的能力对Redis有一定能影响。

  3. 在较新版本的Redis中使用 Jemalloc代替了系统默认的内存分配器。（在Redis5.x中Jemalloc也得到了升级，对内存分配和碎片整理更加高效）

  4. 主动回收内容有两种情况：key过时（expire）、内容到达maxmemory出发驱逐策略

查看是否存在内存碎片

命令：info memory

图片源自《Redis开发与运维》

可以看到内存碎片的公式为：内存碎片 = 进程占用的物理内存总量 / 实际数据内存占用量

该值最好的状态是接近1，该值越大表示存在内存碎片越多。

解决内存碎片问题方案

Redis4.x版本之前，只能通过重启Redis解决，这个就可以重新配置内存。

Redis4.x版本之后，通过设置activedefrag yes开启自动整理内存碎片功能，此后Redis会尽最大努力解决内存碎片问题。

GEO

GEO，地理信息定位。使用zset存储

redis 3.2 开始提供了个有趣的功能，就是GEO。GEO可以实现诸如附近的人、地点距离、摇一摇这样的功能，有趣吧？

[添加地理位置信息]

获取地址的经纬度信息：http://api.map.baidu.com/lbsapi/getpoint/index.html

格式：geoadd key longitude latitude member

  • longitude：经度
  • latitude：纬度
  • member：成员/地点
127.0.0.1:6379> geoadd guangzhou 113.327679 23.125237 zjxc # 珠江新城
(integer) 1
127.0.0.1:6379> geoadd guangzhou 113.327998 23.023749 ds # 大石
(integer) 1
127.0.0.1:6379> geoadd guangzhou 113.27604 22.995208 gznz # 广州南站
(integer) 1

[获取地理位置信息]

格式：geopos key member，例子如下：

127.0.0.1:6379> geopos guangzhou zjxc
1) 1) "113.32767874002456665"
   2) "23.12523573693501788"

[获取两地理位置之间的距离]

格式：geodist key member1 member2 [单位]

  • m：米
  • km：公里
  • mi：英里
  • ft：尺
# 大石到南站
127.0.0.1:6379> geodist guangzhou ds gznz km
"6.1947"
# 大石到珠江新城
127.0.0.1:6379> geodist guangzhou ds zjxc km
"11.2881"

[获取指定距离范围的地里信息]

格式1：georadiusbymember key member 单位

格式2：georadius key longitude latitude 单位

其中member或longitude latitude为中心节点的位置，查询指定半径范围的位置（包含返回中心节点）

127.0.0.1:6379> georadiusbymember guangzhou ds 10 km
1) "gznz" # 广州南站
2) "ds"
127.0.0.1:6379> georadiusbymember guangzhou ds 15 km
1) "gznz" # 广州南站
2) "ds"
3) "zjxc" # 珠江新城

除了上面写法之外，还可以追加以下参数：

  • withcoord：返回结果包含经纬度
  • withdist：返回结果包含距离中心节点位置的距离
  • count 数字参数：执行返回的数量（最近开始）
  • asc|desc：升序和降序（最好指定asc，因为没有默认值，这样做返回的第1个为中心节点）
  • store key：将返回结果的地理位置保存到指定key种
  • storedist key：将返回结果的经纬度保存到指定key种

[删除指定的地里位置信息]

格式：zrem key member

zrem guangzhou zjxc

[返回geo的hash值（经纬度）]

格式：geohash key member

127.0.0.1:6379> geohash guangzhou ds
1) "ws0dfz6gjc0"

Spring Data Redis

官方文档：https://spring.io/projects/spring-data-redis#learn

SpringBoot 2.x 开始默认 Spring Data Redis 使用 Lettuce 作为Redis客户端。（SpringBoot 1.x 默认为Jedis）

[ Lettuce和Jedis区别 ]

  • Jedis是非线程安全的。Jedis的设计和Redis对应都是单线程读写，一个Jedis连接对应一个Redis，多以在多线程场景下应该使用JedisPool来操作。
  • Lettuce基于Netty来实现，连接实例（StatefulRedisConnection）可以在多个线程间并发访问，因为StatefulRedisConnection是线程安全的，所以一个连接实例（StatefulRedisConnection）就可以满足多线程环境下的并发访问。而在Spring Boot 2.x中Lettuce是默认的Redis客户端。

// RedisConnection为Reids客户端的抽象，当RedisTemplate不够使用时，就可以使用以下的方式来使用
@Test
public void testExecuteMethod() {

    Boolean execute = stringRedisTemplate.execute((RedisConnection connection) -> {
        // setnx并指定过期时间
        return connection.set("hi".getBytes(), "test setnx".getBytes(),
                              Expiration.seconds(30), // 过期时间
                              RedisStringCommands.SetOption.SET_IF_ABSENT); // 设置NX
    });

    System.out.println(execute);
}
