微服务开山篇-关于系统架构的演进

 2018-08-13   作者: TanDi   类型：原创

前言

  近年来微服务是越来越火，仿佛聊天中不谈谈微服务都显得有些赶不上潮流。但虽然很多人都在探讨微服务，但我发现大多都只是在聊实现微服务的技术，而很少看见有人聊自己现在做的项目为什么使用微服务。而我的问题是微服务到底是什么，它的存在到底解决了什么问题，这些都是我想知道和了解的，因为我始终认为某项技术或某种架构理念之所以存在肯定是有它的理由而并非空穴来风，而只有了解其存在的意义才可以更好地使用它，而不是只停留在冷冰冰的概念上。

单体架构的进化之路

单体架构适合场景：业务清晰简单，且对并发性、可用性和性能方面的要求不高

应用与数据库分离

  如图所示，为了解决单机负载问题将其一分为二。但显然这不是一个一劳永逸的做法，因为目前而然如果并发再增大的话就可能存在：应用服务器过载、数据库服务器过载

解决应用服务器过载 - 使用集群进行负载均衡

  引入负载均衡的好处是可以解决应用服务器单点负载问题。但是亦引入了不同节点间session如何同步的问题。解决Session问题常见方案如下：

  • 依赖Web服务器的Session共享机制
  • 使用分布式缓存独立存储Session信息
  • 依赖负载均衡服务器来绑定请求与服务器的关系。例如Nginx中的iphash策略。
  • 使用Token来代替Session

解决应用服务器过载 - 前后端分离

解决数据库服务器过载 - 读写分离

适合于读多写少的场景

  前后端分离也现在流行的应用架构。前后端分离能够减轻后端服务器的压力，因为可以清晰划分前端职责和后端职责。对于有前后端团队的公司是一个很好的选择。

解决数据库服务器过载 - 数据库垂直拆分

适合于存在冷热数据的场景

解决数据库服务器过载 - 分布式缓存

  冷热数据场景除了数据库垂直拆分方案之外，还可以使用分布式缓冲来解决。而且使用分布式缓存可以降低数据库的操作量，从而提升系统的吞吐量。

解决数据库服务器过载 - 数据库集群

  对于数据库集群而言就要看数据库本身是否支持集群了。例如MySQL有专门的中间件来协调JDBC的调用。诸如MyCat、Sharding-JDBC等

分布式架构诞生

什么是分布式系统？

  功能分散在不同进程上的系统就叫分布式系统。

  例如，以前有些网络游戏的注册功能是嵌在游戏本身里面的，这时这个游戏就属于单体架构。但发展到现在可以看到诸如腾讯，他们有很多系统单是共用一个账号管理系统。这时某个游戏和这个账号管理系统就构成了分布式系统的关系，因为对于网络游戏而言游戏肯定是需要账号登录的，因此从用户的角度来看，账号管理系统+游戏本身=一个完成的游戏。换句话就是,账号管理系统+游戏本身构成一个分布式架构的游戏。

  那为什么我们有了解决应用服务器过载和数据库服务器过载的方案还需要有分布式呢？它存在的理由是什么？原因如下：

  • 单体架构部署不够灵活。随着并发量增加意味着集群节点要增加，但却不能根据负载模块来扩展节点，显然有些浪费资源
  • 多人参与的单体架构系统随着业务越来越复杂会降低开发效率
  • 单体架构系统代码耦合度高，导致系统难以扩展，存在牵一发而动全身的风险

分布式架构例图

  从图中可以得知，分布式架构其实就是对单体架构的系统进行垂直拆分。垂直拆分之后我们可以根据每个子系统（模块）的负载程度来扩展集群节点数量。这样一来即可以降低单体架构存在的负载问题，同时也提高了部署的灵活程度。

  那么分布式架构在解决单体架构存在的问题同时是不是就没有缺点呢？这当然是否定的。分布式优缺点如下：

【优点】

  • 提高了部署的灵活程度
  • 降低代码的耦合程度
  • 提高系统的可扩展性
  • 在同构多系统的情况下，提高了模块的可重用性
  • 一定程度提高了系统的可靠性

【缺点】

  • 部署相较于单体应用变得复杂
  • 维护成本提高
  • 对开发人员的要求相对提高
  • 存在分布式事务问题

分布式架构 - 异构系统救星SOA

基本概念

  SOA，即面向服务架构。常作为 异构系统协作 的解决方案，如 SOA架构理念 可以解决企业内现有系统的信息孤岛问题。一般的做法是将企业现有的系统看成一个个独立的服务，并且服务之间通过定义良好的接口和契约联系起来 ，最后再对这些服务进行组装和编排来产生一个符合SOA架构理念的”新系统”。

  在SOA技术栈中较为有名的莫过于Web Services。Web Services是一种以XML为基础，且天生跨平台的远程调用技术。Web Services的实现涵盖了诸如 XML-RPC 、UDDI、WSDL、SOAP 等技术。SOA系统可以用Web Services来暴露服务以达到服务可重用的效果。

  除了Web Services之外，SOA还有两大跨平台利器就是 MQ 和 ESB(企业服务总线)。MQ能够实现服务间的异步通信。而ESB则是一个集大成者，是目前传统企业实施 SOA架构 的核心，甚至可以这么说，现今传统企业中SOA架构的实现基本上都是指ESB。

 在ESB组件中通常集成了Web Services和MQ等功能以实现服务重用和消息传递等功能 。ESB组件中还提供 请求路由、服务消费者与提供者松耦合、协议转换 等特性来解决异构系统通信问题。但需要注意的是如果想要从头到尾开发一个 ESB 系统组件其实是较为困难的事情，成本大之余也不见得质量好，所以一般实践 SOA 架构的企业都是向诸如 IBM 这类型的大企业购买 ESB 的产品服务来用的。例如 WebSphere ESB 和 Message Broker 。

SOA（ESB）存在的问题：

  • 复杂的ESB总线处于核心位置，而且存在存在单点问题
  • 学习和使用门口偏高
  • 服务交互过程中传输性能较为厚重
  • SOA中相关协议不够简约，较多没有价值的信息
  • SOA中对规范和标准要求较高，加大了实施和推广的难度

分布式架构 - 微服务(MSA)

基本概念

  到目前为止，其实业界对微服务并没有一个十分明确的定义，从而导致同一个问题不同的人都可能有不同的见解。那么我们应该如何去了解微服务呢？其实现今大多数人对于微服务的认知都源自于架构大师 Martin Fowler 的文章《Microservices》，所以我就随大流阅读了该文章。

  文章大致提及到以上内容，微服务是将一个单体架构应用开发为一组小型服务，这些服务围绕业务功能来构建，服务运行在独立的进程中，服务间通信而言采用的是轻量级通信机制，而且允许服务异构，即限制开发语言和技术，在部署方面微服务架构系统应该支持自动化部署机制来独立部署服务。

图片源自文章《Microservices》

结合文章 Microservices 还有业界为微服务打的标签，基本上我们可以得出以下微服务特性：

  • 允许技术异构
  • 不依赖任何特定技术
  • 轻量级协议
  • 契约接口（规范数据格式）
  • 去中心化治理
  • 去中心化数据管理
  • 独立部署
  • 无状态且职责单一
  • 围绕业务能力来构建
  • 组件化（易替换易扩展升级）
  • 低耦合且高内聚

关于微服务带来的问题

  • 服务划分得过细，就会提供系统的复杂度
  • 服务数量多，会对小团队造成压力
  • 服务数量多，造成调用链过程，从而影响请求处理性能，而且难以定位问题
  • 微服务需要使用容器来做到规范化打包、隔离、规范化部署等操作，所以如果不支持这些则会无法实现快速交付
  • 如果微服务没有进行合理的服务治理且服务数量又多的话，就会导致后期维护成本剧增

个人认为实践微服务架构的关键问题：服务粒度、基础设施服务

服务粒度

  如何拆分服务我认为是一个比较难的问题。但需要强调的是，不应该因为微服务提倡小而精的服务设计而过度细小化服务的体积。例如可以按照 功能 、 业务 、可用性 来拆分服务，这样做可以让服务的职责更加明确。

  除此之外，先前我在一些文章中看到，有人建议按照开发团队的人数来拆分服务，譬如6人团队就拆分为2个服务，如果系统在日后变得复杂就扩充团队到12人并将2个服务拆分变成4个，3个人1个服务这样做的原因是可以让开发者更好地掌控服务细节，因为人越多就会导致代码越复杂，而人少就可能导致复杂度不够造成以后扩展难的问题，还有就是3个人即使1个离职，其余2个都可以把控住服务。对于这种说法，我个人也是比较同意的，因为软件开发是一个协作活，就算你划分得最合理但开发人员把控不了的话出来的东西也不见好，还有团队中个人的技术上和思想都存在差异，所以人越多未必是好事。

基础设施服务（服务治理）

  从本质上说，任何能够提升服务 可用性 和 性能 且 让服务更稳定 的操作都属于服务治理的范畴。而基础设施服务指的就是提供服务治理功能的服务。

  虽然说开发系统是为了完成业务功能，但是如果系统不稳定且时常出现问题的话，那么就会导致用户体现下降，对于互联网产品来说如果一个产品不好用的话用户就会流失，而对于开发运维人员来说经常出现问题就以为着又需要花费时间和精力到决绝bug上。因此，个人认为系统提供完善的基础设施服务是实现微服务架构的重中之重。

MSA与SOA的异同

  表面上 MSA 和 SOA 其实是非常相似的，两者都属于面向服务型的架构，两者都与平台无关。按照个人的理解我认为 微服务 其实是 SOA架构理念 的互联网版实现，而ESB实现则是应对企业现存的多个异构系统之间通信的解决方案，因此ESB实现的SOA更加符合企业的需求。当然，现今的微服务也能解决异构系统问题，但是在企业现有的系统下进行改动的话不见得成本比采购ESB低。

  对于 MSA 来而言，主要应对的是现今互联网系统的版本快速迭代的需求，因此应用在互联网系统上的架构应该包含：可持续交付、可扩展性高、稳定性高、去中心化 等基本要求，且最好能够配合敏捷开发的理念。如此一来如果使用ESB实现的话，首选ESB是一个中心化的重量级系统组件，而且还存在单点问题，所以并不符合现今互联网的需求。可能也是因为这个原因 SOA 才没有被互联网广泛推广。

  除此之外 MSA 和 SOA 还有一个显著的区别，就是服务粒度差别。在 SOA 服务粒度并没有限制，它可以是一个小功能也可以是一整个完成的系统。而 MSA 提倡的是将服务尽可能分割的小（当然也不能过少，否则增加开发和维护成本，所以准确点说应该是小而合理）。
