微服务开山篇-架构的演进之路

内容修订

  • 2019年03月21日13:57:24 — 有新的理解，对文章整体内容进行重构

闲聊

  近年来微服务架构是越来越火，各大论坛、博客、公众号都是相关的内容，以至于貌似不聊聊微服务自己就变得落伍了，但实话实说个人并不太喜欢这样的氛围，因为如何设计一个系统或进行技术选型，其实是应该根据业务、资源、团队等多个因数考量后决定出来的，而并非越新越好或越潮越好，正所谓“合适优于行业领先”就是这个道理。

  虽然现在讨论微服务的确实大有人在，但经常能看见的只是技术类的话题，而聊聊自己公司为什么要使用微服务，又或者用了微服务之后和原来相比有什么好处这类型的话题是少之又少。当然，这也可能是因为大家都是技术出身，所以关注点很自然就落在技术上了。

  个人认为，在学习一门技术之前应该要了解一下它为什么而存在，只有知道它为什么存在才能更直观地知道它解决了什么问题，以及如何更好地使用它。

单体架构

基本介绍

  单体架构是最为直观和简单的系统架构，这样说的理由其实很简单，因为功能都在一个点上，方便了开发、测试和管理。但需要注意的是系统架构本身很多时候是把双刃剑，用得好自然好，用不好也很麻烦。

譬如个人认为以下条件使用单体架构比较有优势：

  • 系统业务清晰和简单
  • 开发团队有着良好的编发规范
  • 资源有限，对性能没要求

而单体架构的优缺点如下：

  • 优点
    • 简单直观
    • 测试和部署简单
  • 缺点
    • 随着业务越来越复杂，系统会变得冗肿和难以维护（譬如牵一发而动全身），导致扩展性差
    • 代码耦合高，不利于团队开发（发团队有良好的编发规范除外）
    • 对开发者要求较高，一般需要前后端兼顾
    • 只能横向扩展，不利于资源的合理利用。譬如著名的二八定理就很好地解析这个问题

传统的单体架构

当前架构存在的问题是单点负载，无法应对业务越来越复杂，用户量越来越多的场景需要。

横向扩展解决单点负载过高问题

虽然横向扩展服务节点能够解决单点负载问题，但与此同时亦带来问题。

  • 多服务节点下会话的管理问题。常见解决方案如下
    • Web服务器会话共享机制
    • 共享存储服务
    • Nginx 的 IP 访问绑定
    • 使用 Token 进行交互
  • 资源不能合理利用，因为在系统中可能只有几个功能被常用，但扩展式要全部部署

前后端分离让技术边界更加明确

随着前端技术的迅速发展，前后端分离架构的普及度已经非常高了。前后端进行分离后，不仅可以让技术分工更加明确，而且相较于最早的单体架构而言，后端服务节点变得更加轻量和简洁。

缺点是使用 Cookie - Sesssion 机制变得复杂，所以一般使用前后端分离架构后都会采用 Token 这种轻量的会话管理机制来维持通信。

拓展：如何前后端分离后继续使用Session？

  • 通过 CORS 允许客户端传递 cookie
  • 通过 CORS 将 SessionId 设置到响应头
  • 将 SessionId 设置到 Token 中

分布式架构

是什么？

  譬如以前一些网络游戏的注册功能是嵌套在游戏本身的，但随着发展公司又开发或代理了更多的游戏，而这时如果玩家每次玩这家公司的新游戏就要记住一套账号那用户体验未免也太差了。因此在往后的日子里很多公司都约定俗成地将用户管理这块功能从软件或游戏中单独抽离了出来让其形成一个立的系统来提供服务。而个时候的游戏或软件就形成了一个分布式系统了，即游戏本身再加上账号管理系统才能算是一套完成的系统。

结论：功能分布在不同进程上的系统就叫分布式系统。

什么时候应该使用分布式架构？

  • 需要应对业务扩展带来的挑战
  • 单体系统过于冗肿，以至于日常维护困难

分布式架构优缺点

优点：

  • 降低了代码和模块的耦合度
  • 提高了系统的性能
    • 可扩展性
    • 伸缩（容错）性
    • 可复用性
  • 部署变得灵活，能够有针对性地对特定的功能模块进行扩容缩容
  • 编码难度降低

缺点：

  • 测试变得复杂，特别是跨团队协作
  • 维护和运维成本变高
  • 引入跨服务问题
    • 分布式事务一致性
    • 可用性降低
  • 要开发和维护一套完善的分布式系统，对开发者自身的知识体系和能力有一定的要求

关于SOA架构

服务化和SOA的关系

  服务化其实是一种在原分布式架构之上的进一步阐述。其强调应该将分布在各个进程上的功能点视作为一种服务，这些服务应该有着可复用、契约接口、模块化、服务自治、松耦合、无状态等特点。而 SOA（面向服务架构）其实就是服务化大环境下的产物，它有着服务化的所有特点，及其相关技术的解决方案。

SOA 的到底是什么，及其价值所在

   个人认为可以站在不同层面来理解 SOA。譬如站在企业层面，SOA 架构是一套用来解决现有异构系统信息孤岛问题的解决方案。企业通常会因为成本效益问题而引入 ESB (SOA) 组件以解决各个系统的通信问题，毕竟与推倒现有系统重构相比，这种方案的风险和成本更低。企业系统在接入 ESB（企业服务总线）后，就可以进行数据互通了。而此时接入 ESB 的所有系统都可将其被视为一个服务，而服务与服务之间就形成了一个 SOA 架构风格的生态系统。

  站在应用系统层面，系统应该进行模块化划分，且让其称为一个可复用和自治的功能单元。模块与模块之间能够以契约接口的方式进行通信协作，以此来达到高内聚地耦合的可扩展性需求。

  最后总的来说，SOA 并不是特指某些技术，而是一套系统服务协作的解决方案。它强调的是服务复用和互通。我们可以通过 SOA 架构思想来提高企业内部系统服务的可复用性和使用价值。

服务化常见技术

  • Web Service：可基于 SOAP 或 REST 进行远程服务调用，因为用的是 XML 进行交行户，所以与平台无关，因此常用于解决异构系统协作问题
  • RPC：服务远程调用。譬如 Dubbo 就称自己为服务化 RPC 框架
  • MQ：提供消息的发布和订阅功能
  • ESB：服务消息总线，是SOA中极具代表性的组件，一般会有以下功能和特点
    • 协议转换
    • 请求路由
    • 可集成 JMS(MQ) 和 Web Service

ESB 存在的问题

  • ESB 是一个重量级组件，存在单点问题
  • 开发和维护成本高，所以企业一般会采购现成的产品（譬如 WebSphere ESB、Message Broker），而不是自己实现
  • 采购的 ESB 服务可能无法全面符合企业当前需求
  • SOA 架构中因为残留系统问题导致其常用的协议不够轻量，交互过程中常有较多没有价值的信息
  • 对规范和标准要求较高，导致实施和推广难度加大

SOA 服务的两种协作模式

第一种

  • 采用契约接口进行点对点通信，解决系统服务的复用问题
  • 缺点是接口管理复杂，维护成本会随着系统服务数量的增加而提升

第二种：

  • 采用 ESB 中间件进行通讯枢纽，以集中式管理和协调通信
  • 缺点是 ESB 重且复杂，存在单点问题，一旦发生故障则瘫痪通信。要解决以上问题导致实施 SOA 变得复杂

为什么有人说 SOA 不适合互联网项目

  首先个人并不完全认同这种说法。在上面已经提到过，SOA 并不是特指某些技术，而是一套系统服务协作的解决方案。譬如企业急需要解决残留系统的通信问题，这时自然就发挥 SOA（ESB）的作用了。譬如有一篇在网上广泛流传的文章《服务端高并发分布式架构演进之路》中就提及到，淘宝在第十二次架构的演进当中就引入了 ESB 来屏蔽接口调用时的协议差异。

而阿里巴巴在国内本就是互联网电商的龙头企业，所以说 SOA 不合适互联网是根本站不住脚的。

  但为什么我要说不完全认同而不是不认同呢？这是因为如果你是站在系统应用层面来看的话，SOA 确实并不符合现今互联网的发展需求，而个人认为导致这种局面的主要原因是 SOA 的通信协议和核心组件（ESB）比较重，而互联网系统对性能、可扩展、稳定、可维护等特性都有较大的需求。

SOA（服务）治理

  当企业内部的系统服务化成 SOA 架构后，系统服务的管理问题就很自然地被放大了，因为除了技术问题之外很可能还会引入跨团队协作。譬如现在 A 系统服务需要和 B 系统服务进行对接，但两个系统由不同公司的团队来维护，因此这时不仅会出现很多业务层面的问题，还有出现团队协作的问题。而所谓的服务治理其实就是用来解决这一问题的。我们也可以简单理解，服务治理其实就是如何让服务运行得更好的过程。它涵盖的范围非常广，譬如开发治理 、测试治理、运维治理、管理治理等多个方面 。

要是从一个开发人员的角度来谈开发治理的话，个人认为应该涵盖了以下这些内容：

  • 服务网关，对服务入口进行统一管理，方便组织和访问
  • 服务的注册和发现，将服务的提供者和消费者解耦，让消费者能感知提供者是否可用
  • 服务容错，采用熔断机制，让系统在处理请求时即使有个别服务不可用，但依然能维持系统的可用性
  • 采用轻量级通信协议，实现服务的松耦合
  • 服务负载均衡，可用过前后端负载均衡，合理地分摊服务的请求负载
  • 服务功能开关，可通过配置中心实现，在必须要可进行功能开关以及逻辑切换
  • 服务限流，对请求或资源进行限额，保证服务能够在高负载下仍然能够继续提供服务
  • 服务的安全性，譬如幂等、鉴权、接口规范等设计和处理
  • 日志、度量、请求链路等监控，实时了解服务运行状态，应对各种突发情况
  • 服务接口版本管理，方便必要时可回滚和记录版本迭代过程
  • 服务版本治理，引入蓝绿发布和灰度发布（金丝雀发布），实现平滑地升级和回滚
  • 实现标准化部署，通过容器镜像实现标准化部署，屏蔽运行环境带来的不确定性
  • 关注服务的编排和伸缩性，依赖现今的流行的分布式支撑平台（譬如 Kubernetes）实现服务的动态伸缩，以及自动管理功能
  • 完善的单元测试用例，单元测试覆盖率越高，就可以大大降低后续集成时带来的问题
  • 合理的开发发布流程，引入持续集成方案，以提供一个可持续发布的应用系统版本
  • 善于运用开发管理工具，譬如 Jira、禅道等，以提高协作效率和系统缺陷反馈工作
  • 良好的源码版本控制，譬如采用 Git Flow 进行合理化的分支管理

可以看到，光从技术角度来看服务治理就涵盖了很多内容，而且不是全部，所以说服务治理话题是相当之大。因为个人目前知识体系的局限性原因，关于服务治理详细的内容就暂且不展开了。

  回到 SOA 的讨论上，合理的服务治理有利于服务的稳定运作和维护。那么我们大概地了解了什么是服务治理之后其实不难发现，如果站在企业层面来实施 SOA 治理其实是一件成本比较大的事情，因为现有的系统通常会有自己的一套治理方案，而这时要想实现标准化就比较困难，而这也是为什么有人说 SOA 不适合互联网项目的一个原因，除了比较重之外还不太利于服务治理的实施。

  然而其实我在列举以上服务治理事项时，是以微服务架构的标准来去列举的，因为微服务架构是一个完全站在应用系统层面的系统架构风格，因此以上列出的内容如果只从应用层面来去实施的话确实是要比 SOA（ESB）要简单和容易很多。

微服务架构(MSA)

我理解的微服务

  微服务是一个完全站在应用系统层面的系统架构风格。指将一个完整的系统划分为多个围绕着业务能力开发的微型系统，而这些微型系统就叫做微服务。在微服务架构中，服务被建议构建成一个拥有独立生态、无状态且职责单一的服务，所以简单理解它其实就是一个微型的系统，只不过这个系统需要围绕着特定的业务场景来组装运作而已。

  个人理解 MSA 其实是 SOA 的一种派生体系，主要区别在于 MSA 在服务的标准上要比 SOA 细致，以及更符合现今架构快速变化的互联网应用系统需求（参考上文 SOA 的介绍）。

  • 图片源自：https://www.linkedin.com/pulse/event-driven-microservices-architecture-using-spring-cloud-bruksha

微服务特点

  • 不依赖特定技术，即服务之间拥抱异构
  • 轻量级协议，指简洁和通用的通信协议
  • 契约接口，通信数据的收发需要遵守契约规范
  • 去中心化，服务应有独立的数据存储单元，以及在系统层面尽量不要过分依赖某个特定服务
  • 无状态，服务自身不记录任何请求行为（落盘数据除外），因为这样会提高系统的不确定性，且无状态有利于提高可扩展性
  • 职责单一，指服务边界明确，一心做好一件事（这件使可以理解为一类、一个功能、甚至是一个业务）
  • 围绕业务能力构建，指服务应该面向业务来设计
  • 组件化，强调服务可复用和可组装（如果觉得难理解，可将微服务理解为一个Java库，这个库在任何项目都可以和库混合使用）
  • 高内聚低耦合，指服务应该尽量做到内循环自给自足而不要依赖其它服务
  • 面向云原生，基于微服务本身的特点，使其在云原生开发和维护方面极具优势，因此在构建微服务时，应尽量采用容器进行标准化部署，以发挥基础设施优势。

微服务带来的问题

  • 如设划分微服务边界以及设计粒度大小，成为一个必须要思考的问题
  • 微服务数量较多时，会导致问题定位困难，提高了系统的维护成本
  • 要想微服务稳定高效运行服务治理是关键，所以微服务的运维成本是比较高的，而且对开发者和设计者的综合能力要求较高

微服务要多微？

  个人认为如何进行粒度和服务边界划分其实是整个微服务架构落地环节中最重要也是极具考验设计者功底的部分，因为其实技术应用方面并没什么困难的地方，只要稍花点时间的话上手开发并不困难，但要想对系统架构、业务能力以及面临的挑战有所理解和洞察却绝非一朝一夕的经验之谈。

  我们可以先从微服务中的“微”字开始理解。那到底要多“微”才叫微服务呢？譬如在《微服务设计》一书中就提到 John Eaves 认为“一个微服务应该可以在两周内完全重写”，而书的作者貌似也认同了这标准，但随后他又给出了自己的理解“足够小即可，但不要太小”。好了，从以上信息个人得出了这样的结论，就是微服务粒度设计其实并不需要过分细致，因为如果划分得太细就会大大增加开发和维护成本，所以只需团队一致认为已经足够小（不冗肿，且职责单一）就行了。

如何判断微服务是否设计合理？

  我觉得可以先进行一次逆向思考，即怎么的微服务架构设计是不合理的？譬如上文中提到《服务端高并发分布式架构演进之路》一文，它表示淘宝到目前为止已经进行了十多次架构变迁，所以从这点上不难得出结论，就是互联网应用系统架构会随着用户流量的不断增加而持续演进（即所谓的演进式架构）。好了，有了这个共识之后，这时我们可以再从大家都认同微服务架构适合现今互联网应用系统这点出发，即既然微服务架构适合，就意味着它能够轻松地应对这一种频繁的架构变迁。所以最后我得出了一个这样的结论来评定目前的微服务架构是否设计得不合理，就是当你设计出来的微服务在进行架构变迁期间让你感到力不从心，这时就证明目前的设计是不合理的，需要进行调整。

  除此之外，现在比较流行的微服务设计方法论莫过于DDD（领域驱动设计）了。DDD是一种从业务角度来设计系统架构以及划分服务边界的一套方法。在 DDD 中需要有一个对当前系统业务足够熟悉的人来充当领域专家一角，领域专家会在架构设计阶段和架构师或开发人员一同进行架构设计。譬如对用户的事件风暴进行收集，通过对其进行领域故事分析提取出领域对象，然后再将领域对象映射成代码模型后将其落地。而且利用 DDD 还很容易划分出服务的边界，例如通过组合聚合来形成限界上下文，而这个限界上下文就可以作为服务的边界了。简单理解限界上下文就好比如语义环境，即即使是相同的话但在不同的语义环境中它的意思就可以大相径庭。例如“衣服能穿多少穿多少”在冬天和夏天的意思是完全相反的。

  当然，DDD 个人认是一个相当复杂的方法论，需要有良好的业务基础才能很好的理解它，特别是要从开发者的角度一下子逾越到架构设计这种思想转变难度是比较大的，所以 DDD 的话题就暂且到这了，等到对其有更深入的理解后我会再做修改或令其一篇文章作通俗的介绍。

如何划分服务边界？

  • 按业务模型划分
  • 按功能分类划分
  • 如果做的是产品，早期还可以按照开发团队能力来划分，按照先大后细的原则
