关于接口故障的应对方案

降级

  降级指的是在系统出现过载现象和功能不可用时的应对方式。例如对服务或者功能进行优先等级划分，当系统出现过载时可以采用优先策略原则进行丢车保帅手段以保持核心服务的可用性。个人理解降级大体可以划分为两种：服务降级 和 功能降级。

  服务降级，通俗点来说就是当系统出现过载时，为了保证核心服务的可用性而削减非核心功能的可用性以求达到为“丢车保帅”。例如博客系统发现系统过载，将博客服务降级为只能读博客而不能写博客，有或者像淘宝这样的大流量电商平台在双十一抢购时可能会暂时先屏蔽不太重要的服务以提高核心服务的可用性等。

  功能降级，当非核心功能不可用时采取备用处理方案。例如电商系统在下单成功后需要发送手机短信通知买家，但由于手机短信服务不可用且重试数次后还是失败，那么就可以进入降级操作，这时可以由原本发送手机短信改为发送邮件，如果发送邮件都不可行则舍弃通知操作。

  功能降级 和 服务降级 的区别在于，功能降级的触发点是服务不可用而不是系统资源短缺。例如，调用A服务时发生了服务熔断，这时就可以采用B服务来代替A服务，又或者进行回退操作。显然这时并不是系统资源短缺，而是服务不可用导致服务质量降低。

常见的降级手段

容错

  容错，从字面意思就很容易理解，就是容许发生错误。通俗点说就是系统能够化解运行时发生的某些错误。容错其实是一种能力，即能够化解多种错误就意味着容错能力较强，反之则较弱。例如接下来要说的 Hystrix 就提供熔断机制来实现系统中服务间调用的容错性。 Hystrix的基本思想是对被调用服务进行可用性评估来判断是否需要开启断路器，断路器的作用是让服务调用能够做到快速失败。所以简单点说就是如果Hystrix认为服务a不可用，那么接下来调用服务a的操作都会快速失败。通过快速失败操作可以避免服务雪崩现象。

限流

  如果说降级是从系统功能优先级的角度考虑如何应对故障和过载，那么限流则是从用户访问压力的角度来考虑如何应对故障和过载。

  限流指的是只允许系统能够承受的访问量限度内进来请求处理，超出系统访问能力的请求将
被丢弃。限流的主要手动可以划分为两种，分别是 基于请求的限流 和 基于资源的限流

  • 基于请求限流：请求阈值、请求频率、接口调用频率

  • 基于资源限流：线程数 、数据库连接数、请求队列长度

请求排队

  请求排队其实是对限流的不同，因为限流会将系统承受范围之外的请求直接拒绝掉，但是在某些场景请求是不允许丢失的，例如经常被人谈论的 秒杀系统 就使用到请求排队来缓解请求处理压力。请求排队可以将请求暂时缓存起来再处理，可以做到 流量削峰 和增大请求处理吞吐量 等效果。例如消息系统Kafka就提供消息持久化功能。但需要注意的是请求排队其实是一种降低系统可用性的操作，因为请求并不会被马上处理，所以用户并不会马上得到处理结果。

关于服务雪崩与断路器

  如图所示，服务雪崩其实是一种正反馈。指的是在调用链中因为某个服务的不可用而导致级联不可用，最后甚至导致整个系统不可用的现象。

  那么如何避险服务雪崩呢？最简单的方法就是引入调用超时机制，即做到尽早地放弃调用从而释放连接资源，以免资源因为长时间等待而耗尽。除了超时机制之外还可以引入断路器，所谓的断路器其实就是熔断的实现，即一旦发现服务不可用，则阻断该服务被调用从而避免不必要的资源消耗。

图解断路器

城池 Server C 被大魔王攻陷。

从Server C来到Server A、Server B必须经一条桥。勇者M挺身而出将桥砍断，以免大魔王进一步的侵略。勇者M驻守在桥的一端待事态平静再考虑把桥修复。

在小故事中，勇者M就是断路器，Server C 就是已被发现的不可用服务。断路器可以阻断 Server A 、Server B 对 Service C 的调用，让其在调用时快速失败来避免不必要的资源消耗。

Netflix Hystrix

基本介绍

  Hystrix是Netflix开源的分布式服务容错框架，提供：超时控制、断路器、资源隔离、降级、请求缓存和合并 等功能。

命令模式实践

  • Hystrix操作采用的是命令模式，既将调用者和执行者进行解耦，因此可以解决依赖服务之间的级联故障。

命令模式优缺点

  • 可做到调用者和执行者解耦。（CQRC架构就基于命令模式，而这里的Student就是聚合对象）
  • 扩展新命令容易
  • 随着新命令的增多会是系统变得复杂

Hystrix工作流程图

  1. 接收到请求后构建command。command通常有以下两种：
    • HystrixCommand：依赖的服务返回单个操作结果
      • execute()：同步执行
      • queue()：异步执行
    • HystrixObservableCommand：依赖的服务返回多个操作结果
      • observe()：返回一个可监听对象Observable， 表了操作的多个结果，但属于Hot Observable，即只能监听到部分
      • toObservable() ：功能和observe()类似，但属于Cold Observable，即全部监听
      • 需要说的是Observable不是JDK中的，而是RxJava中的，RxJava是一个响应式框架
  2. 判断是否开启了command的缓存功能，如果有则直接返回上一次command的响应结果。否则进入下一阶段
  3. 判断是否开启了断路器，如果断路器被打开则直接触发fallback操作。如果断路器没有开启，则判断线程池或信号量是否已经到达预先设置的阈值，如果已经到达阈值则使command直接失败进入fallback操作，反之执行命令
  4. 执行任务时，需要检测调用的依赖服务是否发生异常或是否出现调用超时。如果存在异常或超时，则通知断路器来统计信息以判断是否有必要开启断路器来阻断后面对该服务的调用，随后进入fallback操作。反之返回最终结果
  5. 其中执行fallback操作时也可能会失败，失败则会抛出异常。

断路器工作原理

在Hystrix中断路器有三种状态：

  • CLOSED ：关闭，可以直接请求调用依赖服务
  • OPEN ：开启，请求调用依赖服务会被快速失败
  • HALF-OPEN ：半开启，允许尝试性的请求调用依赖服务
  • 关于滑动窗口可以参考另一篇文章：关于系统的限流问题

调用隔离

  如图所示，Hystrix可以为每个调用依赖服务分配独立的线程池来做到依赖服务之间互相隔离。这种操作也被称为“船舱隔离”。

基于 船舱隔离 的思路Hystrix完成以下操作

线程隔离和信号量隔离的优缺点：

  • 线程池隔离：
    1. 数量可控且受保护，不受客户端恶意调用影响
    2. 待命线程能够加速请求速度
    3. 提供内置的并发性
    4. 多线程都有的问题，就是调度和上下文切换到最后增加计算机开销
  • 信号量隔离：
    1. 信号量实质就是个计数器，比线程池开销要低
    2. 不支持超时和退出

在Hystrix中，提供 execution.isolation.strategy 属性来设置隔离策略。

请求合并

  • Hystrix提供了相同command请求合并的功能
  • 使用请求合并可以有效地减少请求连接，降低负载

请求缓存

  • 请求缓存可以在command相同时，返回上一次的响应内容（成功的）
  • 从图中可以看到，缓存采用ThreadLoad实现。ThreadLoad为线程变量，即只符合单个请求范围内的操作
    1. 例如请x中有a、b、c三个请求都是相同command，且是有序的（a->b->c）
    2. 那么请求缓存会将第一个成功的a返回结果保存到ThreadLoad
    3. 接下来的请求会到ThreadLoad中判断是否为相同的command，如果是则直接返回缓存
  • 请求缓存可以杜绝重复操作
  • 使用缓存必然存在一致性问题，所以需要注意使用

Hystrix提供了清楚缓存的方法，在有必要时可以用于清楚缓存

相关注解

@HystrixCommand
  • commandKey ：命令标识
  • groupKey ：命令集合标识
  • commandProperties ：命令属性
  • threadPoolKey ：线程池标识
  • threadPoolProperties ：线程池属性
  • fallbackMethod ：降级（备用）方法
  • defaultFallback ：默认降级方法，和fallbackMethod一样但方法不能有参数
  • ignoreExceptions ：忽略异常（不触发fallback）
  • observableExecutionMode ：执行RxJava的Observabl执行模式，默认 ObservableExecutionMode.EAGER标识马上执行，而LAZY则是有订阅者时才执行
@HystrixCollapser
  • collapserKey：合并标识
  • scope：操作域。默认为Scope.REQUEST，即请求级别
  • collapserProperties：合并属性
  • batchMethod：批处理命令的方法名称
@HystrixProperty
  • name：属性名称
  • value：属性值
