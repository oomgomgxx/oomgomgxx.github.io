JVM常用调优参数与垃圾回收器相关内容

内容修订

  • 2019年5月11日 16:40:05 — 丰富垃圾回收器内容

常用配置项

基本说明
1）有-XX的为临时参数，可能会随着Java版本升级而改变或被删除
2）有-XX的参数如果赋值的时候需要使用=

参数 说明
-XX:+PrintGC / -Xlog:gc 打印GC信息
-XX:+PrintGCDetails / -Xlog:gc* 打印GC详细信息
-Xloggc: /path/file_name 将GC信息输出到指定文件（分析工具：GC Histo、GC viewer、GCeasy）
-XX:+PrintVMOptions 打印JVM的显式接收到的参数（启动时传入）
-XX:+PrintCommandLineFlags 打印JVM的显式和隐式接收到的配置参数
-XX:+PrintFlagsFinal 打印JVM所有能够设置的配置参数
-XX:+PrintGCApplicationConcurrentTime 打印GC执行期间用户应用执行时间
-XX:+PrintGCApplicationStoppedTime 打印GC执行期间用户应用停顿的时间
-Xms 指定堆的最小值内存。默认为主存的1/64
-Xmx 指定堆的最大值内存。默认为主存的1/4
-Xss 指定Java栈内存大小
-Xmn 指定新生代内存大小（一般为堆的1/3或者1/4）
-XX:NewRatio 指定新生代和老年代的比例，如-XX:NewRatio=老年代/新生代
-XX:NewSize 指定新生代初始化大小
-XX:MaxNewSize 指定新生代最大大小
-XX:PermSize 指定永久代初始化内存(默认20.75MB，大量类的应用该值尽量设置大一些，如128、192或更多)
-XX:MaxPermSize 指定永久代最大内存(默认82MB)
-XX:MetaspaceSize 指定永久代初始化内存(默认20.75MB，大量类的应用该值尽量设置大一些，如128、192或更多)；
-XX:MaxMetaspaceSize 指定永久代最大内存（默认没有限制，即直到主存耗尽；但设置阈值后一旦耗尽就出现OOM）
-XX:SurvivorRatio 指定新生代的eden和from/to的比例，如-XX:SurvivorRatio=eden/s区
-XX:MaxDirectMemorySize 指定最大可用的直接内存大小
-XX:HeapDumpOnOutOfMemoryError 指定发OOM导出相关信息
-XX:HeapDumpPath 指定发 -XX:HeapDumpOnOutOfMemoryError 文件导出路径（使用MAT工具或jstack分析）
-XX:PrintHeapAtGC 打印GC时堆的信息（jdk9或以上可使用-Xlog:gc*）
-XX:-UseTLAB 禁用TLAB策略(线程本地分配缓冲区)，即不为线程预先分配内存，默认开启
-XX:TLABSize 指定TLAB的内存大小
-XX:+DisableExplicitGC 禁用程序的Runtime.getRuntime().gc()/System.gc()，因为该操作会触发Full GC
-XX:MaxTenuringThreshold 指定对象从新生代进入老年代的年龄，默认为15
-XX:PretenureSizeThreshold 指定”大对象“阀值
–XX:TargetSurvivorRatio 该值默认为50，用于幸存者区域满时重新计算晋升年纪。需要注意的是该值越大，计算出来的晋升年龄就越大（不会超过MaxTenuringThreshold）。换个角度说就是当幸存者区域满时，增大该值可以让对象在幸存者区域逗留更久，即这时的晋升率会降低，反之晋升率越高。
-XX:+UseAdaptiveSizePolicy JVM将会动态调整Java堆中各个区域的大小以及进入老年代的年龄，此时–XX:NewRatio和-XX:SurvivorRatio将会失效，而JDK1.8是默认开启 -XX:+UseAdaptiveSizePolicy 配置项的
-XX:+UseStringDeduplication 开启字符串去重（只适用G1垃圾回收器）。开启该功能后 JVM 会在 GC 期间做响应的去重操作
-XX:StringDeduplicationAgeThreshold 字符串需要经历指定次数才会被列为关注对象（默认为3次）。即默认情况下只有经过 3 次 GC 的字符串才会进行去重操作
-XX:+TraceClassLoading 打印类加载器的加载信息
-XX:MinHeapFreeRatio GC 过后如果可用空间少于该阈值则触发扩容，默认为百分之40

即时编译器相关

client模式

  • client编译器也称为c1编译器

  • 比c2编译器更快启动，且编译速度更快

  • 着重低延时，一般用于客户端程序，如GUI程序

  • 使用启动参数-client指定

server模式

  • server编译器也称为c2编译器
  • 因为需要预热（以方法为单位观察其调用次数高的进行编译，热点代码），所以开启编译比c1慢
  • 也因为有预热的原因，所以对执行的代码更佳了解，编译出来的代码更加高效
  • 64位虚拟机默认，也使用启动参数-server指定
  • 需要注意的是，64位虚拟机不能手动设置-client模式，也就是执行使用server模式

混合模式

在Java8开始，默认使用混合模式进行即时编译。

查看模式：java -version

混合模式下有4种编译等级

  1. 解析执行代码
  2. 简单c1编译代码
  3. 受限c1编译代码
  4. 完全c1编译代码
  5. c2编译代码

注：多数情况下方法的第一次编译都是3，但如果方法频繁被调用就会进行4

配置项

修改jvm编译模式

参数 说明
java -Xint 单模式执行（解析执行）
java -Xcomp 单模式执行（编译执行）

即使编译器参数

参数 说明
-XX:+PrintCompileThreshodl 打印JIT日志
-XX:CompileThreshodl 指定编译阀值。既方法调用次数超过该值就进行JIT编译，client模式默认1500，server模式默认10000
-XX:+TieredComilation 开启即时编译器的多级编译（jdk1.7用）
-XX:ReservedCodeCacheSize 指定即时编译器代码缓存大小。64位Java8默认240M、64位Java7默认48M（开启分层编译默认为96M）。需要注意的是，一旦代码缓存耗尽，即时编译器就不会再编译，所以如果机器有足够的内存，可以尽量分配大一些，例如1G~2G
-XX:InitialCodeCacheSize 指定即时编译器代码缓存初始化大小

垃圾收集器相关

垃圾回收前置知识

根节点枚举

  • 查找出用于可达性分析的 GC Root 集合
  • 哪些引用能够做 GC Root (可通过MAT工具查看)
    • 全局实例
      • 常量实例
      • 类静态成员实例
      • ClassLoader
    • JNI实例
    • 线程实例
    • 栈帧中局部变量表中应用的实例
    • 锁对象实例
  • 该过程会发生短暂的 STW

安全点

  • 用户线程需要达到某个点才会停下来让 GC 回收，该点被称为安全点
  • 常见安全点有哪些
    • 方法调用时
    • 循环调转时
    • 异常跳转时
  • 线程停中断的方式
    • 抢占式中断：系统控制所有用户线程到达安全点后停下
    • 主动式中断（Java使用）：设置中断标识。用户线程运行时需要不断检查该标识来判断是否需要主动中断

记忆集（RS）

  • 用于记录从非收集区域指向当前收集区域的引用集合的抽象数据结构
  • Hotspot中记忆集的实现叫卡表，可以简单将其理解为一个标识数组
  • 卡表由一个个卡页组成，而每个卡页则管理着一条快内存（512byte），当卡页发存在非收集区域指向当前收集区域的引用时，对应卡表的位置状态就会改为1（默认0）以表示存在跨区引用
  • 卡表（卡页）的维护由写屏障来完成。可以将写屏障理解为一个AOP操作，当对象存在跨区引用时，写屏障就会将其记录到卡表中

三色标记

  • 实现并发的可达性分析
    • 白色：表示当前对象还没被GC访问过，但如果分析结束对象还是白色则表示该对象不可达
    • 黑色：表示当前对象以及其所有引用都已经被GC访问过了，这种对象是绝对存活的对象，因此无需再进行第二次扫描
    • 灰色：表示当前对象已经被GC访问过，但其引用还没被GC扫描
  • 三色标记会存在漏标的问题，漏标发生后会导致对象被消失。会出现漏标的情况有以下2种：
    • 受到用户线程并发插入引用
    • 受到用户线程并发删除引用
  • 漏标问题解决方案
    • 增量更新（针对黑色对象）：在扫描对象引用的同时，记录下被用户线程并发插入的引用，然后等分析过后再重新从新增了引用的黑色节点开始扫描。例如 CMS 的 remark 阶段
    • 原始快照 snapshot-at-the-beginning （针对灰色对象）：在扫描对象引用的同时，记录下被用户线程并发删除的引用，然后等分析过后再重新从删除了引用的灰色节点开始扫描。例如 G1 的 remark 阶段

G1 收集过程

G1的（逻辑分代）区域类型：

  • E：新生代Region
  • S：幸存者Region
  • O：老年代Region
  • H：大对象Region（默认如果对象大小超过普通Region一半就会存放到这里）
  • 灰色区域：可用Region空间

其它较为新颖的GC

Epsilon收集器（JDK11）

  1. Epsilon垃圾回收器的目标是控制内存分配，但是不执行任何实际的垃圾回收工作。
  2. 它提供一个完全消极的 GC 实现，分配有限的内存资源，最大限度的降低内存占用和内存吞吐延迟时间。
  3. 用途：性能测试、内存压力测试、vm接口测试、极度短暂job任务、延时改进、吞吐量改进
参数 说明
-XX:+UseEpsilonGC 启用Epsilon GC

Shenandoah收集器（JDK12）

  1. 着重停顿，处于实验阶段，和G1一样采用分区算法管理内存
  2. 和G1一样与应用线程并发工作
  3. Shenandoah是Redhat开发的，在商用版JDK中并没有Shenandoah。换句话说Shenandoah只在OpenJDK中出现
参数 说明
-XX:+UseShenandoahGC 开启ShenandoahGC
-XX:+AlwaysPreTouch 使用所有可用的内存分页，减少系统运行停顿，为避免运行时性能损失
-XX:+UseTransparentHugePages 提高大堆的性能，同时建议在Linu上使用

GC性能指标

  • 吞吐量 = 应用程序执行时间 / 运行总时间
  • 执行频率：一段特定时间GC触发的频率，越少越好
  • 反应速度：从对象成为垃圾到被回收的时间
  • 暂停时间，越短越好

手动触发FullGC的方法

  • System.gc()
  • jcmd 进程号 GC.run
  • Jconsole内存面板“执行GC”按钮

锁相关配置

参数 说明
-XX:+UseSpinning 开启自旋锁（jdk1.6或更高版本默认开启）
-XX:PreBlockSpin 指定自旋次数（默认为10次），当线程自旋到这个阀值都没有获取到锁的话就会挂起（挂起并不会释放资源）
-XX:+UseBiasedLocking 开启偏向锁（默认开启），关闭偏向锁 -XX:-UseBiasedLocking
-XX:+BiasedLockingStartupDelay 指定偏向锁的启用延时时间（默认为4秒），可以设置为0，既伴随JVM启动就马上启用 -XX:+BiasedLockingStartupDelay=0
-XX:+UseHeavyMonitors 直接启用重量级锁
-XX:+DoEscapeAnalysis 开启内存逃逸分析
-XX:+EliminateLocks 开启锁消除（需要server模式支持）

监控工具

命令行工具

jps

查看正在运行的 JVM 进程

jstat

根据 PID 输出虚拟机运行时信息（譬如垃圾回收信息）

  • jstat -class -t PID [定时检测时间ms] [n，表示抓取多少次]：显示指定 PID 程序的类加载相关信息
  • jstat -gcutil PID [定时检测时间ms] [n，表示抓取多少次]：显示指定 PID 程序的GC相关信息
  • jstat -compiler PID [定时检测时间ms] [n，表示抓取多少次]：显示指定 PID 程序的 JIT 编译相关信息

例子：以下表示每隔 1s 查询一次

S0：幸存者0内存使用率，既from区

S1：幸存者1内存使用率，既to区

E：eden区内存使用率

O：老年代内存使用率

M：方法区内存使用率

CCS：压缩类内存使用率

YGC：新生代GC，既然 Minor GC 次数

YGCT：执行新生代 GC 所消耗的时间

FGC：Full GC 次数

FGCT：Full GC 所消耗的时间

CGT：GC操作总共消耗的时间

jstack

根据 PID 输出线程栈信息（线程状态，死锁信息）

  • jstack -l PID
  • jstack -m PID：打印包含 native 方法在内的线程栈信息。不推荐普通场景使用，因为可能会导致挂起内核线程从而引起bug。例如类加载死锁时可以用 jstack -m 来观察死锁信息，因为在 JVM 成面来是看不到类加载死锁的，看到的只是等待锁，即线程处于 BLOCKED 状态而不会检测到死锁信息

jinfo

根据 PID 输出当前JVM配置信息

  • jinfo PID：包含 环境变量参数、JVM flags、JVM 参数
    • 环境变量参数：JVM通过系统环境变量读取到的配置
    • JVM flags：当前 PID 进程所显示指定的 JVM 配置项，例如 -XX:+UseParallelGC、-XX:+PrintGC 等
    • JVM 参数：例如 -Xms700m 、-Xmx700m 、-Dfile.encoding=GBK -Duser.language=zh 等
  • jinfo -flags PID：单纯当前 PID 进程所显示指定的 JVM 配置项
  • jinfo -flag 配置项 PID：动态修改或添加 JVM 配置项，例如

jmap

查根据 PID 输出堆信息

  • jmap -heap PID：打印指定 PID 程序的堆信息（JDK1.8）

  • jmap -dump:format=b,live,file=文件名 PID：导出指定 PID 程序的dump（可使用mat工具分析）

    • -dump:表示将堆信息进行转存，其中format=b是参数，即二进制形式。live参数表示只抓取存活对象的信息（因此会触发FullGC）。file参数用于执行输出文件

  • jmap -histo:[对象状态] PID：查看对象信息，histo包含选项：live（存活对象）、all、file（dump文件）
