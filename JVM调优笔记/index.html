JVM基础参数

基本说明
1）有-XX的为临时参数，可能会随着Java版本升级而改变或被删除
2）有-XX的参数如果赋值的时候需要使用=

参数 说明
-XX:+PrintGC/-XX:+Xlog:gc 打印GC信息
-XX:+PrintGCDetails/-XX:+Xlog:gc* 打印GC详细信息
-Xloggc:filename 将GC信息输出到指定文件（分析工具：GC Histo、GC viewer、GCeasy）
-XX:+PrintReference 打印对象引用类型
-XX:+TraceClassLoading 打印类加载
-XX:+TraceClassUnLoading 打印类卸载
-XX:+PrintVMOptions 印jvm的显式参数（启动时传入）
-XX:+PrintCommandLineFlags 打印jvm的显式和隐式参数
-Xms 指定堆的最小值内存
-Xmx 指定堆的最大值内存
-Xss 指定Java栈内存大小
-Xmn 指定新生代内存大小（一般为堆的1/3或者1/4）
-XX:NewRatio 指定新生代和老年代的比例，如-XX:NewRatio=老年代/新生代
-XX:NewSize 指定新生代初始化大小
-XX:MaxNewSize 指定新生代最大大小
-XX:PermSize 指定永久代初始化内存(默认20.75MB，大量类的应用该值尽量设置大一些，如128、192或更多)
-XX:MaxPermSize 指定永久代最大内存(默认82MB)
-XX:MetaspaceSize 指定永久代初始化内存(默认20.75MB，大量类的应用该值尽量设置大一些，如128、192或更多)；
-XX:MaxMetaspaceSize 指定永久代最大内存（默认没有限制，即直到主存耗尽；但设置阈值后一旦耗尽就出现OOM）
-XX:SurvivorRatio 指定新生代的eden和from/to的比例，如-XX:SurvivorRatio=eden/s区
-XX:MaxDirectMemorySize 指定最大可用的直接内存大小
-XX:HeapDumpOnOutOfMemoryError 指定发OOM导出相关信息
-XX:HeapDumpPath 指定发 -XX:HeapDumpOnOutOfMemoryError 文件导出路径（使用MAT工具或jstack分析）
-XX:PrintHeapAtGC 打印GC时堆的信息（jdk9或以上可使用-Xlog:gc*）
-XX:-UseTLAB 禁用TLAB策略(线程本地分配缓冲区)，即不为线程预先分配内存，默认开启
-XX:TLABSize 指定TLAB的内存大小

即时编译器相关

client模式

  • client编译器也称为c1编译器

  • 比c2编译器更快启动，且编译速度更快

  • 着重低延时，一般用于客户端程序，如GUI程序

  • 使用启动参数-client指定

server模式

  • server编译器也称为c2编译器
  • 因为需要预热（以方法为单位观察其调用次数高的进行编译，热点代码），所以开启编译比c1慢
  • 也因为有预热的原因，所以对执行的代码更佳了解，编译出来的代码更加高效
  • 64位虚拟机默认，也使用启动参数-server指定
  • 需要注意的是，64位虚拟机不能手动设置-client模式，也就是执行使用server模式

混合模式

在Java8开始，默认使用混合模式进行即时编译。

查看模式：java -version

混合模式下有4种编译等级

  1. 解析执行代码
  2. 简单c1编译代码
  3. 受限c1编译代码
  4. 完全c1编译代码
  5. c2编译代码

Note：多数情况下方法的第一次编译都是3，但如果方法频繁被调用就会进行4

其他

修改jvm编译模式

参数 说明
java -Xint 单模式执行（解析执行）
java -Xcomp 单模式执行（编译执行）

即使编译器参数

参数 说明
-XX:+PrintCompileThreshodl 打印JIT日志
-XX:CompileThreshodl 指定编译阀值。既方法调用次数超过该值就进行JIT编译，client模式默认1500，server模式默认10000
-XX:+TieredComilation 开启即时编译器的多级编译（jdk1.7用）
-XX:ReservedCodeCacheSize 指定即时编译器代码缓存大小。64位Java8默认240M、64位Java7默认48M（开启分层编译默认为96M）。需要注意的是，一旦代码缓存耗尽，即时编译器就不会再编译，所以如果机器有足够的内存，可以尽量分配大一些，例如1G~2G
-XX:InitialCodeCacheSize 指定即时编译器代码缓存初始化大小

垃圾收集器相关

Serial垃圾收集器

  • 32位JVM默认
  • 单线程回收垃圾
  • MinorGC和FullGC时所有应用线程停顿清理堆空间
  • FullGC时会对老年代进行压缩整理

Throughput（Parallel）垃圾收集器

  • 64位JVM默认
  • 着重于吞吐量（ParallelGC），因此提供了尽力参数，如尽量达到吞吐时间-XX:+MaxGCPauseMillis
  • 多线程回收垃圾(代价是消耗更多CPU资源)
  • MinorGC和FullGC时所有应用线程停顿清理堆空间
  • FullGC时会对老年代进行压缩整理
  • ParallelGC和ParNewGC一样是并行回收GC（回收垃圾），只不过ParallelGC着重程序吞吐量

CMS垃圾收集器

  • 高吞吐垃圾收集器，但需要充足的CPU资源
  • MinorGC时所有应用线程停顿清理堆空间
  • 多线程并发回收垃圾(代价是消耗更多CPU资源)
  • FullGC时不停顿应用线程，使用后台线程间歇对老年代进行扫描和回收
  • 进行垃圾回收时可以做到极地停顿时间
  • 不堆内存做压缩整理，因此会出现内存碎片化
  • 无法获取所需CPU资源 和 堆过度碎片化时，自动转为Serial垃圾收集器，停顿所有线程做垃圾回收并做内存压缩，然后再转回CMS

G1垃圾回收器

  • Java9默认
  • 针对4G以上的大堆
  • 新生代和老年代通用垃圾回收器
  • 将对划分为若干个区域
  • MinorGC时所有应用线程停顿清理堆空间，存活对象复制到survivor区
  • 多线程回收垃圾(代价是消耗更多CPU资源)
  • 后台线程回收老年代垃圾大多无需停顿应用线程
  • 因为堆分为若干个区域，采用的是复制算法，因此可以做到内存压缩整理
  • G1比CMS更不容易遭遇FullGC

GC选择参考

  • 单核CPU采用Serial
  • 多核CPU情况下，应用线程CPU使用率高则选择CMS（因为后台线程只是间歇运作），否则选择Parallel
  • 在CPU资源充足且要求高吞吐量时，选择CMS，且堆在4G以下（FullGC时不停顿应用线程）
  • 4G以上的大堆推荐使用G1（因为堆划分为若干个区域，回收效果更佳），且回收对象时CMS需要扫描整个堆，因此堆越大CMS性能越低

GC参数

参数 说明
-XX:+UseSerialGC 新生代和老年代同时使用串行垃圾收集器
-XX:+UseParNewGC 新生代采用ParNewGC（并行回收），老年代采用SerialOld垃圾收集器
-XX:+UseParallelGC 新生代采用ParallelGC，老年代采用串行垃圾收集器;ParallelGC和ParNewGC一样是并行回收GC，只不过ParallelGC着重程序吞吐量
-XX:+UseParallelOldGC 新生代和老年代都采用ParallelGC
-XX:+MaxGCPauseMillis 指定最大的垃圾回收时间的期望值(0以上值为整数)，如果GC回收时长不满足该配置则会自动调整堆参参数，如减少堆的大小来缩短回收时间
-XX:+UseAdaptiveSizePolicy 开启堆的自动调整功能（默认开启），配合-XX:+MaxGCPauseMillis使用来达到满足回收时间要求，但如果[堆初始化值和最大值一致，新生代始化值和最大值一致]则该配置自动关闭
-XX:+GCTimeTatio 指定吞吐量大小的期望值（0-100整数，默认19）
-XX:+ParallelGCThreads 指定回收垃圾时的工作线程数
-XX:+UseConcMarkSweepGC 新生代采用ParNewGC（并行回收），老年代采用CMS
-XX:+ConcGCThreads 指定CMS并发线程数
-XX:CMSInitiatingOccupancyFraction 指定CMS回收阀值（默认68），即当老年代内存使用率到达68%就会触发一次GC; 需要注意的时，如果老年代的内存使用率增长得过快，则CMS可能会回收失败，这时jvm会使用串行回收GC来回收
-XX:UseCMSCompactAtFullCollection CMS回收过收过后进行内存碎片整理（CMS，标记清除算法），需要注意的是这个操作不是并发的
-XX:CMSFullCGsBeforeCompaction 指定多少次GC后进行一次内存碎片整理
-XX:+CMSClassUnloadingEnabled 指定CMS可以回收永久代中的类信息
-XX:+UseG1GC 使用G1垃圾回收器（JDK9默认）
-XX:MaxGCPauseMillis 指定最大停顿期望值
-XX:+ParallelGCThreads 指定回收时的工作线程数
-XX:InitiatingHeapOccupancyPercent 指定堆内存使用率阀值（默认45，即45%）触发并发标记周期
-XX:-+DisableExplicitGC 禁用程序的Runtime.getRuntime().gc()/System.gc()，因为该操作会触发Full GC
-XX:MaxTenuringThreshold 指定对象从新生代进入老年代的年龄，默认为15
-XX:PretenureSizeThreshold 指定大对象进入老年代的阀值

Epsilon收集器（JDK11）

  1. Epsilon垃圾回收器的目标是控制内存分配，但是不执行任何实际的垃圾回收工作。
  2. 它提供一个完全消极的 GC 实现，分配有限的内存资源，最大限度的降低内存占用和内存吞吐延迟时间。
  3. 用途：性能测试、内存压力测试、vm接口测试、极度短暂job任务、延时改进、吞吐量改进
参数 说明
-XX:+UseEpsilonGC 启用Epsilon GC

Shenandoah收集器（JDK12）

  1. 着重停顿，处于实验阶段，和G1一样采用分区算法管理内存
  2. 和G1一样与应用线程并发工作
  3. Shenandoah是Redhat开发的，在商用版JDK中并没有Shenandoah。换句话说Shenandoah只在开源版本中存在
参数 说明
-XX:+UseShenandoahGC 开启ShenandoahGC
-XX:+AlwaysPreTouch 使用所有可用的内存分页，减少系统运行停顿，为避免运行时性能损失
-XX:+UseTransparentHugePages 提高大堆的性能，同时建议在Linu上使用

手动触发FullGC方法

  • System.gc()
  • jcmd 进程号 GC.run
  • Jconsole内存面板“执行GC”按钮

JDK1.6h后堆Synchronized锁的优化

自旋锁

  1. jdk1.4引入，jdk1.6默认开启
  2. 所谓的自旋锁通俗点说就是尽量避免线程每次锁竞争锁失败都要挂起（对于线程和进程的调度是需要内核进行的，所以该操作需要切换到内核态。用户态和内核态之间的切换和线程/进程的调度对于应用来说会降低应用的执行效率）而采取循环尝试获取锁的操作
  3. 适合使用自旋锁的条件：预知线程持锁时间短，换句话就是业务代码简单不耗时
  4. 从上面几点可以的出一个简单的结论，就是自旋锁是建立在预知线程持锁时间短的基础上，如果线程长时间不释放锁（如处理业务时间较长）的话在高并发的场景就会导致大量线程自旋，这样就会对CPU造成压力，因此而得不偿失。这时可以考虑关闭自旋锁功能-XX:-UseSpinning 和 -XX:PreBlockSpin 指定自选次数（默认10次，当有线程自旋超过10次，锁就会从轻量级升级为重量级）
  5. jdk1.6加入了自适应自旋，简单来说就是，根据上一条获取到锁的线程的等锁时间来做自旋次数的相应调整

锁消除

  1. 锁消除是指JVM即时编译器检测到方法中有不可能线程共享的资源，然后消除该资源上的锁。
  2. 锁消除的主要依据是通过逃逸分析进行判断，既资源如果不可能逃逸的话那么即时编译器就会认为这是线程独占的而无需加锁因此而讲锁消除

锁升级

锁模式-偏向锁

  1. （开启偏向锁功能后）第一条线程加锁后的状态，该状态下的锁有偏向性，通俗点说就是偏心与第一条获取到它（锁）的线程，偏向锁性能相当于没有加锁，因为这时JVM不会进行加解锁操作（没有竞争的前提下）
  2. 当偏向锁有两条以上线程竞争时，锁对象会退出偏向锁模式（偏向失败）并退会无所状态再根据竞争情况升级为轻量级锁
  3. 从上面可以得出结论，其实在高并发场景下偏向锁并不会带来实质的优化效果，因此在高并发环境下可以尝试禁用偏向锁以减少锁升级的开销

锁模式-轻量级锁

  1. 当锁偏向失败锁就会升级为轻量级锁，这时表示有两条以上的线程竞争同一把锁
  2. 轻量级锁状态下，线程抢锁失败将会进入自旋操作（死循环获取锁），直到获取到锁或者是自旋次数超过-XX:PreBlockSpin 后升级为重量级锁（Monitor Lock）
  3. 自旋的目的是避免线程抢锁失败后马上进入内核态从而被阻塞，这是因为频繁的用户态和内核态切换会造成系统资源开销从而导致应用程序阻塞时间加长后事务处理能力下降
  4. 轻量级锁由 JVM 中一个叫 BasicObjectLock 的对象实现，该对象包含：一个BasicLock（锁对象）、一个持有该锁的 Java 对象指针（线程对象）; BasicObjectLock 对象存放在栈帧中，其中BasicLock锁对象又维护了一个_displaced_header字段用于备份BasicObjectLock对象头的Mark Word
  5. 当一个线程持有轻量级锁时，它的对象头Mark Word 为：
    • [ptr (轻量级锁的指针)| 00（轻量级锁模式的标记位值）]
  6. 因为 BasicObjectLock 对象存放在栈帧中，所以如果判断线程是否持有该对象锁只需要查看该对象锁是否存在于线程栈中即可

锁模式-重量级锁/膨胀锁

  1. 在轻量级锁状态下，如果有某条线程自旋次数超过-XX:PreBlockSpin 就会升级至重量级锁
  2. 重量级锁的重在于需要进入内核态进行锁竞争，线程一旦竞争失败则会被挂起（不会释放资源，这是Thread的状态为BLOCKED），当持锁线程释放锁时又会被唤醒，所以这就是它重的原因，因为需要进行用户态和内核态切换并执行线程调度操作，这样会大大提高系统开销从而降低应用程序的事务处理能力

锁参数

参数 说明
-XX:+UseSpinning 开启自旋锁（jdk1.6或更高版本默认开启）
-XX:PreBlockSpin 指定自旋次数（默认为10次），当线程自旋到这个阀值都没有获取到锁的话就会挂起（挂起并不会释放资源）
-XX:+UseBiasedLocking 开启偏向锁（默认开启），关闭偏向锁 -XX:-UseBiasedLocking
-XX:+BiasedLockingStartupDelay 指定偏向锁的启用延时时间（默认为4秒），可以设置为0，既伴随JVM启动就马上启用 -XX:+BiasedLockingStartupDelay=0
-XX:+UseHeavyMonitors 直接启用重量级锁
-XX:+DoEscapeAnalysis 开启内存逃逸分析
-XX:+EliminateLocks 开启锁消除（需要server模式支持）

命令行工具

jps：

查看当前系统中正在运行的 Java 进程

jstat：

根据 PID 输出指定Java进程的虚拟机运行时信息

  • jstat -class -t PID [定时检测时间ms]：显示指定 PID 程序的类加载的相关信息
  • jstat -gcutil PID [定时检测时间ms]：显示指定 PID 程序的 GC 的相关信息
  • jstat -compiler PID [定时检测时间ms]：显示指定 PID 程序的 JIT 编译的相关信息

M：方法区内存使用率

YGC：新生代GC，既然 Minor GC 次数

    • JVM flags：当前 PID 进程所显示指定的 JVM 配置项，例如 -XX:+UseParallelGC、-XX:+PrintGC 等
    • JVM 参数：例如 -Xms700m 、-Xmx700m 、-Dfile.encoding=GBK -Duser.language=zh 等
  • jinfo -flags PID：单纯当前 PID 进程所显示指定的 JVM 配置项
  • jinfo -flag 配置项 PID：动态修改或添加 JVM 配置项，例如

jmap（注意该命令经常变动）：

  • jmap -heap PID：打印指定 PID 程序的堆信息（JDK1.8）
