分布式基础理论

CAP定理

CAP定理在2007年7月被加州大学伯克利分校的Eric Brewer教授所提出，所以CAP定理又叫做 布鲁尔定理 。而该定理的结论是分布式系统并不能同时满足：一致性、可用性、分区容错性，而只能是三者选其二。

C A P概念理解

  • （C）一致性：客户端在任意节点上读取同一个值所返回的结果应该是一致的。
    • 可以通过延迟请求应答来提高一致性。譬如在同步复制复制中，只有等待特定数量的节点同步成功才响应用户
  • （A）可用性：节点应该在合理的时间内返回合理的客户端请求结果，但结果不应包含异常和超时。
    • 可以通过 主备、集群、服务体验降级（腾出资源保障核心服务） 等手段来提高可用性
  • （P）分区容错性：即使出现网络分区，系统也依然能够提供服务。
    • 例如熔断机制就是用来提高分区容错性的

注：

  • ACID中的一致性保证的是事务前后的数据库完整性，而CAP中的一致性强调的是状态，所以”一致性”其实并不是一个泛指
  • 强一致性在网络环境中是很难做到的，因此存在网络的不定性因素影响

为什么三者只能选其二？

  • 对于分布式系统而言数据通常不会只存放在单个节点上，因为在网络环境下难以避免不会出现网络分区，所以如果数据值保存在单个节点上的话一旦出现网络分区就会导致系统不可用，所以我们可以得出结论就是分布式系统首要应该支持的就是分区容错性（P）
  • 对于如何保证分布式系统的可用性，无非就是实现系统的高可用（HA）。最常见的就是横向扩展服务节点，但是扩展节点的话就意味着增加了同步状态的成本，既节点越多同步状态所花费的时间就越多。所以换句话来说，系统可用性越高的话，那么要保证一致性就越困难，所以现今的分布式系统实际上一般都只是在一致性（C）和可用性（A）之间权衡，因为分区容错性对于分布式系统而言是必须的，如果分布式系统没有分区容错性那么就失去的分布式的意义了。
  • 譬如 Zookeeper、Etcd、MongoDB 等分布式高可用集群采用的是 CP 架构，这与其使用的共识算法/协议以及策略有关系。而 Cassandra、CounchDB、Redis 等则是 AP 架构
  • 拿 Redis 来说，Redis 的高可用方案都是在复制的基础上实现的，而复制因为网络原因而存在数据丢失的风险，譬如Master宕机导致不能及时复制以及脑裂等原因导致数据丢失。因此对于 Redis 来说它并不能保证数据的一致性，所以按目前来说一般我们只会用 Redis 来做一些对一致性要求不高的功能。除此之外个人还认为这与 Redis 的设计理念有关，因为AP架构实质是一种可用优先的数据处理方案，而 CP架构则是一致性优先。
  • 这里需要特别强调一下，CP架构的系统并不是指能够保证强一致性，它只是在可用性和一致性之间优先考虑一致性而已。譬如 Zookeeper 中的过半可用机制保证出现网络分区时少于过半的分区集群不再提供服务，然而这时一种降低可用性的方案，因为如果不这么做的话，出现网络分区后就会导致脑裂，如果继续提供服务就会导致数据不一致问题。

网络分区：指系统节点之间因为某些原因而导致不能通信

BASE理论

关于一致性的分类

  • 强一致性：写操作后的读操作一定能够读到上一次写操作（不论是在哪个节点上写）的结果
  • 弱一致性：写操作后的读操作既可能读到，也可能读不到上一次写操作的结果
  • 最终一致性：写操作后的读操作需要经过一段短暂的时间才能读到上一次写操作的结果

注意：其实个人理解一致性只有两个级别，要么是强一致要么是最终一致。而所谓的弱一致、顺序一致这些都是最终一致的其中一种特例。例如弱一致指的是达到数据版本一致的时间较为短暂，譬如第一次读没有读到最新版本而第二次读就读到了。又例如顺序一致其实是指集群处理命令是按照 FIFO 形式进行的，它并没有强调时间，而 Zookeeper 就是这种形式（强调，Zookeeper 只是处理命令 FIFO，而节点同步是没有优先级别的，所以顺序一致只体现在命令处理上）。

基本概念

  • BA：基本可用，当系统出现故障或特定需要时允许适当地延长响应时间（即允许适当地降低系统的可用性）
    • 例如 流量削峰、过载保护、服务(体验)降级、延迟响应 等手段虽然能保护系统，但同时会降低系统的可用性
  • S：软状态，指的是允许对客户端呈现过度状态，即并不要强求数据的强一致性，换句话说就是允许数据在节点间出现不一致
    • 注意：在互联网上基本是不可能做到强一致的，因为可能会出现网络分区问题。但是撇开网络分区来实现强一致性的唯一方案就是延长给客户端响应的时间，而用这段时间来同步节点间的数据，等到同步完成后才响应客户端。也就是说这是一种降低系统可用性来换取一致性的方案
  • E：最终一致性，个人理解这时对 S（软状态）的追加条件，既即使允许对客户端呈现过渡状态，但必须在一定时间后让所有节点达到状态一致

小结

个人理解 BASE 理论其实是 CAP 的一种权衡方案。即如果你不知道怎样选择 CAP 的话，就可以参考 BASE 理论

分布式事务一致性解决方案

介绍

  • 用于保障请求数据在多个服务节点之间的完整性
  • 分布式事务一致性协议大体可分为两类：强一致性协议 和 最终一致性协议
    • 强一致性协议代表：2PC 、3PC
    • 最终一致性协议代表：TCC、可靠消息模式、Event Soucing
  • 在网络环境中基本上不可能实现强一致性，这时因为存在网络分区的隐患，其次是网络传输可能受到波动影响从而导致不能保证传输的可靠性。而上面提到的 2PC 和 3PC 是强一致性协议，那它们是如何实现的呢？简单说就是降低可用性来提高一致性，既需要等待所有事务节点完成操作后才会响应用户
  • 最终一致性协议是建立在 BASE理论的基础之上的，既允许数据过渡状态的出现以及合理情况下允许适当地降低分布式系统的可用性。最终一致性协议比较适合一些对事务要求不高的场景，譬如博客系统，对于用户而言即使修改操作后不能马上得到结果也是可以接受的

强一致性方案

二段提交协议（2pc）

2pc将事务提交操作分为两个阶段： 事务预提交（第一阶段）和 提交事务（第二阶段）

事务预提交（第一阶段）：

  • 事务协调者向所有事务参与者发出事务预提交请求
  • 事务参与者接收到事务协调者发来的事务预提交请求后就进行事务日志记录（undo 和 redo）
  • 各个事务参与者响应事务协调者，告诉事务协调者是否成功记录事务日志

图片源自 servicecomb.apache.org

提交事务（第二阶段）：

  • 事务协调者向所有事务参与者发出 commit 请求（根据第一阶段事务参与者的响应结果来判断发起 commit 还是 rollback）
  • 事务参与者接收到 commit 请求后进行真正的事务提交
  • 各个事务参与者响应事务协调者
  • 事务提交完成，事务协调者响应客户端事务提交成功

中断事务(第二阶段)：

假设第一阶段中有事务参与者预提交事务失败，则事务协调者会在第二阶段发起 rollback/abort 请求。

  • 事务协调者向所有事务参与者发出 rollback/abort 请求
  • 事务参与者接收到 rollback/abort 请求后依赖 undo 日志进行回滚操作，完成后响应事务协调者
  • 事务中断完成，事务协调者告知客户端事务操作失败

2pc存在的问题：

  • 同步阻塞：在第二阶段完成前，事务参与者无法进行其他操作（即全局锁），同时客户端也是阻塞的
  • 单点问题：事务协调者存在单点故障隐患，一旦发生故障就需要人工干预才能重新启动事务
  • 存在数据不一致隐患：事务协调者发出 commit 请求，但因为网络导致只有部参与者接收命令，因此就会导致一致性问题
  • 实现成本较大：需要资源支持XA规范

2pc实践方案：

  在实际开发当中，如果想要确保多数据源操作的ACID事务就可以使用 2PC 来实现，如JTA事务 + XA规范(数据源需支持)。XA规范描述了事务管理器和资源管理器的接口，其实这里的事务管理器就是事务协调者，而资源管理器就是事务参与者。一般事务协调者是需要程序实现的（重量级web服务器除外），譬如可以用实现了 JTA 事务规范的atomikos库和Bitronix库。而XA规范譬如MySQL就实现了，因此加上JTA事务实现就能完成2pc协议了。

三段提交协议（3pc）

3pc对2pc进行了改进，主要解决的是2pc同步阻塞问题。基本原理是将2pc第一阶段进拆分，以及引入了超时机制（协调者等待超时，参与者等待超时）

CanCommit（第一阶段）：

  • 事务协调者询问事务参与者是否可以执行事务预提交操作
  • 事务参与者响应事务协调者结果

PreCommit（第二阶段）：

  • 事务协调者向所有事务参与者发出 PreCommit 预提交请求
  • 事务参与者接收到 PreCommit 后进行事务预提交操作（undo和redo）
  • 事务参与者响应事务协调者结果

DoCommit（第三阶段）：

  • 事务协调者向所有事务参与者发出 DoCommit 提交事务请求
  • 事务参与者接收到 DoCommit 后执行事务提交
  • 事务参与者响应事务协调者结果后，事务协调者响应用户结果

事务中断（第二阶段）：

如果在第二阶段事务参与者预提交失败，那么事务协调者会做以下操作

  • 给事务参与者发出中断请求让其恢复状态
  • 中断整个事务操作再响应客户端，因为这时还没有事务参与者正式提交事务

事务中断（第三阶段）：

第三阶段事务参与者执行事务提交失败，那么事务协调者会做以下操作

  • 给事务参与者发出中断请求让其恢复状态
  • 发起事务回滚请求，事务参与者会依照 undo 记录进行回滚操作
  • 获取到回滚反馈后中断整个事务操作再响应客户端

3pc的优缺点：

  • 通过询问降低了事务参与者的阻塞的几率
  • 依然存在数据不一致隐患，例如当事务协调者发出 PreCommit 之后出现了网络分区，那么事务参与者超时机制发生后依然会提交自己的事务，这样就会导致数据不一致的情况发生
  • 据目前个人了解 3pc 没有开源的实现，因此使用的普遍性不高

最终一致性方案

前置小知识

查询模式：为服务设置查询接口，在需要时可以通过该接口来查看服务的处理状态

定时校对模式：基于查询模式之上，通过定时任务主动检查服务处理状态。

补偿模式：补偿的意思实质就是重试或恢复事务：

  • 自动恢复：通过 重试机制 再次执行失败的事务操作
  • 通知运维：实在是无法回滚或者重试，则可以通过人工干预进行补偿（如一些系统有专门的运维模块）
  • 技术干预：如果以上两种手段都不可行，则可以尝试手动修改数据进行补偿（这是最糟糕的情况）

可靠消息模式+补偿+幂等 (本地消息表方案)

基本概念

可靠消息模式：

  • 用消息系统来驱动事务。如发送消息之前先将消息持久化并标注为未发送，然后再进行消息发送。一旦发送成功就将持久化的消息状态改为已发送
  • 使用补偿模式来处理未成功发送的消息
    • 使用一个定时任务来轮询消息状态，如果发现消息状态为未发送就再次进行发送。这样做的用意是避免消息发送的丢失，即保证消息发送的可靠性。当然，现在的消息系统一般都支持重试机制，即在消息发送失败后会重试一定的次数来确保消息成功发出（例如 kafka 就是使用最少成功发送一次的模式）。但要注意的是，一旦引入重试机制就可能存发生消息重复发送的隐患，所以应该在消息消费方进行幂等处理
    • 除了定时任务轮询消息表之外还可以实现 CDC 功能来进行消息发送补偿。例如 RedHat 的 Debezium 和 阿里的 Canal 这类日志挖掘工具，它们都可以捕获 MySQL 的 BinLog 来进行消息记录发送。该方式要比定时任务轮询要高效一些

解决消费幂等问题：

  • 保证消费者只消费1次，可以使用 防重表、去重token 等方案

TCC协议模式（补偿事务）

  • TCC的操作比较像2PC。TCC 主要分为2个操作：Try、 Confirm / Cancel
  • 可以对分布式事务进行回滚操作

基本原理图

从图中可以看出事务的提交和回滚是由TCC管理器来完成。还有就是使用 TCC 后要求将原来的一个方法拆分为三个。譬如付费服务pay()，拆分成 tryPay()、confirmPay()、cancelPay() 来分别代表预提交、确认提交、取消操作（回滚）。所以 TCC 是一种侵入式的事务实现，其中 confirm 和 cancel 操作均由TCC管理器自动调用。

理解例子

// 预付费
public void tryPay(){

    // 更新订单状态
    orderService.updateOrderStatus("正在付费")

    // 预付费操作
    //1. 查看是否有足够的余额，例如还要100,那么20元商品就满足扣费要求
    //2. 冻结等价于商品的费用，防止中途因为某些原因导致余额不足
    userService.freezeMoney(20); // 将商品金额保存到冻结金钱字段
}

// 正式付费
public void confirmPay(){

    // 正式扣费，从冻结金钱中扣费
    userService.deductionFreezeMoney(20); 

    // 更新订单状态
    orderService.updateOrderStatus("付费成功")
}

// 取消付费
public void cancelPay(){

    // 解除冻结金钱
    userService.relieveFreezeMoney(20); 

    // 更新订单状态
    orderService.updateOrderStatus("等待付费")
}

TCC存在的问题

  • 空补偿/空回滚

    • 意思是没有执行 try 步骤就执行 cancel 了
    • 出现该问题的原因是分支服务因为某些原因宕机了或者出现网络异常了，这时事务协调者触发回滚操作。这时故障分支服务回顾就会出现空补偿问题，即没有执行 try 就执行 cancel
    • 解决思路：因为TM（事务管理器）发起全局事务时会生成全局事务记录，而其中全局ID贯穿整个分布式事务调用链，因此可以再额外增加一个表来记录全局事务ID和分支事务ID。在执行完 try 后往表中插入一条记录以示 try 已经执行完，那么在执行 cancel 时就可以检查是否有记录，如果没有则表示当前操作就是一个空回滚操作

  • 幂等问题

    • 由于 TCC 事务框架可能提供重试机制，因此需要对 try、confirm、cancel 等操作进行幂等处理

  • 悬挂问题

    • 意思是 二阶段 要比一阶段的 try 先执行
    • 出现该问题的原因是协调事务期间因为网络问题事务 RPC 调用发生的乱序
    • 解决思路：和上面的空补偿处理思路一样，执行 try 时先判断表中是否存在 cancel 记录，有则无需执行 try

Event Soucing

事件溯源指的是使用事件对数据状态进行回溯重现的操作。因此Event Soucing和CQRS（命令查询职责分离）是不谋而合的，所以通常 Event Soucing和CQRS 都是同时存在的。在 Event Soucing 中事件才是一等数据，我们可以通过事件回溯得到业务数据在任何时候的状态。

一站式事务解决方案 - Seata

源码：https://github.com/seata/seata

  • Seata（一站式分布式事务解决方案）是 2019 年 1 月份蚂蚁金服和阿里巴巴共同开源的分布式事务解决方案
  • Seata 支持 AT、TCC、SAGA 和 XA 等多种模式的分布式事务实现

Seata中有3个重要角色：

  • 事务协调器（TC） ： 维护全局事务和分支事务的状态
  • 事务管理器（TM） ： 控制全局事务的开始、提交、回滚
  • 资源管理器（RM） ： 向 TC 注册分支事务 和 报告分支事务状态，管理分支事务的提交或回滚

Seata事务管理例图

一个典型的分布式事务过程：

  1. TM 向 TC 申请开启一个全局事务，全局事务创建成功后会生成一个全局唯一的 XID （XID 会在微服务调用链路的上下文中传播）
  2. RM 向 TC 注册分支事务，主要是请求TC将分支事务纳入到指定 XID 的全局事务中
  3. TM 向 TC 发起指定 XID 的全局提交或回滚决议
  4. TC 调度指定 XID 全局事务下的所有分支事务以完成提交或回滚操作

与XA规范的区别：

架构区别

区别在于seata将rm管理操作从资源层面抽离了出来，这样提高seata的通用性和资源要实现分布式协议的成本要求

与2pc区别

可以看到在seata中，如果第一阶段是成功的，那么它就不会锁定资源，相当于缩短了阻塞时间。

分布式共识性（针对故障容错）

  很多人将 Gossip、Paxos、Raft、ZAB 等算法和协议跟 2PC、3PC、TCC 等混为一谈将其统称为一致性协议。虽然这样称呼并没有什么大碍，但按个人认为这样并不利于理解他们，因为它们在于用途上并不太一样。譬如前者多见于用来解决集群节点之间的通信容错，而后者则时解决分布式事务的一致性问题，所以前者用共识性来形容而后者则是一致性这样更加贴切。

注：

  • Gossip、Paxos、Raft、ZAB 等算法和协议也被称做非拜占庭容错算法（Crash Fault Tolerance，简称CFT）。原因是一般的系统并不存在伪造和篡改消息，但如果是区块链服务就可能会存在伪造数据的节点，这时就符合 拜占庭将军问题 这种极端情况

基本介绍

  Gossip是一个非常有趣的分布式共识性协议，因为其工作原理的缘故亦被叫做流行病协议、谣言协议。常用作信息广播、叠加网络 、数据交换、故障检测等用途。譬如 Redis集群 和 ElasticSearch集群 用它来实现节点状态信息的交换，Consul 用它来管理集群成员和广播信息。

  Gossip的工作原理非常简单。 例如A节点希望与集群中的其它节点进行数据状态共享，那么它就会定期从集群中随机选取N个节点，然后与其进行数据交换。而且与A节点交换了信息的节点会变得和A节点一样会定期选取N个节点来广播自己得到的数据状态（这就是为什么叫它流行病协议的原因）。在 Gossip 协议中，一个数据状态扩散到所有节点的时间复杂度为 log(节点数)(底数=扇出数)

特点

  • 代码实现简单
  • 使用该协议的集群扩展节点比较容易。但随着节点数增加，达到最终一致性的时间就会变长
  • 容错较高。即使出现网络分区或节点宕机都不会影响集群正常工作，而且节点恢复后依然能够达到最终一致性
  • 最终一致。节点间状态要达到一致至少需要log(节点数)(底数=扇出数)这么长的时间

支持的传播方案

  • Anti-Entropy（反熵）：传播节点的所有信息。因为节点需要比对信息来达成一致性，因此成本较高
  • Rumor-Mongering（谣言传播）：仅传播节点新接收的信息（增量传播）。会自动检查信息是否已被删除，如果被删除则不进行传播

支持的行为模式

  • push模式： 传播自己接收到的新信息（可以减少网络开销）
  • pull模式：积极想其它节点拉取新的信息
  • push & pull模式： 传播自己接收到的新信息，并积极想其它节点拉取新的信息

从图中可以得知的信息是，Redis 集群是通过 Gossip 协议来广播节点状态（包含槽位信息和节点状态）。

在 Redis 集群中每一个节点都对应着不同的槽位段。例如现在 Client 向 A 节点获取一个缓存，但是 A 节点根据 key 计算后发现这个缓存并不在自己的槽位内，这时它就会根据先前通过 Gossip 协议交换回来的信息来通知 Client ，让其重定向到对应的节点中查找缓存。

Paxos算法

算法的诞生

故事摘自 Leslie Lamport 的《The Part-Time Parliamen》

  在这个千年的早期（公元 10 世纪初），爱琴海上的小岛 Paxos 是一个兴盛的商业贸易中心。经济发展带来了政治的进步。Paxos 的公民们用议会形式的政府取代了原先的神权统治。但是对 Paxos 人来说，做生意才是头等大事，城市职责反在其次，没有 Paxos 人愿意把他全部的时间投入到议会事务中。所以 Paxos 的议会必须在每个议员都可能随时缺席的情况下也能工作下去。（从这点可以看出议会所面对的问题正好对应着现今容错式分布式系统所面对的问题）

  在 Paxos 岛的议会中，每一个议员都会维护一个律簿（ledger），用来记录一系列已通过的法令，每个法令会带有一个编号。议会协议的第一个需求就是在议会议事期间律簿必须保持一致性。例如议事期间通过了法令《132:只有橄榄油可以用来做灯》，那么议员的律簿就不能篡改132编号的内容（可以为空，但如果存在132编号的法令就必须一致）。但仅仅有律簿的一致性是不够的，因为让每个律簿都保留空白也能满足一致性。所以需要另外一些要求来保证法令能最终通过并被记录在律簿中。Paxos这里的人盛行的是相互信任的氛围，所以议员愿意通过任何被提交的法令。但是他们好游历的特性却造成了问题。例如A组议员通过了法令《133：禁止在圣殿的墙上种植》后来离开了议会。随后B组议员进入议会，因为他们并不知道之前发生了什么事，所以又通过了法令《133：允许自由的艺术表达》。因此前后通过的两个法令的编号发生了冲突，导致AB两组议员的律簿失去了一致性。因此议会作出规定，就是如果议员中的多数议员都在议会厅中，并且在一个足够长的时间内没有人进出议会厅，那么任一被某个议员提议的法令都将会被通过，并且每一个被通过的法令都会出现在议会厅中每个议员的律簿上（引入多数派概念）。

  好了，Paxos岛的故事就摘到这里，故事当然是还有后续的，但到这里个人觉得已经足够了解Paxos算法的诞生和Paxos中存在的基本问题了。譬如 Paxos 这个名字源自 Paxos 岛（不确定是否存在），而 Paxos 算法实质就是用来解决议会立法顺利的一种方法论。

Paxos特点

  • 支持多个节点同时发出提案
  • 支持 Leader 选举（注意Leader选举在Paxos算法中不是必须的，选举Leader有两种情况，一是Proposer Leader，二是Learner Leader，两者职责不同，详细参考下文）
  • 强一致性/最终一致性（算法实现方式不同，效果就不同）
  • 高可用。只要节点过半就能够正常提供服务

关系模型

  • 议会：相当于分布式服务应用
  • 议员：服务节点
  • 公民：客户端
  • 法令：服务节点数据状态

算法角色

注意：在 Paxos 算法中一个线程可以同时拥有多个角色的身份

  • Proposer：负责接收处理 Client 请求和发起提案的议员
  • Acceptor：负责接收处理 Proposer 的 prepare请求 和 accept请求（通过提案）的议员
  • Learner：记录通过的提案，但不参与决策。相当于故事中的律簿

算法要求

提案格式：[M，V]。M为提案编号，V为提案Value

P1：Acceptor 必须批准它收到的第一个提案，且 Acceptor 可以批准多个提案。

P1a：只要 Acceptor 尚未响应过任何编号大于 Mn 的 prepare请求，那么它就可以接受这个编号为 Mn 的提案。

P2：提案 [M0，V0] 被选定，那么所有比编号 M0 更高且被选定的提案，其 Value 值必须也是V0。

P2a：提案 [M0，V0] 被选定，那么所有比编号 M0 更高且被 Acceptor 批准的提案，其值必须也是V0。

P2b：提案 [M0，V0] 被选定，那么之后 Proposer 产生编号更高的提案其 Value 都是V0。

生成提案和接收提案

Proposer生成提案

当 Proposer 生成一个 Mn 提案时，必须知道当前议会中某个将要或已被半数批准的编号小于 Mn 但为最大编号的提案。并且 Proposer 会要求 Acceptor 不能再接收编号小于 Mn 的提案。

  1. Proposer 选择一个新的提案编号 Mn，并向 Acceptor 集合成员发送请求（该请求称为prepare请求）。当Acceptor 接收到 prepare请求 后会作出如下回应：
    • Acceptor 向 Proposer 承诺不再批准编号小于 Mn 的提案
    • 如果 Acceptor 在这之前已经批准过提案，那么将返回编号小于Mn 且提案编号为最大的提案给Proposer（主要是获取其中的value值）
  2. 如果 Proposer 接收到半数以上的 Acceptor 响应自己发出的prepare请求，Proposer就可以生成编号为 Mn，Value 为 Vn 的提案了。注意，这里的 Vn 是 Acceptor 响应中得到的。不过也可能会出现一种情况，就是 Acceptor 在这之前并没有批准过提案，这时 Proposer 可以为编号为 Mn 的提案设置任意的 Value 值

成功确立和生成提案后 Proposer 就会将其发送给某个 Acceptor 集合并期待他们通过（该请求称为 accept请求）。注意，这时接收 accept 请求的 Acceptor 集合不一定是原来处理 prepare 请求的集合。这样行的通的原因基于集群中任意2个过半数的集合都肯定至少有一个公共的 Acceptor 成员（因为集合中成员的状态肯定是一致的）

Acceptor请求接收

  1. 可以在任何时候响应 prepare请求
  2. 在不违背对 Proposer 的承诺的前提下，响应 accept请求

算法陈述

Paxos算法分为两个阶段。其中第一阶段因为需要达到半数以上的Acceptor响应才能进入第二阶段，所以第一阶段也称为 投票阶段。而第二阶段其实就是 提交阶段

第一个阶段（投票阶段）：

  1. Proposer选择一个提案编号Mn，然后向Acceptor集合中某一个过半的子集发送一个编号为Mn的perpare请求
  2. 当Acceptor接收到编号为Mn的prepare请求后，如果编号大于当前Acceptor已经响应的所有prepare请求的编号的话，Acceptor 就会向当前发出prepare请求的Proposer承诺不会再批准任何编号小于Mn的提案，并将编号小于Mn但为目前最大的编号的提案反馈给Proposer。否则拒绝请求同时返回目前最大的编号的提案反馈给Proposer

第二阶段（提交阶段）：

  1. Proposer如果得到半数以上的Acceptor响应后 Proposer 就会再发送一个针对 [Mn，Vn] 提案的accept请求给某个过半的Acceptor子群
  2. 当Acceptor收到针对 [Mn，Vn] 提案的accept请求，如果Acceptor在这之前尚未对大于该编号的提案请求作出过响应，那么它就会通过当前的提案

注意，Proposer是可以在任何时候丢弃某个提案的。例如 Proposer发送针对 [Mn，Vn] 的提案给Acceptor，但是Acceptor在这之前已经通过了一个比编号Mn大的提案，那么这时Proposer就会丢弃该提案，与此同时 Acceptor亦应该有义务告诉 Proposer 当前自己通过的最大编号的提案信息，以便Proposer生成新的prepare提案请求

冲协议发生突执（会导致死循环/活锁问题）

Learner获取已通过提案的方案

Learner获取提案可以有3种方法

方案1：

一旦Acceptor批准了某个提案就马上发送给所有Learner。这种方案瞬速但是不高效，因为Acceptor需要的通信次数至少是两边节点数的乘积

方案2：

所有Acceptor在批准了某个提案后集体发送给某个主Learner，然后由主Learner分发给其他Learner。虽然这样减少了Acceptor在通信上的开销，但是主Learner存在单点故障问题

方案3（推荐）：

在方案2的基础上让主Learner组成一个集群以解决单点问题。当然，主Learner的数量肯定是少于普通Learner的（只要能避免单点问题），如果不是的话就和方案1没什么分别了。

  从图中可以得出的信息是，只有 Learner 成功响应 Client 后操作才算成功。那么就会出现以下2种情况：

  • 情况1：算法角色都是独立的节点。当过半 Acceptor 通过某个提案后出现了网络分区，这时应该如果响应Client？还有就是过半 Learner 响应算成功还是全部 Learner 响应算成功？如果过半 Learner 算成功那么数据就是最终一致，如果所有 learner 算成功，那么数据是强一致
  • 情况2：算法角色在节点中重叠，即存在一个节点多个角色。这时因为多数派的缘故，数据对于 Client 来说为最终一致

解决Proposer活锁问题

活锁指的是任务或者执行者没有被阻塞，由于某些条件没有满足，导致一直重复尝试—失败—尝试—失败的过程。 — 百度百科

  在极端的情况下，可能会出现这样的现象：Proposer-1提出了编号为M1的提案，且完成了第一阶段。于此同时Proposer-2提出了编号为M2的提案亦完成了第一阶段。那么这时就出现问题了，因为M2通过第一阶段就意味着Acceptor向Proposer-2承诺不会再批准编号小于M2的请求，因此Proposer-1的accept请求会在第二阶段被Acceptor忽略。这时Proposer-1就会从新进入第一阶段，提出编号为M3的提案并通过第一阶段。而这时Acceptor又承诺Proposer-1不再批准编号小于M3的提案，Proposer-2第二阶段失败。如此类推进入了死循环，即活锁。

  解决活锁的方案是选出一个Leader Proposer，通过只有 Leader Proposer 能发起请求来解决。而且使用Leader Proposer 进行请求，因为只有 Leader Proposer 一个能请求，所以在 Leader Proposer 任期期间可以省略 prepare阶段，而这一种方案通常被叫做 Multi-Paxos （即经过第一轮的 Basic Paxos 成功得到多数派 accept的 Proposer 即成为Leader Proposer）。但是这种方案并非完美，因为选举 Leader Proposer 依然需要进行一轮 Basic Paxos ，因此仍然存在活锁问题。因此可以在 Leander Proposer 的基础上进行优化，就是为每个 Proposer 加上随机超时器功能，即在选举 Leader Proposer 时最先超时的 Proposer 先发出请求，这就避免了活锁问题了。但其实如果做到这种程度的话，实质和Raft算法就十分类似了。

Raft算法

Raft模型理解： http://thesecretlivesofdata.com/raft/

算法的诞生

  Raft是一个以日志为核心的分布式共识协议，是Raft作者针对 Paxos 算法的改良。意在比 Paxos 容易理解，而性能和容错方面与Paxos是对等。

  据了解 Raft 作者当时为了比较 Paxos 和 Raft 两种算法的可理解性还专门在一个班里进行过实验。他分别为班里的每个学生讲授 Paxos 和 Raft 两种算法。然后进行测试，总分为60分。Raft算法平均得分为25.7，而 Paxos算法平均得分为20.8。在测试过后还采访了参与实验的学生，结果回答如出一辙，都表示 Raft 算法比 Paxos 更加容易理解和实现。当然事实也的确如此，Paxos算法确实有些复杂和晦涩，从上面对Paxos的了解我们可以感受到Paxos更像是一个方法论，不同的实现对算法的效果有较大的差别。

  那么 Raft 是如何提高算法的可理解性的呢？分别是：问题拆分 和 减少状态空间。问题拆分很容易理解，就是将大问题化为小问题然后再将其一一击破。而减少状态空间是指减少节点主动记录的数据内容。例如日志条目在 Raft 中只能从Leader节点复制到非Leader节点，是一个单向传输过程，用意是减少集群节点的不确定性，因为如果节点各自都乐于主动记录状态的话就会提高实现状态一致性的成本。

Raft特点

  • 容易理解和实现
  • 支持 Leader 选举（与Paxos相比，Raft中的Leader较为重要，用于主导节点状态日志的复制）
  • 和 ZAB 协议一样，主节点负责读写操作，而其它节点只负责读操作
  • 支持崩溃恢复：节点宕机重启可以重新自动加入到集群中
  • 高可用：只要节点数过半正常就能对外提供服务（不能马上检测到，因此存在脑裂风险）
  • 最终一致性。脑裂和过半应答导致。但可以在实现Raft时参考ZAB协议过半可用来解决脑裂，以及做到只能从 Leader 读写后，则能做到强一致。但如果读写都只能载 Leader 中进行则会降低集群的性能，因此也可以使用NWR思想来解决一致性问题但同时可用性会相较降低，

问题拆分和减少状态空间

Raft将共识性问题拆分成以下4个小问题

  • Leader选举：节点集合启动后需要进行Leader选举，且 Leader 宕机后需要重新选举 Leader
  • 日志复制：Leader 接收到 Client 请求后会将其操作日志复制到集群中的其它节点，并要求这些节点的日志必须和自己的日志在状态上保持一致
  • 安全性：某节点已经将给定索引位置中的日志条目应用到了自己的状态中，那么其节点就不会在相同索引位置中应用其他日志条目内容
  • 成员变更：指的是在提供服务的集群节点中，节点配置的变更不会影响正常提供服务

减少状态空间

  • 强Leader：日志条目只能是从 Leader 节点传给其它集群节点。系统的所有变更都通过 Leader 进行，采用二段提交方式，即 Leader 接收到客户端请求变更状态后并不会马上应用日志条目，而是 Leader 将其日志条目广播出去后得到过半节点响应才由 Leader 率先应用日志条目再发送信息要求其他节点应用条目
  • Leader选举：使用随机超时器来协作选举（解决活锁问题），实质就是按照随机时间来发竞选数据包

算法角色

在 Raft 算法中有以下 3 中角色

  • Follower：跟随者。功能是参与决策（该角色是所有节点的初始化角色）
  • Candidate：领导者候选人。功能是参与决策（发生选举超时的Follower节点）
  • Leader：领导者。负责接收客户端请求以及日志条目复制给非

其中 Raft 算法中有两个重要的超时机制，分别是：选举超时、心跳超时。

选举超时：指 Follower 成为 Candidate 之前所等待的时间，一旦超时将会进行 Leader 选举。

  • 如果在这段时间内没有收到Leader心跳包， Follower 将成为 Candidate，同时 Candidate 会产生一个新的选举任期（Term）并发起 Leader 选举（即通过广播来拉票）
  • 其它 Follower 节点接收到 Candidate 的拉票心跳必会作出选票支持，而率先拿到过半票的 Candidate 就会成为 Leader

心跳超时：指 Leader 会以一定的时间间隔来广播日志条目信息，而这个时间就是心跳超时的时间，一旦心跳超时就会进入选举超时环节

  • 一旦心跳超时节点将会从 Follower 成为 Candidate 并会发起 Leader 选举

  1. 集群节点刚启动，所有节点均为 Follower
  2. 因为一开始是没有 Leader 节点的，所以会发生选举超时，节点从 Follower 变成 Candidate
  3. 节点成为 Candidate 后会马上发起 Leader 选举，如果发起选举后得到集群过半的节点响应（包含Candidate自己的票数），那么 Candidate 就会成为 Leader。其中 Leader 有一个任期的概念，它被称为term。term 代表的是当前这界选举的编号。

Candidate发起选举，但如果此时集群中存在某个比当前 Candidate 更高版本的节点(应用条目的index高)的话，那么 Raft 将不会允许当前 Candidate 成为Leader，而是要求高版本的 Follower 成为 Leader

可以看到S4节点选举超时先发起选举，但是并没有成为Leader，这是因为S4只应用了log index为1中的条目，而其他节点都应用了log index为2的条目，因此都比S4高，所以S4不能成为Leader。这样做的原因是因为在Raft中日志只能从Leader复制到Follower，那么如果S4成为了 Leader 的话也就意味着丢失了log index为2上的日志条目状态，而且按照上面提到的 Raft 安全性，即某节点已经应用了某个索引位置的条目，那么其他节点就不能在该索引位置中应用其他的内容，因此S4成为Leader这件事与Raft定义相违背。

日志由有序编号（log index）的日志条目组成。每个日志条目包含它被创建时的任期号（term），和用于状态机执行的命令。如果一个日志条目被复制到过半数服务器上，那么这个日志条目就会被认为可以提交（commit）了。

  • 除此之外也有人提出 leader lease 方案解决脑裂问题，但按照个人的理解认为其实采用心跳机制来检测是否过半节点正常通信后再决定是否对外提供服务会更加简单和优雅一点，而且 leader lease 引入的 region leader 需要额外地恢复大大增加了 Raft 的复杂性和实现难度

ZooKeeper Atomic Broadcast（ZAB协议）

基本简介

  首先 ZAB 协议并不是 Paxos 算法的一个实现，这点可以从目的上看出差异，但 ZAB 协议和 Paxos 算法一样都存在过半（多数派）和提案等元素（Raft也一样），所以个人认为 ZAB 协议或多或少都有借鉴 Paxos 算法的成分，可能因为这所以有些人也叫 ZAB 和 Raft 为 Multi-Paxos。除此之外，ZAB 和 Raft 也非常相似（ZAB 协议要比 Raft 算法诞生得要早一些）。所以如果你懂 Paxos 或者 Raft 的话，那么再来理解 ZAB 就是件比较容易的事儿了。

  ZAB 协议如其名，是一个专门为 ZooKeeper 设计的协议，是为了解决早期 Zookeeper 崩溃恢复需求而设计的。ZAB 实现了一种主备模式的系统架构，可以维持备份节点与主节点之间的数据一致。除此之外不难看出 ZAB 协议并不像 Paxos 和 Raft 两种算法一样，它是耦合在 Zookeeper 中的协议，并不通用。

ZAB特点

  • 支持 Leader 选举
  • 支持崩溃恢复。节点宕机恢复后可以自动重新加入集群
  • 可用性高。只要过半节点正常就能提供服务
  • 主节点负责读写操作，而其它节点只负责读操作
  • 最终一致性，过半应答导致。虽然集群过半才可用解决了脑裂问题，但是在数据提交过程中只需过半即可进行下一步操作，也就是说极端情况下可能出现某个节点还没响应Leader且还没脱离集群，这时因为已经得到过半Follower响应，ZooKeeper响应用户操作数据成功，但接着进行查询时正好查询到还没响应Leader的那个节点，这时就出现数据不一致问题了

协议角色

Observer官方介绍： https://zookeeper.apache.org/doc/current/zookeeperObservers.html

在 ZAB 协议中存在3种角色：Leader、Follower、Observer。其中 Observer 是单纯为了扩展集群的连接数（读性能）而设计的，它的功能和 Follower 差不多，但它并不参与 Leader 选举，简单地说就是它只会默默地同步 Leader 的状态。

那为什么要用 Observer 来扩展而不直接用 Follower 呢？官方的解释是：如果添加更多的 Follower 就会引起集群的写性能下降，原因是写操作需要过半才算成功写入，因此使用 Follower 来扩展的话写操作就需要询问更多的 Follower 从而导致集群可用性降低。所以到最后其实在 ZAB 协议中只有两个核心角色，它们分别是 Leader 和 Follower 。

算法核心角色

Leader：负责Client的读写请求、给Follower发事务广播、接收Follower的写转发

Follower：备份Leader数据、扩展Leader的读性能、转发来自Client的写请求给Leader、参加Leader选举

通信模型

在ZAB协议中所有的通信通道均为FIFO，因此所有操作均按顺序执行。 步骤如下：

  1. Leader 接收到Client请求之后会将每个请求打包成 proposal 并为其分配一个全局递增的事务ID（zxid），然后按照原接收顺序广播给所有Follower
  2. Follower 接受到 proposal 后会顺序处理然后响应 Leader。从另一方面来说，响应给 Leader 的 ACK也是有序的
  3. 当 Leader 接收到过半Follower 的ACK响应，Leader 就会广播 commit 要求所有 Follower 提交对应zxid的 proposal 。commit 亦会以 Follower 的 ACK 响应顺序发出
  4. Follower 接收到 commit 后会对 proposal 进行顺序处理

ZAB节点的两种模式

  • 恢复模式：选举Leader时、新节点加入集群时
  • 广播模式：集群中过半节点正常提供服务时

例如集群刚启动，因为还没有Leader的缘故所以所有节点都会进入恢复模式后选举Leader，直到选举完成（选举成功的前提是过半节点是正常的）。

又例如，在集群正常运作期间有s新节点加入集群，这时新节点进入恢复模式找到正在广播消息的Leader，然后从Leader中同步状态，直到同步完成后才会退出恢复模式并进入广播模式参与集群通信。

ZXID格式

Epoch：

epoch 是一个时间戳。新任 Leader 会从本地查出最新的一次 proposal，取出其 ZXID 后解析出 Epoch（上一个任期）然后递增1作为新任期编号。这里的 epoch 值相当于 Raft 中的Term。

Counter：

每接收一个请求递增1，从0开始。

注意，Counter是存在溢出问题的。溢出后会导致重新选举Leader， 然后ZXID会变为0 。但因为选举会导致集群短暂无法提供服务，所以这主要看你能不能接受，如果不能接受还可以修改 ZooKeeper 源码来修改ZXID中Epoch和Counter在 64bit 中的比例，详情请异步到参考。

简单粗暴的二阶段提交

ZAB的二阶段提交和2PC的不太一样。在ZAB中 Leader 广播 commit 后，如果 Follower 进行 commit 失败，那么Follower 只会断开与当前 Leader 的连接后再重新进入恢复模式来同步数据。换句话来说ZAB通过恢复模式避免了事务回滚。

协议陈述

zab协议中节点的两种模式

过半概念：n/2+1

两种模式实质可以细分为三个步骤，分别是：发现、同步、广播。

在zab协议中节点的三种状态：LOOKING、FOLLOWING、LEADEING

  • LOOKING：处于选举阶段。集群中节点的初始状态均为 LOOKING
  • FOLLOWING：Follower 的状态
  • LEADEING：Leader 的状态

发现：Zookeeper选举过程（FastLeaderElection）

投票过程如下：

如图所示，选票结构是 vote ( myid，zxid )。myid为节点在集群中的唯一标识，而zxid则是节点接收到的最后一个zxid值。

选举步骤如下：

  1. 集群进入恢复模式，节点切换成 LOOKING 状态并互相发送vote
  2. 接收到 vote 后首先对比zxid，如果自己的zxid比接收到的小就将自己的投标目标改为最大zxid的那个节点。但如果zxid一样（例如集群第一次启动），那么就会继续对比myid，即改投给myid最大的节点。例如图片上的示例，最后Leader将会是 Server1（1，6），而且可以发现，如果在第一轮投票中自己的vote是最大的，那么就不会再发起投票而是等待其节点投票给自己。
  3. 经过第二次投票后，因为得到过半选票，因此 Leader 就可以得到确立，这时 Leader 节点会切换为 Leading，而其他节点则切换为 Following。随后集群会进入同步阶段

注意：

  • 选举过程是可能出现短暂的脑裂时间的。譬如投票时延时大，出现 A 认为 B 是 Leader，B 认为 C 是 Leader 的情况。当然，这时因为 B 知道自己不是 Leader 所以它并不会响应 A 的请求，最后 A 会断开重连恢复。但以上这种情况无疑会降低集群的可用性。为了解决这一问题 FastLeaderElection 引入了 FinalizeWait 机制，既在投票期间，节点会等待一段时间以避免延时带来不必要的问题

同步：节点状态同步

在选举阶段中随着 Leader 的确立，周期值 epoch 也会被重新赋值。如下：

  1. 在刚诞生准 Leader 后（同步Leader节点状态前的这段时间），Leader 会在最新的 zxid 中提取出 epoch 值然后在此基础上加 1 左为新一任 Leader 的任期标识new epoch
  2. 生成新 epoch 值后，Leader 就会将新 epoch 值和事务状态集合打包一同发送给所有 Follower 让其同步自己的节点状态
  3. Follower 成功接收到 Leader 的要求后会通过请求应答机制响应 Leader
  4. 在这期间如果 Leader 得到过半 Follower 的响应，那么 Leader 就会继续向 Follower 发出 commit 操作要求，这时Follower 将会把从 Leader 处同步过来的内容状态应用到自己的节点中

广播详解：集群处于正常服务状态

  1. Leader 接收到 Client 请求后将其封装为 proposal 并赋予一个全局唯一的事务id值（zxid），然后记录 log 并将其广播给 Follower
  2. 得到过半 Follower 响应后，Leader首先自己 commit 该 proposal ，然后再将 commit 操作广播给 Follower
  3. Follower 接收到 commit 要求后就会将 proposal 应用到当前节点中并响应 Leader。但如果 Follower 节点自身 commit 失败的话，则会断开与当前 Leader 的连接，并重新进入恢复模式来同步数据直到最后同步成功为止

Gossip、Paxos、Raft、ZAB之间的区别

注：

  • 算法可以简单理解为解决某个问题的方法，但算法并不是具体实现，所以具体实现不同可能就会造成根本性的差异
  • MongoDB 的 Master 选举以及 Redis Sentinel 的 Master 选举均使用 Raft 算法
  • MongoDB 可以通过调节 writeConcern、readPreference、readConcern 等参数实现强一致的 CP 架构
  • 还值得关注的是据了解 Redis 正在研发一个全新的集群模块，通过采用 Raft 算法实现节点间的共识，既相当于提供多一种集群模式供开发者选择，而这种新的模式将以 一致性 优先而不是 可用性 优先
  • 在 Paxos 中 Leader 选举并不是必须的，它只是一种优化手段
  • ZAB 协议和 Raft 算法其实非常相似。较为突出的不同点有如下这些：
    • 选举条件：ZAB 会参照 MYID 和 ZXID 作为选票，而 Raft 参考 Term 和 日志条目
    • Term和Epoch：Raft 中的 Term 只代表发起选举次数，在节点成为候选者时产生，而且当前 Term 内没有选举成功则直接跳过当前 Term 进入下一个 Term。而 ZAB 中的 Epoch 是在选举成功之后产生的
    • 脑裂问题：ZAB 集群过半才可用，解决了网络分区导致脑裂带来的数据不一致问题。而 Raft 算法是存在脑裂问题的，只能等待网络分区恢复后才能达成最终一致性
    • 一致性：ZAB 协议是一种耦合在 Zookeeper 中的共识协会，属于最终一致协议，导致这种原因是因为在 Zookeeper 集群中副本节点是允许提供读服务的。而 Raft 是一种算法，即如果实现时允许副本节点提供读服务就属于最终一致，而如果读写都只能在 Leader 中进行，就可以做到强一致，但这样集群的性能就大大降低了，譬如 Consul 的服务注册发现就是这样做法
  • 从特性来看个人觉得 Raft 更像通用版的 ZAB 协议

Quorum NWR 一致性控制策略

基本介绍

介绍

Quorum NWR 是一种控制读写一致性策略，是一种以鸽巢原理为理论基础的一致性解决方案。它适用于拥有数据副本形式集群的分布式系统。其中 NWR 的含义如下：

  • N：集群中节点的个数量
  • W：写操作必须等待 W 个副本节点持久化成功后才能告知客户端写操作成功。W 不能大于 N
  • R：读操作必须读取 R 个集群节点上的数据后进行版本比对才能得出最终数据。R 不能大于 N

注意：鸽巢原理是一种证明存在性的论述。例如上述的 W 和 R 代表鸽子，而 N 则代表鸽笼，那么当 W+R > N 的时候就必定能够读取到已经保存最新写入的那个数据节点，从而实现强一致。例如当 N = 5，W = 3， R=3，那么当读操作时就必然读到一种含有最新数据的节点。

使用 Quorum NWR 该如何调节一致性级别？

  • W + R > N：强一致
  • W + R = N：弱一致
  • W + R < N：最终一致

注意事项

  • 不难看出 Quorum NWR 其实是一种通过降低集群可用性来提高集群一致性的方案
  • 按个人理解，读操作一般情况下都是既可以在主节点中也可以在副本节点中完成的，所以R 解释为一次读操作必须读取集群中 R 个节点，这样更加合理一些而不是解释为只读取副本节点。N 亦同理
  • Quorum NWR 只是一种策略，更是一种思路，所以不同的实现方式可能存在一些概念性的出入，但思路的本质是万变不离其宗的，因此了解了以上内容之后基本上就能举一反三了

MongoDB

如何保证写不丢失数据？

通过配置 wirteConcern 参数设置持久化应答策略，几种模式如下：

  • 0：只要主库写入成功就马上响应用户而无需关系从库是否落盘
  • n（n少于集群节点数且大于0）：除了主库落盘成功之外还需要等待n个从库落盘成功才响应用户
  • majority：集群节点数过半，要求集群过半节点落盘成功才响应用户

控制读操作应用由那种节点处理

通过配置 readPreference 参数设置，几种模式如下：

  • primary：只在主库中读取（这样会降低集群的读性能以及增加主库的负载）
    • 注意：这个一个默认选项，也就是说 MongoDB 集群时一个强一致的CP系统
  • primaryPreferred：优先选择主库，如果不可用则选择从库
  • secondary：只在从库中读取
  • secondaryPreferred：优先选择从库，不可用再选主库
  • nearest：选择举例最近的节点进行读取操作（即ping响应速度决定，即ping time）

控制读操作的隔离模式（MongoDB ACID中的隔离性实现）

通过配置readConcern参数设置，几种模式如下：

  • local（默认）：可以读取到所有可用的数据
    • 简单说就是从当前所连接的节点上读取，因此不能保证数据被多数节点所持久化
  • available：可以读取到所有可用的数据
    • 3.6 版本开始支持，针对分片集群
    • 注意，如果使用的是复制集 local 和 available 两种配置时一样效果的
  • majority：读取在大多数节点上提交成功的数据
    • 注意，majority 不代表能够一定读到最新写入的数据，但可以避免脏读。该配置只建议在事务上临时设置否则会影响集群的可用性
  • linearizable：线性化读取文档
    • 意思是读操作能够保证一定读取到当前节点的上一个写操作的值
  • snapshot：读取快照中的数据。隔离性最强，相当于 RDBMS 中的序列化读写

Kafka

确保生产者消息不丢失

  在网上经常可以看到有说 Kafka 不可靠而导致消息丢失的说法，其实这完全是因为对 Kafka 不熟悉所致。例如可以在Producer方调节acks参数级别 以及在Broker方设置min.insync.replicas数量来保证消息日志落盘，但需要注acks级别越高那么集群的可用性和吞吐量就越低。

Producer中的 acks 配置级别：

  • 0：Producer 完全不理会 Broker 的是否已经日志落盘（返回ack响应），发送完一批消息后会马上发下一批消息，这时 Producer 的吞吐量最高，但因为完全不理会日志是否落盘，所以就算出现发送失败也不会进行重试操作，也就是说存在丢失日志的风险
  • 1（默认）：Producer 会等待 Leader Broker 的日志落盘成功（返回ack响应）。换句话就是只要 Leader Broker 持久化日志成功，Producer 就会视消息数据发送成功
  • -1或all：Producer 会等待集群中所有 Broker 的日志落盘成功（返回ack响应）才会发送下一批消息。当前配置模式能够百分百成功持久化日志，但吞吐量是3种配置中最低的

Broker中的 min.insync.replicas 配置：

  • 用于设置必须应答成功的 Replica 数。该配置只在 Producer 中 acks 值为 -1/all 时才生效
  • 譬如某个 Topic 的 Partition 数为 3，而现在将 min.insync.replicas 设置为 2，则表示 Kafka 集群只允许1台 Broker 响应不成功，否者 Producer 就需要进入重试操作

关于 Kafka 中的 ISR 集合（in-sync replica）

ISR 集合是 Kafka 集群所维护的一个健康节点集合，默认情况下只有 ISR 集合中的成员才有资格竞选成为 Leader，除此之外还有一个重要的概念，就是当客户端发送消息到 Kafka 集群，那么这条消息持久化成功与否是需要 ISR 集合是否都接收到该消息而定的，既 ISR 集合都接收到该消息日志就视为成功，否者视为失败。

Redis

如何降低数据复制丢失和延时所带来的风险？

  因为Redis高可用的基础是复制，而且 Redis 高可用方案目前来说属于 AP 系统（据悉 Redis 正在开发基于 Raft 算法的一种新集群模式，而如果采用 Raft 算法，集群可用性将会降低，但会从 AP 摇身一变为 CP 系统）。

  首先要给出结论：目前版本的 Redis 高可用解决方案不能完全避免数据丢失，而只能做到尽量降低丢失数据可能。在 Redis 复制机制中有如下两个关键配置项，我们可以利用它们来降低复制时丢失数据的风险。

# 要求复制延迟低于指定时间值（秒），否则拒绝提供写服务
min-replicas-max-lag 10
# 要求健康节点数（副本节点）要达到一定的数量，否则拒绝提供写服务
min-replicas-to-write 3

以上配置项特性表面上和 Quorum NWR 不是很占边，但事实上亦属于提高读一致的操作。即通过确保副本健康节点的数量和限制网络延时来间接实现 W 功能。换句话说 min-replicas-max-lag 的值越低，以及 min-replicas-to-write 的值越大时，那么读一致性就越高。而在前面已经提到过，Redis 目前并不能完全解决复制延时和数据丢失等问题。

  • 主节点在未复制成功的前提下宕机了，然后副本节点成为主节点提供服务，这时就会造成丢失数据
  • 通过副本节点来提高读性能时，因为网络的不定性必然会出现一致性问题

个人认为，Redis 之所以没有直接支持 Quorum NWR 的原因是与其设计理念有关，因为 Redis 是一个可用性优先的系统，而如果使用 Quorum NWR 就会导致可用性降低（是哟功能 min-replicas-max-lag、min-replicas-to-write 可会一定程度降低可用性 ）。
