Redis高可用解决方案

 2019-01-05   作者: TanDi

前言

官方为Redis提供了两个高可用解决方案，分别是：Replica 和 Cluster

其中 Replica 能够解决单点故障、实时备份和负载均衡等问题。而 Sentinel 和 Cluster 均基于 Replica。

Replica

Redis版本：5.0.5

Replica存在问题：无法做到故障转移

复制的过程

  1. 保存主节点信息
# 复制节点
127.0.0.1:6381> info replication
...
master_host:127.0.0.1  # 主节点地址
master_port:6379 # 主节点端口
master_link_status:up # 连接状态（up/down）
master_sync_in_progress:0 # 从节点是否正在进行全量复制（大于0表示是）
master_repl_offset:4958 # 复制偏移量
...
  1. 主从建立socket连接

  复制节点内部有一个定时任务，作用是每隔1秒检测发现是否有新的主节点。

  当发现有新的主节点后，复制节点就会建立一个端口为24555的套接字来接收主节点发送过来的复制命令（从主节点复制过来的命令）。

  1. 发送ping命令

成功建立socket连接之后，复制节点会使用ping命令首次和主节点通信。

目的如下：

  • 检测主从节点之间的套接字是否可用
  • 检测主节点是否可以接收处理命令

注意：如果ping命令后没有收到pong命令（每隔1秒轮询），则复制节点会先断开socket连接，等到下次定时任务启动再建立。

  1. 权限验证

  如果主节点设置了requirepass参数的话，则需要进行密码验证。这时复制节点必须设置masterauth参数来设置连接密码。

  1. 同步数据集

  在成功通信后，主节点会将自己所有持有的数据全部复制（发送）给从节点（通过复制命令psync）。

包含2种复制情况：

  • 全量复制：当数据量大时，会对主节点和网络造成很大的开销

    1. 第一次连接上主节点
    2. 主节点运行ID改变（例如重启。可以通过debug reload重启加载redis而不改变运行ID）
    3. 复制积压缓冲区（FIFO）没有复制节点需要的数据（例如主从断开时间过长，期间主节点进行了大量写操作）

  • 局部复制

    1. 由于网络原因最近的数据没有成功同步，而主节点复制积压缓冲区存在该命令（通过对比复制偏移量再次同步这部分数据）

关于复制偏移量，从节点会每秒上传自己的复制偏移量给主节点，通过主从节点的复制偏移量对比来判断数据是否一直

  1. 命令持续复制

主节点会将当前接受到的写命令实时复制给复制节点。

复制积压缓冲区是一个FIFO队列，用于从节点丢失数据补救。

# 复制积压缓冲区信息

关于Redis复制的使用问题

读写分离

不推荐直接使用Redis的复制来做读写分离，而是应该考虑Redis集群

  读写分离可以解决Redis的高负载问题，通常在读多写少的场景使用，通过主节点写，复制节点读解决。

Redis读写分离存在以下问题：

  • 复制数据延迟

  复制延迟在网络上是一个不可避免的问题，延迟程序取决与网络速度和命令阻塞程度

  • 读到过期数据

  存在这个问题是因为Redis的删除策略：惰性删除、定时删除

  惰性删除：主节点在获取key的时候会检查key是否过期，如果过期则删除key并将删除命令同步给复制节点。

  定时删除：除了惰性删除，Redis内部还维护了一个定时任务，用于轮询检测并删除过期的key，删除后会通过主节点同步删除命令给复制节点。

  那么就可能会发生这么一种情况，在某时刻大量key失效，主节点无法及时将删除命令同步给复制节点，而这是用户在复制节点对某个过期（将要被删除）的key进行了访问，这就造成了读到过期数据的问题。

  • 节点故障

  Redis复制有个非常大的缺点，就是无法做到故障转移（需要借助Sentinel）。且主从复制模式中，复制节点通常为了避免数据不一致的问题而被设置成只读，在以上基础上就可能出现以下情况：

主节点故障：用户无法写入，只能进行读操作

复制节点故障：用户无法读取数据

  之所以会出现上面两个问题的主要原因是用户无法感知节点停止服务和复制节点断开主从后无法主动变更角色来提供服务。

配置问题

  在Redis的复制中，复制节点配置是可以和主节点不一致的，比如通常的做法是：主节点关闭AOF备份在复制节点上开启，这用可以提高主节点的工作效率。

  还有就是从Reids 5开始提供参数replica-ignore-maxmemory来决定复制节点是否忽略maxmemory剔除策略，默认为yes。表示复制节点不会自己剔除key，而必须通过主节点同步操作来删除。这样做的用意是保证复制节点不会因为maxmemory策略而导致key与主节点不一致。也由于这个原因，复制节点的maxmemory就必须比主节点的maxmemory大，因为复制节点不会自己剔除key。

复制风暴

  复制风暴指的是主节点在短时间内被多个复制节点发起全量复制。复制风暴会导致主节点服务器CPU升高、内存和宽带大量消耗。

  针对这个问题Redis的做法是，主节点为复制节点生成RDB时，如果这时有多个复制节点发起全量复制，那么这些复制节点就会共享这个RDB文件，使用这种做法来降低生成多个RDB的性能开销。

  1）减少主节点上的从节点数而使用树结构

Sentinel

是什么又有什么用？

如图所示，Sentinel用于监控Redis的复制。

Sentinel其实是Redis的一种特殊节点，功能如下：

  • 发现故障
  • 故障转移
  • 通知客户端程序

Sentinel节点可以监控数据节点（包含主从） 和 其余Sentinel节点。做法如下：

  • 检测节点是否可达，不可达则对其做下线标示
  • 检测到主节点不可达，Sentinel节点之间会进行投票，在大多数Sentinel节点都r认定主节点不可达时，Sentinel领导节点就会发起故障转移操作 并 通知客户端程序

相关原理

3个定时任务

  • 每隔10秒，Sentinel节点会获取一次info replication信息，用此感知是否由新的数据节点加入。

  • 每隔2秒，Sentinel节点会向__sentinel__：hello频道发送：对数据主节点判断和自己的sentinel信息。当然，在发送的同时也会订阅该频道来接受其他Sentinel节点的信息。

  • 每隔1秒，Sentinel节点会向 数据节点 和 其他Sentinel节点发送一次ping命令来确定节点是否可达。

主观下线和客观下线

  • 主观下线

当Sentinel节点去ping其他节点时发现没有在down-after-milliseconds内回应，这是Sentinel节点就会对其做下线标记，而这种标记是非常主观的，因为并没有和其它节点商量，因为可能存在这么一种情况，就是只有当前Sentinel节点和某个数据节点通信缓慢（如网络问题），而其它Sentinel节点并没有这种问题。

  • 客观下线

在某个数据节点被主观下线后，且该节点是主节点的话。

这时，标记主节点下线的Sentinel节点就会使用命令sentinel is-master-down-by-addr向其它的Sentinel节点发起仲裁，要求其它Sentinel节点对主节点进行主观下线判断，并收集其信息对该主节点进行下线判断，当主观下线数量满足quorum时，则主节点就会被客观下线，即多个Sentinel节点都认为下线了。

Sentinel故障转移过程

  1. 选择一个能够胜任master复制节点

  2. 向该复制节点执行replicaof no one命令来断开原来的主从关系（此时节点就晋升为master）

  3. 再向其余的复制节点发起主节点变更

  4. 将原来主节点变更为从节点，并在其恢复后让它去复制新master的数据

Sentinel使用需要注意的问题

  • 需要注意的是，Sentinel的功能只针对主节点，而对于复制节点故障依然是无法解决
  • 如果想要在Sentinel环境下实现读写分离，思路是获取到所有复制节点来实现复制节点池，但因为读写分离存在一定的问题，所以不是很推荐，建议使用Cluster来解决

Cluster

关于数据分区策略 - 负载均衡算法

  在了解Redis Cluster之前，需要理解什么是数据分区策略。掌握数据分区策略对于理解Redis Cluster的数据分布和故障转移有很大的帮助。

  常见的数据分区策略有哈希取余、一致性哈希，例如MySQL中还支持诸如range和ksy等策略，而Redis Cluster使用的则是虚拟槽。

哈希取余

  哈希取余指的是从数据中选取一个适合的key进行哈希运算，并取余数据存储节点的数量后按照该结果来选择对应的数据存储节点来存取数据。

公式：hash(key) % 数据节点数量

哈希取余策略通常使用在分库分表上。但必须强调一下它的缺点，就是如果想要扩展节点数量，那么就需要重新将数据分配，这是一个相当大的代价，所以通常使用哈希取余策略进行分区前都应该预计充足的库表数量，确保在往后一段时间来都够用。而如果确实需要扩容的话，通常都是翻倍扩容。

一致性哈希

  一致性哈希的思想是，先计算出存储节点的token值（例如哈希值），这个token值位于0到2的32次幂范围。而0到2的32次幂形成了一个闭合的hash环，用于确立数据应该落在那个token值对应的存储节点上，如下：

图中的例子有4个数据存储节点，那么数据的存储位置需要以下步骤来确立：

  1. 使用数据的key计算出哈希值
  2. 根据哈希值来查找存储节点；例如哈希值位于node1和node2之间，则需要按照顺时针选择原则，即这时数据存储节点应该是node2

  相较于哈希取余，一致性哈希在增加或删减节点时，只会影响到原来存储在左边或右边节点的数据，再加上Hash函数的离散，从而大大降低了因扩容和缩容后导致数据不命中的数据范围。

  其中Redis客户端Jedis提供的ShardedJedis就使用一致性哈希来进行分片操作的。除此之外Google Guava也同样提供一致性哈希的实现。

一致性哈希存在的问题：

  1）增加和减少节点时，会影响左边或右边节点的数据命中。这时需要人工解决或舍弃这部分数据。也因此，一致性哈希通常用在分布式缓存中，因为数据足够离散，所以这部分数据其实对整体来说影响并不大。

  2）当节点数量少，节点数量发生变化就会对数据造成大影响，因为数据不够离散。这时体现不到一致性哈希的价值，因此不推荐使用。

虚拟槽

  虚拟槽和一致性哈希类似，不过它不是做成hash环，而是hash槽。在Redis Cluster中有16384个槽（0~16383）。

集群中的每个节点都负责维护部分槽和槽做映射的数据。

数据通过公式slot=CRC16(key)&16383来决定存储的槽位。这样的好处是，做到了数据与数据存储节点之间的解耦，因此有利于以后的数据迁移。也可以简单理解，数据存储节点 - 虚拟槽 - 数据。

步骤1：目标节点执行cluster setslot {slot} importing {sourceNodeId}，表示目标节点准备接收slot；

步骤2：源节点执行cluster setslot{slot} migrating {targetNodeId}，表示源节点准备导出slot；

步骤3：源节点执行cluster getkeysinslot {slot} {count}，每执行一次就可以获取slot中count个key数据；

步骤4：源节点执行migrate {targetIP} {targetPort} key 0 {timeout}，使用管道将步骤3得到的key发送给目标节点进行保存；

步骤5：重复3-4步骤直到slot所有key都发送给目标节点；

步骤6：所有集群节点执行cluster setslot {slot} node {targetNodeId}，达到告诉集群成员节点得知slot的变动；

关于Redis高可用丢失数据问题

需要强调一下的是 Redis 构建出来的高可用架构是一个 AP 系统，既它并不能保证数据的一致性，因此会存储数据丢失问题。例如延迟复制情况下Master宕机后Replica晋升为Master 或出现网络分区时就会导致数据丢失。

延时复制导致数据丢失

网络分区导致脑裂后丢失数据

如何解决数据丢失？

首先给出结论是，Redis高可用不能避免数据丢失，只能是尽量降低丢失数据的量。

在Redis主从中有如下两个关键配置，可以利用其特定尽量减少数据丢失的量。

# 要求主从复制延迟低于指定时间值，否则拒绝提供写服务（客户端这时无法写入数据，相当于将问题抛给客户端处理）
min-replicas-max-lag 10
# 要求连接到当前Master的健康节点要到达一定的数量，否则拒绝提供写服务
min-replicas-to-write 3
  • 针对延时复制导致的数据丢失可以适当低调低 min-replicas-max-lag 值
  • 针对网络分区导致的脑裂问题可以适当低调高 min-replicas-to-write 的值

多说一句，Redis是存在一致性问题的，所以个人并不太理解那些使用Redis做分布式锁的行为。如果需要实现分布式锁，应该使用对客户端来说属于 AP 的高可用系统，例如 ZooKeeper 。
