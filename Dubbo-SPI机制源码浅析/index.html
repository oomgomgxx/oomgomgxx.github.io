Dubbo SPI机制源码浅析

内容修订

前言

  SPI 的全称为 Service Provider Interface（服务提供者接口）。SPI 是模块化编程中重要的基础，我们可以依赖 SPI 机制做到不修改代码的情况下给接口切换实现类，以此来提高应用的可扩展性。

  SPI 并不是 Dubbo 中独有的机制，准确来说它扩展自 JDK SPI。除 Dubbo 之外，其实很多常见的开源框架亦都直接或间接地在使用 SPI 机制。譬如 Hibernate 第三方库加载（缓存、连接池等）， Sping Boot Starter 自动装配（SpringFactoriesLoader），还有 Servlet 3.0 的无 xml 容器配置方式等，都是建基于 SPI 思想之上实现的。而在 JDK 内部对 SPI 的最佳实践莫过于是 JDBC 驱动加载，自 JDBC 4.0 规范（JDK1.6版本）开始就不再需要手动通过 Class.forname 来加载驱动，取而代之的是通过 SPI 机制检索类路径下的关联文件来自动加载驱动实现。

Dubbo SPI 机制

为什么 Dubbo SPI 要扩展 JDK SPI

在 Dubbo 官方文档中我们可以得知它认为 JDK SPI 有如下缺点

  • 需要遍历所有的实现并将其实例化，且加载完后还需要遍历才能找到需要的实现（API 不够友好）
  • JDK SPI 的关联文件没有为实现命名，导致在程序中很难去准确的引用它们（设计不够友好）
  • 扩展如果依赖其他的扩展，做不到自动注入和装配（不够智能）
  • 没提供类似于 Spring 的 IOC 和 AOP 这样的功能（不够智能）
  • 扩展很难和其他的框架集成。譬如扩展依赖了一个Spring Bean 的话原生的 JDK SPI 机制是不支持（扩展性差）

使用测试

普通模式使用：直接依赖 @SPI 操作
  • @SPI 注解用于标注扩展点（SPI扩展接口）
适配模式使用：@SPI + @Adaptive
  • @Adaptive 用于启用适配模式
  • @Adaptive 可以标注在扩展点的抽象方法或扩展点实现类上
    • 当标注在扩展点实现类上时，表示这是一个装饰器类（参考：AdaptiveCompiler 类）
    • 被 @Adaptive 标注的抽象方法，必需在参数列表中添加 Dubbo 提供的 URL 类作为入参，否则会抛出 IllegalStateException 异常。URL 的主要作用是为 AOP 提供条件依据

激活模式使用：@SPI + @Activate
  • 激活模式和适配模式类似，都需要使用 URL 参数来触发实现
  • @Activate 可以标注在扩展点实现类或其方法上
  • @Activate 常见用法有两种：
    1. @Activate (value = “”)：指定激活参数。即URL中如果包含该值就激活对应的实现
    2. @Activate (group= “”)：指定 group 值，当 getActivateExtension 时指定 group 值时，匹配的将会被激活

在 Dubbo 中内置的加载策略有以下这些

  • DubboInternalLoadingStrategy：从 META-INF/dubbo/internal/ 中加载
  • DubboLoadingStrategy：从 META-INF/dubbo/ 中加载
  • ServicesLoadingStrategy：从 META-INF/services/ 中加载

也就是说，Dubbo SPI 的关联文件可以存放在以上文件路径下。除此之外，你也可以通过实现 LoadingStrategy 接口，然后在 META-INF/services 下添加关联文件来扩展 Dubbo 的加载策略。
