IDEA性能调优小记

 2020-05-22   作者: TanDi   类型：原创

前段时间将 IDEA 升级到了 2020 版本，而最近使用发现经常出现卡顿的现象。虽然笔者的机器是15年买的已经有点历史了（CPU双核4线程、8G内存），但也不至于卡成这样，所以打算针对 IDEA 的 JVM 进行一次调优来舒缓一下卡顿感。而个人认为 JVM 调优是一件非常矛盾的事情，因为调优的核心思想就是在有限资源的情况下让应用运行得更顺畅（所以网上哪些连机器配置都不提就开始谈调优的人是非常不靠谱的）。直白点说就是如果你的机器配资足够高的话是根本不需要考虑调优的。

造成 JVM 应用卡顿的原因大体有以下几种：

  1. GC 回收导致 STW
  2. 内存分配不合理导致（资源缺乏）频繁 GC
  3. CPU 使用率过高导致 JVM 线程执行效率降低

虽然以上1点看起来很明确，但是要平衡它们却不是件容易的事儿。例如你可以增加 GC 线程让垃圾回收进行得更加迅速，但线程多了 CPU 使用率就会上升（特别是自用机器肯定会有有其它应用并行的情况出现），这也导致卡顿。又或者你可以增加 JVM 的内存容量来减少 GC 的次数（因为有足够的内存是不会触发 GC 的），但这样又会加重 GC 线程的回收负担从而让 GC 过程更加长，最后亦可能导致卡顿感加重。这就是上面为什么说 JVM 调优其实是件非常矛盾的事情。

JVM参数调优说明

调优目标：舒缓IDEA启动及运行过程的卡顿感
默认环境配置：

  • 双核四线程CPU
  • 8G内存
  • JRE 11 （可以使用 IDEA 的 choose runtime 插件来查看）
  • CMS 垃圾回收器

注：

  • 从 JDK 9 开始默认 GC 从 Parallel 改为了 G1。G1 的优势是可以将大内存划分为若干个 Region 区来细化管理。例如如果能够为应用分配较大的堆内存（有人建议是4G，但个人认为大于1G的话就能体现G1优势了，再者 JDK 9 开始 CMS 已经被标记为废弃），那么 G1 就要比以前的 GC 都要高效。但对于小内存而言性能就和 CMS 差不多了。因为 G1 和 CMS 一样有并发标记操作，而且 STW 时间更长（但 G1 有优先回收策略，因此个人认为可以忽略这点），其次 IDEA 默认的 GC 就是 CMS。那么它为什么不用 G1 呢？个人认为原因有两个，第1是小内存 CMS 更高效，第2是提供商无法保证 IDEA 用户机器都是高配置的（内存方面），所以默认最好的选择就是 CMS + Parallel 了。

优化过后的配置如下：

-server
-Xverify:none
-XX:ParallelGCThreads=4
-XX:ConcGCThreads=2
-XX:MetaspaceSize=1g
-XX:MaxMetaspaceSize=1g
-Xms1024m
-Xmx1024m
-XX:+UseG1GC
-XX:G1HeapRegionSize=1m
-XX:+UseStringDeduplication
-XX:CICompilerCount=2
-XX:+AlwaysPreTouch
-XX:+UseAdaptiveSizePolicy
-XX:ReservedCodeCacheSize=300m

配置说明：

-server

  • 使用 C2 编译器
  • 编译时间更长但出来的机器码更高效，换句话就是运行得更加顺畅

-Xverify:none

  • 不对加载的类进行字节码校验
  • 能够一定程度提升类加载过程的速度
  • 注意，该配置在 JDK 13 被废弃

-Xms1536m

  • 启动时堆的初始化大小

-Xmx1536m

  • 堆的最大值
  • 该值和 -Xms 最好一致，因为这样可以减少内存扩容带来的开销

-XX:MetaspaceSize 和 -XX:MaxMetaspaceSize

  • 指定元空间大小
  • 两个配置最好一致，原因同上
  • 注意，最好是显式指定元空间大小，因为一旦 MetaspaceSize（默认 20 多兆，非常小）被填满就会触发 Full GC 。例如不配置的话可能会出现在 JVM 启动期间就连续执行多次 Full GC 的情况，而且如果应用类定义较多的情况下最好是预留充足的空间给元空间，这样可以有效地降低 Full GC 的出现。

-Xmn512m

  • 新生代内存大小
  • 该内存要特别主要，虽然 Young GC 速度快（因为用的是标记复制算法）但频率也高，所以如果内存设置得很大的话还是会给 GC 线程带来回收压力的。例如频繁 Young GC 导致 CPU 使用率上升同样会造成卡顿
  • 如果有充足的内存分配给新生代，例如 1.5G 以上，个人建议这时可以适当地将幸存者区域内存增大，这样做可以提升 标记-复制 算法的内存使用率
  • 使用 G1 时不推荐显式设置该配置，因为 G1 是一个停顿优先的垃圾回收器，G1 会通过启发式推断来动态调节内存以尽量停顿要求（MaxGCPauseMillis，默认为 200 ms）

-XX:+UseConcMarkSweepGC

  • 老年代使用 CMS，新生代使用 ParNew

-XX:+UseG1GC

  • 启动 Garbage First 垃圾回收器
  • 因为 G1 的 GC 停顿时间更加可控。可以通过 MaxGCPauseMillis 设置 GC 停顿期望值，默认为 200 ms

-XX:G1HeapRegionSize=1m

  • 指定 G1 Region 区的大小，默认值为1m
  • 适当增加该值可以提高内存的使用率（因为 G1 Region 的清理算法采用的是标记复制），但与此同时增大 Region 就意味着增加 GC 线程的回收负担，所以应该根据实际的堆内存来设置该值。例如 Xmx 为 4G 时可以将该值设置为 4M ，这也是比较常见的配置方法
  • 手动指定值时值必须是 2 的幂，而且范围在 1 MB 到 32 MB 之间。默认情况下该值为 1M，即 2 的 0 次幂

-XX:ParallelGCThreads=4

  • GC 回收时的并行线程数
  • 该值不应该超过 CPU 的核数/线程数
  • 我当前机器的 CPU 是双核四线程，所以这里设置为 4
  • 该值默认为0。如果不显式设置且使用的是 G1 的话，则会自动通过 G1 的启发式推断来设置该值

-XX:ConcGCThreads=2

  • 并发标记线程数。默认值为 （ParallelGCThreads + 3）/4
  • 设置为 2

-XX:CMSInitiatingOccupancyFraction=75

  • 触发并发标记的内存使用阈值，默认为92，即老年代使用率92%时触发
  • 设置为 75 是我根据 HashMap 的扩容因子值来设置的，它的理念是即能提高效率也不浪费空间，是一个折中值
  • 建议该值不建议设置的过大。因为触发并发标记后如果预留的空间不足以分配内存给它对象实例的话就会出现并发失败的情况，这时会使用 Serial 进行串行 STW 执行回收（这样会提升卡顿感）

-XX:-CMSClassUnloadingEnabled

  • 禁止 CMS 卸载类
  • 因为 JDK 1.8 开始元空间彻底代替了永久代，元空间默认情况下只受物理内存大小限制，因此出现 OOM 的几率大大降低
  • 这里禁止卸载类是为了减少类加载带来的开销，当开启该配置的前提是元空间的内存足够充裕，否则会出现频繁 Full GC 的现象
  • 如果 JDK 是 1.7 之前的版本，那么开启 -XX:+CMSClassUnloadingEnabled 配置可以有效地降低永久代 OOM 的几率

-XX:CICompilerCount=2

  • JIT 编译线程数
  • 设置为 2
  • 建议该线程数在自用机器上不要过大，因为在 IDEA 启动时会伴随大量的 JIT 编译 和多次 Young GC 出现，因此CPU 非常容易到达100%，这是 IDEA 启动时卡顿的关键点所在（因为考虑到运行过程的 GC 效率，所以 ParallelGCThreads 不能少，所以只能降低 CICompilerCount 来舒缓 CPU 使用率了。当然 2 条线程其实也足够了，不至于会有卡顿）

-XX:MaxTenuringThreshold=10

  • 对象晋升老年代的岁数。默认值为15（CMS默认为6），即对象经历15次 GC 都没被回收的话就会晋升到老年代
  • 设置为 10 是因为经过我观察老年代其实并不缺乏内存，而且 CMS 并发标记可以有效地减少 Full GC ，所以对象尽早进入老年代就可以减少新生代对象数，换言之就是提高 Young GC 回收效率

-XX:ReservedCodeCacheSize=500m

  • 存放 JIT 编译内容的缓存大小
  • 注意该值在 64bit server 模式下默认为240m。如果编译内容到达该阈值的话 JIT 就不会再编译了，这样会降低 IDEA 代码的执行效率，直白点说就是会提高卡顿感

-XX:+AlwaysPreTouch

  • 在 JVM 进程启动时就分配所有的内存（-Xmx）而不是等到访问时才分配。该配置可以有效地提升程序运行期间的稳定性
  • 理论上来说堆越大的话，该配置就会带来一定的启动耗时。但个人认为可以忽略，所以建议配置

-XX:+UseStringDeduplication

  • 开启字符串去重节省内存（注意：该配置只在 G1 上生效）
  • 字符串去重操作会在 GC 期间进行，默认情况下只有经历 3 次 GC 还存活的字符串才会列为关注对象（可以-XX:StringDeduplicationAgeThreshold 设置 GC 阈值）
  • 开启字符串去重功能会加大 GC 垃圾回收的工作量，即会一定程度上延长 GC 停留时间，所以这是一个时间换空间的操作。个人认为如果内存不充裕（例如我当前机器的情况）且对 GC 停留没有严苛的要求时建议开启该功能

-XX:+UseAdaptiveSizePolicy

  • 可以在 Full GC 时打印自适应配置策略信息，以及告知是谁引发了 Full GC

  • 首先启动期间的卡顿感已经明显降低，所以我要的调优效果已经达到
  • 因为一直没有关闭电脑，所以 IDEA 已经运行了将近 65 个小时。在这期间 Young GC 执行了 2232 次，用时将近两分钟，相当于每次 Young GC 花费时间在 53 毫秒左右。而 Full GC 执行了 14 次用时将近 12.6 秒，即相每次 900 毫秒左右。从数据上不难看出一次 Full GC 的用时是要比 Young GC 高很多的。当然，这点要还是要看对中的对象有多少而定
  • Full GC 是不可能避免的。Full GC 是一个有效地防止内存溢出的机制，换句话来说内存越小触发 Full GC 的几率就越大，因为随着程序产生了大量长期驻留在对象导致内存紧张又或者出现了内存泄露问题都会导致频繁地 Full GC 的出现
  • 最后多说一句，如果项目堆内存紧张的话可以考虑一下利用手动形式触发 Full GC 来清理内存以避免内存溢出 。譬如在系统访问流量较低的时间段利用定时任务触发 System.gc() 。一个比较好的例子就是 Tomcat 中有个预防内存泄露的监听类 JreMemoryLeakPreventionListener 就是采用这种方式来释放内存并检测内存泄露的。除此之外如果使用了 CMS 的话可以开启 +XX:+ExplicitGCInvokesConcurrent来提高 Full GC 的并发效率。但是，如果你想禁止这种手动触发 Full GC 的操作，则可以开启 -XX:+DisableExplicitGC 配置（譬如解决 Tomcat 监听类每小时触发一次 Full GC 的问题）

IDEA设置优化

除此之外还可以在 IDEA 右下角点击右键开启内存监控器实时观察堆的内存使用情况，可以在内存占用较大时手动点击触发 Full GC

注意，有些时候可能你会观察到这么一个现象，就是 IDEA 的内存使用量超过了 -xmx 所指定的范围。这是因为 -xmx 控制的只是 JVM 堆内存，而在任务管理器中所看到的是 JVM 进程所占用的全部内存，其中包含 堆内存 和 堆外内存（栈内存（由系统分配管理，速度快且没有泄露问题）、元空间、直接内存、C库内存池） 等。

知识拓展

拓展1 — C库内存池，也叫内存分配器

  • 内存池用于加速内存分配。一般而言应用程序会首先向应用内存池（譬如 JVM 堆）申请内存，如果不足分配则会向C库内存池申请（malloc），只有在 C库内存池 不足分配时才会到系统内核中申请，而系统内核调配资源相对来说是比较低效的
  • 常见的 C库内存池譬如有 Linux 下 JVM 使用的 （glibc）Ptmalloc2 、Google 的 TCMalloc、Redis 用的 Facebook JeMalloc 等等
  • malloc 属于动态内存分配，它的实现分为 brk 和 mmap 两种。它们的区别在于 brk 用于分派较小的内存，同时缓存释放后不会马上归还给内核而是缓存起来以提高后续所需的分配操作，而 mmap 用于较大的内存分配时使用（按需向上取整到整数个页），特点是内存在释放后会马上归还， 所以每次 mmap 都会发生缺页中断（频繁发生会使内核管理负担增大 ） 。这里需要重要注意的是 malloc 并不是将内存马上分配给进程，分配给进程的只是虚拟内存，即真正分配是在进程访问虚拟内存发生缺页中断时分配的
  • 其中 Ptmalloc2 会为每条线程预分配 64M 的内存（用于线程需要时领取），其中又因为 JVM 的线程模型 与 Linux内核线程是单对单的关系，所以在多线程高并发场景下会出现进程所占内存突增的现象。这个问题可以添加环境变量 MALLOC_ARENA_MAX 来控制，亦可以通过环境变量 LD_PRELOAD 来挂载指定的 malloc
  • 因为 glibc 默认的 malloc 容易造成内碎片，影响内存回收，因此 Redis 用 JeMalloc 取而代之。除此之外，如果发生非堆上内存泄露，同时泄露点不在应用程序上时则可以考虑将 malloc 更改为 JeMalloc

拓展2 — 查看进程的内存状态

  1. 先用 pidof 或 jps 得到 PID 号
  2. 再用 pmap -x PID 查看

拓展3 — Linux OOM-Killer （内存溢出杀手）

  • Linux 系统的一种保护机制，在内存不足时用于关闭 /proc/<pid>/oom_score 值最大的进程
  • 可通过命令 grep "Out of memory" /var/log/messages 查看有哪些进程被 Killer 关闭
  • 防止进程被关闭的方法是调整进程oom_score_adj 的值（范围在 -17 至 15 之间） ，该值越小被关闭的机会就越少，其中 -17 表示禁止杀死该进程。例如可通过命令 sudo echo -17 > /proc/进程PID/oom_score_adj 设置
