关于本地缓存

 2019-02-24   作者: TanDi

什么是本地缓存？

本地缓存又称为进程缓存，顾名思义本地缓存是与应用处于统一进程同一空间的，这就与分布式缓存形成了鲜明的区别;

缓存的意义

不论是本地缓存还是分布式缓存，其主要目的都是为了减少不必要的数据库访问;

因为从整个系统的角度来看，往往系统的瓶颈都来自数据库，因为数据库操作是最为耗时的，当大量并发操作都聚集在数据库时，数据库的连接数上限（受限于所在操作系统tcp最大连接数 ulimit -n）、磁盘io操作（优化sql索引解决）都会降低系统的性能;

使用本地缓存需要注意的问题

  • 缓存一致性问题：举个例子，假设现在系统使用Nginx进行反向代理实现负载均衡，既将原来的单个节点横向扩展成多个节点，那么这时对于本地缓存而言因为是线程独立的所以每个节点上的本地缓存内容就可能不一样，这也是使用本地缓存时需要注意的缓存一致性问题;
  • 缓存对象的存活时间：使用本地缓存时为了避免缓存堆积进而占用内存，应该使用特定的缓存清空策略来去将”无用缓存“及时清除
  • 微服务不建议使用本地缓存：因为本地缓存存在一致性问题，而且本地缓存会让节点带有状态，鉴于微服务的设计规范而言并不推荐使用

缓存驱逐策略

  • FIFO(first in first out)：先进先出策略，通俗点来说就是将缓存池看成一个队列且有大小限制;如大小为3，现在队列中有a,b,c三个缓存对象（顺序），现在队列显然已经慢了，现在再添加d进队列，那么缓存列就生在d,a,b，而c因为超出队列限制的缘故被删除; 策略算法主要比较缓存元素的创建时间。在数据实效性要求场景下可选择该类策略，优先保障最新数据可用
  • LFU(less frequently used)：最少使用/最少命中次数策略(着重次数)，根据元素的被使用次数判断，清除使用次数较少的元素释放空间。策略算法主要比较元素的hitCount（命中次数）; 在保证高频数据（经常被访问的数据）有效性场景下，可选择这类策略
  • LRU(least recently used)：最近最少使用策略(着重时间)，通俗点来说就是删除距离某个时间（比较元素最近一次被get使用时间）点最远的缓存; 在热点数据（近段时间经常被访问的数据）场景下较适用，优先保证热点数据的有效性

SpringBoot中本地缓存API应用

使用

启动本地缓存

@EnableCaching
@Configuration
public class CacheConfig {
}

操作注解

// 默认会将方法参数做为保存缓存对象的key
// 如可有多个参数也可以使用key参数来制定，不是的话会采用联合key，既合并作为key
// @Cacheable(cacheNames = "findById", key="#id")
@Cacheable(cacheNames = "findById")
public Optional<Person> findById(Long id) {
    return personRepository.findById(id);
}

@Cacheable(cacheNames = "findAll")
public List<Person> findAll() {
    return personRepository.findAll();
}

// 缓存条件age>100
@Cacheable(value="getPerson", key="#name", condition="#age>100")
public Person getPerson(String name, int age) {
    return new Person(name, age); 
}

// age==100不缓存
@Cacheable(value="getPerson2", unless="#age==100")
public Person getPerson2(String name, int age) {
    return new Person(name, age); 
}

// 和@Cacheable主要却别在于@CachePut会保证方法体执行
// 当@Cacheable发现传入的参数有缓存会直接返回缓存而不会执行方法体
@CachePut(value="getPerson3")
public Person getPerson3(String name) {
    System.out.println("into getPerson3()");
    return new Person(name, age); 
}

// 清楚指定cacheNames的缓存对象
@CacheEvict(cacheNames = "remove")
// 清除在cacheNames在特定key的缓存对象
// @CacheEvict(cacheNames = "remove", key = "#id")
public void remove(Long id) {
    personRepository.deleteById(id);
}

// 清除所有缓存
@CacheEvict(allEntries = true)
public void cleanCache() {
}

基本原理

spring中本地缓存默认采用JDK动态代理来实现

@Target(ElementType.TYPE)
@Retention(RetentionPolicy.RUNTIME)
@Documented
@Import(CachingConfigurationSelector.class)
public @interface EnableCaching {
    // 采用的切面模式
    AdviceMode mode() default AdviceMode.PROXY;
}

保存方式以key-value形式存储（SpringBoot默认情况下使用ConcurrentHashMap来缓存对象）

常用的本地缓存

ConcurrentHashMap（SpringBoot默认）

如果没有添加其他cache的依赖的话，默认SpringBoot启动SimpleCacheConfiguration进行初始化CacheManager

@Configuration
@ConditionalOnMissingBean(CacheManager.class)
@Conditional(CacheCondition.class)
class SimpleCacheConfiguration {
    private final CacheProperties cacheProperties;
    private final CacheManagerCustomizers customizerInvoker;
    SimpleCacheConfiguration(CacheProperties cacheProperties, CacheManagerCustomizers customizerInvoker) {
        this.cacheProperties = cacheProperties;
        this.customizerInvoker = customizerInvoker;
    }

    @Bean
    public ConcurrentMapCacheManager cacheManager() {
        ConcurrentMapCacheManager cacheManager = new ConcurrentMapCacheManager();
        List<String> cacheNames = this.cacheProperties.getCacheNames();
        if (!cacheNames.isEmpty()) {
            cacheManager.setCacheNames(cacheNames);
        }
        return this.customizerInvoker.customize(cacheManager);
    }

}

EhCache

Ehcache是现在最流行的纯Java开源缓存框架，配置简单、结构清晰、功能强大，是一个非常轻量级的缓存实现，我们常用的Hibernate里面就集成了相关缓存功能

SpringBoot中使用EhCache

1）添加依赖并添加@EnableCaching

<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-cache</artifactId>
</dependency>
<dependency>
    <groupId>net.sf.ehcache</groupId>
    <artifactId>ehcache</artifactId>
</dependency>

2）在类路径下添加ehcache.xml配置文件

<ehcache>

<!-- 指定一个文件目录，当Ehcache把数据写到硬盘上时，将把数据写到这个文件目录下 -->
<diskStore path="java.io.tmpdir"/>

<!-- 设定缓存的默认数据过期策略 -->
<defaultCache
        maxElementsInMemory="10000"
        eternal="false"
        overflowToDisk="true"
        timeToIdleSeconds="0"
        timeToLiveSeconds="0"
        diskPersistent="false"
        diskExpiryThreadIntervalSeconds="120"/>

<!-- 设定具体的命名缓存的数据过期策略 -->
<cache name="person1"
       maxElementsInMemory="1000"
       eternal="true"
       overflowToDisk="true"/>  

<cache name="perosn2"
    maxElementsInMemory="1000"
    eternal="false"
    timeToIdleSeconds="200"
    timeToLiveSeconds="4000"
    overflowToDisk="true"/>
</ehcache>

cache标签配置参数：

  • name：缓存名称
  • maxElementsInMemory：内存中最大缓存对象数
  • maxElementsOnDisk：硬盘中最大缓存对象数，若是0表示无穷大
  • eternal：true表示对象永不过期，此时会忽略timeToIdleSeconds和timeToLiveSeconds属性，默认为false
  • overflowToDisk：true表示当内存缓存的对象数目达到了maxElementsInMemory界限后，会把溢出的对象写到硬盘缓存中。注意：如果缓存的对象要写入到硬盘中的话，则该对象必须实现了Serializable接口才行
  • diskSpoolBufferSizeMB：磁盘缓存区大小，默认为30MB。每个Cache都应该有自己的一个缓存区
  • diskPersistent：是否缓存虚拟机重启期数据
  • diskExpiryThreadIntervalSeconds：磁盘失效线程运行时间间隔，默认为120秒
  • timeToIdleSeconds： 设定允许对象处于空闲状态的最长时间，以秒为单位。当对象自从最近一次被访问后，如果处于空闲状态的时间超过了timeToIdleSeconds属性值，这个对象就会过期，EHCache将把它从缓存中清空。只有当eternal属性为false，该属性才有效。如果该属性值为0，则表示对象可以无限期地处于空闲状态
  • timeToLiveSeconds：设定对象允许存在于缓存中的最长时间，以秒为单位。当对象自从被存放到缓存中后，如果处于缓存中的时间超过了 timeToLiveSeconds属性值，这个对象就会过期，Ehcache将把它从缓存中清除。只有当eternal属性为false，该属性才有效。如果该属性值为0，则表示对象可以无限期地存在于缓存中。timeToLiveSeconds必须大于timeToIdleSeconds属性，才有意义
  • memoryStoreEvictionPolicy：当达到maxElementsInMemory限制时，Ehcache将会根据指定的策略去清理内存。可选策略有：LRU（最近最少使用，默认策略）、FIFO（先进先出）、LFU（最少命中次数）

3）使用

@Cacheable(cacheNames = "person1")
public Optional<Person> findById(Long id) {
    return personRepository.findById(id);
}

@Cacheable(cacheNames = "person2")
public List<Person> findAll() {
    return personRepository.findAll();
}

Caffeine（推荐）

事实上在SpringBoot1.x的时候，SpringBoot是提供Guava Cache的自动配置集成的，但是在SpringBoot2.x后就被移除了，取而代之的就是Caffeine（咖啡因）; 这里需要注意的是Caffeine是Java 8重写的Guava Cache，取代了对Guava的支持;

拓展：关于guava cache

Google Guava是一个开源的Java工具库，提供了很多实用的功能实现，因而得到广泛的应用，而Guava Cache只是Google Guava其中的一个功能，Guava Cache实现了LRU（最近最少使用）算法并支持多种缓存过期策略;

Guava Cache使用推荐文章：

https://github.com/google/guava/wiki/CachesExplained

https://segmentfault.com/a/1190000011105644

对于Caffeine的性能而言毋庸置疑是目前最高效的本地缓存，对于这点网上也有很多评测

Caffeine采用的驱逐策略：Window TinyLFU（LRU+LFU结合）

关于Window TinyLFU通俗点来说，它采用一种叫CountMin Sketch的草图来把关（过滤器），先记录一些高命中的缓存对象，当 Sketch中的对象的命中次数高于正要被驱逐的缓存对象时才会真正被缓存，这样做的原因是为了保证在缓存中的对象都是高命中的对象（热点数据）;

除了CountMin Sketch之外，Window TinyLFU还采用分段LRU(Segmented LRU)来存储缓存对象来达到长期缓存的效果

如图所示，其中绿色的被称为试用段（probationary segment），紫色的就是CountMin Sketch，而红色部分叫保护段（protected segment），其中保护段的上限为Segmented LRU的80%;

基本原理是，当处于probationary segment的缓存对象被连续访问，再经过CountMin Sketch的把关后达到晋升到protected segment的机会，而因为protected segment上限可以达到整个Segmented LRU的80%，因此在后续的访问中，就单个缓存对象而言，它会不断晋升，直到80%这个临界点，这时缓存对象可能因为不满足继续停留在protected segment（剩余的LRU空间）会被驱逐出Segmented LRU回到probationary segment，而有资格停留的缓存对象会继续晋升直到够条件被驱逐为止，这样做的原因是确保保留具有最热的缓存数据，并且驱逐在Segmented LRU较冷的缓存数据

上面的内容参考自文章：http://highscalability.com/blog/2016/1/25/design-of-a-modern-cache.html

SpringBoot中使用Caffeine

添加依赖

<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-cache</artifactId>
</dependency>
<dependency>
    <groupId>com.github.ben-manes.caffeine</groupId>
    <artifactId>caffeine</artifactId>
</dependency>

配置Caffeine

spring:
  cache:
    type: caffeine
    cache-names:
      - person1
      - person2
    caffeine:
      spec: maximumSize=100,expireAfterWrite=60s

Caffeine配置说明：

  • initialCapacity=[integer]: 初始的缓存空间大小
  • maximumSize=[long]: 缓存的最大数量
  • maximumWeight=[long]: 缓存的最大权重
  • expireAfterAccess=[duration]: 最后一次写入或访问后经过固定时间过期
  • expireAfterWrite=[duration]: 最后一次写入后经过固定时间过期
  • refreshAfterWrite=[duration]: 创建缓存或者最近一次更新缓存后经过固定的时间间隔，刷新缓存
  • weakKeys: 打开key的弱引用，优化GC回收
  • weakValues：打开value的弱引用，优化GC回收
  • softValues：打开value的软引用，优化GC回收
  • recordStats：开发统计功能

注意：

  • expireAfterWrite和expireAfterAccess同事存在时，以expireAfterWrite为准。
  • maximumSize和maximumWeight不可以同时使用
  • weakValues和softValues不可以同时使用
