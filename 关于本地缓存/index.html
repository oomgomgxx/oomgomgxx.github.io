关于本地缓存

 2019-02-24   作者: TanDi   类型：原创

什么是本地缓存？

本地缓存又称为进程缓存，顾名思义本地缓存是与应用处于统一进程同一空间的。这点与分布式缓存形成了鲜明的对比。

缓存的意义

不论是本地缓存还是分布式缓存，其主要目的都是为了减少不必要的数据库访问。因为从系统角度来看往往较早出现瓶颈的地方都是数据库。这么说的原因有两个，一是因为 JDBC 操作是阻塞式的，二是因为数据库存在大量的磁盘IO操作，而且连接数上也存在局限（受限于所在操作系统tcp最大连接数 ulimit -n）。那么从以上两点来看，降低数据库访问压力是势在必行的事情，而缓存就能有效地做到这点。

使用本地缓存需要注意的问题

  • 缓存一致性问题：举个例子，假设现在系统使用Nginx进行反向代理实现负载均衡，既将原来的单个节点横向扩展成多个节点，那么这时对于本地缓存而言因为它是线程独立的，所以每个节点上的本地缓存内容都可能不一样，这是使用本地缓存时要注意的
  • 缓存对象的存活时间：使用本地缓存时要避免缓存堆积从而占用内存空间，因此应该使用特定的缓存清空策略来定期去除那些无用缓存
  • 微服务不建议使用本地缓存：因为本地缓存存在一致性问题，所以本地缓存会让微服务节点带上状态。鉴于微服务设计原则因此并不推荐在微服务中使用本地缓存（微服务应该是无状态的）

缓存驱逐策略

  • FIFO(first in first out)：先进先出策略。通过一个带有长度限制的 FIFO 队列实现，当队列满了的时候再往队中添加缓存的话，那么队列头的缓存就会率先被剔除
  • LRU(least recently used)：最近最少使用策略(着重时间)。可以使用k-v节点形成固定长度的队列，再使用头插法将超出容量的尾部缓存剔除
  • LFU(less frequently used)：最少使用策略（着重次数，个人认为叫最少命中策略更加合适）。可以根据缓存的被使用次数来判断，通过清除哪些使用次数较少的缓存来释放内存空间

SpringBoot本地缓存API

使用

启动本地缓存

@EnableCaching // 启用本地缓存
@Configuration
public class CacheConfig {
}

操作注解

// 默认会将方法参数列表作为缓存对象的key
// 如可有多个参数的话，也可以使用key参数来指定
// @Cacheable(cacheNames = "findById", key="#id")
@Cacheable(cacheNames = "findById")
public Optional<Person> findById(Long id) {
    return personRepository.findById(id);
}

@Cacheable(cacheNames = "findAll")
public List<Person> findAll() {
    return personRepository.findAll();
}

// 缓存条件age>100
@Cacheable(value="getPerson", key="#name", condition="#age>100")
public Person getPerson(String name, int age) {
    return new Person(name, age); 
}

// age==100不缓存
@Cacheable(value="getPerson2", unless="#age==100")
public Person getPerson2(String name, int age) {
    return new Person(name, age); 
}

// 和@Cacheable区别在于@CachePut会保证方法体执行
// 当@Cacheable发现传入的参数有缓存会直接返回缓存而不会执行方法体
@CachePut(value="getPerson3")
public Person getPerson3(String name) {
    System.out.println("into getPerson3()");
    return new Person(name, age); 
}

// 清楚指定cacheNames的缓存对象
@CacheEvict(cacheNames = "remove")
// 清除在cacheNames在特定key的缓存对象
// @CacheEvict(cacheNames = "remove", key = "#id")
public void remove(Long id) {
    personRepository.deleteById(id);
}

// 清除所有缓存
@CacheEvict(allEntries = true)
public void cleanCache() {
}

基本原理

Spring本地缓存默认采用JDK动态代理来实现

@Target(ElementType.TYPE)
@Retention(RetentionPolicy.RUNTIME)
@Documented
@Import(CachingConfigurationSelector.class)
public @interface EnableCaching {
    // 采用的切面模式
    AdviceMode mode() default AdviceMode.PROXY;
}

保存方式是 key-value 形式（SpringBoot默认情况下会使用 ConcurrentHashMap 来保存缓存对象）

SpringBoot Cache整合Redis（分布式缓存）

整合案例

依赖

<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-data-redis</artifactId>
</dependency>
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-cache</artifactId>
</dependency>

整合配置类

@EnableCaching
@Configuration(proxyBeanMethods = false)
public class CacheConfig {

    /**
     * spring cache 整合 redis
     * @param redisTemplate
     * @return
     */
    @Bean
    public CacheManager cacheManager(RedisTemplate redisTemplate) {

        // k-v序列化器
        final RedisSerializationContext.SerializationPair serializeKeys =
                RedisSerializationContext.SerializationPair.fromSerializer(redisTemplate.getStringSerializer());

        final RedisSerializationContext.SerializationPair serializeValues =
                RedisSerializationContext.SerializationPair.fromSerializer(redisTemplate.getValueSerializer());

        // Redis缓存配置
        RedisCacheConfiguration defaultCacheConfiguration = RedisCacheConfiguration.defaultCacheConfig()
                .serializeKeysWith(serializeKeys)
                .serializeValuesWith(serializeValues)
                .disableCachingNullValues() // 禁止缓存null值
                .entryTtl(Duration.ofHours(1)); // 统一缓存1个小时

        // 根据缓存配置构造出管缓存理器
        RedisCacheManager redisCacheManager = RedisCacheManager.RedisCacheManagerBuilder
                .fromConnectionFactory(redisTemplate.getConnectionFactory())
                .cacheDefaults(defaultCacheConfiguration)
                .transactionAware()
                .build();

        return redisCacheManager;
    }
}

自定义缓存注解 - 解决@Cacheble局限

依赖

<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-aop</artifactId>
</dependency>

自定义缓存注解

@Target(ElementType.METHOD)
@Retention(RetentionPolicy.RUNTIME)
public @interface CustomCache {
    @AliasFor("cacheKey")
    String value() default "";
    String cacheKey() default "";
    int expires(); // 单位:秒
}

注解处理类（切面类）

@Component
@Aspect
public class CustomCacheAspect {

    @Autowired
    private RedisTemplate redisTemplate;

    @Pointcut("@annotation(customCache)") // 拦截注解 TestAnnotation
    public void pointcut(CustomCache customCache) {
    }

    @Around("pointcut(customCache)")
    public Object around(ProceedingJoinPoint proceedingJoinPoint, CustomCache customCache) throws Throwable {

        // 序列化和反序列化工具：Gson
        final Gson gson = new Gson();

        // 表达式解析器 和 参数列表变量名发现器
        final ExpressionParser parser = new SpelExpressionParser();
        final ParameterNameDiscoverer nameDiscoverer = new LocalVariableTableParameterNameDiscoverer();

        // 当前切入点方法
        MethodSignature methodSignature = (MethodSignature) proceedingJoinPoint.getSignature();
        Method method = methodSignature.getMethod();

        // 当前切入点方法的参数列表变量名称和值
        String[] params = nameDiscoverer.getParameterNames(method);
        final Object[] args = proceedingJoinPoint.getArgs();

        // 构建表达式解析目标（根对象）
        // 注意，如果根对象经常变动则不建议使用解析上下万，因为解析上下文会缓存值，如果根对象不变则建议使用解析上下文
        Map<String, Object> methodParams = new HashMap<>();
        for (int i = 0; i < params.length; i++) {
            methodParams.put(params[i], args[i]);
        }

        // 构造解析上下文
        /*
        EvaluationContext context = new StandardEvaluationContext();
        for (int len = 0; len < params.length; len++) {
            context.setVariable(params[len], args[len]);
        }
         */

        String el = null;
        Expression expression = null;
        String condition = null;
        try {
            el = "".equals(customCache.value())?customCache.cacheKey():customCache.value();
            expression = parser.parseExpression(el);// 解析表达式
            condition = expression.getValue(methodParams, String.class);

        } catch (ParseException e1) {
            throw new RuntimeException("解析异常，当前注解的表达式为：" + el);
        } catch (EvaluationException e2) {
            throw new RuntimeException("求值异常，当前注解的表达式为：" + el);
        }

        String jsonObject = null;

        // 判断是否已经存在缓存
        if (redisTemplate.hasKey(condition)) {
            System.out.println("从缓存中获取, ttl = " + redisTemplate.getExpire(condition));
            jsonObject = (String)redisTemplate.opsForValue().get(condition);
            return gson.fromJson(jsonObject, Person.class);
        }

        System.out.println("从数据库中获取");

        // 执行完切入点方法所返回的值
        Person methodInvokeResult = (Person)proceedingJoinPoint.proceed();

        // 将方法返回结果缓存
        jsonObject = gson.toJson(methodInvokeResult);
        redisTemplate.opsForValue().set(condition, jsonObject, customCache.expires(), TimeUnit.SECONDS);

        return methodInvokeResult;
    }

}

常用的本地缓存

ConcurrentHashMap（SpringBoot默认）

如果没有添加第三方 Cache 依赖的话，SpringBoot 默认会启动 SimpleCacheConfiguration 来对CacheManager做初始化操作

@Configuration
@ConditionalOnMissingBean(CacheManager.class)
@Conditional(CacheCondition.class)
class SimpleCacheConfiguration {
    private final CacheProperties cacheProperties;
    private final CacheManagerCustomizers customizerInvoker;
    SimpleCacheConfiguration(CacheProperties cacheProperties, CacheManagerCustomizers customizerInvoker) {
        this.cacheProperties = cacheProperties;
        this.customizerInvoker = customizerInvoker;
    }

    @Bean
    public ConcurrentMapCacheManager cacheManager() {
        ConcurrentMapCacheManager cacheManager = new ConcurrentMapCacheManager();
        List<String> cacheNames = this.cacheProperties.getCacheNames();
        if (!cacheNames.isEmpty()) {
            cacheManager.setCacheNames(cacheNames);
        }
        return this.customizerInvoker.customize(cacheManager);
    }

}

EhCache

Ehcache是目前比较流行的纯 Java 开源缓存框架。它的特点主要体现在配置简单、结构清晰 和 功能强大。是一个非常轻量级的缓存实现。例如我们常使用的ORM框架 Hibernate 就默认使用EhCache来做二级缓存

在SpringBoot中使用EhCache

1）添加依赖并添加@EnableCaching

<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-cache</artifactId>
</dependency>
<dependency>
    <groupId>net.sf.ehcache</groupId>
    <artifactId>ehcache</artifactId>
</dependency>

2）在类路径下添加 ehcache.xml 配置文件

<ehcache>

<!-- 指定一个文件目录，当Ehcache把数据写到硬盘上时，将把数据写到这个文件目录下 -->
<diskStore path="java.io.tmpdir"/>

<!-- 设定缓存的默认数据过期策略 -->
<defaultCache
        maxElementsInMemory="10000"
        eternal="false"
        overflowToDisk="true"
        timeToIdleSeconds="0"
        timeToLiveSeconds="0"
        diskPersistent="false"
        diskExpiryThreadIntervalSeconds="120"/>

<!-- 设定具体的命名缓存的数据过期策略 -->
<cache name="person1"
       maxElementsInMemory="1000"
       eternal="true"
       overflowToDisk="true"/>  

<cache name="perosn2"
    maxElementsInMemory="1000"
    eternal="false"
    timeToIdleSeconds="200"
    timeToLiveSeconds="4000"
    overflowToDisk="true"/>
</ehcache>

cache标签配置参数：

  • name：缓存名称
  • maxElementsInMemory：内存中最大缓存对象数
  • maxElementsOnDisk：硬盘中最大缓存对象数，若是0表示无穷大
  • eternal：true表示对象永不过期，此时会忽略timeToIdleSeconds和timeToLiveSeconds属性，默认为false
  • overflowToDisk：true表示当内存缓存的对象数目达到了maxElementsInMemory界限后，会把溢出的对象写到硬盘缓存中。注意：如果缓存的对象要写入到硬盘中的话，则该对象必须实现了Serializable接口才行
  • diskSpoolBufferSizeMB：磁盘缓存区大小，默认为30MB。每个Cache都应该有自己的一个缓存区
  • diskPersistent：是否缓存虚拟机重启期数据
  • diskExpiryThreadIntervalSeconds：磁盘失效线程运行时间间隔，默认为120秒
  • timeToIdleSeconds： 设定允许对象处于空闲状态的最长时间，以秒为单位。当对象自从最近一次被访问后，如果处于空闲状态的时间超过了timeToIdleSeconds属性值，这个对象就会过期，EHCache将把它从缓存中清空。只有当eternal属性为false，该属性才有效。如果该属性值为0，则表示对象可以无限期地处于空闲状态
  • timeToLiveSeconds：设定对象允许存在于缓存中的最长时间，以秒为单位。当对象自从被存放到缓存中后，如果处于缓存中的时间超过了 timeToLiveSeconds属性值，这个对象就会过期，Ehcache将把它从缓存中清除。只有当eternal属性为false，该属性才有效。如果该属性值为0，则表示对象可以无限期地存在于缓存中。timeToLiveSeconds必须大于timeToIdleSeconds属性，才有意义
  • memoryStoreEvictionPolicy：当达到maxElementsInMemory限制时，Ehcache将会根据指定的策略去清理内存。可选策略有：LRU（最近最少使用，默认策略）、FIFO（先进先出）、LFU（最少命中次数）

3）使用

public Optional<Person> findById(Long id) {
    return personRepository.findById(id);
}

@Cacheable(cacheNames = "person2")
public List<Person> findAll() {
    return personRepository.findAll();

Caffeine（推荐）

简介

本质上来说 Caffeine 是用 Java 8 重写 Guava Cache 的产物。在性能方面 Caffeine 毋庸置疑是目前最高效的本地缓存实现。

Caffeine 采用一种名为 Window TinyLFU（LRU+LFU结合）的缓存驱逐策略来释放内存占用空间。它的基本原理是先用一种叫 CountMin Sketch 的机制过滤缓存（主要是记录下命中次数较高的缓存），然后当 Window TinyLFU 要剔除缓存时会先到 CountMin Sketch 中检查，看看 CountMin Sketch 中有没有命中次数高于当前正要剔除的缓存的缓存，如果有就将其纳入到正式缓存中（可以看到 CountMin Sketch 其实起着缓冲的作用）。这样做的好处是能够保证在缓存的对象都是高命中的热点数据。

除了 CountMin Sketch 机制之外，Window TinyLFU 还用 Segmented LRU 来管理缓存。例如上图所示，绿色部分被称为试用段（probationary segment），紫色部分就是上面提到的 CountMin Sketch ，而红色部分叫保护段（protected segment）。其中保护段的上限为整个 Segmented LRU 的80%。当处于 probationary segment 的缓存对象被连续访问，它就会在 CountMin Sketch 得到晋升 protected segment 的机会，而因为 protected segment 上限可以达到整个 Segmented LRU 的80%，所以在后续的访问中缓存会被不断晋升直到80%这个临界点，这时缓存对象可能因为不满足继续停留就会被驱逐出 Segmented LRU 而回到 probationary segment中，而有资格停留的缓存对象则会继续晋升直到够条件被驱逐为止。这样做的原因是确保热点缓存长期驻留在 Segmented LRU 内，而将那么相对来说较为冷门的缓存驱逐出 Segmented LRU

SpringBoot中使用Caffeine

添加依赖

<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-cache</artifactId>
</dependency>
<dependency>
    <groupId>com.github.ben-manes.caffeine</groupId>
    <artifactId>caffeine</artifactId>
</dependency>

配置Caffeine

spring:
  cache:
    type: caffeine
    cache-names:
      - person1
      - person2
    caffeine:
      spec: maximumSize=100,expireAfterWrite=60s

Caffeine配置说明：

  • initialCapacity=[integer]: 初始的缓存空间大小
  • maximumSize=[long]: 缓存的最大数量
  • maximumWeight=[long]: 缓存的最大权重
  • expireAfterAccess=[duration]: 最后一次写入或访问后经过固定时间过期
  • expireAfterWrite=[duration]: 最后一次写入后经过固定时间过期
  • refreshAfterWrite=[duration]: 创建缓存或者最近一次更新缓存后经过固定的时间间隔，刷新缓存
  • weakKeys: 打开key的弱引用，优化GC回收
  • weakValues：打开value的弱引用，优化GC回收
  • softValues：打开value的软引用，优化GC回收
  • recordStats：开发统计功能

注意：

  • expireAfterWrite和expireAfterAccess同事存在时，以expireAfterWrite为准。
  • maximumSize和maximumWeight不可以同时使用
  • weakValues和softValues不可以同时使用
