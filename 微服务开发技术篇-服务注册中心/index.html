为什么需要服务注册中心

点对点消费

上面这种通信方式的缺点是，当服务提节点数量或状态发生变化时，服务消费者无法感知，并且需要手动修改配置，不利于维护

使用反向代理改良

引入反向代理服务来代理上游服务器集群。这时服务消费者只需和反向代理服务打交道即可，大大降低了服务地址的维护成本。

但这种方法依然存在一定的问题：

  • 反向代理服务存在单点问题，需要往外引入注入Keepalived这样的服务提高其可用性，增加了架构的复杂度
  • 反向代理服务依然无法感知上游服务提供者节点状态的变化，开发者仍然需要手动维护服务提供者配置

使用服务注册/发现中心

  1. 服务提供者启动时将自己暴露的服务列表注册到服务注册中心
  2. 服务消费者启动时到服务注册中心拉取自己所关注的服务提供者列表

服务注册/发现中心不仅解决了服务提供者节点变动时需要手动修改配置的问题，还能够实现动态服务注册和服务状态感知

目前业界常用的服务注册中心

  • ZooKeeper
  • Netflix Eureka
  • Consul
  • Etcd

Netflix Eureka

介绍

  • Eureka是Netflix旗下开源的服务注册/发现中心
  • GitHub首页已明确表明Eureka 2.x已经停止开源工作（Eureka 1.x依然在维护，可以提交Issues）
  • 从CAP定理角度来看，Eureka是一个AP（可用性优先）系统

@Slf4j
@Repository
public class MessageRepository {

    // 要调用的服务
    private final static String SERVEICE_NAME ="MSA-EUREKA-PROVIDER-USER";

    // HTTP请求工具
    @Autowired
    private RestTemplate restTemplate;

    // 服务发现客户端
    @Autowired
    private DiscoveryClient discoveryClient;

    // delete请求
    public User deleteUserById(Long id) {
        String urlPrefix = "http://";
        String urlSuffix = "/user/delete/{id}";
        String url = urlPrefix + SERVEICE_NAME + urlSuffix;

        // restTemplate.delete(url, id); // 无返回值
        ResponseEntity<User> exchange = restTemplate.exchange(url, HttpMethod.DELETE, null, User.class, id);

        return exchange.getBody();
    }

    // get请求
    public List selectAllUser() {
        String urlPrefix = "http://";
        String urlSuffix = "/user/find/all";
        String url = urlPrefix + SERVEICE_NAME + urlSuffix;

        List<ServiceInstance> instances = discoveryClient.getInstances(SERVEICE_NAME);
        List<String> services = discoveryClient.getServices();
        String description = discoveryClient.description();
        log.info("服务名称：{}", services);
        log.info("服务提供者：{}", instances);
        log.info("服务描述：{}", description);

        return restTemplate.getForObject(url, List.class);
    }


    // post请求
    public User addUser(User user) {

        String urlPrefix = "http://";
        String urlSuffix = "/user/add";
        String url = urlPrefix + SERVEICE_NAME + urlSuffix;

        // 请求头
        HttpHeaders headers = new HttpHeaders();
        headers.setContentType(MediaType.APPLICATION_FORM_URLENCODED);

        // 请求体
        MultiValueMap<String, Object> map= new LinkedMultiValueMap<>();
        map.add("id", user.getId());
        map.add("name", user.getName());
        map.add("age", user.getAge());

        // 请求内容
        HttpEntity<MultiValueMap<String, Object>> request = new HttpEntity<>(map, headers);

        return restTemplate.postForEntity(url, request, User.class).getBody();
    }
}

特性和源码浅析

基础架构

Netflix Eureka集群图（源自https://github.com/Netflix/eureka）

上图是 Eureka 1.x 在 Netflix 内部中的部署架构。在 Eureka 架构中有两个较为重要的概念，分别是：Region、Zone。

Region 可以理解为 Eureka 所在的区域，而 Zone 可以理解为 Eureka 所在机房。

一个 Region 可以包含多个Zone，而且每个 Region 中至少含有一个Eureka Server。Zone与Zone、Region与Region之间形成高可用集群。

默认情况下 Eureka 使用的 Region 名为 us-east-1，而 Zone 为http://localhost:8761/eureka/

eureka:
  client:
      # 以下为默认配置，所以可以省略
    serviceUrl:
      defaultZone: http://localhost:8761/eureka/
    region: us-east-1

EurekaClientConfigBean 源码

private String region = "us-east-1";
public static final String DEFAULT_ZONE = "defaultZone";

@Override
public String[] getAvailabilityZones(String region) {
    // 如果没有配置就用默认的
    String value = this.availabilityZones.get(region);
    if (value == null) {
        value = DEFAULT_ZONE;
    }
    return value.split(",");
}

服务注册和发现时机

  • Eureka Client将服务注册到 Eureka Server 后会每隔30秒就发送心跳包给Eureka Server以示对当前服务进行续约。而如果Eureka Server在90s（默认）内都没有收到某个服务提供者的续约心跳包的话，就会从可用服务列表中下线该服务。但前提是没有出现网络分区（譬如短时间内丢失大量客户端连接），因为一旦出现网络分区Eureka Server就会开启保护机制，保护机制一旦开启 Eureka Server 就会暂缓下线服务直到大部分服务可用后恢复

  • Eureka Client 从 Eureka Server 中获取到服务列表后会对其进行缓存，然后再每隔30s（默认）重新抓取一次服务列表。这样做的原因主要是防止 Eureka Server 宕机导致 Eureka Client 无法获取到服务列表，所以通过缓存服务列表来以防 Eureka Server 宕机后无法发现服务的情况发生。

EurekaClientConfigBean 源码

Eureka Client 会使用 Ribbon 对抓取到的服务列表进行负载均衡调用，而在这之前 Eureka Client 需要先确立自己应该从哪个 Eureka Server 中抓取服务列表。

@Override
public List<String> getEurekaServerServiceUrls(String myZone) {
    // 先从自己所在的 Zone 中获取 Eureka Server
    String serviceUrls = this.serviceUrl.get(myZone);

    // 如果自己的所在 Zone 中没有 Eureka Server ，则到默认的 Zone 中查找
    if (serviceUrls == null || serviceUrls.isEmpty()) {
        serviceUrls = this.serviceUrl.get(DEFAULT_ZONE);
    }
    if (!StringUtils.isEmpty(serviceUrls)) {
        final String[] serviceUrlsSplit = StringUtils
            .commaDelimitedListToStringArray(serviceUrls);
        List<String> eurekaServiceUrls = new ArrayList<>(serviceUrlsSplit.length);
        for (String eurekaServiceUrl : serviceUrlsSplit) {
            if (!endsWithSlash(eurekaServiceUrl)) {
                eurekaServiceUrl += "/";
            }
            eurekaServiceUrls.add(eurekaServiceUrl.trim());
        }
        return eurekaServiceUrls;
    }

    return new ArrayList<>();
}

自我保护机制

上面已经提到 Eureka Server 实现了一种自我保护机制，该机制是专门用来应对网络分区的

Eureka中服务状态有以下几种

  • UP：正常提供服务

  • DOWN：服务已下线

  • OUT_OF_SERVICE：已停止服务

  • UNKNOWN：未知

服务注册过程

DiscoveryClient 源码

boolean register() throws Throwable {
    EurekaHttpResponse<Void> httpResponse;
    try {
        // 注册服务
        httpResponse = eurekaTransport.registrationClient.register(instanceInfo);
    } catch (Exception e) {}
    return httpResponse.getStatusCode() == Status.NO_CONTENT.getStatusCode();
}

进入 eurekaTransport.registrationClient.register(instanceInfo);

@Override
public EurekaHttpResponse<Void> register(final InstanceInfo info) {
    return execute(new RequestExecutor<Void>() {
        @Override
        public EurekaHttpResponse<Void> execute(EurekaHttpClient delegate) {
             // 注意，默认情况下Eureka Client会使用Jersey来和Eureka Server通信
            // 可以将其依赖排除，改用RestTemplate
            return delegate.register(info);
        }

        @Override
        public RequestType getRequestType() {
            return RequestType.Register;
        }
    });
}

点进 delegate.register(info)

// 拼接想服务注册中心注册服务的请求
// 通过Eureka Server提供的Restful端点实现
@Override
public EurekaHttpResponse<Void> register(InstanceInfo info) {
    // 请求URL
    String urlPath = serviceUrl + "apps/" + info.getAppName();

    // 请求头
    HttpHeaders headers = new HttpHeaders();
    headers.add(HttpHeaders.ACCEPT_ENCODING, "gzip");
    headers.add(HttpHeaders.CONTENT_TYPE, MediaType.APPLICATION_JSON_VALUE);

    // 发出请求
    ResponseEntity<Void> response = restTemplate.exchange(urlPath, HttpMethod.POST,
                                                          new HttpEntity<>(info, headers), Void.class);

    return anEurekaHttpResponse(response.getStatusCodeValue())
        .headers(headersOf(response)).build();
}

获取服务列表和续约服务的定时任务

DiscoveryClient

private void initScheduledTasks() {

    // 获取服务列表的定时任务
    if (clientConfig.shouldFetchRegistry()) {
        // registry cache refresh timer
        // 重新抓取的时间间隔默认为30s（在EurekaClientConfigBean中定义）
        int registryFetchIntervalSeconds = clientConfig.getRegistryFetchIntervalSeconds();
        int expBackOffBound = clientConfig.getCacheRefreshExecutorExponentialBackOffBound();

        scheduler.schedule(
            new TimedSupervisorTask(
                "cacheRefresh",
                scheduler,
                cacheRefreshExecutor,
                registryFetchIntervalSeconds,
                TimeUnit.SECONDS,
                expBackOffBound,
                new CacheRefreshThread()
            ),
            registryFetchIntervalSeconds, TimeUnit.SECONDS);
    }

    // 续约服务的定时任务
    if (clientConfig.shouldRegisterWithEureka()) {
        // 时间间隔默认为30s（LeaseInfo中定义）
        int renewalIntervalInSecs = instanceInfo.getLeaseInfo().getRenewalIntervalInSecs();
        int expBackOffBound = clientConfig.getHeartbeatExecutorExponentialBackOffBound();

        // Heartbeat timer
        scheduler.schedule(
            new TimedSupervisorTask(
                "heartbeat",
                scheduler,
                heartbeatExecutor,
                renewalIntervalInSecs,
                TimeUnit.SECONDS,
                expBackOffBound,
                new HeartbeatThread()
            ),
            renewalIntervalInSecs, TimeUnit.SECONDS);
    }
}

AgentConsulClient

// 服务注册
@Override
public Response<Void> agentServiceRegister(NewService newService, String token) {
    UrlParameters tokenParam = token != null ? new SingleUrlParameters("token", token) : null;
    // 使用Goolgl Gson序列化服务注册信息
    String json = GsonFactory.getGson().toJson(newService);
    // 服务注册（默认使用HttpClient进行HTTP通信）
    // 可以看到这里和Eureka Server类似，是通过Consul提供的Restful接口注册服务
    RawResponse rawResponse = rawClient.makePutRequest("/v1/agent/service/register", json, tokenParam);
    // 返回注册结果
    return new Response<Void>(null, rawResponse);
}

// 获取服务列表
@Override
public Response<Map<String, Service>> getAgentServices() {
    // 获取注册在Consul上的所有服务
    RawResponse rawResponse = rawClient.makeGetRequest("/v1/agent/services");
    // 封装到Map中
    Map<String, Service> agentServices = GsonFactory.getGson()
        .fromJson(rawResponse.getContent(),
                  new TypeToken<Map<String, Service>>() {}.getType());

    return new Response<Map<String, Service>>(agentServices, rawResponse);
}
