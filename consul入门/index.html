Consul的基本使用

 2018-08-03   作者: TanDi

是什么？

  • Consul 是 HashiCorp 公司推出的开源工具，用于实现分布式系统的服务发现与配置
  • Consul 提供 服务注册与发现、分布一致性协议实现、健康检查、Key/Value存储、多数据中心等功能，而无需要依赖其它工具（比如ZooKeeper等），使用起来也较为简单
  • Consul 使用Golang开发，具有天然的可移植性。其中安装包仅包含一个可执行文件，方便部署之余且和Docker能够无缝配合

下载安装

官网：https://www.consul.io/

consul是一个命令工具，上传linux或直接用wget下载，解压后将其mv到/bin目录即可

命令参数

-advertise        通知展现地址用来改变我们给集群中的其他节点展现的地址，一般情况下-bind地址就是展现地址
-bootstrap        用来控制一个server是否在bootstrap模式，在一个datacenter中只能有一个server处于bootstrap模式，当一个server处于bootstrap模式时，可以自己选举为raft leader。
-bootstrap-expect 在数据中心中期望提供的server节点数目，当该值提供的时候，consul一直等到达到指定sever数目的时候才会引导整个集群，该标记不能和bootstrap公用
-bind             该地址用来在集群内部的通讯，集群内的所有节点到地址都必须是可达的，默认是0.0.0.0
-client           consul绑定的地址，这个地址提供HTTP、DNS、RPC等服务，默认是127.0.0.1
-config-file      明确的指定要加载哪个配置文件
-config-dir       配置文件目录，里面所有以.json结尾的文件都会被加载
-data-dir         提供一个目录用来存放agent的状态，所有的agent允许都需要该目录，该目录必须是稳定的，系统重启后都继续存在
-dc               该标记控制agent允许的datacenter的名称，默认是dc1
-encrypt          指定secret key，使consul在通讯时进行加密，key可以通过consul keygen生成，同一个集群中的节点必须使用相同的key
-join             加入一个已经启动的agent的ip地址，可以多次指定多个agent的地址。如果consul不能加入任何指定的地址中，则agent会启动失败，默认agent启动时不会加入任何节点。
-retry-join       和join类似，但是允许你在第一次失败后进行尝试。
-retry-interval   两次join之间的时间间隔，默认是30s
-retry-max        尝试重复join的次数，默认是0，也就是无限次尝试
-log-level        consul agent启动后显示的日志信息级别。默认是info，可选：trace、debug、info、warn、err。
-node             节点在集群中的名称，在一个集群中必须是唯一的，默认是该节点的主机名
-protocol         consul使用的协议版本
-rejoin           使consul忽略先前的离开，在再次启动后仍旧尝试加入集群中。
-server           定义agent运行在server模式，每个集群至少有一个server，建议每个集群的server不要超过5个
-syslog           开启系统日志功能，只在linux/osx上生效
-ui-dir           提供存放web ui资源的路径，该目录必须是可读的
-pid-file         提供一个路径来存放pid文件，可以使用该文件进行SIGINT/SIGHUP(关闭/更新)agent

服务端口

端口 作用
8300 Used for RPC Server
8301 LAN GOSSIP（局域网传播）
8302 WAN GOSSIP（广域网传播）
8500 Used for HTTP Server
8600 Used for DNS Server
8502 Used for gRPC Server

基础架构

  • Consul 支持多数据中心（官网建议使用1个）
  • Consul Cluster 由多个 Consul Agent 实例组成
  • 其中 Agent 分为： Client模式 和 Server模式。Server模式主要负责对外提供服务。Client模式是一个轻量级的进程，主要用于注册服务、运行健康检查、转发请求等
  • Agent Client 与 Agent Server 通信使用 Gossip协议 和 RPC协议
  • Agent Server 之间通信使用 Raft算法 和 RPC协议
  • 其中 Agent Server 又分为：Leader 和 Follower 两种角色。Leader主要负责 处理请求 和 同步数据
  • 集群的 Agent Server 通过Raft算法来实现Leader选举，所以 Agent Server 数量应该是 2n+1（官方推荐每个集群至少有1个 Agent Server 且不要超过5个）
  • 在Consul的Raft算法实现中，Follower 会将所有请求转发给 Leader 处理，通过这样的方式来提高一致性

Agent的启动和停止

启动agent

➜  ~ consul agent -dev -node agent-01 -client 192.168.31.250
  • -dev：开发模式启动。可以省略数据 bind 和 data_dir 配置
  • -node：节点名称。默认为所在机器名称
  • -client：HTTP、DNS、RPC等服务的通信地址。默认为127.0.0.1

命令方式 - 查看集群成员

➜  ~ consul members -http-addr=192.168.31.250:8500
Node      Address         Status  Type    Build  Protocol  DC   Segment
agent-01  127.0.0.1:8301  alive   server  1.6.2  2         dc1  <all>
  • -http-addr：对应agent启动时设置的 -client 值。默认为127.0.0.1:8500

REST方式 - 查看集群成员

➜  ~ curl 192.168.31.250:8500/v1/catalog/nodes

停止agent

➜  ~ consul leave -http-addr=192.168.31.250:8500

启动一个生产环境的Agent

➜  apps consul agent -server -data-dir /tmp/consul -node=consul-01 -bind=192.168.31.250 -config-dir /etc/consul.d
  • -data-dir：保存Agent运行过程产生的数据
  • -node：节点名称
  • -bind：绑定的机器，即监听那台机器的服务
  • -config-dir：配置文件夹

注册和查询服务

步骤1：手动注册一个服务

需要手动创建文件和写入内容，文件名称一般和服务名称一样

➜  ~ mkdir /etc/consul.d
➜  ~ echo '{"service": {"name": "web", "tags": ["rails"], "port": 80}}' > /etc/consul.d/web.json

步骤2：启动agent并指定配置文件

➜ consul agent -dev -node=agent-01 -config-dir=/etc/consul.d

步骤3：REST方式查询服务。可以追加 ?passing 表明只查询健康的服务

搭建集群

环境说明：两台CentOS7虚拟机，简称cent01，cent02；其中cent02复制cent01（虚拟机复制要小心，特别要注意软件的配置文件和数据）

SSH连接如下：

步骤1：创建一个Server节点（cent01）

➜  ~ consul agent -server -bootstrap-expect 1  -data-dir /tmp/consul -node=server-node -bind=192.168.4.157  -config-dir /etc/consul.d

步骤2：创建一个Client节点（cent02）

➜  ~ consul agent -data-dir /tmp/consul -node=client-node01 -bind=192.168.4.159 -config-dir /etc/consul.d

步骤3：将client-node01节点加入到集群（cent02）

注意：在consul中，节点想要加入到集群中，只需要知道集群中的随便一个服务节点即可

➜  ~ consul join 192.168.4.157

步骤4：查看consul集群信息

➜  ~ consul members
Node(节点名称)  Address(节点位置)    Status  Type    Build  Protocol  DC   Segment
server-node    192.168.4.157:8301  alive   server  1.4.2  2         dc1  <all>
client-node01  192.168.4.159:8301  alive   client  1.4.2  2         dc1  <default>

集群中可能出现的问题

  • 无法加入集群，因为nodeId冲突

    • 虚拟机复制的时候报/tmp/consul也复制了，最大可能是在虚拟机复制之前，使用过/tmp/consul，并产生了nodeid保存到了目录，解决方法将其清除重启consul agent

  • 桥接模式，两台虚拟机之间ping不同

    • 确认网络可用，即虚拟机能联网，主机能ssh操作虚拟机
    • 关闭主机和虚拟机的防火墙

启用WEB-UI

本机访问指定 -ui 即可。非本机访问需要指定 -client 0.0.0.0 -ui

➜  consul agent -server -bootstrap-expect 1 -data-dir /tmp/consul -bind 192.168.31.250 -config-dir /etc/consul.d  -client 
0.0.0.0 -ui

访问：http://localhost:8500

键值对存取

保存 key - value

# 在kv（根）目录创建
➜  ~ curl -X PUT -d '值' http://localhost:8500/v1/kv/键
# 在指定目录下创建
➜  ~ curl -X PUT -d '值' http://localhost:8500/v1/kv/目录名称/键

查看 key - value

# 查询kv（根）目录下所有key-value
➜  ~ curl http://localhost:8500/v1/kv/\?recurse
# 查询指定key
➜  ~ curl http://localhost:8500/v1/kv/键

更新 key - value

➜  ~ curl -X PUT -d '值' http://localhost:8500/v1/kv/键

删除 key - value

➜  ~ curl -X DELETE http://localhost:8500/v1/kv/键
