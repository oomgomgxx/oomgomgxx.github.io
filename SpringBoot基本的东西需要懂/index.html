关于 @Conditional 注解

@Conditional 注解是 Spring 4 提供的实例化谓语，也就是说只有满足满足特定条件下才会实例化 Bean。

以下为常见的 @Conditional 派生注解：

条件注解 说明
@ConditionalOnBean 要求上下文中存在特定的 Bean 实例
@ConditionalOnClass 要求类路径中存在指定的类文件
@ConditionalOnExpression 要求满足表达式要求
@ConditionalOnMissingBean 要求上下文中不存在特定的 Bean 实例
@ConditionalOnMissingClass 要求类路径中不存在指定的class文件
@ConditionalOnNotWebApplication 要求当前应用所在的环境不是 Web 环境（Servlet、WebFlux）
@ConditionalOnProperty 要求环境中有指定的配置

SpringBoot配置的加载顺序

首先，优先级最高的是启动时传入参数，其次是环境变量，而最低优先级别的其实是配置文件。

加载配置文件的工作由org.springframework.boot.context.config.ConfigFileApplicationListener监听器完成（SpringApplication创建后，run之前），其加载顺序如下：

  1. 应用同级目录 config 下的配置文件
  2. 应用同级目录下的配置文件
  3. 类路径中 config 下的配置文件
  4. 类路径下的配置文件(即 reources 下的配置文件)

关于SpringBoot如何弃用默认技术或自动配置

有些时候我们希望使用自己熟悉的技术，譬如 SpringBoot 中默认数据库连接池是 HikariCP，现在我们打算使用Druid，那么只需要将 HikariCP 在 pom 中的依赖排除再将 Druid 的 starter 导入即可。

Druid GitHub：https://github.com/alibaba/druid

详细配置：https://github.com/alibaba/druid/tree/master/druid-spring-boot-starter

除此之外，有时候我们不希望 SpringBoot 对某些功能进行自动配置。譬如多数据源这种常见下，除了使用 @Primary 进行优先级注入之外，还可以直接排除其自动配置改为手动配置。实现方法如下：

  1. 首先将配置排除

  1. 在配置文件中添加自定义的数据源连接配置项

  2. 加载配置项到环境中，再用其进行数据源和事务管理器的配置
