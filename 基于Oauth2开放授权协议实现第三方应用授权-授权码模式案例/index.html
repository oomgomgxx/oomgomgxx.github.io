OAuth2是什么？

  • OAuth2.0是OAuth1.0（Open Authorization，开放授权）协议的延续版本，但不向后兼容OAuth1.0，而OAuth是一个受权的开放标准，即公开的标准，是一个授权协议;
  • 在LETF-rfc6749中的标题为The OAuth 2.0 Authorization Framework，可以看出OAuth2.0是一个授权框架并且能够完成“安全委托访问”，通俗点说就是你可以授权b网站去访问你存储在a网站上的一些信息，而不会泄露除了这个“一些信息”之外的内容；
  • Oauth2的四种授权类型
    • 授权码模式
    • 简单模式
    • 密码模式
    • 客户端模式

网上关于Oauth2协议介绍的文章有很多，这里不做过多的叙述，深入了解推荐以下文章和书籍：

  • 阮一峰的《理解OAuth 2.0》,直接百度名称就有了，这文章是对LETF-rfc6749的白话简述
  • 书籍《OAuth2实战》对OAuth2作出了详细的介绍

关于jwt的那些事

​ jwt的出现无疑是为了解决session占用服务器内存和节点间session需要同步的问题，因为日益增长的用户、成熟的技术和应用架构对于session而言已经慢慢成为了累赘; 如在微服务架构中就提倡节点无状态，其实很容易理解，因为对于灵活的节点扩展而言，如果节点存在状态的话那么在拓展节点之后就会可能会出现问题，所以本地缓存、session等让服务器节点产生状态的操作都是尽量避免的;

​ 如果使用jwt的话，就意味着服务器并不对会话信息进行维护而是让客户端来维护，这是一个职责的转移，所以jwt的相关内容应该是尽量避免持久化的，服务器的职责只是对jwt进行验证，至于jwt是谁发过来的不管，只要jwt合法服务器就理所当然接受请求处理请求，所以如何保证jwt的使用安全是需要客户端合理设计的，如可以使用https协议对通信进行加密处理;为什么突然提到这个是因为看到网上一些文章在讨论这问题，这是我对与jwt的理解，所以如果使用jwt如果不是没办法的话，最合理的做法就是服务器只对jwt做验证操作;

​ 还需要注意的是，jwt只是token的一种，jwt是不应该持久化的，而bearer令牌需要持久化，因为bearer不像jwt那样能够针对结构化内容校验，bearer只是一串字符;

以下注释位于SpringSecurityOauth2，jwt并不进行持久化

如何验证jwt呢？以下是auth0-jwt的验证原理

  1. 将”头部”和“负载”decode，返回一个DecodeJWT
  2. 验证”头部”的加密算法是否和服务器一直，不一直接报错
  3. 使用生产jwt时的密钥和算法对DecodeJWT进行encode对比校验
  4. 校验“负载”中的声明信息，如判断其是否过期

关于自动配置

经过上面对ouath2的简单使用过后，不难发现，对于授权服务器和资源服务器的基本原理和使用是差不多的;

从用例中引出如下问题：

  1. Spring Security OAuth2无疑是基于SpringSecurity的，那么它是如何建立关系的
  2. 如何维护token：TokenStore和TokenConverter
  3. 如何维护客户端注册信息（详细信息）：ClientDetailsService
  4. 如何处理oauth2端点请求

带着以上的疑问进入spring-security-oauth2-autoconfigure包的自动配置类OAuth2AutoConfiguration，以下为主要部分：

OAuth2AuthorizationServerConfiguration：授权服务器

OAuth2ResourceServerConfiguration：资源服务器

OAuth2MethodSecurityConfiguration：对授保护方法增强处理，如Oauth2表达式处理

OAuth2RestOperationsConfiguration：根据配置文件进行client相关的条件配置
