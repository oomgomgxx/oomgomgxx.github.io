Spring MVC 分派器原理

父子容器是如何组装的？

在上面我们通过继承 AbstractAnnotationConfigDispatcherServletInitializer 抽象类成功配置了父子容器。所以我们可以进入其源码了解一下它是如何配置容器的

public abstract class AbstractAnnotationConfigDispatcherServletInitializer
        extends AbstractDispatcherServletInitializer {

    // 创建父容器
    // 该方法返回的容器回用于创建ContextLoaderListener监听器
    @Override
    @Nullable
    protected WebApplicationContext createRootApplicationContext() {
        Class<?>[] configClasses = getRootConfigClasses();
        // 如果提供了父容器配置类，则创建父容器
        if (!ObjectUtils.isEmpty(configClasses)) {
            AnnotationConfigWebApplicationContext context = new AnnotationConfigWebApplicationContext();
            context.register(configClasses); 
            return context;
        }
        else {
            // 作为一个Web项目是可以没有父容器的
            // 即此时Web项目只有一个DispatchServlet
            return null; 
        }
    }

    // 创建子容器
    // 用于提供给DispatchServlet使用
    @Override
    protected WebApplicationContext createServletApplicationContext() {
        AnnotationConfigWebApplicationContext context = new AnnotationConfigWebApplicationContext();
        Class<?>[] configClasses = getServletConfigClasses();
        // 如果提供了子容器配置类，则创建子容器
        if (!ObjectUtils.isEmpty(configClasses)) {
            context.register(configClasses);
        }
        return context; 
    }


    // 父子容器模板，需要实现类来完成

    @Nullable
    protected abstract Class<?>[] getRootConfigClasses();
    @Nullable
    protected abstract Class<?>[] getServletConfigClasses();

}

不难看出 AbstractAnnotationConfigDispatcherServletInitializer 抽象类其实只是一个模板，用于创建父子容器对应的 WebApplicationContext。

当 DispatchServlet 被创建时，子容器 WebApplicationContext 就会被传入以协助 init() 的初始化工作，而父容器就是在初始化 DispatchServlet 时被注入到子容器的（Servlet的init方法由Tomcat的生命周期操作回调）

DispatchServlet#initWebApplicationContext

protected WebApplicationContext initWebApplicationContext() {
    // 从ServletContext中提取出父容器
    WebApplicationContext rootContext =
        WebApplicationContextUtils.getWebApplicationContext(getServletContext());
    WebApplicationContext wac = null;

    if (this.webApplicationContext != null) {
        // 当前DispatchServelt所在的子容器（在创建时被传入）
        wac = this.webApplicationContext;
        if (wac instanceof ConfigurableWebApplicationContext) {
            ConfigurableWebApplicationContext cwac = (ConfigurableWebApplicationContext) wac;
            if (!cwac.isActive()) {
                if (cwac.getParent() == null) {
                    // 注入父容器
                    cwac.setParent(rootContext);
                }
                configureAndRefreshWebApplicationContext(cwac);
            }
        }
    }
    // ...
    return wac;
}

分派器剖析

回顾 Servlet 规范

我们知道 DispatchServlet 的本质就是 Servlet，所以在这先简单回顾一下 Servlet 标准

Servlet

public interface Servlet {
    // 初始化
    public void init(ServletConfig config) throws ServletException;
    // 获取当前Servlet的配置
    public ServletConfig getServletConfig();
    // 请求处理
    public void service(ServletRequest req, ServletResponse res);
    // 获取当前Servlet信息
    public String getServletInfo();
    // 销毁
    public void destroy();
}

ServletConfig

 public interface ServletConfig {
    // 当前Servlet的名字
    public String getServletName();
    // 当前Servlet的上限文对象
    // ServletContext是Servlet的三大域之一，除此之外还有request、session
    public ServletContext getServletContext();
    // 获取指定的初始化参数值
    public String getInitParameter(String name);
    // 获取所有初始化参数值
    public Enumeration<String> getInitParameterNames();
}

但一般我们很少直接使用 Servlet 接口，因为需要实现所有的抽象方法，所以后来就有了 GenericServlet 抽象类。GenericServlet 只是简单复写了 Servlet 接口，让其子类无需实现自己不需要的方法，即所谓的接口隔离原则。而除了 GenericServlet 之外，Servlet API 包还提供了 HttpServlet 抽象类（继承了 GenericServlet），在 HttpServlet 中它将原来的 service 方法细分成了以下这几种操作

private static final String METHOD_DELETE = "DELETE";
private static final String METHOD_HEAD = "HEAD";
private static final String METHOD_GET = "GET";
private static final String METHOD_OPTIONS = "OPTIONS";
private static final String METHOD_POST = "POST";
private static final String METHOD_PUT = "PUT";
private static final String METHOD_TRACE = "TRACE";

而且 HttpServlet 的细分操作也很简单

@Override
public void service(ServletRequest req, ServletResponse res)
    throws ServletException, IOException
{
    HttpServletRequest  request;
    HttpServletResponse response;

    if (!(req instanceof HttpServletRequest &&
          res instanceof HttpServletResponse)) {
        throw new ServletException("non-HTTP request or response");
    }
    // 先将原来的request、response转为HTTP实现
    request = (HttpServletRequest) req;
    response = (HttpServletResponse) res;

    // 进行细分操作
    service(request, response);
}

细分操作如下

protected void service(HttpServletRequest req, HttpServletResponse resp)
    throws ServletException, IOException
{
    String method = req.getMethod();

    if (method.equals(METHOD_GET)) {
        long lastModified = getLastModified(req);
        if (lastModified == -1) { // 不支持 If-Modified-Since，直接响应最新数据
            doGet(req, resp);
        } else {
            long ifModifiedSince = req.getDateHeader(HEADER_IFMODSINCE);
            if (ifModifiedSince < lastModified) { // 请求的资源被修改过，响应最新数据
                maybeSetLastModified(resp, lastModified);
                doGet(req, resp);
            } else { // 请求的资源没被修改过，无需放回，即让浏览器使用缓存
                resp.setStatus(HttpServletResponse.SC_NOT_MODIFIED);
            }
        }

    } else if (method.equals(METHOD_HEAD)) {
        // 请求资源在服务器中最后一次被修改的时间
        long lastModified = getLastModified(req);
        maybeSetLastModified(resp, lastModified);
        doHead(req, resp);
    } else if (method.equals(METHOD_POST)) {
        doPost(req, resp);
    } else if (method.equals(METHOD_PUT)) {
        doPut(req, resp);
    } else if (method.equals(METHOD_DELETE)) {
        doDelete(req, resp);
    } else if (method.equals(METHOD_OPTIONS)) {
        doOptions(req,resp);
    } else if (method.equals(METHOD_TRACE)) {
        doTrace(req,resp);
    } else {
        String errMsg = lStrings.getString("http.method_not_implemented");
        Object[] errArgs = new Object[1];
        errArgs[0] = method;
        errMsg = MessageFormat.format(errMsg, errArgs);
        // 无法处理返回错误信息
        resp.sendError(HttpServletResponse.SC_NOT_IMPLEMENTED, errMsg);
    }
}

而 FrameworkServlet 又继承了 HttpServletBean，如下

至此可以看到，Spring MVC 和 Servlet 的接入点就在 HttpServletBean 抽象类里，所以我们大可先看看它有没有复写父类的方法

可以看到它并没有复写诸如 service、doXXX 这些方法，只是添加了一些获取环境变量的操作而已，所以这时我们可以回看它的实现类 FrameworkServlet （它只一个直属实现）

最后在 FrameworkServlet 中我们发现，它实现了 serivce 方法，这时我们可以断定这就是关键所在

protected void service(HttpServletRequest request, HttpServletResponse response)
    throws ServletException, IOException {

    HttpMethod httpMethod = HttpMethod.resolve(request.getMethod());
    // PATCH请求相当于PUT，但PATCH是非幂等的，即它提供的不是完整的资源
    if (httpMethod == HttpMethod.PATCH || httpMethod == null) {
        processRequest(request, response);
    }
    else {
        // 其他请求由父类处理
        // 在这我们知道，当前FrameworkServlet的父类是HttpServletBean
        // 但HttpServletBean并没有实现service，因此这里的service是位于HttpServlet
        // 而HttpServlet的serivce只作分发工作（按HTTP请求方法分派）
        super.service(request, response);
    }
}

@Override
protected void doService(HttpServletRequest request, HttpServletResponse response) throws Exception {
    logRequest(request);

    // 保存request域的属性快照
    Map<String, Object> attributesSnapshot = null;
    if (WebUtils.isIncludeRequest(request)) {
        attributesSnapshot = new HashMap<>();
        Enumeration<?> attrNames = request.getAttributeNames();
        while (attrNames.hasMoreElements()) {
            String attrName = (String) attrNames.nextElement();
            if (this.cleanupAfterInclude || attrName.startsWith(DEFAULT_STRATEGIES_PREFIX)) {
                attributesSnapshot.put(attrName, request.getAttribute(attrName));
            }
        }
    }

    // 将当前环境的相关引用保存到request域
    // 1. 应用上下文容器
    request.setAttribute(WEB_APPLICATION_CONTEXT_ATTRIBUTE, getWebApplicationContext());
    // 2. 本地解析器（国际化相关）
    request.setAttribute(LOCALE_RESOLVER_ATTRIBUTE, this.localeResolver);
    // 3. 主题解析器（支持页面主题样式，需要配合spring提供的taglib）
    // <%@ taglib prefix="spring" uri="http://www.springframework.org/tags"%>
    request.setAttribute(THEME_RESOLVER_ATTRIBUTE, this.themeResolver);
    request.setAttribute(THEME_SOURCE_ATTRIBUTE, getThemeSource());

    //...

    RequestPath requestPath = null;
    if (this.parseRequestPath && !ServletRequestPathUtils.hasParsedRequestPath(request)) {
        requestPath = ServletRequestPathUtils.parseAndCache(request);
    }

    try {
        // 将请求分派下去处理
        doDispatch(request, response);
    }
    finally {
        // 处理异步结果
        // 在Servlet3开始，Servlet支持异步请求处理
        // request.getAsyncContext().start(Runnable)
        if (!WebAsyncUtils.getAsyncManager(request).isConcurrentHandlingStarted()) {
            // Restore the original attribute snapshot, in case of an include.
            if (attributesSnapshot != null) {
                restoreAttributesAfterInclude(request, attributesSnapshot);
            }
        }
        if (requestPath != null) {
            ServletRequestPathUtils.clearParsedRequestPath(request);
        }
    }
}

DispatchServlet#doDispatch ，分派器的核心代码

protected void doDispatch(HttpServletRequest request, HttpServletResponse response) throws Exception {

    HttpServletRequest processedRequest = request;
    // 在HandlerExecutionChain中包含了“处理器”和“拦截器链”
    HandlerExecutionChain mappedHandler = null;
    boolean multipartRequestParsed = false;

    WebAsyncManager asyncManager = WebAsyncUtils.getAsyncManager(request);

    try {
        // 从处理器返回的ModelAndView（数据和视图信息）
        ModelAndView mv = null;
        Exception dispatchException = null;

        try {
            // 检查是否是文件上传操作（譬如表单的enctype="multipart/form-data"）
            processedRequest = checkMultipart(request);
            multipartRequestParsed = (processedRequest != request);

            // 一、通过映射器（HandlerMapping）匹配并返回具体的处理器实现和拦截器链
            // 以Spring 5.3.1为例有以下3个内置的处理器映射器
            // 1. BeanNameUrlHandlerMapping: 让url和bean名字进行映射匹配，常用于Controller接口实现方式的处理器定义
            // 2. RequestMappingHandlerMapping：让url和@RequestMapping进行映射匹配（包含其余的Restful风格注解，其实本质只是@RequestMapping的派生注解）
            // 3. RouterFunctionMapping：让url和RouterFunction映射（Spring5.2新特性，用于简化Controller编写）
            mappedHandler = getHandler(processedRequest);
            if (mappedHandler == null) {
                noHandlerFound(processedRequest, response);
                return;
            }

            // 二、根据处理器返回对应的适配器
            // 以Spring 5.3.1为例有以下4个内置的处理器适配器
            // 1. HandlerFunctionAdapter：处理器通过RouterFunctions实现
            // 2. HttpRequestHandlerAdapter：处理器通过@RequestMapping实现
            // 3. SimpleControllerHandlerAdapter：处理器通过Controller接口实现
            // 4. SimpleServletHandlerAdapter：处理器通过Servlet接口实现
            HandlerAdapter ha = getHandlerAdapter(mappedHandler.getHandler());

            // 检查是否有必要真的执行
            String method = request.getMethod();
            boolean isGet = "GET".equals(method);
            if (isGet || "HEAD".equals(method)) {
                long lastModified = ha.getLastModified(request, mappedHandler.getHandler());
                if (new ServletWebRequest(request, response).checkNotModified(lastModified) && isGet) {
                    return;
                }
            }

            // 三、执行拦截器链的preHandle方法
            // 这时可以对HttpServletRequest和HttpServletResponse修改
            if (!mappedHandler.applyPreHandle(processedRequest, response)) {
                return;
            }

            // 四、通过适配器执行处理器
            mv = ha.handle(processedRequest, response, mappedHandler.getHandler());

            if (asyncManager.isConcurrentHandlingStarted()) {
                return;
            }
            // 如果没有指定视图名称则返回默认的
            applyDefaultViewName(processedRequest, mv);

            // 五、执行拦截器链postHandle方法
            mappedHandler.applyPostHandle(processedRequest, response, mv);
        }
         //...

        // 六、执行响应操作
        // 1. 根据视图解析器解析出视图View，然后对数据进行渲染
        // 2. 执行拦截器链postHandle方法afterCompletion
        processDispatchResult(processedRequest, response, mappedHandler, mv, dispatchException);
    }
    //...
    finally {
        if (asyncManager.isConcurrentHandlingStarted()) {
            // Instead of postHandle and afterCompletion
            if (mappedHandler != null) {   
                mappedHandler.applyAfterConcurrentHandlingStarted(processedRequest, response);
            }
        }
        else {
            // Clean up any resources used by a multipart request.
            if (multipartRequestParsed) {
                cleanupMultipart(processedRequest);
            }
        }
    }
}

流程图

装配过程

  1. WebApplicationContext 上下文容器启动
  2. 实例化 Singleton 域的 Bean（自定义的 WebContainerConfig，其实现了WebMvcConfigurer），顺序规则如下
    1. 先实例化配置类
    2. 定义类配置类上的注解，譬如@EnableWebMvc、@Import之类
    3. 按照配置类的定义从上到下实
    4. 按需实例化。譬如优先级较低的bean被以上bean依赖了，就会提前实例化
  3. 因为 WebContainerConfig 被 @EnableWebMvc 标注，所以在 WebContainerConfig 实例化后会马上实例化DelegatingWebMvcConfiguration
  4. 继续回到 WebContainerConfig，按定义顺序和需要继续实例化 Bean
  5. 如果 WebContainerConfig 复写了 WebMvcConfigurer 的方法，则必要时就会实例化 WebMvcConfigurationSupport 中对应的Bean，因为它是 DelegatingWebMvcConfiguration 的父类。除此之外还会触发 DelegatingWebMvcConfiguration 的委派操作，这时如果环境中有多个 WebMvcConfigurer 实现，则只要其中一个 WebMvcConfigurer 的方法被执行，其他所有 WebMvcConfigurer 对应的方法都会被执行。

private List<HandlerMethodArgumentResolver> getDefaultArgumentResolvers() {
    List<HandlerMethodArgumentResolver> resolvers = new ArrayList<>(30);

    // Annotation-based argument resolution

    // 支持@RequestParam和MultipartFile解析
    resolvers.add(new RequestParamMethodArgumentResolver(getBeanFactory(), false));
    resolvers.add(new RequestParamMapMethodArgumentResolver());
    // 支持@PathVariable解析
    resolvers.add(new PathVariableMethodArgumentResolver());
    resolvers.add(new PathVariableMapMethodArgumentResolver());
    // 支持@MatrixVariable解析
    resolvers.add(new MatrixVariableMethodArgumentResolver());
    resolvers.add(new MatrixVariableMapMethodArgumentResolver());
    // 支持@ModelAttribute解析
    resolvers.add(new ServletModelAttributeMethodProcessor(false));
    // 支持@RequestBody和@ResponseBody解析
    resolvers.add(new RequestResponseBodyMethodProcessor(getMessageConverters(), this.requestResponseBodyAdvice));
    // 支持@RequestPart解析
    resolvers.add(new RequestPartMethodArgumentResolver(getMessageConverters(), this.requestResponseBodyAdvice));
    // 支持@RequestHeader解析
    resolvers.add(new RequestHeaderMethodArgumentResolver(getBeanFactory()));
    resolvers.add(new RequestHeaderMapMethodArgumentResolver());
    // 支持@CookieValue解析
    resolvers.add(new ServletCookieValueMethodArgumentResolver(getBeanFactory()));
    // 支持@Value解析
    resolvers.add(new ExpressionValueMethodArgumentResolver(getBeanFactory()));
    // 支持@SessionAttribute解析
    resolvers.add(new SessionAttributeMethodArgumentResolver());
    // 支持@RequestAttribute解析
    resolvers.add(new RequestAttributeMethodArgumentResolver());



    // Type-based argument resolution

    // 支持一下类型解析
    // WebRequest、ServletRequest、MultipartRequest、HttpSession
    // Principal、InputStream、Reader、HttpMethod、Locale、TimeZone、ZoneId
    resolvers.add(new ServletRequestMethodArgumentResolver());
    // 支持 ServletResponse、OutputStream、Writer 解析
    resolvers.add(new ServletResponseMethodArgumentResolver());
    // 支持 HttpEntity、RequestEntity 解析
    resolvers.add(new HttpEntityMethodProcessor(getMessageConverters(), this.requestResponseBodyAdvice));
    // 支持 RedirectAttributes 解析
    resolvers.add(new RedirectAttributesMethodArgumentResolver());
    // 支持 Model 解析
    resolvers.add(new ModelMethodProcessor());
    // 支持 Map 解析
    resolvers.add(new MapMethodProcessor());
    // 支持 Errors 解析
    resolvers.add(new ErrorsMethodArgumentResolver());
    // 支持 SessionStatus 解析
    resolvers.add(new SessionStatusMethodArgumentResolver());
    // 支持 UriComponentsBuilder 解析
    resolvers.add(new UriComponentsBuilderMethodArgumentResolver());
    if (KotlinDetector.isKotlinPresent()) {
        resolvers.add(new ContinuationHandlerMethodArgumentResolver());
    }

    // 自定义解析器
    // 在 WebMvcConfigurationSupport#requestMappingHandlerAdapter 中调用
    // 添加操作可通过实现WebMvcConfigurer接口的addArgumentResolvers方法实现
    if (getCustomArgumentResolvers() != null) {
        resolvers.addAll(getCustomArgumentResolvers());
    }

    // Catch-all
    resolvers.add(new RequestParamMethodArgumentResolver(getBeanFactory(), true));
    resolvers.add(new ServletModelAttributeMethodProcessor(true));

    return resolvers;
}

Listener

基本介绍

  观察者是一种用于构建一对多关系模型的行为型设计模式，作用是当“一”方发生状态改变时，“多”方就会感知到并自动执行处理逻辑，所以 Listener 的主要作用其实是为了监听某些特定对象，而这些对象常见有如下这些：

  • ServletContextListener：监听 ServletContext 的初始化和销毁事件
  • ServletContextAttributeListener：监听 ServletContext 属性的更新事件（包含增删改）
  • ServletRequestListener：监听 ServletRequestListener 的初始化和销毁事件
  • ServletRequestAttributeListener：监听 ServletRequestAttributeListener 属性的更新事件（包含增删改）
  • ReadListener：监听输入流状态事件（已读、异常、可能在读 javax.servlet.ServletInputStream#isReady() 被调用时）
  • WriteListener：监听输出流状态事件（可能在写，在 javax.servlet.ServletOutputStream#isReady() 被调用时）

Filter

基本介绍

  过滤器顾名思义就是用来筛选信息的，开发者可以通过实现 Filter 来对 Request、Response 进行处理。过滤器通常不会单个存在，所以它时常会伴随着责任链模式出现。即多个过滤器按顺序形成一条过滤链，而每个过滤器都有各自的责任，所以这条链就叫责任链。责任链模式非常常见，除了当前 Filter 外，上面提到的管道模式其实也是一种责任链，除此之外后面将要聊到的 Interceptor、AOP 都用到了责任链。

快速入门责任链模式

Filter 接口

public interface Filter {
    // 初始化
    public void init(FilterConfig filterConfig) throws ServletException;
    // 执行拦截
    public void doFilter(ServletRequest request, ServletResponse response,
                         FilterChain chain)
            throws IOException, ServletException;
    // 销毁
    public void destroy();
}
  • 可以看到，在 doFilter 的参数列表中就有一个 FilterChain 类型的变量，而这个类就是责任链模式实现

FilterChain 接口

public interface FilterChain {
    // 执行拦截
    public void doFilter ( ServletRequest request, ServletResponse response ) throws IOException, ServletException;
}
  • 该接口看上去很简单，只有一个 doFilter 方法
  • 注意，到这里我们发现了件有趣的事，不知道你有没有感觉 FilterChain#doFilter 方法看上去更加适合 Filter，所以我们在这可以发挥一下想象力，就是认为责任链模式的实现技巧就是将 Chain（“链”）添加到自己“原本”的方法中，而“原本”的方法就转移到 Chain（“链”） 中
  • 在 Tomcat 中 FilterCain 的实现只有一个 ApplicationFilterChain

以下为去除了多余操作的 ApplicationFilterChain 源码

public final class ApplicationFilterChain implements FilterChain {

    //...

    // 拦截器包装类（责任链）
    private ApplicationFilterConfig[] filters = new ApplicationFilterConfig[0];

    private int pos = 0;
    private int n = 0;

    // 往责任链添加Filter
    void addFilter(ApplicationFilterConfig filterConfig) {
        //...

        filters[n++] = filterConfig;
    }

    // 执行完责任链后要接着执行的Servlet
    private Servlet servlet = null;
    void setServlet(Servlet servlet) {
        this.servlet = servlet;
    }

    // 执行责任链
    @Override
    public void doFilter(ServletRequest request, ServletResponse response)
        throws IOException, ServletException {
        if( Globals.IS_SECURITY_ENABLED ) {
            //...
        } else {
            internalDoFilter(request,response);
        }
    }

    private void internalDoFilter(ServletRequest request,
                                  ServletResponse response)
        throws IOException, ServletException {

        // 执行责任链
        if (pos < n) {
            ApplicationFilterConfig filterConfig = filters[pos++];
            try {
                Filter filter = filterConfig.getFilter();
                //...
                if( Globals.IS_SECURITY_ENABLED ) {
                    //...
                } else {

                    // 执行Filter
                    filter.doFilter(request, response, this);
                }
            } catch (IOException | ServletException | RuntimeException e) {
                //...
            }
            return;
        }

        // 执行完责任链了，开始执行Servlet（譬如DispatchServlet）
        try {
            //...
            if (/**/) {
                //...
            } else {
                // 执行Servlet
                servlet.service(request, response);
            }
        } catch (IOException | ServletException | RuntimeException e) {
            //...
        } finally {
            //...
        }
    }

    // 释放责任链
    void release() {
        for (int i = 0; i < n; i++) {
            filters[i] = null;
        }
        n = 0;
        pos = 0;
        servlet = null;
        servletSupportsAsync = false;
    }

    void reuse() {
        pos = 0;
    }
}

Interceptor

基本介绍

  在 Spring Web 模块中提供了两种类型的拦截器，分别是 HandlerInterceptor、WebRequestInterceptor。在处理器被执行前后 Spring 会对其进行适配调用，而两者的区别是 WebRequestInterceptor 只能获取WebRequest 、ModelMap 进行修改而不能阻断请求，而 HandlerInterceptor 更强大，更像 Filter 。

拦截器责任链

public class HandlerExecutionChain {

    private final Object handler;

    private final List<HandlerInterceptor> interceptorList = new ArrayList<>();

    private int interceptorIndex = -1;

    boolean applyPreHandle(HttpServletRequest request, HttpServletResponse response) throws Exception {

        for (int i = 0; i < this.interceptorList.size(); i++) {
            HandlerInterceptor interceptor = this.interceptorList.get(i);
            // 将 request, response, handler 让所有拦截器进行“预”处理（譬如预统一处理或阻截操作）
            if (!interceptor.preHandle(request, response, this.handler)) {
                triggerAfterCompletion(request, response, null);
                return false;
            }
            this.interceptorIndex = i;
        }
        return true;
    }

    void applyPostHandle(HttpServletRequest request, HttpServletResponse response, @Nullable ModelAndView mv)
            throws Exception {

        for (int i = this.interceptorList.size() - 1; i >= 0; i--) {
            HandlerInterceptor interceptor = this.interceptorList.get(i);
            // 将 request, response, handler 让所有拦截器进行“后”处理（譬如结果统一处理）
            interceptor.postHandle(request, response, this.handler, mv);
        }
    }

    void triggerAfterCompletion(HttpServletRequest request, HttpServletResponse response, @Nullable Exception ex) {

        for (int i = this.interceptorIndex; i >= 0; i--) {
            HandlerInterceptor interceptor = this.interceptorList.get(i);
            // 将 request, response, handler 让所有拦截器进行“渲染后”处理（譬如释放资源、记录处理后日志）
            try {
                interceptor.afterCompletion(request, response, this.handler, ex);
            }
            catch (Throwable ex2) {}
        }
    }    
    //...
}
  • HandlerExecutionChain 可以简单理解为是对请求以及处理数据的处理链
  • 责任链的处理器可以是严格有序的，但也可以是互不相干的

AOP（ Aspect Oriented Programming ）

AOP 术语

  • 关注点代码
    • 在多个方法中出现，且与业务关系不大的代码
  • 通知（Advice/Interceptor）
    • 其实就是关注点代码的别名
  • 切入点
    • 需要添加关注点代码的方法
    • 在Spring AOP 中切入点单位只能是方法
  • 切入点表达式
    • 定义切入点的表达式
  • 织入
    • 指将通知添加到切入点的这个过程

基本介绍

  AOP 是一种编程思想，它旨在将重复出现在多个方法中的代码抽离并单独存放，然后在需要时将其织入到方法中。它是一种降低代码耦合性和无侵入扩展的实现方案，而且 AOP 的思想和前面提到的拦截器（Interceptor）很像，你可以简单理解只要是拦截器能做到的 AOP 都能够做到，而且能做得更好。

  除此之外，你还可以使用 AOP 实现诸如日志记录、权限管理、代码埋点方式的信息提取、性能监控、事务管理、缓存管理、统一异常管理 等等。

在 Java 中 AOP 织入时机可以分为以下这几种，及其代表技术：

  • 编译时织入：AspectJ
  • 类加载时织入： AspectJ 5
  • 运行时织入：Spring AOP、Java Agent

AOP 在 Spring 的常见应用案例：

  • 事务管理
  • 缓存管理
  • 统一异常处理

Spring AOP 支持的几种配置方式：

  • 基于 ProxyFactoryBean
  • XML 的 AOP 标签（xmlns:aop=”http://www.springframework.org/schema/aop"）
  • @Aspect 注解（推荐）

Spring AOP 原理以及执行时机

  虽然关于 Spring AOP 的原理已经在以前的文章中详细地聊过了，但为了当前文章更加饱满，以及践行温故而知新的道理，所以接下来还是打算再次看看 Spring AOP 是执行原理。但再次之前需要多说一句，这部分内容的前置知识点是 Bean 的生命周期过程，所以应该要有先后次序地学习。

  Spring 会在 Bean 生命周期的初始化阶段（initializeBean）回调默认及其依赖的 BeanPostProcessor ，在回调后置处理方法期间会通过一个叫 AnnotationAwareAspectJAutoProxyCreator 的 BeanPostProcessor 来判断当前 Bean 是否有必要生成代理，而其判断的主要依据如下

这是因为先前我们定义了切入点表达式锁定了 TestController 的所有方法的原因

@Pointcut("execution(* com.td.controller.TestController.*(..))")
public void pointcut() {}

因此，在 AbstractAdvisorAutoProxyCreator#getAdvicesAndAdvisorsForBean 时就会通过切入点表达式返回匹配的通知。

但现在的问题是，Spring 是如何知道有哪些通知的？其过程大致如下：

  1. 在 Spring 容器启动时，检测到配置类上有 @EnableAspectJAutoProxy 注解，因此就会在上下文中注册一个 AnnotationAwareAspectJAutoProxyCreator 的 BeanPostProcessor 用于判断每一个 Bean 是否有必要生成切面代理实例
  2. 经过步骤 1 之后，当有 Bean 触发 AnnotationAwareAspectJAutoProxyCreator 的后处理方法 postProcessBeforeInstantiation 回调时，就会判断哪些是切面类。具体逻辑位于 BeanFactoryAspectJAdvisorsBuilder#buildAspectJAdvisors 方法中
  3. 是否为切面类的判断步骤如下：
    1. 先通过 BeanFactory 获取到所以 Bean 的 BeanName
    2. 循环地根据 BeanName 获取到与其对应的 Bean Type
    3. 最后通过 AspectJAdvisorFactory#isAspect 方法判断其类是否被 @Aspect 注解
  4. 判断出切面类之后，会通过 AspectJAdvisorFactory#getAdvisors 返回内中的所有通知方法的定义

至此，所有的通知都被找了出来，它们会在判断是否需要为 Bean 生成切面代理实例或代理目标方法被执行时被应用。当代理实例生成后，它会像其他 Bean 一样被添加到容器中以提供给使用者使用。

  • 之所以一直称通知为拦截器，是因为通知其实际是 Advice ，而在 AOP 中 Interceptor 就继承自 Advice 。而它比较常用的一个实现就是进行事务管理的 TransactionInterceptor
