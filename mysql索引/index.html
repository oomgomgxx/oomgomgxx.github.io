什么是索引？

  1. 使用目录来去理解索引是最好的方法
  2. 不同类型的索引实现方式不同
  3. 就算是相同类型的索引在不同的存储引擎中实现方式也可能有所不同

什么情况需要添加索引？

  1. 当表的数据量越来越大时，有人建议超过300条记录就该加上索引了
  2. 频繁被用作条件查询、表连接条件、分组、排序的字段
  3. 索引性高的字段，通俗点说就是重复性小的字段，最好是唯一字段

索引是不是越多越好和那些字段不适合用索引？

  1. 答案是否定的。原因是索引是独立的数据结构，需要花费额外的时间和磁盘空间来维护
  2. 频繁进行修改的字段不适合做索引，原因同上
  3. 值较大的字段不适合作为索引，上面已经提到索引是需要额外的磁盘空间存放的，所以如果字段值大且如果表的记录较多的时，索引所占的磁盘空间就会急速增加
  4. 离散度低的字段不适合

索引类型

常见索引数据结构

散列表

图片源自网络

  • 散列表数据结构由 数组 和 链表 组成，通过对关键字段 key 进行哈希计算后再用其计算出数组的下标值以确立存储位置。
  • 散列表最理想的查询时间复杂度是O(1)，即没有任何的哈希冲突，且表示存储的数据离散度非常高（如身份证号码），此时散列表优势明显。但如果存储时出现哈希冲突，就需要使用拉链法来形成链表解决冲突，而因为链表查找需要一个个遍历，因此这时查询复杂度将退化为O（n）。
  • 使用散列表做为索引数据结构时需要注意其以下特性：
    • 散列表并不支持多列（联合）索引的最佳左前缀原则，因为在计算数组下标时会将多个索引列一同计算以达到尽可能离散
    • 散列表因为采用哈希算法来确立数组下标，因此是没有顺序可言的，所以散列表并不支持区间操作和排序
    • 散列表只适合简单的等值操作，而其他操作并不能发挥散列表的特性，因此在使用前应考虑场合

有序数组

图片源自网络

  • 有序数组的特点是内存连续，所以在等值查询和区间查询上效率是非常高的
  • 但需要意识到的是，有序数组因为要求内存连续的特性，所以在进行增删操作时需要挪动后续的数据以保持连续性，因此有序数组会随着数据量的增大，效率会逐渐降低
  • 因此，从上可以得出结论就是，有序数组比较适用于读多写少的场景

二叉查找树、平衡二叉查找树

图片源自网络

  • 二叉查找树的特点是左边子节点小于右边子节点，因此除了有序之外还可以使用二分查找的高效检索方法

  • 虽然二叉查找树从综合性来看效率不错，但是也存在一些隐患，例如极端情况下如果数据呈递增表现，那么就会出现数据结构退化的现象，即二叉树退化为链表，从时间复杂度O（logn）退化成O（n），因此会影响检索数据的效率

  • 对于二叉树数据结构退化这种现象可以通过使用平衡版二叉查找来解决。所谓的平衡二叉树指的是根节点到叶节点的路径长度一样（绝对平衡）。例如插入数据时，平衡二叉树判断到路径长度不一致就会触发节点自旋操作以恢复到平衡以确保数据查询的时间复杂度。但平衡二叉树并不是十全十美的，因为如果发生大量修改节点操作的话就会出大量的自旋操作导致性能下降

  • 以个人目前的观察，其实直接使用二叉树数和绝对平衡二叉树来做存储结果的系统甚少，用的较多的都是红黑树，因为红黑树首先它是平衡二叉查找树，但并非绝对平衡，因为红黑树只要求左右子节点的黑色节点数量相等即可，所以光看这个特性就可以得出结论，红黑树可以降低绝对平衡二叉树的大自旋操作和普通二叉树可能会退化成链表的隐患，因此红黑树其实是 二叉查找树和绝对平衡二叉树的折中选项

B+树

图片源自网络

  • B+树的B是balance的意思，即B+Tree也是平衡树，且是棵多叉树，一般树高保持在3-4左右，即通常一次查找只需3-4次IO即可

  • B+树的特点是普通节点不记录数据，而数据全部位于叶子节点上，即所谓的聚簇索引，而且B+树的叶子节点的数据呈有序递增之余每个叶子节点都包含左右两边叶子节点的引用，因此天然支持区间和排序等操作

  • 首先是 B+树 和 普通二叉查找树、平衡二叉树的对比。B+树 因为是平衡树所以不存在数据结退化的现象，而且 B+树 一般高度比较小因此在效率和自旋等方面都要比 普通二叉查找树和平衡二叉树 稳定和高效

  • 还有需要谨记的一件事就是，B+树思想是希望普通节点尽可能记录多些自己子节点的信息。如在MySQL中，MySQL默认的内存页大小是16k，因此如果每个节点都只保存左右子节点的信息的话实在是浪费内存，也可以说1次IO操作查询到的数据就变少了，因此 B+树 普通节点并没有拿来存储当前节点的数据，而是存储了多个子节点的引用信息以减少IO操作来提高操作效率。B+树这种做法其实就是空间换时间，也因此现在的关系型数据库如果不是有特别场景的话都是用的B+树来实现索引的

  • 从上面的图可以看出B树的数据和一般二叉查找数一样是存储在每个节点上的

  • 在先前已经提到过，B+树 是希望节点多存储些子节点的信息以此来降低IO的检索次数来提高效率。那么不禁地有个这样的问题，就是B+树这么好，那么B树还有存在的意义吗？答案是肯定的，以本人了解，在NoSQL的MongoDB中就采用B+树来存储数据，主要的原因是MongoDB是一种文档存储类型的NoSQL，而且操作的都是半结构化数据（数据结构可以变动），因此数据呈聚合型。所以如果数据不存储在节点上的话就需要更多的额外操作来将数据聚合，因此会效率低下，所以B树就成为了B+数之下的首选

小知识：

  • 聚簇索引指数据和索引顺序是一致的，且保存在一起。 聚簇索引并不是索引类型，而是一种数据的存储方式
  • MySQL存储引擎InnoDB的主键就是聚簇索引，而二级索引（也有人叫辅助索引）虽然也是使用B+树实现，但是二级索引的叶子节点中保存的只是主键索引的引用，所以如果使用二级索引去检索记录的步骤是，先根据二级索引查找的主键索引，然后根据主键索引引用进行回表查询

关于索引和约束

  • 当添加primary key时，会自动创建主键索引且同时添加unique和not null两个约束

  • 当添加foreign key时，则处理添加外键约束之外，同时还会创建普通索引（原因是避免死锁）

  • 当添加unique key时，会自动添加唯一约束且同时创建普通索引

拓展小知识：

  • 首先外键是一个跨表关联机制，外键字段的操作表现为，可以为null和可以重复，但是外键字段的值不能超出所关联的表的字段值。例如 student.tid 外键引用了 teacher.id，那么 student.tid 的值就不能超出 teacher.id 的返回，例如设置一个 teacher.id 不存在的值，这是可以的。

  • 那么为什么说外键如果不加索引的话就可能会死锁呢？如上所述外键是一种约束机制，控制着子表（student）的值不能超过父表（teacher）字段值的范围。那么如何做到在增删改父表期间又能保证不破坏约束呢？答案就是加锁，通过悲观锁进行互斥从而避免同步问题。但如果外键没有建立索引，在操作父表时就不能准确地知道子表的那条记录引用了自己，因此而不能进行行级别的加锁而被迫对整个表进行加锁，而这就是导致死锁的根源。案例如下（旧版本oracle的外键是不加索引的）：

  • 死锁指的是在并发时，持有锁的一方因为某些原因而导致未能释放锁从而造成其他想要获取该锁的线程或事物一致处于阻塞状态

  • 分析案例：当session-1请求往student加表锁时，因为此时session-2持有student的行锁，所以session-1会阻塞等待session-2释放行锁，但由于session-2的事物还没提交所以session-2并不会释放锁。
    此时同样的理由，session-2在删除id为2的teacher记录是也会申请往student加表锁，但session-1已经处于阻塞状态所以不可能释放锁，从而导致session-1和session-2交叉等待，导致死锁的发生。但如果此时外键有索引的话，那么session-1和session-2都不需要互相等待其释放锁，因为行锁粒度小，所以session-1和session-2要不是操作同一条记录的话是不需要等待对方释放锁的，从而也降低了死锁出现的记录。

  • 需要多说一句的是，锁的粒度越大的话其实发生死锁的记录就越少，如案例中一开始就加表锁的话是不可能存在两个会话同时持有锁的情况的（悲观锁），所以就不会有交叉等待的情况。但表锁会降低事务并发的吞吐量，这点需要明白

索引是不是越多越好

  • 索引本身不属于我们存储的数据的一部分，而是从中独立出来的一个’目录’，所以索引越多的话就会占据越多的磁盘空间
  • 当我们往表中插入数据时，数据库需要额外的操作来维护索引结构，因此如果索引越多的话，数据库消耗的额外操作就越多，从而导致性能的损耗

什么是覆盖索引

  • 覆盖索引和聚簇索引一样都不是索引的类型，覆盖索引准确点说其实是一种现象。通俗点说就是通过索引字段查找就能直接得到结果的查询，所以覆盖索引的效率是非常高的，例如使用主键来查询，因为上面我们已经分析过B+数，在B+数中主键为聚簇索引，所以可以直接在其叶子节点中获取到整条记录
  • 因为覆盖索引效率非常高，所以网上经常有人建议尽量少用select *其实大体有以下两个原因：
    • 在实际工作中很多情况下其实都不需要一次查询所有字段，所以select *就返回了与本次业务无关的冗余数据，同时也增加了数据的传输开销
    • 减少覆盖索引的出现。例如使用二级索引需来查询记录的id值，又或者用组合索引的左前缀查找右边的字段等这些操作都是覆盖索引，但如果使用select *的话因为是二级索引，所以叶子节点不可能存储了整条记录的数据，因此肯定是需要进行回表查询的

小结

  • 如果想要提高覆盖索引的出现，就应该少用select *，即能不用就不用
  • 如果某几个字段经常被查询或者组合使用，则应该建立组合索引，而字段的顺序可以由字段的使用频率从高到底由左往右排来组成组合索引

什么情况下索引会失效

  • 索引值类型为字符型时，使用值进行运算以及不使用''包裹都会导致索引失效
  • 索引值类型为字符型或者索引类型为组合索引时，使用 !=和 <>会导致索引失效
  • 索引列上使用函数或进行运算会导致索引失效。注意这里不是指值，而是索引列，例如 left(‘empId’, 5) = 100
  • 索引值类型为字符型时，使用了左模糊匹配，既 like ‘%索引列’ 会导致索引失效。如果值类型为数值型则使用模糊查询会直接使索引失效。
  • 使用组合索引时失效情况：
    • 没有遵循最佳左前缀
    • 使用 !=和 <>操作，即使符合最佳左前缀也会导致索引失效
