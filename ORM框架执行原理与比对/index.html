ORM框架执行原理与比对

 2020-04-22   作者: TanDi   类型：原创

今天心血来潮地在网上搜了一下有关 ORM 框架的使用对比，发现其实国内外对于ORM 框架的使用还是有着很大差别的。例如国内比较流行 MyBatis ，而国外则是 Hibernate。至于为什么这样，个人认为不同的人有着不同的考虑。其实对于技术本身而言并没有最好也没有最坏之说，有的只是合不合适而已。因此从今晚开始，打算写一篇有关ORM 框架对比相关的文章以梳理一下自己对 ORM 框架的理解，以及顺便对相关知识进行查漏补缺一下。

ORM是什么

  ORM的全称叫 Object Relational Mapping ，既对象关系映射。最初 ORM 的概念起始于面向对象编程，因为在面向对象编程中有句这样的话，就是“万物皆对象”，但是我们知道数据最终是要被持久化到硬盘上的，例如通过关系型数据库做介质来做持久化操作，因此也有人将 ORM 中的 R 解释为关系型数据库，那么当时存在的问题就是在关系型数据库中数据是以一条条记录的形式来存储的，这种存储方式叫行存储，行存储的特点是可以将多个不同类型的数据列存储为一条行记录。因此问题的关键点就在于关系型数据库并不是以对象的形式来持久化数据的。那么面对这样一个问题该如何解决呢？这时就诞生了 ORM 。通俗点来说 ORM 其实就是将数据库表映射为对象的操作，那么程序要想操作表记录的话就只需操作其映射对象就可以了，而完成这个 ORM 操作或者说提供 ORM 功能支持的就叫 ORM 框架。

  现今主流的 ORM 框架可以大致分为两种，既 完全ORM 和 非完全ORM。完全ORM指的是在ORM框架的加持下开发人员无需再维护SQL语句而只需要关心其对象就可以了。而非完全ORM指的是虽然提供ORM功能，但SQL语句还是需要开发人员来维护的。

  对于目前而言，据个人的了解提供完全 ORM 功能的框架有两个，Hibernate 和 TopLink。但是实话实说 TopLink 我是没有使用过，而且目前使用 TopLink 的人实在是少，所以不太好评价。而半完全 ORM 的代表莫过于 MyBatis 了。

  最后还是那句，对于技术而言并没有最好和最好，有的只是合不合适。

完全ORM-Hibernate

特点

  • 无需管理 SQL
  • 支持 JPA 规范，能够自动生成数据库表
  • 除了映射数据库表外，还支持映射视图
  • 支持 Native SQL、HQL、Criteria 三种操作方式
  • 支持 Annotation 和 XML 两种映射配置方式
  • 支持 一级缓存（Session级别）、二级缓存（SessionFactory级别）、查询缓存（需要二级缓存支持）
  • 默认会缓存 SQL （全列）语句模板来提高操作效率（可使用 @DynamicUpdate/@DynamicInsert 改为动态非全列操作）
  • 支持多种 ID 自生成策略
  • 支持批量更新和插入操作（与主键 IDENTITY 自增长策略一起使用会失效）
  • 支持的懒加载配置：FetchType.LAZY、FetchType.EAGER、@Fetch
  • 支持多种存储过程调用方式：
    • Session#createStoredProcedureCall()
    • Session#doWork
    • EntityManager#createStoredProcedureQuery
  • 对并发操作支持较为完善，提供 悲观锁（LockMode/LockModeType）、@Version 乐观锁、无版本乐观锁（@OptimisticLocking） 等实现
  • 对第三方技术有良好支持，例如可以模块化组装第三方本地缓存以及线程池等
  • 内置分页方法
  • 对实体的关联关系（数据库关联查询）有较为全面的支持
  • 能做到切换数据源时代码无感知
  • 因为是完全 ORM 框架，所以有效地提高了开发效率
  • 支持多租户场景的数据隔离操作

缺点

  • 封装程度较高，导致架构略显复杂，所以学习成本要比其它 ORM 要高
  • 无法对 SQL 进行统一管理，如果存在修改 SQL 的需求时需要修改源码后并重新编译
  • 批量操作不够灵活

半完全ORM-MyBatis

特点

  • 支持 XML 统一管理 SQL 语句，方便修改和调优需要
  • 架构清晰简单，小巧易上手
  • 开放拦截器接口(插件)，可按需定制和扩展功能，灵活性较高
  • 支持 Annotation 和 XML 两种映射配置方式
  • 支持 一级缓存(Session/ Statement级别)、二级缓存(Namespace)
  • 提供 动态SQL 标签，可实现逻辑拼装 SQL
  • 提供 ResultMap 标签定制映射结果集，以及 association(1-1) 和 collection(1-n) 两种关联配置
  • 映射配置文件支持 OGNL 表达式

缺点

  • 需要维护 SQL，也因此导致切换数据库时成本增大，且降低了开发效率
  • 不支持物理分页操作，需要依赖插件来完成
  • 级联操作与 Hibernate 相比支持较差
  • 引入第三方连接池需要手动实现接口
  • 不支持 JPA 规范，无法自动生成数据库表
  • 因为映射模板 ID 唯一，所以导致关联接口时方法不能重载
  • 二级缓存使用不当容易导致脏读现象
  • ID自生成策略较为薄弱，只支持 Jdbc3KeyGenerator(依赖数据库) 和 SelectKeyGenerator 两种

纯手动ORM-Spring JDBC

特点

  • 仅仅在原生JDBC的基础上进行了简单的封装，让 JDBC 操作边的更加方便好用
  • 简单易上手，只要有原生 JDBC 知识就可以直接使用
  • 提供简单的 ORM 功能（RowMapper）

缺点

  • 对 ORM 映射功能支持薄弱，复杂查询一般需要实现自定义 RowMapper
  • 无法对 SQL 进行统一管理，如果存在修改 SQL 的需求时需要修改源码后并重新编译
  • 开发人员除了代码之余还要维护 SQL，降低了开发效率
  • 耦合程度较高，导致切换数据库困难
  • 不支持缓存，数据量较大时效率偏低
  • 不支持级联操作

JPA规范升华-Spring Data JPA

基本介绍

  Spring Data JPA 其实并不是 ORM 框架，而是 JPA 规范的再封装。它是 Spring Data 成员，Spring Data 致力于统一持久化操作，可以简单理解为，它在现今五花八门的持久化API库的基础上提供了一套抽象API，让其变得简单且开发起来更加高效。开发人员通过 Spring Data 可以大大提高持久层的开发效率。并且凭借着优秀的抽象模型可以做到屏蔽底层实现，为底层技术切换提供了良好的支持。

Spring Data JPA 架构图

从上图我们可以大体看到 Spring Data JPA 位于 Jakarta EE JPA 规范之上。所以可以得出的结论是，只要作为底层的 ORM 框架支持 Jakarta EE JPA 规范，那么就能够被 Spring Data JPA 所支持。

优点

  • 继承了 Hibernate 的优点
  • 支持约定优于配置的 CRUD 方法定义，极大程度地提高了开发效率
  • 支持 JPQL(Spring版HQL) 、Native SQL 两种方法定义SQL操作
  • 简单易上手
  • 因为基础是 Spring Framework ，所以灵活程度较高
  • 适合单表操作并支持 DDD（领域驱动设计）聚合事件操作（ @DomainEvents 、 AbstractAggregateRoot ）

缺点

  • 无法对 SQL 进行统一管理，如果存在修改 SQL 的需求时需要修改源码后并重新编译
  • 批量操作(批量操作SQL语法)不够灵活。要么全部开启，要么通过 EntityManager 独立实现
  • 默认不支持动态SQL（即根据条件拼装SQL查询），因为Spring Data JPA符合的是DDD设计思想，其优点在于单表操作，符合现今微服务单一职责理念，但是如果是单体应用或者确实有复杂查询和动态查询的需求，那么亦有以下6种解决：
    1. 定义多个不同条件的 Repository 方法，然后据入参条件再判断应该使用哪个 Repository 方法
    2. 自定义 JPA Repository 接口实现。通过注入 EntityManager 进行动态查询，可选择 Native SQL 、JPQL、Criteria 其中一种实现
    3. 在 @Query 注解中使用 MySQL if函数进行动态拼接 SQL
    4. Example API 查询。局限性较大所以不推荐
    5. （非内置方式）使用 QueryDSL JPA 进行查询，类似 Hibernate 的 Criteria API

与JPA同台竞争-MyBatis Plus

基本介绍

  MyBatis Plus 是国内开发团队 baomidou 在原 MyBatis 基础框架上做的扩展。其中特别强调了“ 只做增强不做改变 ”，所以如果是从 MyBatis 切换到 MyBatis Plus 的话，只需要简单地 将 SqlSessionFactory 切换为 MybatisSqlSession 即可。MyBatis Plus 做的扩展在原理上解析其实很简单，主要是重写了 MyBatis 的以下部分：

  • 资源解析 - MapperXXXBuilder（注入通用SQL位置）
  • 代理处理 - MapperProxy
  • 方法分派 - MapperMethod

通过以上组件的重写可以支持通用 CRUD 注入和 ActiveRecord 模式等功能。

那么 MyBatis Plus 除了封装通用 CRUD 之外其实还继承了一些插件到自己身上。例如以前如果需进行要物理分页的话，是需要单独引入和配置分页插件的，而现在这个分页插件已经内嵌在 MyBatis Plus 中，那么用的时候只需要配置一下就可以了。所以，可以简单地说 MyBatis Plus 让 MyBatis 更加人性化了。

  还有就是其实从了解 MyBatis Plus 开始，它给我的感觉就是它和 Spring Data JPA 在功能上十分相似，甚至可以明确地看到存在借鉴 JPA 规范的操作，例如实体注解以及通用 CRUD 思想。最后个人有个疑问，就是为什么 baomidou 不尝试把在 MyBatis Plus 上做的扩展 pull requests 到现今的 MyBatis 中（难道是为了团队利益？），因为如果后期 MyBatis 升级导致功能发生了重叠的话，那么这就是个需要解决的问题了。

MyBatis Plus 的依赖结构

优点

  • 继承了 MyBatis 的优点
  • 内嵌分页插件支持物理分页操作
  • 新增通用 CRUD 模板。主要分为 Service 和 Mapper 两种，再配合新增的 Wrapper 实现对进行条件封装，该方案其实就是等于 Spring Data JPA 中的约定优于配置自定义Repository CRUD 方法
  • 新增 ActiveRecord 模式，实体通过继承 Model 类可以将 CRUD 操作内聚到实体自身中
  • 新增 ID 自生成策略
  • 新增 SQL 阻断器功能，支持拦截恶意或误操作等性质的全表 delete、update 行为
  • 新增字段映射类型转换器。例如：JacksonTypeHandler、GsonTypeHandler 等

缺点

  • 缺乏日志信息
  • 不是正统 MyBatis 版本
  • 需要维护 SQL，也因此导致切换数据库时成本增大，且降低了开发效率
  • 级联操作与 Hibernate 相比支持较差
  • 引入第三方连接池需要手动实现接口
  • 不支持 JPA 规范，无法自动生成数据库表
  • 因为映射模板 ID 唯一，所以导致关联接口时方法不能重载
  • 二级缓存使用不当容易导致脏读现象

拓展小知识

ORM 中的 N + 1 问题

  首先关于 N + 1 这个名字个人认为叫做 1 + N 会更加适合，原始是1条主 SQL导致了n条从SQL的诞生。

需要注意的是 1 + N 问题是建立在 ORM 框架支持关联查询的基础上才会发生的。指的是当我们只想查询主模型时，ORM 框架(默认)会连同主模型关联的从模型都查出来。简单地说就是原本只需要1条SQL的功夫，现在变成了1+n条，而 n 代表的是第1条SQL所查出来的模型个数。n操作会根据第1条SQL所查出来的每一个模型的id值再去查询其关联的从模型。

例子：假设 A 模型关联了 B 模型

  • A 进行 findAll（select * from A） 操作，假设现在返回了 2 个模型，既这时 n = 2
  • 接着，ORM框架(默认)会根据 findAll 查询出来的每个模型的 id 值再到 B 表中将其关联的从模型查询出来。因为现在 n = 2，所以ORM框架会再多执行 2 条SQL语句来查询关联模型

以上就是 1 + N 问题的叙述。其实 1 + N 问题早已不是什么新鲜的东西，网上也有很多关于该问题的文章，但是说到点上的却很少，而大多数都是像我上面写的这样，简单地告诉你使用 ORM 的关联查询时会出现该现象，然后就急忙着手告诉你如何解决了。

  • 那么有没有思考过为什么我们需要讨论 1+N 问题呢？ 或者说 1+N 问题的本质其实想表达点什么呢？

针对以上我提我简单地说说我自己的理解。1 + N 问题的本质其实是执行了我们认为多余的SQL操作，例如上述例子中原本只需要1条SQL就能得到结果，而现在却执行了1+n条，显然这不是我们想要。

而且你应该需要知道的是数据库通常是整个应用中最容易出现瓶颈的地方，所以如果存在大量的 1 + N 操作，就会导致大量的 SQL 语句执行，这无疑会对数据库造成一定的负担。除此之外还因为 JDBC 在获取返回数据时是阻塞的，因此大批量地执行 SQL 的话效率不言而喻。所以，因为以上这些原因我们应该尽量地避免 1 + N 操作，因为它会导致应用性能下降。

  最后在罗列解决方案之前需要多说一句，1 + N 问题只要 ORM 支持关联操作就一定会出现，因此我们能够做的只是一些补救的操作

  • 使用懒加载来延迟n操作执行，既只有我们通过代理实例访问其关联模型时才会进行实质的 SQL 查询
  • 将 1+n 操作直接合并为 1条 SQL，例如在 Hibernate 的 HQL 中就支持 join fetch 以示一次过连同关联模型都查询出来，那么这样就避免了后续需要执行多 n 条 SQL 的问题了
  • 在 Hibernate 中还提供一种叫 Criteria 的查询方式，也有人将其称为 QBC （Query by Criteria ），该操作效果和 join fetch 一样
  • 在 Hibernate 中还提供了一个叫 @BatchSize 的注解用来限制关联模型加载的个数，可以用于限制立即执行n条SQL的数量

除了以上这些方案之外，我还看到有些文章建议使用二级缓存来解决，我想大概的思想是使用了缓存之后虽然第一次findAll操作会有 1 + N 问题，但因为查询出来后会被缓存，那么后续操作就能相应地降低额外的 SQL 数量了。首先个人认为这种方案并不是什么机灵的做法，因为 1 + N 问题的本质是执行了多余的 SQL ，所以我们要从如何减少多余SQL执行的角度出发。而且缓存这东西其实一开始就是针对那些常用而且少修改的对象来设计的，因此如果缓存的对象如果经常被修改，那么缓存的意义就没有了，所以个人更加推荐懒加载方案或者是合并为1条SQL的方案
