闲聊

今天心血来潮地在网上搜了一下有关 ORM 框架的使用对比，发现其实国内外对于ORM 框架的使用还是有着很大差别的。例如国内比较流行 MyBatis ，而国外则是 Hibernate。至于为什么这样，个人认为不同的人有着不同的考虑。其实对于技术本身而言并没有最好也没有最坏之说，有的只是合不合适而已。因此从今晚开始，打算写一篇有关ORM 框架对比相关的文章以梳理一下自己对 ORM 框架的理解，以及顺便对相关知识进行查漏补缺一下。

ORM是什么

  ORM的全称叫 Object Relational Mapping ，既对象关系映射。最初 ORM 的概念起始于面向对象编程，因为在面向对象编程中有句这样的话，就是“万物皆对象”，但是我们知道数据最终是要被持久化到硬盘上的，例如通过关系型数据库做介质来做持久化操作，因此也有人将 ORM 中的 R 解释为关系型数据库，那么当时存在的问题就是在关系型数据库中数据是以一条条记录的形式来存储的，这种存储方式叫行存储，行存储的特点是可以将多个不同类型的数据列存储为一条行记录。因此问题的关键点就在于关系型数据库并不是以对象的形式来持久化数据的。那么面对这样一个问题该如何解决呢？这时就诞生了 ORM 。通俗点来说 ORM 其实就是将数据库表映射为对象的操作，那么程序要想操作表记录的话就只需操作其映射对象就可以了，而完成这个 ORM 操作或者说提供 ORM 功能支持的就叫 ORM 框架。

  现今主流的 ORM 框架可以大致分为两种，既 完全ORM 和 非完全ORM。完全ORM指的是在ORM框架的加持下开发人员无需再维护SQL语句而只需要关心其对象就可以了。而非完全ORM指的是虽然提供ORM功能，但SQL语句还是需要开发人员来维护的。

  对于目前而言，据个人的了解提供完全 ORM 功能的框架有两个，Hibernate 和 TopLink。但是实话实说 TopLink 我是没有使用过，而且目前使用 TopLink 的人实在是少，所以不太好评价。而半完全 ORM 的代表莫过于 MyBatis 了。

  最后还是那句，对于技术而言并没有最好和最好，有的只是合不合适。

完全ORM-Hibernate

特点

  • 无需管理 SQL
  • 支持 JPA 规范，能够自动生成数据库表
  • 除了映射数据库表外，还支持映射视图
  • 支持 Native SQL、HQL、Criteria 三种操作方式
  • 支持 Annotation 和 XML 两种映射配置方式
  • 支持 一级缓存（Session级别）、二级缓存（SessionFactory级别）、查询缓存（需要二级缓存支持）
  • 默认会缓存 SQL （全列）语句模板来提高操作效率（可使用 @DynamicUpdate/@DynamicInsert 改为动态非全列操作）
  • 支持多种 ID 自生成策略
  • 支持批量更新和插入操作（与主键 IDENTITY 自增长策略一起使用会失效）
  • 支持的懒加载配置：FetchType.LAZY、FetchType.EAGER、@Fetch
  • 支持多种存储过程调用方式：
    • Session#createStoredProcedureCall()
    • Session#doWork
    • EntityManager#createStoredProcedureQuery
  • 对并发操作支持较为完善，提供 悲观锁（LockMode/LockModeType）、@Version 乐观锁、无版本乐观锁（@OptimisticLocking） 等实现
  • 对第三方技术有良好支持，例如可以模块化组装第三方本地缓存以及线程池等
  • 内置分页方法
  • 对实体的关联关系（数据库关联查询）有较为全面的支持
  • 能做到切换数据源时代码无感知
  • 因为是完全 ORM 框架，所以有效地提高了开发效率
  • 支持多租户场景的数据隔离操作

缺点

  • 封装程度较高，导致架构略显复杂，所以学习成本要比其它 ORM 要高
  • 无法对 SQL 进行统一管理，如果存在修改 SQL 的需求时需要修改源码后并重新编译
  • 批量操作不够灵活

执行原理

一、持久化（瞬态转持久态）

  1. 使用 Session/EntityManager 执行 CRUD 操作，譬如 save(Object)
  2. 内部通过 SessionImpl#fireEventOnEachListener 发布 SaveOrUpdateEvent 事件
    • SaveOrUpdateEvent 包含了入参和 EventSource（SessionImpl）等信息
  3. DefaultSaveEventListener（DefaultSaveOrUpdateEventListener） 接收到事件进行处理，会进行一下操作：
    1. 使用 IdentifierGenerator 生成主键
    2. 将主键设置到入参实体中
    3. 将实体保存到一级缓存（StatefulPersistenceContext）
  4. 返回 ID 值给调用层
  5. 提交事务（start）
  6. 执行 JdbcResourceLocalTransactionCoordinatorImpl#beforeCompletionCallback，而其主要逻辑位于 SessionImpl#flushBeforeTransactionCompletion
  7. 发布 FlushEvent 事件
  8. DefaultFlushEventListener 接收事件执行 onFlush 方法
    1. 刷新缓存
    2. 执行 SQL
      1. 通过 AbstractFlushingEventListener#performExecutions 执行 ActionQueue（当前 Session 的 CRUD 操作）
      2. 执行具体 Action 的 execute 方法，主要做以下事情
        1. 构建 SQL 语句（PreparedStatement 从具体连接池中连接中返回）
        2. 将入参设置进 PreparedStatement
        3. 执行 SQL
  9. 提交事务（end）

二、查询操作

1）get 操作

  1. Session#get（class，id）
  2. SessionImpl#doLoad（id）发布 LoadEvent 事件。LoadEvent 包含 id、实例类型限定名、SessionImpl
  3. DefaultLoadEventListener 接收到事件执行 onLoad 方法
    1. 尝试从一级缓存（StatefulPersistenceContext）中找，如果没有则到二级缓存中找（CacheEntityLoaderHelper#loadFromSecondLevelCache）
    2. 如果实例依然被 Session 管理则返回，否则返回null

2）query#iterator 操作

  1. Session#createQuery（HQL）返回 Query 实例
  2. Query#iterator 返回迭代器。期间会保存查询策略，并先查出 HQL 对应的所有 ID 值
  3. Iterator#hasNext 判断是否有数据
  4. Iterator#next 返回代理实例。但 next 返回过程会根据具体的 ID 值执行 SQL 查询返回具体的实例

半完全ORM-MyBatis

特点

  • 支持 XML 统一管理 SQL 语句，方便修改和调优需要
  • 架构清晰简单，小巧易上手
  • 开放拦截器接口(插件)，可按需定制和扩展功能，灵活性较高
  • 支持 Annotation 和 XML 两种映射配置方式
  • 支持 一级缓存(Session/ Statement级别)、二级缓存(Namespace)
  • 提供 动态SQL 标签，可实现逻辑拼装 SQL
  • 提供 ResultMap 标签定制映射结果集，以及 association(1-1) 和 collection(1-n) 两种关联配置
  • 映射配置文件支持 OGNL 表达式

缺点

  • 需要维护 SQL，也因此导致切换数据库时成本增大，且降低了开发效率
  • 不支持物理分页操作，需要依赖插件来完成
  • 级联操作与 Hibernate 相比支持较差
  • 引入第三方连接池需要手动实现接口
  • 不支持 JPA 规范，无法自动生成数据库表
  • 因为映射模板 ID 唯一，所以导致关联接口时方法不能重载
  • 二级缓存使用不当容易导致脏读现象
  • ID自生成策略较为薄弱，只支持 Jdbc3KeyGenerator(依赖数据库) 和 SelectKeyGenerator 两种

执行原理

图解

过程

一、初始化过程

  1. 容器启动，实例化 ConfigurationAnnotationProcessor，解析@Configuration 配置类上 @MapperScan 上的信息
2. 实例化 MapperScannerConfigurer 并执行其 postProcessBeanDefinitionRegistry 方法
* 该方法源自 BeanDefinitionRegistryPostProcessor 
3. 创建一个 ClassPathMapperScanner，用来扫描 @MapperScan 中指定包上的 Mapper，并为其生成与之对应的 MapperFactoryBean 的 BeanDefinition，添加到容器中
4. 实例化 MapperFactoryBean
5. 执行 MapperFactoryBean#afterPropertiesSet（初始化方法）
6. 执行 MapperFactoryBean#checkDaoConfig，将抽象方法上面的注解信息，解析为 MappedStatement，并保存在 Configuration 中提供后续 MapperProxy 操作使用
* MappedStatement 包含了 sqlSource、keyGenerator 的一些信息。其中 sqlSource 用于返回 BoundSql ，而 BoundSql 是 SQL 相关的元信息（包含解析出来的 SQL 语句以及入参名字）

二、使用过程

  1. 有其他 Bean 依赖了 Mapper 接口，那么在该 Bean 填充属性阶段就会用对应的 MapperFactoryBean#getObject 方法返回一个 Mapper 的代理实例（JDK动态代理）

  1. 用 Mapper 代理（MapperProxy）执行 CRUD 方法

  2. 根据当前所被代理 Method 实例的限定名，从 Configuration#getMappedStatement 中返回 MappedStatement 实例

    • MappedStatement 保存在 StrictMap 中，该类继承自 HashMap。特点是 Key 不能重复，一旦在添加元素时检查到已经存在，就会直接抛异常。这样是为什么 Mapper 方法不能重载的注意原因，是因为其 key 使用抽象方法的全限定名来命名的（不包含参数列表）

  3. 利用 SqlSession 创建对应的 Executor ，并将执行权交给 Executor 来处理（CachingExecutor -> SimpleExecutor）。

    • 注意，Executor 实例是通过 Configuration 创建的，在返回实例前会执行 InterceptorChain 拦截操作

  4. Executor 会将 this 引用、入参、对应的 MappedStatement 等变量作为入参通过 Configuration 生成一个 RoutingStatementHandler 实例，并将执行权交给它。

    • RoutingStatementHandler 是一个路由器。实例化时会根据 SQL 类型来创建具体的 StatementHandler

    • 注意

      1. 创建 StatementHandler 时，会先后创建 ParameterHandler 和 ResultSetHandler。而 ParameterHandler、ResultSetHandler 都是通过 Configuration 创建的，所以在返回实例前会执行 InterceptorChain 拦截操作
      2. StatementHandler 也是通过 Configuration 创建的，所以在返回实例前以会执行 InterceptorChain 拦截操作
      3. 即 InterceptorChain 会先后拦截 Executor 、ParameterHandler、ResultSetHandler、StatementHandler

  5. RoutingStatementHandler 会使用 Connection 返回一个 PrepareStatement 实例（注意 PrepareStatement 是由具体的线程池提供的）

    • 即先从 BoundSql 中获取具体的 SQL 语句，然后通过 connection#prepareStatement 返回 PrepareStatement

      1. RoutingStatementHandler 将设置参数的任务路由给具体的 StatementHandler 类型来处理
      • 即从 BoundSql 中返回入参的名称和类型，然后通过 TypeHandler 将值设入 PrepareStatement
      • TypeHandler 对应着不同参数类型，而 MyBatis 提供了四十多个 TypeHandler 实例

  6. 执行 SQL 语句

  7. 返回 SQL 影响到的记录数量。在这之前如果使用了指定的 KeyGenerator，则在当前返回之前会先将主键封装到实体中再返回

获取返回主键值

基本介绍

在 MyBatis 中，默认执行完 Insert 语句后返回的是更新记录的数量，所以如果想有获取其自增的主键值时，就需要依赖 MyBatis 提供的 KeyGenerator 接口来实现。KeyGenerator 可以做到在执行 SQL 后将其 ID 值保存回实体中。

KeyGenerator 有 3 个实现

  • NoKeyGenerator：默认，即不需要返回 ID 值
  • Jdbc3KeyGenerator：用于支持自增主键的数据库，例如 MySQL
  • SelectKeyGenerator：支持可以手动获取最新 ID 主键值的数据库，例如 Oracle、MySQL
    • 即 SelectKey 动态查询标签

关于SqlSource类型

介绍

SqlSource 的主要作用是用来返回 BoundSql。BoundSql 实例保存了具体的 SQL 语句和入参的类型。但不同 SqlSource 所产生的 BoundSql 是不一样的。譬如 SqlSource 它 有以下 4 个具体实现：

  • DynamicSqlSource：SQL 用了 ${} 时使用
  • RawSqlSource：SQL 没有用 ${} 时使用
  • StaticSqlSource：其实上面 2 种类型中都存在一个 StaticSqlSource 实例。
    • StaticSqlSource 提供了 getBoundSql 实现
    • 可以简单理解上面 3 种类型都是 StaticSqlSource 的装饰器
  • ProviderSqlSource：使用了 Provider 相关注解时使用
    • 譬如 @InsertProvider 是在 MyBatis 3.4.5 版本后新增的，用于注解方式配置动态 SQL

SqlSource 的生成过程

Spring 容器启动时会例化 MapperFactoryBean 并执行其初始化方法 afterPropertiesSet （该方法实现自生命周期接口 InitializingBean）。

在 MapperFactoryBean#afterPropertiesSet 中会先通过 MapperFactoryBean#checkDaoConfig 方法对对应的 Mapper 接口进行解析。即如果这时使用了注解配置就会对其进行解析并得到 SQL 相关修改信息（如果是 XML 配置，则在构建创建 SqlSessionFactory 时解析）。

以注解配置方式为例，这时就会通过 MapperAnnotationBuilder#parseStatement 方法对接口中的各个 Method 进行解析，而 SqlSource 就是在这期间产生的。

buildSqlSourceFromStrings 返回 SqlSrouce 的主要逻辑实现位于 XMLLanguageDriver#createSqlSource

经过以上操作过后，SqlSource 最终会被封装到对应的 MappedStatement 实例中。而 MappedStatement 则在执行 CURD 操作时被使用。

Executor的生成时机

Executor 会在返回 SqlSession 实例时被创建。SqlSession 主要用于对外提供 CRUD 操作，而内部实质是由 Executor 来处理的 ，即 SqlSession 相当于 Executor 的一个代理。值得留意的是 Executor 使用了装饰器模式来扩展功能，譬如二级缓存实现。

纯手动ORM-Spring JDBC

特点

  • 仅仅在原生JDBC的基础上进行了简单的封装，让 JDBC 操作边的更加方便好用
  • 简单易上手，只要有原生 JDBC 知识就可以直接使用

缺点

  • 对 ORM 映射功能支持薄弱，复杂查询一般需要实现自定义 RowMapper
  • 无法对 SQL 进行统一管理，如果存在修改 SQL 的需求时需要修改源码后并重新编译
  • 开发人员除了代码之余还要维护 SQL，降低了开发效率
  • 耦合程度较高，导致切换数据库困难
  • 不支持缓存，数据量较大时效率偏低
  • 不支持级联操作

JPA规范升华-Spring Data JPA

基本介绍

  Spring Data JPA 其实并不是 ORM 框架，而是 JPA 规范的再封装。它是 Spring Data 成员，Spring Data 致力于统一持久化操作，可以简单理解为，它在现今五花八门的持久化API库的基础上提供了一套抽象API，让其变得简单且开发起来更加高效。开发人员通过 Spring Data 可以大大提高持久层的开发效率。并且凭借着优秀的抽象模型可以做到屏蔽底层实现，为底层技术切换提供了良好的支持。

Spring Data JPA 架构图

从上图我们可以大体看到 Spring Data JPA 位于 Jakarta EE JPA 规范之上。所以可以得出的结论是，只要作为底层的 ORM 框架支持 Jakarta EE JPA 规范，那么就能够被 Spring Data JPA 所支持。

优点

  • 继承了 Hibernate 的优点
  • 支持约定优于配置的 CRUD 方法定义，极大程度地提高了开发效率
  • 支持 JPQL(Spring版HQL) 、Native SQL 两种方法定义SQL操作
  • 简单易上手
  • 因为基础是 Spring Framework ，所以灵活程度较高
  • 适合单表操作并支持 DDD（领域驱动设计）聚合事件操作（ @DomainEvents 、 AbstractAggregateRoot ）

缺点

  • 无法对 SQL 进行统一管理，如果存在修改 SQL 的需求时需要修改源码后并重新编译
  • 批量操作(批量操作SQL语法)不够灵活。要么全部开启，要么通过 EntityManager 独立实现
  • 默认不支持动态SQL（即根据条件拼装SQL查询），因为Spring Data JPA符合的是DDD设计思想，其优点在于单表操作，符合现今微服务单一职责理念，但是如果是单体应用或者确实有复杂查询和动态查询的需求，那么亦有以下6种解决：
    1. 定义多个不同条件的 Repository 方法，然后根据入参条件再判断应该使用哪个方法
    2. 自定义 JPA Repository 接口以及其实现。通过注入 EntityManager 进行动态查询，可选择 Native SQL 、JPQL、Criteria 其中一种实现
    3. 在 @Query 注解中使用 MySQL if 函数进行动态拼接 SQL
    4. Example API 查询。局限性较大所以不推荐
    5. （非内置方式）使用 QueryDSL JPA 进行查询，类似 Hibernate 的 Criteria API

与JPA同台竞争-MyBatis Plus

基本介绍

  MyBatis Plus 是国内开发团队 baomidou 在原 MyBatis 基础框架上做的扩展。其中特别强调了“ 只做增强不做改变 ”，所以如果是从 MyBatis 切换到 MyBatis Plus 的话，只需要简单地 将 SqlSessionFactory 切换为 MybatisSqlSession 即可。MyBatis Plus 做的扩展在原理上解析其实很简单，主要是重写了 MyBatis 的以下部分：

  • 资源解析 - MapperXXXBuilder（注入通用SQL位置）
  • 代理处理 - MapperProxy
  • 方法分派 - MapperMethod

通过以上组件的重写可以支持通用 CRUD 注入和 ActiveRecord 模式等功能。

那么 MyBatis Plus 除了封装通用 CRUD 之外其实还继承了一些插件到自己身上。例如以前如果需进行要物理分页的话，是需要单独引入和配置分页插件的，而现在这个分页插件已经内嵌在 MyBatis Plus 中，那么用的时候只需要配置一下就可以了。所以，可以简单地说 MyBatis Plus 让 MyBatis 更加人性化了。

  还有就是其实从了解 MyBatis Plus 开始，它给我的感觉就是它和 Spring Data JPA 在功能上十分相似，甚至可以明确地看到存在借鉴 JPA 规范的操作，例如实体注解以及通用 CRUD 思想。最后个人有个疑问，就是为什么 baomidou 不尝试把在 MyBatis Plus 上做的扩展 pull requests 到现今的 MyBatis 中（难道是为了团队利益？），因为如果后期 MyBatis 升级导致功能发生了重叠的话，那么这就是个需要解决的问题了。

MyBatis Plus 的依赖结构

优点

  • 继承了 MyBatis 的优点
  • 内嵌分页插件支持物理分页操作
  • 新增通用 CRUD 模板。主要分为 Service 和 Mapper 两种，再配合新增的 Wrapper 实现对进行条件封装，该方案其实就是等于 Spring Data JPA 中的约定优于配置自定义Repository CRUD 方法
  • 新增 ActiveRecord 模式，实体通过继承 Model 类可以将 CRUD 操作内聚到实体自身中
  • 新增 ID 自生成策略
  • 新增 SQL 阻断器功能，支持拦截恶意或误操作等性质的全表 delete、update 行为
  • 新增字段映射类型转换器。例如：JacksonTypeHandler、GsonTypeHandler 等

缺点

  • 缺乏日志信息
  • 不是正统 MyBatis 版本
  • 需要维护 SQL，也因此导致切换数据库时成本增大，且降低了开发效率
  • 级联操作与 Hibernate 相比支持较差
  • 引入第三方连接池需要手动实现接口
  • 不支持 JPA 规范，无法自动生成数据库表
  • 因为映射模板 ID 唯一，所以导致关联接口时方法不能重载
  • 二级缓存使用不当容易导致脏读现象

Spring中ORM的注意事项

事务问题

在 Spring 单元测试环境中，使用 @Transactional 注解后，数据库的 CRUD 操作在执行完后，均会自动回滚。

即如果想要持久化，可以直接去掉 @Transactional 注解即可，但并不推荐，因为单元测试数据没必要做落盘操作。

EntityManager问题

EntityManager 本身是非线程安全的，因此如果在 Singleton 实例中使用则应该通过 @PersistenceContex 注解注入，而非 @Autowired、@Resource，因为 @PersistenceContex 可以为每条线程注入一个线程安全的 EntityManager。

ORM 中的 N + 1 问题

基本介绍

  首先关于 N + 1 名字而言，个人认为叫 1 + N 更加合适。指的是当我们只想查询主模型时 ORM 框架默认会连同其关联也查出来。即原本只需 1 条 SQL 的功夫，现在却变成了 1 + N 条。其中 N 代表的是 1 这条 SQL 所查出来的模型个数，因为 ORM 会根据这些模型的 ID 值再次去查询其关联模型。

例子：假设 A 模型关联了 B 模型

  • A 进行 findAll（select * from A） 操作，假设现在返回了 2 个模型，既这时 n = 2
  • 接着 ORM 框架(默认)会根据 findAll 查询出来的每个模型的 ID 值再到 B 表中将其关联的模型再查询出来。因为现在 n = 2，所以会再多执行 2 条SQL语句来查询其关联模型

问题思考

其实 1 + N 问题早已不是什么新鲜的东西，网上也有很多相关的文章，但是说到点上的却很少。那么我们有没有思考过为什么需要避免 1+N 问题呢？

个人理解1 + N 问题的本质其实就是执行了我们认为多余的SQL操作。譬如上述例子中原本我们只需要 1 条 SQL 的数据，但却执行了 1 + N 条，这显然造成了无畏的消耗，不是我们想要的。而且应该需要知道的是，数据库通常是整个应用中最容易出现瓶颈的地方，所以如果存在大量的 1 + N 操作，就会导致大量的 SQL 语句执行，从而给数据库造成负担。除此之外，JDBC在获取返回数据时是阻塞的，所以如果大批量地执行 SQL 的话其效率不言而喻。因此为了解决上述问题，我们就应该尽量地避免 1 + N 操作。

  最后在罗列解决方案之前需要多说一句，1 + N 问题只要 ORM 支持关联操作就一定会出现，所以我们能做的只有一些补救的操作：

  • 使用懒加载延迟 N 操作的执行，既只有我们通过代理实例访问其关联模型时，才真正去执行关联 SQL 查询
  • 将 1 + N 操作直接合并为 1 条 SQL。譬如在 Hibernate 的 HQL 中支持使用 join fetch 操作一次过连同关联模型一同查询出来，这样做可以减少连接多次执行 SQL 所带来的资源竞争和网络开销
  • 在 Hibernate 中还提供一种叫 Criteria 的查询方式，即所谓的 QBC （Query by Criteria ）。该操作效果和 join fetch 一样，会连同关联模型一同查出来
  • 除了 Criteria 外，Hibernate 还提供了一个叫 @BatchSize 的注解，该注解可以用来限制关联模型查询的批次，可以用于限制一次性执行 N 条 SQL （即分拆执行）

除了以上方案外还看到网上有些文章建议使用二级缓存来解决，我想大概的思路是，第一次 findAll 操作虽然会有 1 + N 问题，但因为查询出来后会被缓存，那么后续操作就能相应地降低额外的 SQL 查询数量了。

首先个人认为这种方法实属是投机取消，因为 1 + N 问题的本质是执行了多余的 SQL ，所以我们要从如何减少多余 SQL 执行的角度出发。而且缓存这种机制一开始就是针对那些常用而且少修改的数据设计的，所以如果对象频繁地被修改的话，那么缓存就没什么意义了，所以个人更加推荐懒加载或合并为1条SQL的方案
