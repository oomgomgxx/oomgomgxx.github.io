闲聊

前段时间阅读了部分 Netty 4 的源码，但当时并没有趁热打铁地将理解过程记录下来，以至于现在忘得七七八八。起初阅读 Netty 源码的原因是想知道它为什么高效，因为在 Spring Web Flux 和 Spring Cloud Gateway 中都看到了它的身影。虽然自己以前在公司项目中使用过 Netty 来实现功能，但是我个人很清楚，会用和懂其实是两种截然不同的境界。通常会用技术的人很多但真正懂技术的人却是少数，所以学习技术应该尽量做到知其然而知其所以然，亦只有这样才能在众多同行中增大自己的核心竞争力。

除此之外，个人对学习技术也有一定的看法。譬如我认为技术都是万变不离其宗的，所以学习技术应该要抓住根本。例如 Java 生态中开源技术繁多，所以我们不太可能将它们每一个都弄得透彻，但它们的根本是相同的，即都是 JavaSE、设计模式、数据结构的堆砌。所以接下来本文并不是要聊 Netty，而是 Netty 的本质，即IO模型和事件驱动。

在类 Unix 系统中 IO 模型分别有阻塞IO、非阻塞IO、复用IO、信号驱动IO、异步IO 5 种，它们各有各的特点，没有好坏之分。而谈到事件驱动的话，如果没有记错第一次接触到这个词是很久以前我学习 Node.js 时看到的，因为 Node.js 的三大特性之一就是事件驱动。当时其实很好奇，主要是想知道事件是如何产生及何时被回调函数处理的，但这个疑问一直等到看了 Netty 源码后才变得明朗。例如在 Netty 中采用的是 Reactor 模式实现的事件驱动处理（Node.js、Readis、Nginx、Memcache 等也是），而除了 Reactor 之外还可以通过 Proactor 来实现，两者的区别在于前者是 复用IO 后者是 异步IO，但目前 Linux 对异步 IO 模型支持较差，所以 Java 的 AIO 其实也是采用复用 IO 来实现的。

IO的5种模型

有关Linux的系统调用，可参考 syscalls.h

同步阻塞IO

1）图解

2）特点

  • IO 操作全程阻塞
  • 在低并发 IO 密集型场景，阻塞可有效减少不必要的用户态内核态切换，所以能够一定程度地提高人物线程的处理效率
  • 过程中用户程序只需进行一次系统调用
  • 为了解决用户程序处理阻塞问题，需要为每个套接字分配独立的一条处理线程
  • 不适合高并发场景，例如如下：
    • 阻塞会导致吞吐量下降
    • 多线程会导致大量 CPU 上下文切换从而导致线程处理能力下降，效率不佳

3）阻塞IO例子：Java BIO

同步非阻塞IO

1）图解

2）特点

  • IO操作无需阻塞
  • 用户程序需要主动轮询来判断是否有就绪文件描述符
  • 过程中用户程序需要 n 次系统调用
  • 不适合高并发场景，例如如下：
    • 用户程序需要主动轮询，高并发场景会导致 CPU 使用率极速上升而导致程序卡顿
    • 因为无法逃避主动轮询，所以和阻塞IO一样需要为每个套接字分配独立的一条处理线程
  • 虽然 IO 操作无需阻塞，但优势并不明显，个人认为甚至有些鸡肋，所以很少见会有程序直接使用非阻塞 IO 模型

复用IO

1）图解

2）特点

  • 可以将复用 IO 理解为非阻塞 IO 的升级版
  • 复用 IO 相当于将非阻塞 IO 的主动轮询操作迁移到了内核中进行，这样大大减少了系统调用带来的消耗
  • IO 操作一般情况下只会在 Selector 多路复用器上发生短暂的阻塞
  • 过程中用户程序需要 2 次系统调用
  • 多路复用器可以用 select/poll/epoll/kqueue 等方案实现，而在 Linux 中 epoll 效率最佳，所以一般只要提到复用 IO 的话基本上都是采用 epoll 实现的
  • 适合高并发场景，理由如下：
    • IO 操作阻塞短暂
    • 一条线程可以同时处理多个用户请求（负责注册感兴趣事件），大大降低了 CPU 资源竞争消耗

3）复用 IO 例子：Java NIO

Selector 多路复用器实现（内核版本：linux-5.4.81）

1）select 函数

  • 无状态，因为 fd_set 只有一个字段用来记录关注和就绪的事件，所以每次调用都会导致 fd_set 初始化，即相当于每次调用 select 都需要将感兴趣的文件描述符内容重新拷贝到内核
  • 正常情况下，只要有就绪的文件描述符或 select 超时就会返回就绪文件描述符数量，但返回后需要再轮询集合找出具体的文件描述符，所以随着文件描述符（连接数）增多而效率会降低
  • 因为随着文件描述符（连接数）增多而会导致效率下降，所以通常单条进程 select 时的最大感兴趣文件描述符数量会受到限制，一般为 2014（FD_SETSIZE），可以执行 ulimit -n 查看。所以 select 在并发能力上是存在局限性的

2）poll 函数

  • 和 select 相比，仍然是无状态，因为在每次系统调用时还是需要将所有感兴趣的文件描述符重新拷贝到内核，但 poll 还是相对于 select 有所优化的， 譬如使用链表代替了数组来解决单进程文件描述符数量限制问题，以及将感兴趣和就绪集合分开从而避免了不必要的初始化操作
  • 返回的依然是就绪文件描述符的数量，所以还是需要重新轮询集合得到具体的文件描述符，即时间复杂度和 select 一样是 O(n)

3）epoll 函数

epoll 和 select、poll 不一样，它其实有三个函数：epoll_create、epoll_ctl、epoll_wait，而且 epoll 内部是基于事件驱动的。

  • 无最大文件描述符数量限制
  • 存在状态，内部维护着一颗红黑树和双向链表，所以用户线程无需多次拷贝感兴趣的文件描述符
  • epoll 将用户程序感兴趣的文件描述符注册在红黑树上，当有就绪的文件描述符时内核会通过事件驱动方式将其添加到双向链表（rdllist）中，即当执行 epoll_wait 时就只需要从 rdllist 中获取就绪文件描述符即可，而无需遍历整个红黑树
  • epoll 整个过程是通过事件驱动的方式来完成的，即当有就绪的文件描述符时内核会自动回调 ep_poll_callback 方法将就绪的文件描述符添加到双向链表（rdllist）中并唤醒等待的内核线程，所以避免了繁重的轮询操作，大大降低了系统的开销
  • epoll 的时间复杂度为 O(1)，从上面的源码可以得出结论就是 epoll 不会随着并发连接数的增多而降低效率，因为它是基事件驱动的，而且没有轮询操作

信号驱动IO

1）图解

2）特点

  • 读取数据时需要先往注册一个信号处理器，当存在就绪 fd 时内核就会发出信号来通知处理
  • 过程中用户程序需要 2 次系统调用
  • 读取数据时是非阻塞的，且无需主动轮询，因此一定程度地降低 CPU 资源消耗
  • 注意，该模型一般只在 UDP 协议场景下应用，例如监听数据报到来或是异常通知等一些简单需要。而在 TCP 协议场景基本没人用，原因有以下两个：
    • 响应信号过于太频率，因此对处理能力要求较高
    • 响应信号不能表示出具体的事件（Linux支持的信号有限），所以并发信号时无法辨识事件

异步IO

1）图解

2）特点

  • IO 操作无需阻塞，亦无需主动轮询
  • 过程中用户程序只需发起一次系统调用
  • 适合高并发高吞吐量场景
  • 注意：
    • 异步 IO 是需要系统内核支持的。例如 Windows 上的 IOCP（Input/Output Completion Port）才是真正的异步 IO，而目前 Linux 上暂时还不能真正实现的异步 IO，所以在 Java 的 AIO 中实质用的是 epoll （可参考 LinuxAsynchronousChannelProvider 类）
    • 与此同时我们也不能忽视 Linux 在 AIO 方面作出的努力。譬如在 Linux 5.1 这个内核版本中就发布了一套全新的异步操作API io_uring，所以 Linux 完善异步 IO 我们就拭目而待吧

事件驱动

Reactor

单Reactor单线程模式

1）图解

2）角色

  • selector：复用 IO 模型中的多路复用器，对应着 select/poll/epoll/kqueue 。主要负责接收用户感兴趣的事件
  • dispatch：属于 Reactor 职责，用于在 selector 返回非 accept 就绪事件后将其分派给 Handler 处理
  • Acceptor：accept 事件处理器，以及负责其它事件Handler的创建工作
  • Handler：事件处理器

3）例子说明：日志服务程序

  1. 程序启动时，先向 Reactor 注册一个 Acceptor 处理器用于处理就绪的 accpet 事件
  2. Reactor 将对应的 Acceptor 处理器感兴趣的事件（accept）注册到 Selector 多路复用器中
  3. 当有客户端连接到来时，Selector 多路复用器会返回 accept 事件的 key 给 Reactor，表示存在就绪事件让Reactor 进行分配处理
  4. Reactor 按照 key 的类型将其分派给对应的 Handler 。当前是 accept 事件所以会分派给 Acceptor 来处理
  5. Acceptor 接收到分派任务之后就会处理 accept 事件
  6. 当 Acceptor 处理完 accept 事件后就会为后续的操作作准备，例如创建其它类型的 Handler 来监听和处理各种事件，譬如 read 和 write。
  7. Handler 被 Acceptor 创建后需要注册到 Reactor 上，除此之外和上面的 Acceptor 一样还需要注册相应感兴趣的事件。譬如 ReadHandler 就注册 read 事件，WriteHandler 就注册 write 事件。

4）模式最佳实践以及优缺点

  • Redis 采用的就是单 Reactor 单线 (6.x.x 版本前)
  • 优点1：单线程不存在资源竞争，所以这是线程的处理能力是最高的
  • 优点2：实现起来简单
  • 缺点1：单线程无法充分发挥多核心 CPU 的性能
  • 缺点2：单线程在执行 Handler 时无法处理其它事件，所以 Handler 执行时间较长时就会很容易导致系统出现瓶颈。也因为这个缘故所以 Redis 是不建议使用诸如 keys 这样较为耗时的命令来操作的。
  • 总的来说，单Reactor单线只适合请求处理迅速的场景

单Reactor多线程模式

1）图解

2）差异

在当前模式中 Handler 位于 Reactor 线程，但它只负责接收事件，而真正的任务处理会交给其它线程来完成。例如可以创建一个单独的 Worker 线程池来处理。

3）模式最佳实践以及优缺点

  • Redis 6.x 版本开始支持多线程处理网络IO，而开启该功能后 Redis 就数据单Reactor多线程模型
  • 优点1：从 Reactor 线程中抽离了最为耗时的读写事件处理操作，大大地提高程序的任务处理能力
  • 优点2：引入了多线程处理，可以充分发挥多核 CPU 性能
  • 缺点1：多线程操作引入资源同步以及 CPU 资源竞争激烈等问题
  • 缺点2：因为 Reactor 依然是单线程，所以大量客户端发起 accept 操作时亦可能会出现瓶颈（当然这量要很大）

多Reactor多线程（主从Reactor）

1）图解

2）差异

Reactor 引入了多线程，将 accept 和 非accept 两种事件进行了更细的拆分。

主 Reactor 线程负责 accept 事件，而从 Reactor 则负责非accept 事件。

每个 Reactor 线程都有自己的 Selector 。

3）模式最佳实践以及优缺点

  • Memcache 以及 Nginx(Worker进程) 都采用了主从 Reactor 模式
    • 在 Nginx 中该模型叫 Run-Loop 而不叫 Reactor，但只是名字有别而已
    • 要注意的是 Nginx 使用的是多进程而不是线程。虽然多线程模型能够节省资源，但缺点是位于同一个进程内，线程之间能够访问同一地址空间，彼此容易互相影响，譬如虚拟内存发生缺页中断则有可能影响所有的线程。Nginx 使用多进程加异步操作处理网络请求，这种高可用高可靠的服务器需求
  • 优点1：Reactor 通过多线程对职责进行了进一步的划分，这样领到模式的整体结构更加清晰以及实现起来比单Reactor多线程更加容易
  • 优点2：父子 Reactor 线程交互简单，且每个线程都有独自的 Selector ，所以处理非同步资源时性能要比前两种都好
  • 缺点：多线程会导致 CPU 资源竞争激烈，因此线程并不是越多越好。
  • 若果你不知道如何设置线程数的话可以参考以下原则
    • CPU密集型：大量操作落在 CPU 和 内存 上，意味着 CPU 资源竞争激烈，这样会容易导致 CPU 使用率急速上升从而导致系统瓶颈，甚至会出现卡顿。这时线程数应该尽量少一些，例如CPU核心数 + 1
    • IO密集型：大量操作落在 IO 上，因为 IO 存在阻塞现象（异步IO除外），所以线程可能会被短暂阻塞，这样可以起到缓解 CPU 资源竞争带来的压力。这时线程数可以稍微高一些，例如CPU核心数x2+1

Proactor

1）图解

2）角色

  • Proactor Initiator：Proactor 引导程序
  • Acceptor：accept 事件处理器
  • Handler：非 accept 事件处理器
  • Asynchronous Operation Processor：内核异步操作处理
  • Proactor：异步IO事件分派器

3）模式最佳实践和优缺点

  • ACE Proactor （由 Windows IOCP实现）
  • 在 Linux 中所谓的 AIO 操作是由 GNU libc 库函数在用户空间借由 pthread 方式实现的，而且仅仅针对磁盘类 I/O，套接字 I/O 不支持
