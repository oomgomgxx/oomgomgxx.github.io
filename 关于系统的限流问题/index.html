代码限流操作

 2018-08-27   作者: TanDi

前言

  限流其实很好理解，就是一种保护系统有效地正常运作的手段。这样的操作在日常生活中也很常见，例如车流限 和 人流限制。

  对系统而言，限流通常是指只允许系统能够承受的访问量限度内进来请求处理，超出系统访问能力的请求将
被丢弃。限流的主要手动可以划分为两种，分别是 基于请求的限流 和 基于资源的限流

  • 基于请求限流：请求阈值、请求频率、接口调用频率

  • 基于资源限流：线程数 、数据库连接数、请求队列长度

信号量：Semaphore

有什么用

  • 限制同一时刻最大执行线程数

基本原理

Semaphore类位于JDK的JUC包。看到 JUC 基本可以断定是基于AQS来实现的（自旋+CAS）。

从上图源码中很直观地得知，Semaphore使用共享锁来实现功能，并支持公平和非公平两种抢锁模式。其中Semaphore需要接收有一个名为permits(许可证)的参数来指定可用许可证数量（实质就是共享锁数量）。当线程获取到许可证就可以继续往下操作，如果获取不到许可证就会一直自旋。

以非公平锁为例，源码如下：

// tryAcquireShared(1)
final int nonfairTryAcquireShared(int acquires) {
    for (;;) {
        // 获取可用许可剩余数量
        int available = getState();
        // 减去被获取的许可数
        int remaining = available - acquires;
        // 当可用许可小于0 或 成功获取到许可 就返回remaining变量
        if (remaining < 0 || compareAndSetState(available, remaining))
            return remaining;
    }
}

// sync.acquireSharedInterruptibly(1)
public final void acquireSharedInterruptibly(int arg)
    throws InterruptedException {
    if (Thread.interrupted())
        throw new InterruptedException();
    // 小于0表示许可已经被获取完，进入链表等待其他线程释放许可
    if (tryAcquireShared(arg) < 0)
        doAcquireSharedInterruptibly(arg);
}

// Semaphore.acquire()
public void acquire() throws InterruptedException {
    sync.acquireSharedInterruptibly(1);
}

用法

可以观察到，当许可被用完，其他线程就没办法执行临界代码了，而是一直在尝试获取许可。

速率限制器：RateLimiter

有什么用

  • 限制处理速率

常见的限制速率算法

Fixed Window（固定窗口）

  固定窗口算法很容易理解，例如接口1分钟内只能处理100个请求，这里用变量counter来记录请求数，当counter到达100后当前这1分钟将不再接收请求（丢弃）。直到下一个1分钟counter自动回零再继续接收请求，当然峰值依然是100，如此类推。

  但该算法有个比较严重的临界问题。举个较为极端点的例子，如第一分钟的前58s都没有请求，到59s时突然来了100个请求（counter最大100）。但是59s过后意味着进入下一个一分钟，这时counter归零可以重新再接收100个请求。因此在59s到0s这个间隙期间最大的请求峰值是200而不是100。

Slide Window（滑动窗口）

  滑动窗口算法是固定窗口算法的改良版本。

图片源自网络

如图所示，红色框表示滑动窗口。特点是可移动，且它将窗口按照时间来分段。例如1分钟分为6段。每段都含有各自的counter。换句话来说就是对固定窗口进行了拆分，通过这种方式来缩小固定窗口的缺陷。譬如接口峰值为60个请求，所以固定窗口临界点会出现120个请求。但这点在滑动窗口就不一样了，在滑动窗口中每一段都有自己的counter，如图所示有6段，所以有6个counter，即每个counter在滑动窗口中（一分钟）都可以接受10个请求。虽然滑动窗口依然存在临界问题，但数量降低至20而不是120。

Leaky Bucket（漏桶）

  漏桶算法中的漏桶通常用一个FIFO队列来实现，如下图：

图片源自网络

水滴就是请求，当请求数量大于漏桶（队列）容量时，就溢出（丢弃）。换句话来说就是通过漏桶的容量来限制速率。

Token Bucket（令牌桶）

图片源自网络

令牌桶的思想如下：

  1. 存放token的桶固定容量为k。
  2. 每秒往桶中放入x个token。如果在放入过程中token数等于k，则丢弃剩余的token。
  3. 请求按照自己的需要到桶中申请指定数量的token。申请成功就进行请求处理，申请失败则进入等待。

令牌桶的优点在于即限制了请求速率的同时亦允许突发增长的情况，这点体现在请求可以按自己实际需要来获取token数量。例如桶中放有10碗水（token），现在有2个人要用水，这两个人我们用A和B表示，那么令牌桶的思想就是这10碗水给A和B自由分配，即不固定获取水的碗数，如果桶不够水就等待令牌桶往桶里加水知道满足自己所需要的数量再获取。

漏桶 vs 令牌桶

漏桶 令牌桶
不依赖令牌 依赖令牌
桶满时丢弃多余请求 桶满时丢弃令牌，请求获取不到令牌只需要等待
固定速率，不允许突发 可以按照请求所需来申请token来应付突发

Guava令牌桶算法实现

  在开源 Google Guava 库中就提供令牌桶算法实现，且提供两种不同的突发模式。分别是SmoothBursty（平滑突发）和SmoothWarmingUp（平滑预热突发）。后者可以通过预热让请求获取token的时间平稳，从而降低突发性。

引入依赖

<dependency>
    <groupId>com.google.guava</groupId>
    <artifactId>guava</artifactId>
    <version>28.0-jre</version>
</dependency>

使用例子

public static void main(String[] args) throws Exception {

    // 平滑突发
    //RateLimiter rateLimiter = RateLimiter.create(10);
    // 平滑预热突发。预热2秒钟
    RateLimiter rateLimiter = RateLimiter.create(10, 2, TimeUnit.SECONDS);

    for (;;) {
            // 有足够令牌返回0，否则阻塞。
            // 阻塞过后获取到令牌返回等待令牌的那段艰苦时间
            System.out.println(rateLimiter.acquire());
            //System.out.println(rateLimiter.acquire(3)); // 获取指定数量的令牌
            //System.out.println(rateLimiter.tryAcquire()); // 尝试获取1个令牌
            //System.out.println(rateLimiter.tryAcquire(2, TimeUnit.SECONDS)); //尝试获取1个令牌但只等待2秒
    }

}
