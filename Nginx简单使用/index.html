基本介绍

Nginx（ 发音：Engine X）是一个基于多进程模型及事件驱动的轻量服务器。

本文不会对 Nginx 的配置作过多的阐述，各模块的配置建议参考官方文档：https://nginx.org/en/docs/

特性

  • 高并发
    • 使用 Reactor 模型处理网络请求（在 Nginx 中叫 Run-Loop）
  • 高性能
    • 确保每一个连接消耗的内存最小化
  • 高可靠
    • Master 进程可以在 Worker 进程出错后快速拉起新的 Worker 进程继续提供服务
  • 可扩展性高
    • 模块化设计，第三方模块丰富
  • 热部署
    • 支持不停机(信号)维护进程，以及更新配置项和日志文件
  • 源码使用BSD许可证，给予使用者非常高的使用自由度
  • 提供HTTP/TCP/UDP等7层和四层(ngx_stream_core_module)负载均衡实现

常见使用场景

  • HTTP静态资源服务器
    • 当前文件系统作为静态资源存储点提供服务
  • 反向代理服务器（最常用）
    • 基于多进程事件驱动性能强大
    • 提供静态资源缓存功能，能一定程度提高响应速度
    • 提供负载均衡能力，可合理地分摊上游服务器负载
  • 网站请求动静分离
  • 微服务服务API网关
    • Nginx + Lua
  • 流媒体服务器
    • 可通过 nginx-rtmp 模块实现

编译安装

1）在官网选择需要的版本

  • https://nginx.org/en/download.html

2）下载后解压

> wget https://nginx.org/download/nginx-1.14.2.tar.gz
> tar -zxvf nginx-1.14.2.tar.gz

3）生成编译文件 Makefile

> cd nginx-1.14.2
> ./configure --prefix=/opt/nginx
  • 可以通过./configure --help查看配置参数
  • –prefix 用于指定安装路径
  • 如果需要安装指定模块，可以参考 –with- 开头的参数；反之参考 –without- 开头的参数
  • 执行完./configure后会在当前目录生成一个objs文件夹，其中 ngx_modules.c 中包含将要安装的模块

4）安装 Nginx 需要的依赖

> yum install -y gcc gcc-c++ autoconf pcre pcre-devel make automake httpd-tools zlib zlib-devel openssl openssl-devel

5）根据 Makefile 编译和安装

> make && make install

6）将Nginx的命令添加到环境变量（可选操作）

  • 将 sbin 目录配置到 /etc/profile 中的 PATH 环境变量尾部
  • 修改完执行source /etc/profile刷新配置即可

7）使用 vim 编辑 Nginx 配置文件时高亮显示（可选操作）

> pwd
/root/nginx-1.14.2
> mkdir ~/.vim
> cp -r contrib/vim/* ~/.vim

8）将 Nginx 添加到系统服务中，并设置开机启动（CentOS 8）

  1. 添加服务文件 vim /lib/systemd/system/nginx.service

    [Unit]
Description=nginx service
After=network.target

[Service]
Type=forking
ExecStart=/opt/nginx/sbin/nginx
ExecReload=/opt/nginx/sbin/nginx -s reload
ExecStop=/opt/nginx/sbin/nginx -s stop
PrivateTmp=true

[Install]
WantedBy=multi-user.target

  2. 启动服务并设置开机启动

    > systemctl start nginx.service
> systemctl enable nginx.service

命令使用

nginx命令

> nginx -h
nginx [-?hvVtq] [-s signal] [-c filename] [-p prefix] [-g directives]

-?,-h           : 打印帮助信息
-v              : 打印显示版本信息
-V              : 打印显示版本信息，以及 configure 配置项
-t              : 测试配置文件
-q              : 在检测配置文件期间屏蔽非错误信息
-s signal       : 给Master进程发送信号：stop（停止）, quit（退出）, reopen（开启新日志）, reload（重新加载配置）
-p prefix       : 指定nginx目录
-c filename     : 指定配置文件
-g directives   : 指定附加配置路径

kill命令

在 Nginx 中除了可以使通过 nginx -s 给进程发送信号外，其实还可以通过命令 kill 发送

1）kill 支持的操作信号：

> kill -l
HUP INT QUIT ILL TRAP ABRT BUS FPE KILL USR1 SEGV USR2 PIPE ALRM TERM STKFLT CHLD CONT STOP TSTP TTIN TTOU URG XCPU XFSZ VTALRM PROF WINCH POLL PWR SYS
  • 注意以上信号都有与之对应的编号，从1开始往右递增。

2）常用信号：

编号 信号 功能
1 HUP 平滑启动/重启信号。用新配置启动新进程，然后关闭旧进程
2 INT 中断信号（ctrl+c）
3 QUIT 平滑退出信号
9 KILL 强制关闭信号（程序无时间回收资源）
15 TERM （默认）终止信号，正常退出（程序有时间回收资源）
18 CONT 从 STOP 恢复运行信号
19 STOP 暂停信号
10 USR1 开启新日志。常用来做日志切割
12 USR2 平滑升级。使用新版本启动进程，然后关闭旧版本进程
28 WINCH 平滑地关闭 Worker 进程。常和 USR2 一同使用来实现平滑升级

案例：为正在运行的 Nginx 添加新模块

1）下载第三方模块

> git clone https://github.com/replay/ngx_http_consistent_hash.git

2）进入源文件夹重新生成 Makefile

> nginx -V # 查看之前的配置项，做到在原来的基础上添加
nginx version: nginx/1.14.2
built by gcc 8.3.1 20191121 (Red Hat 8.3.1-5) (GCC) 
configure arguments: --prefix=/opt/nginx
> ./configure --prefix=/opt/nginx --add-module=/root/ngx_http_consistent_hash

3）重新根据 Makefile 编译

> make
  • 注意无需执行make install 重新安装，否则现有的 Nginx 目录会被覆盖

4）将 objs 中编译好的二进制 Nginx 命令拷贝到安装目录

# 先备份预防万一
> mv /opt/nginx/sbin/nginx /opt/nginx/sbin/nginx.bak
> cp objs/nginx /opt/nginx/sbin

5）验证

> nginx -V
nginx version: nginx/1.14.2
built by gcc 8.3.1 20191121 (Red Hat 8.3.1-5) (GCC) 
configure arguments: --prefix=/opt/nginx --add-module=/root/ngx_http_consistent_hash # 可以看到新模块已经加入

6）因为修改了二进制文件而不是配置文件，因此需要通过平滑升级方式重启，而不是 nginx -t; nginx -s reload

> kill -USR2 14180 # 平滑升级，这里会使用新的二进制文件启动Nginx
> kill -WINCH 14180 # 通知旧Master进程让其关闭Worker进程
> kill -QUIT 14180 # 平滑地关闭旧Master进程

配置模块的基本作用

全局块（对整个Nginx起作用）

  • 指定 Nginx 的管理用户及其组信息
  • 指定启动时拉起的 Worker 进程的数量
  • 进程 PID 号的存放位置
  • 日志存放路径和日志类型配置
  • 引入其它配置

events块

  • 指定 Worker 进程同一时刻能够连接客户端的最大数量
    • 最大连接数 = worker_processes * worker_connections
    • 注意：worker_connections 不能超过文件系统所允许打开的最大文件数量。可以通过 cat /proc/sys/fs/file-max 查看
  • 事件驱动模型选择
  • 是否允许同时接受多个网络连接

http模块

  • 媒体类型
  • 代理配置
  • 缓存配置
  • 日志定义
  • 连接配置
  • 压缩配置
  • 零拷贝配置

server块

  • 虚拟主机。可配置多个
  • 监听端口
  • 域名关联
  • 错误码页面配置
  • 安全通信配置

  • 虚拟主机资源。可配置多个

使用 http_stub_status_module 进行基本状态监控

1）添加模块

2）配置 localtion 

location /status {
    stub_status;

4）访问资源

> curl localhost/status
Active connections: 1 
server accepts handled requests
 13 13 35 
Reading: 0 Writing: 1 Waiting: 0
  • Active connections：当前活动的客户端连接数，包括Waiting连接数
  • accepts：已接受的客户端连接总数
  • handled：已处理的连接总数。通常该值与 accepts 一致，除非达到了worker_connections资源限制
  • requests：客户端请求总数
  • Reading：正在向客户端进行读操作的连接数
  • Writing：正在向客户端进行写操作的连接数
  • Waiting：正在空闲等待客户端请求的连接数

使用keepalived提供高可用解决方案

VRRP简介

  虚拟路由器冗余协议（VRRP）可用于解决路由器单点故障问题。该协议会在指定网卡上创建一个虚拟IP地址，然后通过自动切其换网关来完成高可用实现。譬如虚拟IP开始会以Master作为网关来处理请求，而一旦Master宕机，则虚拟IP的网关将会被切换到热备节点上，以顶替Master节点的工作。

  Keepalived 提供了 VRRP 功能实现，因此 Keepalived 可以为任何服务提供高可用解决方案。换句话说如果服务本身没有高可用解决方案的话，就可以考虑使用 Keepalived 来实现。

在Nginx服务上安装整合Keepalived

0）例子环境

192.168.0.210 nginx、keepalived1
192.168.0.211 nginx、keepalived2

1）安装keepalived

yum install -y keepalived

2）配置keepalived的vrrp

vim /etc/keepalived/keepalived.conf

keepalived1配置

global_defs {
   router_id nginx_master # keepalived集群节点唯一键
}
vrrp_instance VI_1 {
    state MASTER # 配置状态
    interface enp0s3 # 绑定网卡，和Nginx一致即可
    virtual_router_id 51
    priority 100 # 提供服务的优先级别
    advert_int 1
    authentication {
        auth_type PASS
        auth_pass 1111
    }
    # 虚拟IP
    # 注意：网段必须可以被外接访问
    virtual_ipaddress {
        192.168.0.222
    }
}

keepalived2配置

global_defs {
   router_id nginx_backup
}

vrrp_instance VI_1 {
    state BACKUP
    interface enp0s3
    virtual_router_id 51
    priority 50
    advert_int 1
    authentication {
        auth_type PASS
        auth_pass 1111
    }
    virtual_ipaddress {
        192.168.0.222
    }
}

3）应用并启动Keepalived服务

systemctl enable keepalived.service
systemctl start keepalived.service

4）测试步骤：启动两台Keepalived服务器上的Nginx，并设置测试标记

192.168.0.210 > echo "nginx 192.168.0.210" > /opt/nginx/html/index.html
192.168.0.211 > echo "nginx 192.168.0.211" > /opt/nginx/html/index.html

5）测试步骤：在随便一台机器上使用虚拟IP访问，譬如：http://192.168.0.222/

两台Keepalived正常服务时

> curl 192.168.0.222                                      
nginx 192.168.0.210

关闭 192.168.0.210 上的Keepalived服务后

> systemctl stop keepalived.service
> curl 192.168.0.222
nginx 192.168.0.211

重启 192.168.0.210 上的Keepalived服务后

> systemctl start keepalived.service
> curl 192.168.0.222
nginx 192.168.0.210

为 Keepalived 实现故障检测功能

需要注意的是 Keepalived 并不能自主检测服务是否还处于可用状态。譬如以当前案例为例，如果 Nginx 因为某些原因被停掉了，Keepalived 是无法感知的，因此它不会为虚拟IP切换网关，从而导致整个 Nginx 代理的上游服务不可用。

为了解决上面的问题，我们可以为 Keepalived 提供一个服务的健康监测脚本（/root/nginx_health_check.sh）：

#!/bin/bash
if [ `systemctl status nginx.service | grep "running" | wc -l` -ne 1 ]
    then
        echo "Nginx服务已停止"
        systemctl stop keepalived.service
else
    echo "Nginx正常提供服务"
fi

将脚本文件配置到 Keepalived，在配置文件中追加以下内容：

# 脚本位置
vrrp_script check_nginx {
    script "/root/nginx_health_check.sh"
    interval 2 # 每两秒执行一次脚本
}
# 应用脚本
track_script {
    check_nginx
}
