ConcurrentHashMap中的位运算应用

 2019-11-28   作者: TanDi

闲聊

  先前写过一篇针对 HashMap 源码进行浅析的文章，现在回看觉得内容实在有些简陋，所以借着最近时间比较充裕，打算记录一下自己对 JDK 1.8 后 ConcurrentHashMap 的一些理解。

  位运算是计算机中最为高效的计算方式，如果你平时有翻阅源码的爱好那么肯定不难发现其实很多大神写的代码中的计算操作都是用位运算来完成的。因此接下来我们就从位运算的角度切入 ConcurrentHashMap 源码，分析一下在 ConcurrentHashMap 中位运算的应用。

自定义初始化容量

  先前在Q群里看到有小伙伴说面试被问到一个这样的问题，就是如果将 HashMap 初始化容量设置为 10000，然后往里头存储 10000 个元素，问 HashMap 需要扩容几次？对于这样一个问题如果你没有阅读过源码你就很容易被引入坑里，因为一些人他只知道 HashMap 中加载因为 0.75 ，所以 10000 的容量实质只能存储 7500 个，因此需要扩容1次。但是实际情况真的是这样的？答案肯定是否定的，对于 HashMap 而言它的容量必须是2的n次幂，且n不能大于30。也就是说如果你设置初始化容量为 10000 的话，那么 HashMap 会先判断 10000 是不是2的n次幂，如果不是则转变为2的n次幂再作为初始化容量。当然，前提是传入的这个初始化值是大于或等于零且小于或等于 MAXIMUM_CAPACITY（1 << 30，即2的30次方）。

以下我们就承接着上面的问题来阅读以下初始化容量的构造方法，代码如下：

public ConcurrentHashMap(int initialCapacity) {

    if (initialCapacity < 0)
        throw new IllegalArgumentException();

    int cap = ((initialCapacity >= (MAXIMUM_CAPACITY >>> 1)) ?
               MAXIMUM_CAPACITY :
               // 扩容为原来的1.5配再加1
               tableSizeFor(initialCapacity + (initialCapacity >>> 1) + 1)); 
    this.sizeCtl = cap;
}
  • initialCapacity 必须大于或者等于0
  • 如果 initialCapacity 大于或等于(MAXIMUM_CAPACITY >>> 1)则直接将其设置为MAXIMUM_CAPACITY。这样做理由视情况而定：
    • 大于时，意味着 initialCapacity 可能超过了 MAXIMUM_CAPACITY 或者并不是2的n次幂，因此避免越界和减少计算操作而直接设置为MAXIMUM_CAPACITY会更加高效
    • 等于时，我对此的理解是，如果 initialCapacity 等于 (MAXIMUM_CAPACITY >>> 1) 的话其实对于后续扩容只能有且仅有地进行1次，而且到了那个时候的数据量再进行扩容操作代价就有些大了，因此直接设置为 MAXIMUM_CAPACITY 效果可能更佳
  • 如果 initialCapacity 不符合上面的条件，那么 initialCapacity 会增加为原来的 1.5 倍再加 1。操作时 initialCapacity + (initialCapacity >>> 1) + 1 。这里采用无符号右移的好处是当 initialCapacity 为基数时可以截断其小数位。

将 initialCapacity 扩容为原来的1.5配再加1后操作还没有完成，这时还需要将其结果转换为2的n次幂。操作如下：

private static final int tableSizeFor(int c) {
    int n = c - 1;
    n |= n >>> 1;
    n |= n >>> 2;
    n |= n >>> 4;
    n |= n >>> 8;
    n |= n >>> 16;
    return (n < 0) ? 1 : (n >= MAXIMUM_CAPACITY) ? MAXIMUM_CAPACITY : n + 1;
}

上面这这个操作主要分为两部分：

  1. return 对上的操作，主要是将数值 c 的二进制最高位1后的所有0置换为1
  2. return行，判断结果值 n 是否越界，大于或等于 MAXIMUM_CAPACITY 时则最终结果为MAXIMUM_CAPACITY。否则为 n + 1。

接下来是分析时间，我们分析一下 return 对上的操作是如何进行的。

现在假设传入的 initialCapacity 值为 10000。过程如下：

第一步：进入构造器

initialCapacity + (initialCapacity >>> 1) + 1 = 1501

第二部：进行 return 对上的操作，即将数值的二进制最高位1后的所有0置换为1

n = 1501 - 1 = 1500

哈希值

  哈希值对于散列表数据结构来说是至关重要的存在。这是因为元素要想存储到散列表时必须通过哈希值（也叫散列值）来确定自己的存储下标。因为这个缘故，所以大多数情况下要是我们相拥散列表来存储自己编写的引用类型时都应该重载从 Object 类继承过来的 hashCode 和 equals 两个方法。

  在讲解 hashCode 和 equals 两个方法之前我们先简单回顾以下什么是散列表。散列表也叫哈希表，一般数据结构表现为 数组 + 链表。但其实这只是一种常见实现而已（例如JDK1.8加入了红黑树），因为散列表在最理想的情况下仅仅只需要数组来实现即可，即这个时候的查询时间复杂度为O(1)，这是最理想的情况，前提是数据足够离散。但是生活告诉我们很多情况都是事与愿违的，如果数据不够离散的话，那么元素使用哈希值来定位存储下标时就会出现一种叫”哈希冲突/碰撞”的现象，既元素之间出现了相同的哈希值。出现这种情况的原因有3种：

  • 重复保存了同一个元素
  • 不同元素间存在相同哈希值
  • 大小值位运算导致哈希冲突（下面有案例）

  好了，我们已经回顾了散列表数据结构，还提到什么是”哈希冲突/碰撞”。那么现在就说说 hashCode 和 equals 两个方法的作用：

  • hashCode：获取实例的哈希值
  • equals：判断实例是否相同

  hashCode 和 equals 的关系是这样的，例如：Person A 和 Person B 的 hashCode 相等，既发生了哈希碰撞，这时散列表会调用实例的 equals 方法进一步判断 Person A 和 Person B 是不是同一个实例。这就为什么需要重载 hashCode 和 equals 方法的原因。也因为这个缘故所以重载方法时需要重点注意，既如果两个实例进行 equals 操作返回 true 的话，那么他们的 hashCode 肯定是相等的，反之不然。

  至此，该回顾的内容我们都已经回顾了，接下来是时候看看 ConcurrentHashMap 中它是如何使用位运算来优化元素实例的哈希值的。我们从它的 put 方法开始观察，源码如下：

final V putVal(K key, V value, boolean onlyIfAbsent) {
    // 省略
    int hash = spread(key.hashCode());
    // 省略
}

可以看到，在存储元素时它先获取了元素自身的 hashCode 值，然后传递给了 spread 方法。对于 spread 而言，它的主要功能有两个：

  1. 降低元素的哈希冲突
  2. 将元素的负哈希值改为正

以下为 spread 方法源码：

static final int spread(int h) {
    return (h ^ (h >>> 16)) & HASH_BITS;
}

第一步：h ^ (h >>> 16)

  首先需要知道的是 hashCode 它是一个int类型，即32位。这里无符号右移16位的意思就是将高16位的值挪到低16位位置中，这样做的目的是降低哈希冲突。因为散列表下标计算公式为：h & (n-1) ，而 n 的最大值为 2 的 30 次方（散列表最大容量），但又因为 hashCode 是一个32位的int类型，其最大值是 2 的 31 次方减1，因此存在 h 大于 (n-1) 的情况。如果不做任何处理的话就会加大哈希冲突的出现。举个简单的例子：

# 例如有两个哈希值a 和 b，分别如下：
#    0000 0000 0000 1000 0000 0000 0000 0010 = a
#    0000 0000 0000 0000 0000 0000 0000 0010 = b
# 可以看到a和b的低16位是相同的，而高16位是存在差异的。
# 接下来以HashMap默认容量值16来计算一下a和b的存储下标位置：
a:
    0000 0000 0000 0000 0000 0000 0000 1111 = 16 - 1，高位以0补全
        &
    0000 0000 0000 1000 0000 0000 0000 0010 = a
    ---- 结果
    0000 0000 0000 0000 0000 0000 0000 0010 = 下标2

b:
    0000 0000 0000 0000 0000 0000 0000 1111 = 16 - 1，高位以0补全
        &
    0000 0000 0000 0000 0000 0000 0000 0010 = b
    ---- 结果
    0000 0000 0000 0000 0000 0000 0000 0010 = 下标同样为2

从例子中可以看到，如果不对原来的哈希值做任何处理的话就会出现上面的情况，既1高位不同但地位相同的两个哈希值计算出来的下标发生了哈希冲突现象，而 h ^ (h >>> 16) 就是解决这个问题的。接下来我们还是使用上面a和b这两个值来进行一次 h ^ (h >>> 16) 计算以验证一下是不是上面所说的那样可以解决上面的冲突问题。

原值
0000 0000 0000 1000 0000 0000 0000 0010 = a
0000 0000 0000 0000 0000 0000 0000 0010 = b

>>> 16 后
0000 0000 0000 0000 0000 0000 0000 1000 = a
0000 0000 0000 0000 0000 0000 0000 0000 = b

^ 后
0000 0000 0000 1000 0000 0000 0000 1010 = a // 相当于原基础上加了8
0000 0000 0000 0000 0000 0000 0000 0010 = b

经过计算后可以看到 a 的低位变成了1010，而b则原封不动还是0010，因此解决了不处理时的冲突现象。

LastRun高低位机制

了解

  LastRun机制是用来提高 ConcurrentHashMap 的扩容效率的，主要体现在它可以减少扩容时需要重新计算散列表下标值的工作量。但需要注意的是 LastRun 机制在 JDK1.7 和 JDK1.8 两个版本中是存在差异的，如 JDK1.7 中是没有高低位操作，既只有 LastRun 机制。

  对于高低位机制的理解个人认为应该溯源到 JDK1.7 的 HashMap 身上，因为高低位机制解决了 HashMap 并发扩容后 get 操作的死循环问题（实在不太理解为什么会有人用多线程并发操作HashMap，真是黑人问号.jpg）。那么这个问题的点位于 JDK1.7 版 HashMap 的 transfer 方法中，源码如下：

java.util.HashMap

void resize(int newCapacity) {
    // 省略
    transfer(newTable, initHashSeedAsNeeded(newCapacity));
    table = newTable; // 注意：这里需要先迁移完才会切断OldTable的引用
    // 省略
}

void transfer(Entry[] newTable, boolean rehash) {
    int newCapacity = newTable.length;
    for (Entry<K,V> e : table) {
        while(null != e) { // 关键变量 e 和 next
            Entry<K,V> next = e.next;
            if (rehash) {
                e.hash = null == e.key ? 0 : hash(e.key);
            }
            int i = indexFor(e.hash, newCapacity);
            // 问题代码
            e.next = newTable[i];
            newTable[i] = e;
            e = next;
        }
    }
}

从问题代码中可以得知迁移过程使用的是 头插法 迁移方式，过程如下：

  1. 先将目标位置下的冲突链表赋值给自己(e)的next指向
  2. 再将自己(e)设置到这个目标位置下

好了，现在我们知道了 JDK1.7 的 HashMap 并发扩容时会导致 get 死循环问题后我们再看看 JDK1.8 中HashMap 是如何解决这个问题的，迁移源码如下：

java.util.HashMap#resize

Node<K,V> loHead = null, loTail = null;
Node<K,V> hiHead = null, hiTail = null;
Node<K,V> next;

do { 
    next = e.next; // 记录下一个迭代节点
    if ((e.hash & oldCap) == 0) { // 记录低位
        if (loTail == null) 
            loHead = e;
        else 
            loTail.next = e;
        loTail = e;
    }
    else { // 记录高位
        if (hiTail == null)
            hiHead = e;
        else
            hiTail.next = e;
        hiTail = e;
    }
    // 迭代冲突链表
} while ((e = next) != null); 

if (loTail != null) {
    loTail.next = null;
    newTab[j] = loHead;
}

if (hiTail != null) {
    hiTail.next = null;
    newTab[j + oldCap] = hiHead;
}

可以看到，相较于 JDK 1.7 而言除了e和next之外 JDK1.8 还新增了4个变量值：

  • 记录低位节点：loHead、loTail
  • 记录高位节点：hiHead、hiTail。

那么它是如何解决循环引用问题的呢？如果你有认真看上面那张循环引用分析图的话其实很容易就能得出答案。JDK 1.7 之所以并发扩容会导致循环引用，主要是因为使用了头插法后直接操作变量e所导致的，因为这样会导致原来的 A -> B 引用转换为 B -> A ，那么多线程并发操作时就会存在循环引用的隐患。而 JDK 1.8 后采用的是尾插法之余还新增了4个变量来避免直接操作变量e，既不存在将 A -> B 引用转换为 B -> A 的情况，具体迁移步骤是先用 hiHead 和 loHead 按原来顺序拼接链表，等拼接完了才设置到新散列表中。

但在这里需要强调的是，即使 JDK 1.8 的 HashMap 解决了循环引用的问题，但它依然是非线程安全得多，例如在并发拼接链表时节点的next指向会被修改，这样一来就可能会直接导致其它线程拼装链表时丢失某部分节点，所以如果是多线程环境的话还是需要使用线程安全的 ConcurrentHashMap 才能保证安全。

  在 JDK 1.8 的 HashMap 中扩容时其实是用到了LastRun机制的，如上面这段代码，只不过它没有明确表述出来而已，其中 (e.hash & oldCap) == 0 就是 LastRun机制操作，稍后我会从位运算角度来分析一下为什么LastRun机制可以降低扩容时需要重新计算哈希值的操作量，以及它的巧妙之处。

分析

  接下来将视角切换到 JDK 1.8 的 ConcurrentHashMap 中。我们先了解一个 ConcurrentHashMap 扩容时是如何使用LastRun高低位机制的。迁移源码如下：

java.util.concurrent.ConcurrentHashMap#transfer

if (tabAt(tab, i) == f) { // 普通节点
    Node<K,V> ln, hn;
    if (fh >= 0) { // n为当前散列表容量，fh为当前节点的哈希值，runBit用于判断高低位
        int runBit = fh & n; 
        Node<K,V> lastRun = f; // 预定义LastRun节点为冲突链表的head节点
        for (Node<K,V> p = f.next; p != null; p = p.next) {
            int b = p.hash & n;
            if (b != runBit) { // 迭代出LastRun，判断依据是runBit存在差异
                runBit = b; // 注意：如果runBit存在差异，证明在迁移后当前冲突链表需要拆分
                lastRun = p;
            }
        }
        if (runBit == 0) { // 低位
            ln = lastRun;
            hn = null;
        }
        else { // 高位
            hn = lastRun;
            ln = null;
        }
        // 拼装高低位链表，只需迭代到lastRun即可
        for (Node<K,V> p = f; p != lastRun; p = p.next) { 
            int ph = p.hash; K pk = p.key; V pv = p.val;
            if ((ph & n) == 0) // 低位
                ln = new Node<K,V>(ph, pk, pv, ln);
            else // 高位
                hn = new Node<K,V>(ph, pk, pv, hn);
        }
        setTabAt(nextTab, i, ln); // 低位链表迁移无需修改下标
        setTabAt(nextTab, i + n, hn); // 高位链表新下标= 当前下标 + OldTable.length
        setTabAt(tab, i, fwd); 
        advance = true;
    }

可能跟随着上面注释的步伐一时之间你还是不明白 LastRun 到底在哪里降低了下标值的计算工作。所以我画了一张图来帮组理解：

LastRun的巧妙之处是使用 runBit = fh & n 来区分高低位以确定节点迁移的新下标值。而runBit的结果只有两种：

  • 等于0，迁移后 newIndex = oldIndex
  • 非0，迁移后 newIndex = oldIndex + old

其实一开始看源码的时候我自己是十分困惑的，我的疑问是，为什么可以这样确定新下标值，感觉很是神奇。Google了一番都没有得到结果。
