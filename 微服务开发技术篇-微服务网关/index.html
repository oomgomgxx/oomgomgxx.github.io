微服务开发技术篇-微服务网关

为什么需要网关

没有用网关之前

  • 客户端需要维护众多服务的地址
  • 每个服务都需要对客户端请求做权限验证
  • 跨域请求处理变得复杂
  • 拦截请求责任落在服务上领到服务变得沉重
  • 对用户数据采集不方便

引入服务网关后

  • 易于监控
  • 易于认证
  • 解决了没有网关前的各种问题，网关就像面向客户端的Facede模式

网关可以完成的功能

  • 请求路由
  • API组合
  • 协议转换
  • 身份验证
  • 访问授权
  • 速率限制
  • 缓存
  • 指标收集
  • 请求日志
  • 浏览器CORS跨域响应处理

注意事项

  • 因为 API 网关代理了后端的多个服务因此而变得职责模糊，这时可以使用后端前置模式，即不同的服务类型采用不用的 API 网关
  • 开发者在开发完后端服务之后必须更新 API 网关，否者客户端就会被逼等待
  • 如果使用 API 网关实现 API 组合功能，则建议使用反应式编程来实现并行请求以提高可用性，例如使用 RxJava、Project Reactor 等框架实现
  • API 网关尽可能使用断路器来辅助服务的调用以提高本身的可靠性，以免发生调用等待则阻塞后续请求
  • 建议微服务在 API 网关中进行身份验证操作，再配合授权服务发放 token 的形式来完会话功能。在微服务中并不推荐使用共享存储方式来实现共享会话（例如Redis实现共享会话），因为这样会违反松耦合的原则

微服务网关和Nginx有什么区别？

  首先要说的是Nginx也可以作为微服务网关来使用，而且性能也非常不错，但需要借助Lua脚本来手动实现各种功能需求，因此增加了学习和开发的成本(当然，如果确实不嫌麻烦和喜欢定制的话其实也可以使用Node.js来手动实现一个微服务网关)。随着微服务的兴起其实诞生了不少既简单又好用的微服务网关，目前也有不少企业把Nginx和微服务网关结合来使用。

企业开发常用的微服务 API 网关

  • Netflix Zuul 1.x：SpringCloud集成，已停止更新
  • Netflix Zuul 2.x ：无SpringCloud集成
  • Mashape Kong ：面向Cloud Native，基于Nginx_Lua模块开发
  • SpringCloud Gateway ：Zuul的代替品， 基于Sprint Boot2、WebFlux、Project Reactor构建
  • Linkerd ：纳入了CNCF，即面向Cloud Native

目前而言个人较为推荐 Kong 和 SpringCloud Gateway。

基础概念和处理流程

  • Route：是 ID、URI、Predicates、Filters 的集合
  • Predicate：根据配置来匹配HTTP请求，HTTP请求必须满足Predicate要求
  • Filter：过滤请求，可以在发送请求之前或之后修改请求和响应
